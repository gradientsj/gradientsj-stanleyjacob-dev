Operating systems: processes, virtual memory, file systems, and the kernel boundary

An operating system is a resource multiplexer built on three abstractions: the process, which virtualizes the CPU; the address space, which virtualizes memory; and the file, which virtualizes storage. This page works through all three with the arithmetic shown: the cost of crossing the user/kernel boundary measured on this site's build machine, CFS vruntime and EEVDF deadlines computed by hand and then verified against the live scheduler, a mutex built up from a single atomic instruction to the futex system call, a 4-level x86-64 page walk decomposed address-by-address, a measured 1.9x penalty for translating 32 GiB through 4 KiB pages instead of huge pages, inode pointer arithmetic, the crash-consistency argument behind journaling, and the ring buffers of io_uring driven through raw system calls. It closes where the kernel's own boundaries are drawn today: namespaces and cgroups below, hypervisors beneath.

Why this subject matters now

Operating systems knowledge went through a strange decade. The textbook material, processes, paging, file systems, stabilized in the 1990s, and for a while it was fashionable to treat the kernel as a solved layer one merely sits on. Three shifts ended that. First, the cloud made the kernel's isolation machinery into product surface: containers are nothing but namespaces plus cgroups, serverless platforms boot a hardware virtual machine per function, and the difference between a 130 ns system call and a 4 µs trap into a hypervisor is a line item on someone's bill. Second, hardware outran the kernel's old interfaces: NVMe devices that sustain millions of I/O operations per second made the one-syscall-per-operation model a bottleneck, which is why io_uring exists, and terabyte-scale memory made 4 KiB pages a measurable tax, which is why huge pages and multi-level TLB design are now application-level concerns. Third, the schedulers changed under everyone's feet: Linux replaced the O(1) scheduler with CFS in 2007, replaced CFS's heuristics with EEVDF in 2023, and since 6.12 lets operators load entire scheduling policies as BPF programs. An engineer who last looked at scheduling in the CFS era is two generations behind the kernel their service runs on.

The practitioner bar has risen accordingly. Debugging a tail latency regression today routinely bottoms out in kernel territory: a transparent-huge-page compaction stall, a cgroup CPU quota throttling a container at exactly the wrong moment, a journal commit serializing behind an fsync, a page cache that looks like a memory leak to a naive dashboard. The material on this page is the vocabulary those investigations are conducted in. It is anchored on Arpaci-Dusseau and Arpaci-Dusseau's Operating Systems: Three Easy Pieces, Tanenbaum and Bos's Modern Operating Systems, and Love's Linux Kernel Development, with the original papers, Ritchie and Thompson on UNIX, McKusick on FFS, Rosenblum and Ousterhout on LFS, Denning on working sets, cited where the ideas were born.

Every number presented as a measurement below was taken on this site's build machine: an Intel Xeon Platinum 8480+ (Sapphire Rapids, 52 logical CPUs), Ubuntu 22.04 on Linux 6.8, gcc 11.4 at -O2, recorded in classes/data/os.json. The host is itself a KVM guest on shared multi-tenant hardware, a fact worth keeping in view twice over: absolute numbers carry guest-exit and neighbor noise, so they are order-of-magnitude engineering data rather than microarchitectural truth, and the machine is a living example of the virtualization stack the last section describes. Context-switch and lock-contention numbers measured in the same environment live in classes/data/concurrency.json and are dissected at length on the concurrency page; cache and TLB hardware internals are covered on the architecture page. This page owns the kernel's side of both stories.

The process: the kernel's unit of accounting

A process is a running program: one or more threads of execution bound to an address space, a table of open file descriptors, credentials, signal state, and, on a modern Linux, a set of namespaces and a cgroup membership. The definition matters because every one of those components is a kernel data structure with a cost and a sharing rule. In Linux the whole bundle hangs off struct task_struct in include/linux/sched.h, one per thread, several kilobytes each: the pid, the run state, a pointer to mm_struct (the address space: page-table root, VMA tree, RSS counters), a pointer to files_struct (the fd table), cred (uid, gid, capabilities), the scheduling entity that CFS/EEVDF sorts, and the namespace proxy. The teaching kernel xv6-riscv (MIT's modern rewrite of v6 UNIX) collapses this to a 64-line struct proc in kernel/proc.h, pid, state, page-table pointer, kernel stack, open files, and reading it first makes the production structure legible: everything Linux adds is accounting, not concept.

The state machine is small. A task is created, becomes runnable, runs when the scheduler picks it, blocks when it waits for I/O or a lock, and on exit becomes a zombie: dead but still occupying a process-table slot so its parent can read the exit status with wait. Only after the parent reaps it does the kernel free the slot; a parent that never waits leaks zombies, which is why long-running daemons either catch SIGCHLD or double-fork so init inherits and reaps. Linux's D state, uninterruptible sleep, is the operationally famous one: a task blocked in the kernel, usually on disk I/O, that not even kill -9 can touch until the I/O completes, because the kernel cannot safely unwind a half-finished operation on its stack.

The UNIX creation interface splits creation into two calls, and the split is a design decision worth defending because it looks wasteful at first sight. fork() duplicates the calling process, same code, same (copy-on-write) memory, same open files; exec() replaces the program image within the existing process. Ritchie and Thompson's 1974 paper shows what the gap between them buys: the shell implements redirection by forking, then, in the child, between fork and exec, closing fd 1 and opening the output file in its place, then exec-ing the command. No special kernel support for redirection, pipes, or environment setup is needed; the child rearranges its own inheritance using ordinary calls before it becomes the new program. The alternative, a single spawn call, needs a parameter block describing every possible rearrangement, which is exactly what posix_spawn's file-actions API looks like. The cost side, measured on this machine and recorded in concurrency.json: fork plus wait for an exiting child is 118.9 µs, fork plus exec of /bin/true is 499.7 µs, posix_spawn of the same 432.6 µs, and a pthread create/join, which shares the address space instead of copying page tables, is 33.9 µs. The copy-on-write machinery that makes fork cheap enough to survive is measured in the virtual memory section below.

Linux implements all of these over one primitive: clone(), whose flag bits decide what the child shares. fork is clone sharing almost nothing; pthread_create is clone with CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD, sharing the address space, fd table, and signal handlers; a container runtime's first act is clone with CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWNET, sharing nothing and additionally unsharing the kernel's own naming. Process, thread, and container are not three kernel concepts; they are three settings of one bitmask.

System calls: what the boundary costs

User code runs at privilege level 3, the kernel at level 0, and the only sanctioned way across is a trap. On x86-64 the syscall instruction does the crossing in hardware: it copies the return address into rcx and the flags into r11, loads the instruction pointer from the LSTAR machine-specific register, which the kernel set at boot to point at its entry stub, and raises the privilege level, all without touching memory. The kernel stub then does the work the instruction skipped: swapgs to find the per-CPU area, a switch to the task's kernel stack, a spill of user registers into a struct pt_regs, and an indirect call through the system-call table indexed by rax. The return path (sysret) mirrors it. Two things are worth separating immediately. This is a mode switch, not a context switch: the address space does not change, no scheduler runs, and the same task continues in kernel clothes. And the cost is not the instruction, which is tens of cycles, but the round trip: entry bookkeeping, the work itself, exit checks (pending signals, need-resched), and the indirect cache and branch-predictor damage on both sides.

Some "system calls" never cross at all. The kernel maps a shared page of code and data, the vDSO, into every process, and calls whose answer the kernel can precompute, clock_gettime, gettimeofday, getcpu, are satisfied there: the vDSO reads the timekeeping variables the kernel publishes under a sequence lock, scales the CPU's timestamp counter, and returns without a trap. The measured gap below is the whole argument for the mechanism.

/* syscall_cost.c: gcc -O2 syscall_cost.c && taskset -c 2 ./a.out
 * Measures the user/kernel boundary four ways. Numbers quoted on this
 * page: 1.4 / 32.2 / 133.8 / 188.6 ns on a Xeon 8480+ guest, Linux 6.8. */
#define _GNU_SOURCE
#include <stdio.h>
#include <stdint.h>
#include <time.h>
#include <unistd.h>
#include <sys/syscall.h>

static inline uint64_t now_ns(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);          /* itself vDSO */
    return (uint64_t)ts.tv_sec * 1000000000ull + ts.tv_nsec;
}

__attribute__((noinline)) long plain_call(long x) {
    __asm__ volatile("" : "+r"(x));               /* defeat inlining */
    return x + 1;
}

int main(void) {
    const long N = 10 * 1000 * 1000;
    volatile long sink = 0;
    struct timespec ts;
    uint64_t t0;

    for (long i = 0; i < 200000; i++) sink += getpid();   /* warmup */

    t0 = now_ns();
    for (long i = 0; i < N; i++) sink += plain_call(i);
    printf("plain call:          %6.2f ns\n", (double)(now_ns()-t0)/N);

    t0 = now_ns();
    for (long i = 0; i < N; i++) { clock_gettime(CLOCK_MONOTONIC, &ts); sink += ts.tv_nsec; }
    printf("clock_gettime vDSO:  %6.2f ns\n", (double)(now_ns()-t0)/N);

    t0 = now_ns();
    for (long i = 0; i < N; i++) sink += syscall(SYS_getpid);
    printf("raw getpid trap:     %6.2f ns\n", (double)(now_ns()-t0)/N);

    t0 = now_ns();                                /* deny the vDSO */
    for (long i = 0; i < N; i++) { syscall(SYS_clock_gettime, CLOCK_MONOTONIC, &ts); sink += ts.tv_nsec; }
    printf("clock_gettime trap:  %6.2f ns\n", (double)(now_ns()-t0)/N);
    return (int)(sink & 1);
}

Problem 1

A tracing framework timestamps function entry and exit, two clock reads per traced call, and the traced service executes 2,000,000 traced calls per second per core. Using this machine's measurements (32.2 ns vDSO, 188.6 ns forced trap), compute the fraction of a core consumed by timestamping in each configuration, and the break-even call rate at which trap-based timestamps alone would consume an entire core.

Solution. Clock reads per second: \( 2 \times 2{,}000{,}000 = 4{,}000{,}000 \). Via the vDSO: \( 4 \times 10^6 \times 32.2\,\text{ns} = 0.1288\,\text{s} \) of CPU per second, i.e. 12.9% of a core. Via forced traps: \( 4 \times 10^6 \times 188.6\,\text{ns} = 0.754\,\text{s} \), i.e. 75.4% of a core, and the service has lost three quarters of its capacity to observability. Break-even for a full core with traps: \( 1 / 188.6\,\text{ns} = 5.30 \times 10^6 \) reads/s, or 2.65 million traced calls per second. The design conclusion: high-frequency instrumentation is only viable because the vDSO exists, and even then 12.9% is a price one states in a design review rather than discovers in production. This is why serious tracers read the TSC directly (about 7 ns) and defer conversion to wall time.

Context switches, measured

A context switch is the scheduler's act of replacing the running task: save the outgoing task's registers into its kernel stack, switch stacks, switch mm_struct if the incoming task lives in a different address space (a CR3 write, which on this hardware does not flush the TLB thanks to PCID tags, but does invalidate the cheap fiction that the TLB content is yours), restore the incoming registers, and return into a different program. The direct cost is small; the indirect cost, refilling caches, TLB entries, and branch predictors that the other task evicted, is the part that scales with the workload and dominates in practice.

Both were measured here. The direct cost: a 1-byte ping-pong between two processes over a pair of pipes, both pinned to the same logical CPU so every round trip forces two switches, 199,000 round trips. Result: 3,770 ns per round trip, so about 1,885 ns per process switch; the same experiment with threads (shared address space, no CR3 write) gives 1,749 ns, and a sched_yield handoff pair costs 1,583 ns. The concurrency page's independent run of the same experiment (concurrency.json) got 3,730 ns per round trip, agreement well within this host's noise. Cross-core, where each wakeup needs an inter-processor interrupt instead of a local schedule, the round trip balloons to about 15.9 µs on this virtualized box. The indirect cost: the same ping-pong, but each peer touches 512 random cache lines of a private working set before yielding. As the working set grows from 32 KiB to 64 MiB, the measured overhead per turn versus running alone grows from 5.1 µs to 11.3 µs: the switch itself did not change, but every turn now begins by reloading state the peer evicted. "How much does a context switch cost" has no single answer; it has a cache footprint parameter.

And the 1,885 ns switch cost sets the floor for every blocking primitive: a mutex that parks its waiter cannot hand off in less than two switches, which is why the futex design below spends so much effort never entering the kernel at all.

Scheduling: from round-robin to EEVDF

Metrics, and the quantum trade-off

A scheduler optimizes under a conflict. Turnaround time, \( T_{\text{turnaround}} = T_{\text{completion}} - T_{\text{arrival}} \), rewards running each job to completion; response time, \( T_{\text{response}} = T_{\text{first run}} - T_{\text{arrival}} \), rewards slicing finely. Shortest Job First minimizes mean turnaround among non-preemptive policies, and the proof is a one-line exchange argument: if a longer job runs immediately before a shorter one, swapping them reduces the shorter job's completion time by the longer job's length and increases the longer job's by the shorter one's, a net decrease in the sum; repeat until sorted. But SJF needs future knowledge and starves long jobs, so interactive systems slice: round-robin with quantum \( q \) bounds response time by \( (n-1)q \) for \( n \) runnable tasks at the cost of paying one context switch per quantum.

The quantum trade-off becomes concrete with this machine's 1,885 ns switch cost. Overhead fraction is \( s/(q+s) \): at \( q = 10 \) ms it is \( 1.885/10001.885 = 0.019\% \), irrelevant; at \( q = 1 \) ms, 0.186%; at \( q = 100 \) µs, 1.83%, and that still ignores the cache-refill tax measured above, which at a 64 MiB working set adds several microseconds per switch. Solving \( s/(q+s) \le 1\% \) gives \( q \ge 99s = 186.6 \) µs. This bracket, a few hundred microseconds to a few milliseconds, is where every production scheduler's slice lands, and now the reader knows why.

MLFQ: learning priority from behavior

The multi-level feedback queue, descended from Corbató's CTSS (1962), the first system that had to timeshare interactive users, approximates SJF without future knowledge by using the past. The rules: maintain a stack of queues by priority; always run from the highest non-empty queue, round-robin within it; new jobs enter on top; a job that exhausts its allotment at a level drops one level; and periodically every job is boosted back to the top. The scheme learns: an interactive job that sleeps before its allotment expires keeps its priority, a CPU-bound job sinks to the bottom queues where quanta are long and switches rare. Each rule patches a real failure mode. The periodic boost fixes both starvation (a bottom-queue job under a stream of interactive arrivals would otherwise never run) and behavior change (a compile that becomes a REPL is re-observed). The "allotment across the level, not per burst" accounting fixes gaming: without it a job could sleep for a millisecond just before each quantum expired and camp on the top queue forever. Windows and the BSDs run recognizable MLFQ descendants; Linux went a different way.

Lottery and stride: sharing by arithmetic

Waldspurger and Weihl's lottery scheduling (OSDI 1994, MIT) reframed the goal: not priority, but proportional share. Give each task tickets; each quantum, draw a uniform ticket and run the holder. A task holding \( t_i \) of \( T \) tickets gets each quantum with probability \( p_i = t_i / T \), so over \( n \) quanta its allocation is binomial with mean \( n p_i \) and standard deviation \( \sqrt{n p_i (1-p_i)} \): the relative error decays as \( 1/\sqrt{n} \), fairness that is only asymptotic. Stride scheduling, Waldspurger's deterministic follow-up, removes the variance. Each task gets \( \text{stride}_i = L / t_i \) for a large constant \( L \), and a pass counter; always run the task with the smallest pass, then advance it by its stride. With tickets 250, 100, 50 and \( L = 10{,}000 \): strides 40, 100, 200. Simulating from zero (ties broken by name) gives the schedule A B C, A, A, B, A, A, A, B: in ten quanta A ran 6 times, B 3, C 1, against exact entitlements 6.25, 2.5, 1.25, and the error is provably bounded by one quantum forever, never \( \sqrt{n} \). These two papers are the direct ancestors of everything below: CFS's vruntime is a pass counter with weights, and cgroup CPU shares are tickets under another name.

CFS: the vruntime invariant, worked and then verified

Linux's Completely Fair Scheduler (2007-2023) implements weighted proportional share by bookkeeping a single quantity per task, virtual runtime. While a task runs for \( \Delta t \) of wall time, its vruntime advances by $$ \Delta v_i \;=\; \Delta t \cdot \frac{w_0}{w_i}, \qquad w_0 = 1024, $$ where \( w_i \) is the task's weight from the kernel's nice table (sched_prio_to_weight in kernel/sched/core.c): nice 0 is 1024, and each nice step multiplies the weight by about 1.25, so nice -5 is 3121, nice 5 is 335, nice 19 is 15, nice -20 is 88761. The scheduler keeps runnable tasks in a red-black tree ordered by vruntime and always runs the leftmost, the task that has received the least weighted service. Heavy tasks accumulate vruntime slowly, so they hold the CPU longer before the tree rotates them out.

Worked numerically: tasks A at nice 0 (\( w_A = 1024 \)) and B at nice -5 (\( w_B = 3121 \)) share one CPU for 100 ms. In equilibrium CFS equalizes vruntime advance, so CPU time divides in weight proportion: \( 1024 + 3121 = 4145 \), A gets \( 100 \times 1024/4145 = 24.70 \) ms and B gets \( 100 \times 3121/4145 = 75.30 \) ms. Check the invariant: A's vruntime advances \( 24.70 \times 1024/1024 = 24.70 \) ms; B's advances \( 75.30 \times 1024/3121 = 24.70 \) ms. Equal, as required, and that equality is not an assumption but the fixed point the leftmost-first rule drives the system toward: any task whose vruntime lags gets picked until it catches up (Problem 2 makes this a derivation).

The prediction was then tested against the live scheduler: pairs of spin loops pinned to one logical CPU of this machine for 4 seconds, CPU time read from getrusage. Nice 0 versus nice 5 predicts a share of \( 1024/(1024+335) = 0.753 \) for the nice-0 task; measured, 0.753. Nice 0 versus nice 3: predicted 0.661, measured 0.660. Nice 0 versus nice 10: predicted 0.903, measured 0.902. The weight table is not a suggestion; the kernel hits it to three digits.

EEVDF: adding deadlines to fairness

CFS answered "who is owed CPU" but had no principled answer to "who needs it soon": interactivity came from a pile of heuristics (sleeper credits, wakeup preemption tunables) that were a constant source of regressions. The replacement, merged as the default in Linux 6.6 and running on this machine's 6.8 kernel, is Stoica and Abdel-Wahab's Earliest Eligible Virtual Deadline First (1995-96), a scheduler with a fairness and a latency variable. Define system virtual time \( V(t) \) as the weighted average of all runnable tasks' virtual runtimes; task \( i \)'s lag is \( \ell_i = w_i\,(V - v_i) \), the service it is owed (positive) or has over-received (negative). Two rules:

First, eligibility: only tasks with \( \ell_i \ge 0 \), i.e. \( v_i \le V \), may run. A task that has already over-consumed cannot be picked again merely because it holds a large weight. Second, among eligible tasks, run the one with the earliest virtual deadline $$ d_i \;=\; v_{e,i} + \frac{r_i}{w_i}, $$ where \( v_{e,i} \) is the virtual time at which the task's current request became eligible and \( r_i \) is its slice, the amount it asks for before rescheduling. The consequence, worked with numbers: two equal-weight tasks become eligible at \( v_e = 0 \), one a latency-sensitive task with a 2 ms slice, one a batch task with a 10 ms slice. Deadlines: 2 versus 10. The interactive task runs first, not because it holds more weight, over any window both still get 50%, but because it asked for less at a time. Latency and share have been decoupled: nice adjusts \( w_i \) (how much you get), slice adjusts \( r_i \) (how urgently you get it), and the old heuristics are simply deleted. Stoica and Abdel-Wahab prove lag stays bounded by one maximal request in their 1996 technical report; the proof is out of scope here but the bounded-lag claim is the whole warranty. The implementation is readable in kernel/sched/fair.c, where entity_eligible and pick_eevdf are near-literal transcriptions of the two rules.

One machine-level footnote: all of the above is per-CPU. Linux runs one runqueue per logical CPU and migrates tasks between them under a separate load-balancing regime that respects cache and NUMA domains, because a globally fair queue would serialize 52 CPUs on one lock. Fairness is therefore local and approximate across cores, a deliberate trade the parallel computing page revisits from the application side.

Problem 2

(a) Prove that under CFS's rule \( \Delta v_i = \Delta t \cdot w_0 / w_i \), equal vruntime advance across runnable tasks over a window implies each task \( i \) received CPU time proportional to \( w_i \), and conversely. (b) Three tasks at nice 0, nice 5, and nice 10 (weights 1024, 335, 110) share one CPU for 3 seconds with no blocking. Compute each task's CPU time to the millisecond and each task's common vruntime advance.

Solution. (a) Let task \( i \) receive wall time \( c_i \) in the window. Its vruntime advance is \( \Delta v_i = c_i w_0 / w_i \). Suppose all advances are equal to a common \( \Delta v \). Then \( c_i = \Delta v \cdot w_i / w_0 \), which is proportionality \( c_i \propto w_i \) with constant \( \Delta v / w_0 \). Conversely if \( c_i = K w_i \) for all \( i \), then \( \Delta v_i = K w_i \cdot w_0 / w_i = K w_0 \), independent of \( i \), hence equal. So the leftmost-first rule, which by always advancing the minimum vruntime forces all vruntimes to track each other to within one slice, is exactly weighted fair sharing, up to one-slice granularity.

(b) Total weight \( 1024 + 335 + 110 = 1469 \). Shares of 3,000 ms: nice 0 gets \( 3000 \times 1024/1469 = 2091.2 \) ms; nice 5 gets \( 3000 \times 335/1469 = 684.1 \) ms; nice 10 gets \( 3000 \times 110/1469 = 224.6 \) ms. (Sum: 2999.9 ms, rounding.) Common vruntime advance, using any task: \( 2091.2 \times 1024/1024 = 2091.2 \) ms; check with nice 10: \( 224.6 \times 1024/110 = 2090.9 \) ms, equal to rounding. Note the compounding: two tasks five nice levels apart differ by 3.06x, ten levels 9.3x, consistent with the measured table above.

Locks from atomics, and the futex

Spinning: correct in three lines, wrong for user space

Every mutual-exclusion primitive bottoms out in one hardware capability: an atomic read-modify-write on a shared word. With xchg or compare-and-swap, a spinlock is three lines: loop exchanging 1 into the lock word until the old value comes back 0; store 0 to release. Add the test-and-test-and-set refinement, spin on a plain load and only attempt the exchange when the lock looks free, so that waiters spin in their own caches instead of stampeding the line, plus a pause hint; the cache-coherence mechanics of why this matters are traced on the architecture page, and the memory-ordering contract (acquire on lock, release on unlock) is derived on the concurrency page. The kernel uses spinlocks pervasively, and may: it disables preemption while holding one, so the holder cannot be descheduled and hold-time is bounded by code inspection.

User space has no such guarantee, and that changes the verdict entirely. If a thread is preempted while holding a spinlock, every other thread spins through its whole quantum accomplishing nothing; with the measured numbers, waiters burn milliseconds of CPU to wait out a 1,885 ns switch. On a single core it is strictly worse: spinning prevents the holder from running. What user space needs is a lock that costs nothing when uncontended and sleeps when contended, and that is precisely the futex design.

The futex: kernel help, only on the slow path

The futex system call (Franke, Russell, and Kirkwood, Ottawa Linux Symposium 2002; the definitive user guide is Drepper's "Futexes Are Tricky") provides exactly two things over a user-space 32-bit word. FUTEX_WAIT(addr, val): the kernel locks the hash bucket for addr, re-reads *addr, and only if it still equals val enqueues the caller and sleeps; otherwise it returns EAGAIN immediately. FUTEX_WAKE(addr, n): wake up to \( n \) sleepers queued on addr. The compare inside WAIT is the entire intellectual content: it makes "check the lock word and go to sleep" atomic against a concurrent unlocker, which closes the lost-wakeup window that would exist if user space checked and then slept in two steps. The kernel keeps no state for an uncontended futex; there is no registration, no per-lock kernel object, nothing until someone actually waits.

A production-shaped mutex needs three states, not two: 0 unlocked, 1 locked with no waiters, 2 locked with possible waiters. The third state is what lets the unlock fast path skip the FUTEX_WAKE syscall when nobody is sleeping, and its maintenance, waiters must re-lock to state 2, not 1, because after one wakeup other sleepers may remain, is where hand-rolled futex code classically goes wrong. 

/* futex_mutex.c: a three-state futex mutex, after Drepper's analysis.
 * states: 0 = unlocked, 1 = locked, 2 = locked with (possible) waiters.
 * Fast paths are one CAS / one XCHG; the kernel is entered only under
 * contention. gcc -O2 -pthread futex_mutex.c */
#define _GNU_SOURCE
#include <stdatomic.h>
#include <linux/futex.h>
#include <sys/syscall.h>
#include <unistd.h>
#include <limits.h>

typedef struct { _Atomic unsigned v; } mutex_t;   /* 32-bit word */

static long futex(_Atomic unsigned *addr, int op, unsigned val) {
    return syscall(SYS_futex, addr, op, val, NULL, NULL, 0);
}

void mutex_lock(mutex_t *m) {
    unsigned c = 0;
    /* fast path: 0 -> 1, one CAS, ~6 ns uncontended on this machine */
    if (atomic_compare_exchange_strong(&m->v, &c, 1)) return;
    /* slow path: force state to 2 so unlock knows waiters may exist.
     * If the xchg returns 0 the lock was just released and we own it
     * (as 2, conservatively: other sleepers may remain). */
    if (c != 2) c = atomic_exchange(&m->v, 2);
    while (c != 0) {
        /* sleep iff the word is STILL 2: the kernel re-checks under
         * its own lock, closing the lost-wakeup race */
        futex(&m->v, FUTEX_WAIT_PRIVATE, 2);
        /* on wakeup, re-acquire as 2, NOT 1: see comment above.
         * This mirrors Drepper's third, correct mutex exactly. */
        c = atomic_exchange(&m->v, 2);
    }
}

void mutex_unlock(mutex_t *m) {
    /* fast path: if we were in state 1, nobody waits; just store 0 */
    if (atomic_exchange(&m->v, 0) == 2)
        futex(&m->v, FUTEX_WAKE_PRIVATE, 1);   /* wake one waiter */
}

// Same three-state futex mutex in Rust. std::sync::Mutex has been
// futex-based on Linux since Rust 1.62 and follows this exact shape;
// this version exposes the mechanics. cargo add libc
use std::sync::atomic::{AtomicU32, Ordering::{Acquire, Release, Relaxed}};

pub struct FutexMutex { v: AtomicU32 }   // 0 free, 1 locked, 2 contended

fn futex(addr: &AtomicU32, op: i32, val: u32) {
    unsafe {
        libc::syscall(libc::SYS_futex, addr as *const AtomicU32,
                      op, val, 0usize, 0usize, 0u32);
    }
}

impl FutexMutex {
    pub const fn new() -> Self { Self { v: AtomicU32::new(0) } }

    pub fn lock(&self) {
        if self.v.compare_exchange(0, 1, Acquire, Relaxed).is_ok() {
            return;                                   // fast path
        }
        loop {
            let prev = self.v.swap(2, Acquire);       // mark contended
            if prev == 0 { return; }                  // got it (as 2)
            futex(&self.v, libc::FUTEX_WAIT | libc::FUTEX_PRIVATE_FLAG, 2);
            // loop: re-attempt as contended; spurious wakeups are fine
        }
    }

    pub fn unlock(&self) {
        if self.v.swap(0, Release) == 2 {             // waiters exist
            futex(&self.v, libc::FUTEX_WAKE | libc::FUTEX_PRIVATE_FLAG, 1);
        }
    }
}

The cost model, entirely from this machine's measurements: the uncontended fast path is one CAS, 5.5 ns (concurrency.json measures an uncontended mutex lock/unlock at 5.53 ns per operation, 180.8 M ops/s). The contended slow path pays one futex syscall in the waiter (~134 ns of boundary plus queueing), a context switch out (~1,885 ns), a switch back in on wake, and the cache refill tax. The ratio between the paths is three orders of magnitude, which is the quantitative case for the golden rule of lock design: make critical sections short enough that contention stays rare, because the primitive's cost is bimodal, not average. Under real contention on this machine an 8-thread mutex-protected counter collapses to 11.5 M ops/s aggregate, 255x worse than uncontended arithmetic; that cliff, and the sharded designs that avoid it, are the concurrency page's subject.

Deadlock: the Coffman conditions and the banker

Coffman, Elphick, and Shoshani (1971) reduced deadlock to four simultaneously necessary conditions: mutual exclusion (resources are held exclusively), hold-and-wait (holders can request more), no preemption (resources cannot be revoked), and circular wait (a cycle in the who-waits-for-whom graph). The taxonomy is useful because it is a menu: break any one condition and deadlock is impossible. Practice almost always breaks circular wait with a global lock ordering, acquire in a fixed order, and the concurrency page demonstrates both the hang (two transfers locking accounts in argument order, killed by timeout on this machine) and the fix (std::scoped_lock's all-or-nothing acquisition). The kernel's own lockdep validator does the same thing dynamically: it records the acquisition order classes observed and reports the first cycle-creating acquisition, deadlock detection at development time.

Deadlock avoidance, deciding at request time whether a grant could lead to deadlock, is Dijkstra's banker's algorithm: grant a request only if the resulting state is safe, meaning there exists some order in which all processes can run to their declared maximum and finish. The test is a simulation: repeatedly find a process whose remaining need fits in the currently available pool, pretend it finishes and returns its holdings, and see whether everyone eventually finishes. Its production footprint is small, it needs declared maxima and a closed world, but it is the canonical safety-versus-liveness computation, and worth doing by hand once.

Problem 3

A system has three resource types with totals \( (9, 6, 4) \). Current allocation and declared maxima: A holds \( (1,1,0) \) of max \( (4,2,1) \); B holds \( (2,1,1) \) of max \( (5,2,2) \); C holds \( (2,0,1) \) of max \( (3,1,3) \); D holds \( (1,1,0) \) of max \( (2,3,1) \). (a) Show the state is safe. (b) B requests \( (1,0,1) \): grant or defer? (c) Separately, in a one-resource system with 12 units, A holds 5 of max 10, B holds 2 of max 4, C holds 2 of max 9. C requests 1 unit while 3 are free. Grant or defer?

Solution. (a) Held total is \( (6,3,2) \), so available is \( (9,6,4) - (6,3,2) = (3,3,2) \). Needs (max minus held): A \( (3,1,1) \), B \( (3,1,1) \), C \( (1,1,2) \), D \( (1,2,1) \). C's need fits in \( (3,3,2) \): run C, reclaim its \( (2,0,1) \), available becomes \( (5,3,3) \). D fits: available \( (5,3,3)+(1,1,0) = (6,4,3) \). A fits: \( (7,5,3) \). B fits: \( (9,6,4) \), the full pool. Safe sequence C, D, A, B exists, so the state is safe.

(b) Pretend-grant: available \( (3,3,2)-(1,0,1) = (2,3,1) \); B now holds \( (3,1,2) \), need \( (2,1,0) \). Search for a safe sequence: A needs \( (3,1,1) \), but only 2 units of the first resource are free, blocked. B's new need \( (2,1,0) \le (2,3,1) \): run B, reclaim \( (3,1,2) \), available \( (5,4,3) \). Then A: \( (6,5,3) \); then C \( (1,1,2) \le (6,5,3) \): \( (8,5,4) \); then D: \( (9,6,4) \). Sequence B, A, C, D finishes everyone: the post-grant state is safe, so grant.

(c) Pretend-grant: available \( 3 - 1 = 2 \); C holds 3, needs \( 9 - 3 = 6 \). Who can finish with 2 free? A needs \( 10-5 = 5 \): no. B needs \( 4-2 = 2 \): yes; run B, reclaim its 4 held (2 original plus... B held 2, so available \( 2 + 2 = 4 \)). Now A needs 5: no. C needs 6: no. No process can finish: the state after the grant is unsafe, so defer, even though a unit is physically free. This is the algorithm's whole point: it refuses states from which an unlucky but legal future cannot be survived, trading utilization for a guarantee.

Virtual memory: paging and the 4-level walk

Why paging won

The address-space abstraction promises each process a private, contiguous, zero-based memory while the kernel packs everyone into shared physical DRAM. Translation at some granularity is forced by that promise; the granularity choices were segments (variable-size, matching program structure) or pages (fixed-size, matching nothing). Paging won because external fragmentation is the killer: with variable-size segments, free physical memory shatters into unusable slivers and compaction means copying, while fixed 4 KiB pages make every free frame interchangeable, allocation a bitmap operation, and eviction a per-page decision. The costs paging accepts in exchange, internal fragmentation of at most one page per region, and a translation structure that must map a 48-bit space without occupying it, define the rest of this section.

The naive structure fails arithmetic immediately: a flat array of 8-byte entries for every 4 KiB page of a 48-bit space is \( 2^{48-12} \times 8 = 2^{39} \) bytes, 512 GiB, per process. The fix is to make the table a radix tree and allocate only the paths actually used. x86-64 splits the 48-bit virtual address into four 9-bit indices plus a 12-bit page offset, giving a 4-level tree in which each node is itself exactly one 4 KiB page holding 512 8-byte entries; a page of a process that maps almost nothing costs four pages of table, not 512 GiB. (Ice Lake and later, including this machine's Sapphire Rapids, optionally extend to 5 levels and 57 bits when the kernel enables LA57.)

Problem 4

Decompose the user virtual address 0x00007f3bd1c04a20 into its x86-64 4-level indices, give the byte offset the hardware reads within each table page, count the memory references the full translation costs, and state where the split would occur if the mapping used a 2 MiB huge page.

Solution. Write the address as bits and peel fields from the top. \( \text{PML4} = (\text{addr} \gg 39) \,\&\, 511 = 254 \); \( \text{PDPT} = (\text{addr} \gg 30) \,\&\, 511 = 239 \); \( \text{PD} = (\text{addr} \gg 21) \,\&\, 511 = 142 \); \( \text{PT} = (\text{addr} \gg 12) \,\&\, 511 = 4 \); offset \( = \text{addr} \,\&\, 4095 = 0\text{xa20} = 2592 \). Reassembling \( (254 \ll 39) + (239 \ll 30) + (142 \ll 21) + (4 \ll 12) + 2592 \) reproduces 0x7f3bd1c04a20 exactly, confirming the split. Since each entry is 8 bytes, the hardware reads the PML4 page at byte offset \( 254 \times 8 = 2032 \), the PDPT at \( 239 \times 8 = 1912 \), the PD at \( 142 \times 8 = 1136 \), and the PT at \( 4 \times 8 = 32 \): four dependent memory reads before the data byte itself, five total for one load. On a 2 MiB huge page the PD entry has the PS bit set and is the leaf: translation stops after three reads, and the low \( 21 \) bits, \( \text{addr} \,\&\, 0\text{x1FFFFF} = 0\text{x4a20} = 18{,}976 \), become the in-page offset. One fewer level, and, far more importantly, 512x more reach per TLB entry.

Each entry carries control bits beside the frame number: Present (clear means any access faults, the hook demand paging hangs on), Writable and User (checked at every level; a supervisor-only bit anywhere on the path protects the kernel half of the space), Accessed and Dirty (set by hardware, harvested by replacement), NX (no instruction fetch, the cheap end of exploit mitigation), and PS (this entry is a 2 MiB or 1 GiB leaf). A page fault delivers the faulting address in CR2 plus an error code, and the kernel's fault handler (mm/memory.c on Linux, handle_mm_fault) decides among: allocate and zero a page (first touch of anonymous memory), read from the page cache (file mapping), copy a shared page (copy-on-write), grow the stack, or deliver SIGSEGV. The measured cost of the benign minor fault on this machine: 1,509 ns per first-touch of a 4 KiB anonymous page, against 21.5 ns for a store to an already-mapped page, a 70x gap that explains why allocators recycle memory rather than returning it to the kernel.

TLBs and huge pages: the 1.9x measurement

The four dependent reads of Problem 4 would make every load five loads, so the translation lookaside buffer caches final translations, and hardware page-walk caches hold the upper levels; the microarchitecture is on the architecture page. What the kernel controls is reach. A typical two-level TLB of order a few thousand entries covers only a few MiB at 4 KiB per entry: this is tiny against a multi-GiB heap, and once the working set of translations exceeds it, every access pays a walk. At 2 MiB per entry the same TLB covers hundreds of times more, which is the entire case for huge pages.

Measured here, with a benchmark built to isolate translation: an anonymous region is faulted in, then a random Hamiltonian cycle visiting one cache line per 4 KiB page is chased with fully dependent loads, so neither the prefetcher nor out-of-order overlap can hide anything. Over 32 GiB with 4 KiB pages: 255 ns per access. Same region backed by 2 MiB transparent huge pages: 134.6 ns, a 1.9x difference with identical DRAM traffic, pure translation overhead. Over a 4 GiB region the gap narrows (178.7 versus 122.8 ns, 1.45x) because 4 GiB of PTEs is 8 MiB, which mostly fits in this machine's caches, while 32 GiB of PTEs is 64 MiB, which does not: the walk itself starts missing cache. Sequential controls confirm the mechanism: a 64 B stride costs 4.5 ns/access regardless of page size (prefetch hides everything), and a 4,096 B sequential stride costs ~19 ns, cheap because 512 consecutive PTEs share one table page and eight of them share one cache line.

Copy-on-write, fork, and mmap

fork's survival depends on not copying memory. The child gets a copy of the page tables with both copies' PTEs marked read-only; the first write on either side faults, and the handler copies just that page. Measured on this machine, the scaling law is visible: forking a process with 1 MiB resident costs 39.2 µs, 16 MiB costs 122.2 µs, 128 MiB costs 798.6 µs, 512 MiB costs 3.27 ms, and 2 GiB costs 14.27 ms, an asymptotic ~25-27 ns per resident page of pure page-table duplication and refcounting. The deferred price also measured: the first write to an inherited page costs about 2.0 µs (fault, allocate, copy 4 KiB, remap), roughly the same as a minor fault plus a page copy, paid only for pages actually written. The design consequence: fork-then-exec from a large process wastes the whole table copy, 14 ms for a 2 GiB parent, which is why posix_spawn (measured 432.6 µs versus 499.7 for fork+exec of a small parent, and unboundedly better for large ones) exists, and why Redis-style fork-for-snapshot pays a known COW tax proportional to the write rate during the snapshot.

mmap generalizes the same machinery to files: a mapping is a VMA whose faults are satisfied from the page cache instead of by zero pages, giving file I/O with load instructions. It is not automatically faster than read(): measured here on a warm 2 GiB file, read() streams at 9.02 GB/s while touching the same data through a mapping runs at 7.35 GB/s and takes 32,768 minor faults, one per 64 KiB faultahead cluster; the copy that mmap avoids costs less than the faults it adds at this access pattern. mmap wins for random access to large files (no syscall per access, no double buffering) and for sharing (two processes mapping one file share physical pages), not as a universal read replacement.

Page replacement: Belady, clock, and the working set

When memory is scarce the kernel must choose a page to evict, and the theory has one exact result: Belady's OPT (1966, IBM), evict the page whose next use is farthest in the future, is optimal and unimplementable, existing to calibrate everything else. LRU approximates OPT by using the past as the future's proxy and inherits a valuable structural property: it is a stack algorithm, the set of pages LRU holds with \( k \) frames is always a subset of what it holds with \( k+1 \), so more memory can never cause more faults. FIFO is not a stack algorithm, and Belady, Nelson, and Shedler (1969) showed the consequence, an anomaly worth verifying by hand once in a lifetime.

Problem 5

Run FIFO on the reference string 1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5 with 3 frames and with 4 frames, counting faults. Then state what LRU gives on both sizes and why the FIFO behavior cannot happen to LRU.

Solution. Three frames (queue order, oldest first; F = fault, H = hit): 1→F [1]; 2→F [1 2]; 3→F [1 2 3]; 4→F [2 3 4]; 1→F [3 4 1]; 2→F [4 1 2]; 5→F [1 2 5]; 1→H; 2→H; 3→F [2 5 3]; 4→F [5 3 4]; 5→H. Total: 9 faults. Four frames: 1, 2, 3, 4 all fault [1 2 3 4]; 1→H; 2→H; 5→F [2 3 4 5]; 1→F [3 4 5 1]; 2→F [4 5 1 2]; 3→F [5 1 2 3]; 4→F [1 2 3 4]; 5→F [2 3 4 5]. Total: 10 faults. Adding a frame made FIFO worse, because the early hits on 1 and 2 left them ancient in the queue exactly when the string returned to them. LRU gives 10 faults at 3 frames and 8 at 4, monotone as the stack property requires: with LRU the \( k \)-frame content is the \( k \) most recently used pages, which is by construction a subset of the \( k+1 \) most recent, so a hit at \( k \) frames is a hit at \( k+1 \). The same simulation run in this repository (os.json) also checked clock on this string: 9 and 10 faults, inheriting FIFO's anomaly, and OPT: 7 and 6.

True LRU is too expensive to maintain per-access in hardware-assisted paging, so real kernels run clock: pages in a circular list, one reference bit each, set by hardware on access; the hand clears set bits and evicts the first clear one, giving each page one lap of grace. Simulated on a 20,000-reference trace with a moving 8-page hot set (85% of references) over 64 pages, this repository's runs rank the policies as theory predicts: at 16 frames, OPT 1,553 faults, LRU 2,303, clock 2,550, FIFO 3,818; and on a uniformly random trace all three practical policies collapse together (15,080-15,096 faults at 16 frames) because without locality there is nothing to learn. That collapse is the important negative result: replacement policy is a bet on locality, not a free win.

Denning's working set model (1968) supplies the vocabulary for when paging works at all: \( W(t, \theta) \), the set of pages referenced in the last \( \theta \) references, and its size on the simulated trace grows from 6.7 pages at \( \theta = 10 \) to 19.5 at \( \theta = 100 \) to only 54.5 at \( \theta = 1000 \), the flattening curve that defines a working set. The policy consequence: give each process frames for its working set or do not run it, because a process below that threshold faults continuously, and a system of such processes spends its time paging instead of computing, thrashing, the failure mode where throughput falls as load rises. Modern Linux descendants of these ideas: the split active/inactive LRU lists, and since 6.1 the multi-generational LRU contributed by Google, which tracks page age in generations precisely because two lists quantize recency too coarsely.

File systems: from inodes to journals

The VFS layer and the inode

Linux interposes a virtual file system layer between the syscall interface and any concrete file system: four object types, superblock (one per mount: block size, root, operations), inode (one per file: metadata and the map from file offsets to blocks, but no name), dentry (a name component mapping to an inode, cached aggressively because path lookup is the hottest metadata operation in the system), and file (an open instance: position, mode, a pointer into the dentry cache). The separation of name from file is load-bearing: hard links are just multiple dentries for one inode, and unlinking while a process holds the file open works because the inode's life is governed by link count plus open count, not by any name. Every concrete file system, ext4, XFS, Btrfs, tmpfs, FUSE, supplies operation tables for these objects, which is why cat works identically over all of them.

The classic UNIX inode maps offsets to blocks with a small fixed array biased toward small files: 12 direct block pointers, then one single-indirect (a block full of pointers), one double-indirect, one triple-indirect. With 4 KiB blocks and 4-byte pointers, one indirect block holds 1,024 pointers, and the tiers cover: direct \( 12 \times 4\,\text{KiB} = 48 \) KiB; single \( 1024 \times 4\,\text{KiB} = 4 \) MiB; double \( 1024^2 \times 4\,\text{KiB} = 4 \) GiB; triple \( 1024^3 \times 4\,\text{KiB} = 4 \) TiB; total maximum file size \( 48\,\text{KiB} + 4\,\text{MiB} + 4\,\text{GiB} + 4\,\text{TiB} = 4{,}402{,}345{,}721{,}856 \) bytes. The asymmetry is deliberate: measurement has always shown most files are small (so 12 direct pointers usually suffice and metadata stays one block), while the tree depth grows only logarithmically for the rare huge file.

Problem 6

In such a file system (4 KiB blocks, 4-byte pointers, 12 direct), a process reads one byte at offset 1,300,000,000 of a cold file. Which pointer chain resolves it, with exact indices at each level, and how many disk reads does the byte cost, assuming the inode itself is cached but no data or indirect blocks are?

Solution. The file block index is \( \lfloor 1{,}300{,}000{,}000 / 4096 \rfloor = 317{,}382 \), with in-block offset \( 1{,}300{,}000{,}000 - 317{,}382 \times 4096 = 3{,}328 \). Subtract the direct tier: \( 317{,}382 - 12 = 317{,}370 \). Subtract the single indirect tier's 1,024 blocks: \( 317{,}370 - 1024 = 316{,}346 \), which is less than the double tier's \( 1024^2 = 1{,}048{,}576 \), so the double-indirect pointer resolves it. Outer index: \( \lfloor 316{,}346 / 1024 \rfloor = 308 \); inner index: \( 316{,}346 - 308 \times 1024 = 954 \). The chain: inode's double-indirect pointer → outer indirect block, entry 308 → inner indirect block, entry 954 → data block, byte 3,328. Three disk reads (outer, inner, data). The general lesson: offset arithmetic is pure integer division, no search, which is what makes the inode's map O(depth) rather than O(file size), and why a byte at offset 1.3 GB costs two more reads than a byte at offset 0.

Modern file systems replaced per-block pointers with extents, ext4's being a representative design (Mathur et al., Ottawa Linux Symposium 2007): an extent is (logical block, length up to 32,768 blocks, physical start), so a contiguous 128 MiB file is described by one 12-byte record instead of 32,768 pointers, with a B-tree of extents for fragmented files. Extents win threefold: metadata shrinks by orders of magnitude, contiguity is encoded rather than incidental (enabling large sequential I/O requests), and allocation can be delayed until writeback so the allocator sees whole files, not single blocks.

FFS: putting layout near the arm

McKusick, Joy, Leffler, and Fabry's Fast File System (Berkeley, 1984) is the paper that made file-system design an empirical performance discipline. The original UNIX file system put all inodes at the start of the disk and treated blocks as interchangeable, so reading any file seeked across the whole platter between metadata and data, and the paper reports it delivering only a few percent of disk bandwidth. FFS's move was structural: divide the disk into cylinder groups, each with its own inode table, bitmap, and data blocks; place a file's inode and data in the same group, files of a directory together, and spread directories across groups; plus the pragmatic changes, larger blocks with fragments for the tails of small files, and a ~10% free-space reserve because allocation quality collapses on a full disk (the paper measured up to 47% of disk bandwidth after these changes). Every rotating-disk file system since is recognizably FFS-shaped, and the ideas survive SSDs in modified form: locality still matters, but for erase blocks and mapping-table pressure rather than seek arms.

Crash consistency: the argument for the journal

Appending one block to a file updates three on-disk structures: the data block itself, the inode (new size, new pointer), and the block bitmap. A crash can strike between any pair of writes, and enumerating the six orderings gives the whole problem. Data alone: harmless, an orphaned write. Bitmap alone: a space leak. Inode alone: the file now points at a block containing whatever was there before, someone else's old data, both a corruption and a security hole. Inode+bitmap without data: the file durably contains garbage. The old answer, fsck, scans the entire disk at boot reconciling structures; it repairs invariants but cannot restore data, and its cost scales with disk size, hours on modern volumes.

Journaling imports write-ahead logging from databases: before touching home locations, write a description of the whole update into a log, ending with a commit record; only after the commit is durable, checkpoint the changes to their home locations; after a crash, replay committed transactions and discard uncommitted ones. The invariant is atomicity at the transaction level: the append either happens entirely or not at all, and recovery time is proportional to the journal, not the disk. ext4's jbd2 makes the trade-offs selectable. data=journal logs data and metadata both: full atomicity, everything written twice. data=ordered (the default) logs only metadata but enforces one ordering rule, data blocks reach disk before the metadata transaction that references them commits: the inode can never durably point at garbage, at half the write cost; what remains possible is a file whose data landed but whose metadata did not (the append vanishes, harmlessly). data=writeback drops the ordering rule and re-admits the stale-data window in exchange for scheduling freedom. The rule of thumb this generates: ordered mode protects the file system's integrity and metadata's atomicity, but an application that needs its own data durable must still call fsync, and the price was measured here: a 4 KiB append plus fsync costs 608.7 µs on this machine's virtio disk, against 1.15 µs without, a 530x durability tax that batching (group commit) exists to amortize.

Ganger, McKusick, Soules, and Patt's soft updates (Michigan and Berkeley, published in full in TOCS 2000) is the competing line: instead of a log, track fine-grained dependencies between in-memory metadata blocks and order writeback so that the disk only ever transitions between consistent states (initialize an inode before the directory entry that names it; clear the entry before freeing the inode). No double-write, near-fsck-free recovery; the cost is fearsome implementation complexity, dependency cycles must be broken by rolling back parts of blocks at write time, and journaling's comparative simplicity is why it won everywhere but FreeBSD's FFS.

LFS: the log becomes the file system

Rosenblum and Ousterhout (Berkeley, 1992) pushed the logic to its limit: if the journal must be written anyway, make it the only structure. LFS buffers all writes, data, inodes, indirect blocks, into memory and writes them as one large sequential segment; nothing is ever overwritten in place. Finding things back requires one level of indirection, the inode map (inode number → latest inode location), itself written into the log and rooted in a periodically-updated checkpoint region. Every write is sequential, so write bandwidth approaches disk bandwidth by construction; the price is that old segments fill with dead versions and a cleaner must compact live data to reclaim space. The cleaning cost is derivable in three lines. To reclaim a segment whose live fraction is \( u \): read the whole segment (1 segment of I/O), write back the live \( u \), netting \( 1 - u \) of fresh space; new data then fills that space, another \( 1 - u \) of writes. Total I/O per unit of new data: $$ \text{write cost} \;=\; \frac{1 + u + (1-u)}{1-u} \;=\; \frac{2}{1-u}. $$ At \( u = 0.5 \) every byte costs 4 bytes of I/O; at \( u = 0.8 \), 10; at \( u = 0.9 \), 20. The design consequence is bimodality: LFS must clean cold segments when they are nearly empty and avoid touching hot ones, and the paper's cost-benefit cleaner does exactly that. The idea outlived the file system spectacularly: every SSD firmware is an LFS (the FTL remaps writes into erase blocks and garbage-collects, with \( 2/(1-u) \) reappearing as write amplification), as are F2FS, most key-value store compaction schemes, and every copy-on-write file system's allocation strategy.

I/O: the page cache, epoll, and io_uring

The path of a read, measured layer by layer

A buffered read traverses: syscall → VFS → page cache (hit ends here) → file system → block layer (bio assembly, per-queue scheduling, request merging) → driver → device, with blk-mq providing one submission queue per CPU so 52 cores do not serialize on one lock. The page cache dominates the economics, measured on this machine's virtio disk with a 2 GiB file: cold sequential read 2.28 GB/s, warm 9.02 GB/s; cold random 4 KiB reads 47.0 µs each, warm 1.0 µs, a 47x gap. O_DIRECT, which bypasses the cache entirely, reads at 2.11 GB/s, the device's own speed; databases that maintain their own buffer pools choose it precisely to stop paying for two caches holding the same bytes.

For many concurrent I/O sources the question becomes how to wait. Blocking a thread per fd costs a stack and two context switches per operation (3.7-5.9 µs measured round trips above). Readiness models multiplex: select and poll hand the kernel the whole fd list every call, O(n) each time, while epoll registers interest once and each wait returns only ready fds. The per-event price measured here: a socketpair ping-pong costs 5,885 ns blocking versus 6,342 with an epoll_wait before each read, about 460 ns for the extra syscall; epoll's win is not per event but per idle fd, it makes 100,000 mostly-idle connections cost nothing per wait, which readiness models with per-call registration cannot.

io_uring: completion, batched, on shared rings

Readiness has two structural residues: one syscall per actual I/O remains, and regular files are always "ready," so epoll cannot express asynchronous disk reads at all. Axboe's io_uring (2019, "Efficient IO with io_uring") is the completion-model answer: two ring buffers shared between user space and kernel via mmap, a submission queue of 64-byte SQEs describing operations and a completion queue of CQEs carrying results. User space writes SQEs and advances the tail with a release store; one io_uring_enter submits any number of them; completions appear in the CQ readable with acquire loads and no syscall whatsoever. Optional flags remove even the submit call (SQPOLL: a kernel thread polls the ring) and per-op overheads (registered buffers and files). The memory-ordering discipline on the ring indices is the acquire/release pattern derived on the concurrency page, deployed at the kernel boundary. 

              shared memory (mmap'd from the ring fd)
   user space                                        kernel
  ──────────────  SQ ring: [ SQE | SQE | SQE | ... ] ──────────────
  write SQEs,  ─→ tail (release store)      head ─→  consume SQEs,
  bump tail                                          start I/O
                  CQ ring: [ CQE | CQE | ... ]
  read CQEs,   ←─ head (acquire load)       tail ←─  post results
  bump head
  one io_uring_enter() submits N and/or waits for M completions;
  with SQPOLL, steady state needs zero syscalls

Measured here, driving the rings through raw io_uring_setup/io_uring_enter syscalls (this machine has no liburing headers, which forced the instructive path): random 4 KiB reads from a 512 MiB file, cold cache, queue depth 64 versus a pread loop: 151.4 kIOPS versus 37.4 kIOPS, 4.0x, because the device finally sees a queue instead of one request at a time. Warm cache, where there is no device to overlap: pread 851 ns, io_uring 790 ns, and a separate queue-depth sweep on a 1 GiB warm file measured 1,078 ns for pread, 1,069 at depth 1, 956 at depth 32, batching recovers only the ~130 ns syscall share of an operation dominated by the 4 KiB copy itself. The honest summary: io_uring is transformative when there is real device parallelism or syscall count is the bottleneck, and near-neutral for cached reads, which is exactly what the mechanism predicts. 

/* uring_read.c (core): io_uring with no liburing, so the actual ABI
 * is visible. Error handling elided here; full source compiled and
 * run for the numbers quoted on this page. */
#include <linux/io_uring.h>
#include <sys/syscall.h>
#include <sys/mman.h>
#include <stdatomic.h>

struct io_uring_params p = {0};
int ring = syscall(__NR_io_uring_setup, 64, &p);

/* map the SQ ring (indices + array), CQ ring (indices + CQEs), and
 * the SQE array; FEAT_SINGLE_MMAP lets SQ and CQ share one mapping */
size_t sq_sz = p.sq_off.array + p.sq_entries * sizeof(unsigned);
size_t cq_sz = p.cq_off.cqes  + p.cq_entries * sizeof(struct io_uring_cqe);
char *sq = mmap(NULL, sq_sz, PROT_READ|PROT_WRITE,
                MAP_SHARED|MAP_POPULATE, ring, IORING_OFF_SQ_RING);
char *cq = (p.features & IORING_FEAT_SINGLE_MMAP) ? sq
         : mmap(NULL, cq_sz, PROT_READ|PROT_WRITE,
                MAP_SHARED|MAP_POPULATE, ring, IORING_OFF_CQ_RING);
struct io_uring_sqe *sqes =
    mmap(NULL, p.sq_entries * sizeof(*sqes), PROT_READ|PROT_WRITE,
         MAP_SHARED|MAP_POPULATE, ring, IORING_OFF_SQES);
unsigned *sq_tail = (unsigned *)(sq + p.sq_off.tail);
unsigned *sq_mask = (unsigned *)(sq + p.sq_off.ring_mask);
unsigned *sq_arr  = (unsigned *)(sq + p.sq_off.array);

/* submit a batch of `depth` reads, then wait for all of them */
unsigned tail = *sq_tail;
for (unsigned i = 0; i < depth; i++) {
    unsigned idx = tail & *sq_mask;
    struct io_uring_sqe *s = &sqes[idx];
    memset(s, 0, sizeof *s);
    s->opcode = IORING_OP_READ;
    s->fd     = file;
    s->addr   = (uint64_t)bufs[i];
    s->len    = 4096;
    s->off    = random_page_offset();
    sq_arr[idx] = idx;
    tail++;
}
/* publish: the kernel's acquire load must see completed SQEs */
atomic_store_explicit((_Atomic unsigned *)sq_tail, tail,
                      memory_order_release);
syscall(__NR_io_uring_enter, ring, depth, depth,
        IORING_ENTER_GETEVENTS, NULL, 0);
/* CQEs are now readable between cq head (ours) and tail (kernel's,
 * loaded with memory_order_acquire) */

Containers: namespaces plus cgroups

A container is not a kernel object. It is a process created with an unusual set of clone/unshare flags (namespaces: what the process can see) and placed in a cgroup (what it can consume), usually with a changed root (pivot_root), dropped capabilities, and a seccomp filter (measured above: ~11 ns per syscall). The namespaces virtualize the kernel's naming: pid (process 1 inside is a normal pid outside), mnt (private mount table), net (own interfaces, ports, routing), uts (hostname), ipc, user (uid 0 inside maps to an unprivileged uid outside, the flag that made rootless containers possible), cgroup, and time. All of it is demonstrable with stock tools, run here as an unprivileged user:

Reading a container runtime confirms there is no other magic: runc's libcontainer issues exactly these unshares, writes uid_map, mounts a new /proc (so ps inside sees the pid namespace), pivot_roots into the image, writes the cgroup files, and execs the entrypoint. The economic consequence of "container = process" is the entire density story: starting one is a fork+exec (measured ~500 µs) plus namespace setup, not a kernel boot, and a thousand containers share one page cache and one scheduler. The security consequence cuts the other way: every container shares the host kernel's ~350-syscall attack surface, one kernel exploit is a full escape, which is precisely the gap gVisor (a user-space kernel intercepting syscalls) and firecracker-style microVMs exist to close.

The cgroup side deserves one mechanical note: cpu.max is a quota, not a share. A container with cpu.max = 50000 100000 gets 50 ms of CPU per 100 ms period and is throttled for the remainder even if the machine is idle, a famous source of tail latency for multithreaded services whose N threads burn the quota in period/N wall time. cpu.weight is the work-conserving alternative, EEVDF weights (the same 1024 arithmetic worked above) applied to groups; the pressure stall information files (cpu.pressure, memory.pressure) are the modern signal for "this cgroup is starving," and Kubernetes' CPU-limit debates are this paragraph with a larger cast.

Virtualization: one more level of the same idea

A hypervisor applies the process abstraction one level down: to a guest kernel, the "hardware" is virtual, and the host multiplexes real hardware among guests. Popek and Goldberg (1974) gave the criterion under which naive trap-and-emulate works: every sensitive instruction (one that reads or changes privileged machine state) must be privileged (trap when executed without privilege). Run the guest kernel deprivileged; sensitive instructions trap; the hypervisor emulates them against a virtual machine state block. Classic x86 fails the criterion, instructions like popf silently do less instead of trapping when deprivileged, so the first x86 hypervisors could not trap-and-emulate. VMware's answer was binary translation, rewriting guest kernel code on the fly (Adams and Agesen's 2006 paper is the definitive account); Xen's was paravirtualization (Barham et al., SOSP 2003, Cambridge), modifying the guest to call the hypervisor explicitly, with reported overheads of a few percent. Intel VT-x and AMD-V then fixed the architecture: a new root/non-root mode distinction, a hardware-defined control structure (VMCS) holding guest state, and defined exits, restoring Popek-Goldberg by construction and enabling KVM, a kernel module that turns Linux itself into the hypervisor, with QEMU or a minimal VMM supplying device models from user space.

The memory story composes with everything above: the guest translates virtual to guest-physical through its own 4-level tree, and hardware then translates each guest-physical address to host-physical through a second 4-level tree (EPT/NPT). The composition is quadratic in the worst case: each of the guest walk's 4 loads is itself a guest-physical address needing a 4-level EPT walk, plus EPT translation of the final data address, so a fully cold translation costs up to \( 5 \times 5 - 1 = 24 \) memory references against 4 native. TLBs cache the composed translation so the steady state is native-speed, but TLB-miss-heavy workloads, exactly the 32 GiB random chase measured above, are the ones that hurt most inside VMs, and huge pages on both levels are the standard remedy. I/O composes similarly: trapping on every device register access costs a VM exit (microseconds) per operation, so virtio defines paravirtual devices around shared-memory descriptor rings, the io_uring shape again, with exits only for doorbells, and this machine's own disk is a virtio-blk device behind exactly that design. The endpoint of the minimalist line is Firecracker (Amazon, NSDI 2020): a ~50k-line Rust VMM offering only virtio-net, virtio-blk, and a serial console, booting a microVM in about 125 ms at ~5 MiB overhead, cheap enough to give every serverless function hardware isolation, the container density story with the shared-kernel risk removed.

Worked problems

Six problems are worked in context above (syscall budgets, the CFS derivation, the banker, the page walk, Belady, and inode arithmetic). Three more round out the set.

Problem 7

Using this machine's measured 1,885 ns per context switch: (a) compute scheduling overhead at quanta of 10 ms, 1 ms, and 100 µs; (b) find the smallest quantum keeping direct overhead under 0.5%; (c) with 20 runnable tasks and a 5 ms quantum, bound the worst-case response time for a newly woken task, and explain which EEVDF parameter shrinks it without changing anyone's long-run share.

Solution. (a) Overhead is \( s/(q+s) \) with \( s = 1.885 \) µs. At \( q = 10 \) ms: \( 1.885/10001.885 = 0.019\% \). At \( q = 1 \) ms: \( 1.885/1001.885 = 0.188\% \). At \( q = 100 \) µs: \( 1.885/101.885 = 1.85\% \). (b) Require \( s/(q+s) \le 0.005 \Rightarrow q \ge 199\,s = 375.1 \) µs. (c) Worst case, the woken task waits behind 19 full quanta: \( 19 \times 5 = 95 \) ms, plus 19 switches (\( 35.8 \) µs, negligible). Under EEVDF the task's slice \( r_i \) sets its virtual deadline \( d_i = v_e + r_i/w_i \): a smaller slice gives an earlier deadline and therefore earlier selection among eligible tasks, improving wakeup latency, while its weight, and hence its long-run share, is untouched. This is the decoupling CFS could not express: under CFS the only latency lever was priority, which also changed the share.

Problem 8

A process on x86-64 (4-level, 4 KiB pages) maps four regions: 64 MiB of text+data, a 32 MiB heap, an 8 MiB stack, and a 16 MiB shared library, each contiguous, placed far apart, and fully populated. Compute the exact page-table storage, compare it to the flat-table alternative and to the mapped data, and recompute for the same regions backed by 2 MiB huge pages.

Solution. Leaf PT pages: each covers 2 MiB (512 entries x 4 KiB), so \( 64/2 + 32/2 + 8/2 + 16/2 = 32 + 16 + 4 + 8 = 60 \) PT pages. PD pages: each covers 1 GiB; each region needs one and they are far apart, so 4 (each holding 32, 16, 4, 8 live entries respectively). PDPT pages: each covers 512 GiB; user space spans 128 TiB, so assume the regions land under 2 distinct PDPTs (text low, stack and mmap regions high): 2. One PML4. Total \( 60 + 4 + 2 + 1 = 67 \) pages \( = 274{,}432 \) bytes, about 268 KiB. Against the mapped 120 MiB this is 0.22% overhead; against the flat table's \( 2^{36} \times 8 = 512 \) GiB it is seven orders of magnitude, the radix tree's sparsity doing exactly its job. With 2 MiB huge pages the PD entries become leaves: the 60 PT pages vanish, leaving \( 4 + 2 + 1 = 7 \) pages, 28 KiB, and every translation is one level shorter. The general formula: populated span divided by 2 MiB gives PT pages, and each higher level divides its child count by 512, so table storage is a geometric series dominated by the leaf level, which huge pages delete.

Problem 9

Model effective memory access time as \( \text{EAT} = t + (1-h)\,w \), where \( t \) is the access cost with a TLB hit, \( h \) the TLB hit rate, and \( w \) the added walk penalty on a miss. (a) Use this machine's 32 GiB chase measurements (255.0 ns at 4 KiB pages where essentially every access misses the TLB, 134.6 ns with 2 MiB THP where the residual translation cost is small because the walk is one level shorter and its 128 KiB of PMD entries stay cache-resident) to estimate \( w \). (b) Predict the slowdown for a workload with \( h = 0.98 \) and \( t = 100 \) ns, and for the same workload inside a VM if a cold composed walk is 24/4 times as expensive. (c) State the huge-page reach argument in one line of arithmetic.

Solution. (a) Treat the THP number as \( t \approx 134.6 \) ns (hit-dominated) and the 4 KiB number as \( t + w \) (miss-dominated): \( w \approx 255.0 - 134.6 = 120.4 \) ns, a walk penalty comparable to the DRAM access itself, which is what "the walk misses cache too" means quantitatively. (b) Native: \( \text{EAT} = 100 + 0.02 \times 120.4 = 102.4 \) ns, a 2.4% tax. Virtualized, scaling the penalty by \( 24/4 = 6 \): \( w' = 722.4 \) ns and \( \text{EAT} = 100 + 0.02 \times 722.4 = 114.4 \) ns, 14.4%: the same hit rate hurts six times more, which is why TLB behavior is the first thing to profile when a workload "mysteriously" runs slower in a VM. (c) A 1,536-entry TLB reaches \( 1536 \times 4\,\text{KiB} = 6 \) MiB with base pages but \( 1536 \times 2\,\text{MiB} = 3 \) GiB with huge pages: a 512x reach change from a flags bit.

Implementation: watching translation happen

The kernel exports enough introspection to verify the virtual-memory story empirically, no kernel module required. /proc/self/pagemap is an array of 64-bit records, one per virtual page, holding the physical frame number and flag bits; reading entry \( \text{va} / 4096 \) translates any address the process owns. The programs below map memory, translate it, and make copy-on-write directly observable: after fork, parent and child report the same physical frame for a page until one writes to it, at which point the writer's frame number changes. (Reading frame numbers needs CAP_SYS_ADMIN on hardened kernels; the soft-dirty and present bits work unprivileged.) 

/* vatophys.c: translate a virtual address via /proc/self/pagemap and
 * watch COW break sharing after fork. gcc -O2 vatophys.c (run as root
 * to see frame numbers; kernels hide them from unprivileged readers) */
#define _GNU_SOURCE
#include <stdio.h>
#include <stdint.h>
#include <stdlib.h>
#include <fcntl.h>
#include <unistd.h>
#include <sys/wait.h>

static uint64_t pagemap_entry(void *va) {
    int fd = open("/proc/self/pagemap", O_RDONLY);
    uint64_t ent = 0;
    off_t off = ((uintptr_t)va / 4096) * sizeof(uint64_t);
    pread(fd, &ent, sizeof ent, off);
    close(fd);
    return ent;               /* bit 63: present; bits 0-54: PFN */
}

int main(void) {
    char *p = aligned_alloc(4096, 4096);
    *p = 'x';                 /* fault it in */
    uint64_t e = pagemap_entry(p);
    printf("va=%p present=%llu pfn=%#llx\n", (void *)p,
           (unsigned long long)(e >> 63),
           (unsigned long long)(e & ((1ull << 55) - 1)));

    if (fork() == 0) {        /* child: same PFN until the write */
        printf("child  pre-write pfn=%#llx\n",
               (unsigned long long)(pagemap_entry(p) & ((1ull<<55)-1)));
        *p = 'y';             /* COW fault: ~2.0 us measured here */
        printf("child post-write pfn=%#llx\n",
               (unsigned long long)(pagemap_entry(p) & ((1ull<<55)-1)));
        _exit(0);
    }
    wait(NULL);
    return 0;
}

How it is done in practice

The production Linux stack in 2026, mapped to this page's sections: scheduling is EEVDF (since 6.6; this machine's 6.8 runs it), with cgroup cpu.weight carrying the same weight arithmetic across container boundaries and sched_ext (6.12) letting fleets load BPF schedulers without kernel forks. Memory is 4 KiB pages with THP in madvise mode almost everywhere, explicit hugetlbfs for databases and hypervisors that preallocate, and the multi-generational LRU handling reclaim on Android, ChromeOS, and increasingly servers. Storage is ext4 or XFS in ordered journaling for general fleets, Btrfs where snapshots pay their way, F2FS on phones, with io_uring behind high-performance proxies and databases and its syscall surface restricted in several managed platforms after a run of privilege-escalation bugs, security and performance negotiating in public, as usual.

The gap between the textbook and the fleet shows up in the incidents this material lets one read correctly. The 2018 PostgreSQL fsync affair: kernels historically could mark pages clean after an fsync failure, so a database that retried fsync and saw success had durably lost data; the fix was application-level (treat fsync failure as fatal) and kernel-level (error reporting reworked), and the lesson is this page's journaling section applied adversarially. Cgroup CPU quota throttling: services observing 100 ms latency spikes at p99.9 traced to cpu.max periods, fixed by switching limits to weights, the arithmetic worked above. THP compaction stalls in latency-sensitive stores, diagnosed by exactly the AnonHugePages bookkeeping used in the TLB section. And the standing cloud design choice, containers versus microVMs, is a table on this page: share the kernel and its 1,885 ns switches, or take Firecracker's 125 ms boots and hardware isolation. None of these yield to intuition; all yield to the measured numbers and the data structures.

The current research frontier

Scheduling has left the kernel. Google's ghOSt (SOSP 2021) delegates policy to user space, with BPF and agents making microsecond-scale decisions for fleets; Linux mainlined the idea's spirit as sched_ext, and published schedulers from Meta and Valve (latency-sensitive gaming) now iterate in weeks instead of kernel-release years. The kernel-bypass line runs in parallel: Shenango (NSDI 2019) and Caladan (OSDI 2020) from MIT reallocate cores between latency-critical and batch tasks at 5 µs granularity, treating the kernel scheduler itself as too slow, and Microsoft's Demikernel (SOSP 2021) proposes a library-OS shape for kernel-bypass I/O. Memory management research concentrates on tiering now that CXL makes far-memory a hardware product: Meta's TPP (ASPLOS 2023) and Microsoft's Pond (ASPLOS 2023) place pages across DRAM and CXL by temperature, the working-set model wearing new clothes; DAMON, contributed by Amazon and in-tree, makes access-pattern monitoring first-class. Huge-page policy remains open, the Ingens/HawkEye line continuing against THP's bloat/latency trade. On the assurance side, seL4 (SOSP 2009, NICTA/UNSW) remains the benchmark: a machine-checked proof from specification to C of a capability microkernel, with the total effort the paper reports, on the order of twenty person-years across kernel and proof, defining the current price of certainty. And the boundary itself is being renegotiated: eBPF programs now implement schedulers, security policy, and network functions inside the kernel safely, Rust is accepted for drivers since 6.1, and io_uring keeps absorbing syscalls, three different answers to the same question of how much power to grant which side of the 133 ns line measured at the top of this page.

Open source to read

The kernel is unusually rewarding to read source-first, because the papers above map to specific files.

  • torvalds/linux: the reference for everything here. Start with kernel/sched/fair.c (EEVDF: entity_eligible, pick_eevdf, and the vruntime bookkeeping), then mm/memory.c (handle_mm_fault: demand paging, COW, and huge-page faults in one file), then fs/ext4/extents.c for the extent tree this page's inode arithmetic evolved into.
  • mit-pdos/xv6-riscv: MIT's teaching reimplementation of v6 UNIX, a complete multiprocessor OS in about 10k lines. Read kernel/proc.c (fork, exit, wait, scheduler) and kernel/vm.c (a page-table walker in a page of code); it is the fastest route to holding a whole kernel in one's head.
  • axboe/liburing: the io_uring userspace library by the interface's author. examples/io_uring-cp.c first, then test/ as an encyclopedia of ring idioms.
  • opencontainers/runc: the reference container runtime. libcontainer/ shows the namespace, cgroup, and pivot_root choreography of this page's container section as shipping code.
  • containerd/containerd: the layer above runc that registries, snapshots, and Kubernetes actually talk to; read it to see where images end and processes begin.
  • firecracker-microvm/firecracker: the minimal KVM VMM in Rust. src/vmm/ is a legible tour of what a hypervisor must actually do: VMCS setup via KVM ioctls, virtio queues, and nothing else.
  • qemu/qemu: the maximal device model, useful as the contrast: hw/virtio/ implements the ring protocol this page sketches.
  • google/gvisor: a user-space kernel in Go that intercepts syscalls to shrink the shared-kernel attack surface; pkg/sentry/ is an OS textbook written defensively.
  • sched-ext/scx: production BPF schedulers. scheds/c/scx_simple.bpf.c is a complete scheduling policy in a few hundred lines, the frontier section made runnable.

Common misconceptions

"A system call is a context switch." It is a mode switch: privilege changes, the address space and task do not, and no scheduler runs. The measured gap is an order of magnitude, 134 ns for a trap against 1,885 ns for an actual switch, and the working-set tax makes real switches more expensive still. Conflating them misprices every design that trades syscalls against threads.

"fork copies the process's memory." It copies page tables and marks both sides read-only; memory copies happen later, one page per first write, at ~2 µs each. Measured here: forking a 2 GiB process costs 14.3 ms of table work, about 27 ns per resident page, an order of magnitude cheaper than the roughly 200 ms a 2 GiB copy would cost at a typical 10 GB/s single-thread memcpy rate, and pages never written are never copied at all.

"The OS is a program running in the background, scheduling things." The kernel is not a process and does not run between processes; it is code that runs within some task's context on traps, interrupts, and returns. When nothing traps, the kernel is not running anywhere, which is why a spinning userspace loop with interrupts masked off (in a guest, say) can wedge a CPU: there is no scheduler daemon to intervene, only the timer interrupt that was masked.

"More memory can only reduce page faults." True for stack algorithms like LRU, false in general: FIFO on the string worked in Problem 5 faults 9 times with 3 frames and 10 with 4. The anomaly is not a curiosity; it is the reason replacement-policy structure (the inclusion property) matters, and the same non-monotonicity reappears in cache hierarchies and TLB sizing.

"mmap is faster than read()." Not on this machine for warm sequential access: 7.35 GB/s through the mapping versus 9.02 GB/s through read(), because 32,768 minor faults outcost the copies they avoid. mmap earns its keep for sparse random access, page-granular sharing, and files larger than memory, an access-pattern decision, not a universal upgrade.

"Huge pages are free performance." The measured upside here is real, 1.9x on a 32 GiB random chase, but so are the downsides: up to 512x internal fragmentation for sparse regions, a larger first-touch stall, compaction-induced latency spikes, and, as this machine demonstrated, a silent per-process prctl veto that made THP a no-op until explicitly cleared. Measure AnonHugePages; never assume.

"A journaling file system means my data is safe after a crash." Default ordered-mode journaling makes metadata atomic and guarantees pointers never reference garbage; it does not make your write durable. Durability requires fsync, measured at 608.7 µs for a 4 KiB append here, 530x the unsynced cost, and applications that skip it are choosing performance over durability whether they know it or not.

"Containers are lightweight VMs." Containers are processes with restricted naming (namespaces) and metered consumption (cgroups) under the same kernel; a VM runs its own kernel above hardware-enforced translation. The measured consequences differ in both directions: container "boot" is a ~500 µs fork+exec while a microVM boots in ~125 ms, but one kernel exploit escapes every container on the host, which no VMCS misconfiguration short of a hypervisor bug can match. gVisor and Firecracker exist precisely because the two models are not interchangeable.

Key takeaway

An operating system is three virtualizations and a boundary, and every one of them was measured on this page rather than asserted. The boundary costs 134 ns to cross and 1,885 ns to switch across, numbers that price every design above them: the vDSO exists to skip the first, the futex to skip both, io_uring to amortize the first over a queue. The CPU virtualization is arithmetic, vruntime and virtual deadlines, and the live kernel matches the hand-computed shares to three digits. The memory virtualization is a radix tree whose sparsity makes it affordable and whose depth makes the TLB mandatory; huge pages bought back 1.9x on this machine precisely when reach ran out, and copy-on-write turned a 2 GiB copy into 14 ms of page-table work. The storage virtualization is an inode map plus a write-ahead argument about crash orderings, with the log-structured endpoint reappearing inside every SSD. Namespaces, cgroups, and hypervisors are the same ideas applied recursively: rename, meter, translate, one level down. The craft is knowing which of these costs dominates the system in front of you, and the only reliable way to know is the one used throughout: measure it.
