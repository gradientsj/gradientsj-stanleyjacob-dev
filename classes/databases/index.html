Databases: relational algebra, query optimization, transactions, and modern engines

A database engine is a stack of theorems with an I/O budget. This page works through the stack in order: the relational model and its algebra, with SQL translated to algebra and back; functional dependencies and BCNF decomposition traced step by step on a concrete schema; storage layouts, with a row store and a column store benchmarked against each other on five million rows on this machine; B+trees and LSM trees with the fanout and write-amplification arithmetic done in full; the three join algorithms and a Selinger-style dynamic program traced over a concrete catalog; serializability, two-phase locking, snapshot isolation and its write-skew anomaly demonstrated in a real experiment; ARIES recovery replayed through a small log; and the distributed and modern-engine landscape, from two-phase commit and consistent hashing to vectorized execution, compiled queries, disaggregated storage, and vector indexes.

Why this subject matters now

Databases are the rare corner of computing where a fifty-year-old paper still describes the production systems of today. Codd's 1970 relational model and the System R optimizer design that Selinger and colleagues published in 1979 are not history; they are the architecture of PostgreSQL, MySQL, and every cloud warehouse, and the reason is that both papers got the abstraction boundary right: describe what data is wanted, let the system decide how. What has changed in the last decade is everything beneath that boundary, and a practitioner today is expected to know the new floor plan. Analytics moved to column stores executing vectorized or compiled plans, a line of work running from MonetDB and X100 at CWI through HyPer at TU Munich to DuckDB and every cloud warehouse. Storage and compute were pulled apart: Aurora ships only the redo log across the network, Snowflake keeps tables in object storage and rents compute by the second. The write path of most new storage engines is an LSM tree rather than a B+tree, which trades read amplification for sequential writes in a way that can be computed exactly. Distributed transactions went from a research embarrassment to a product category: Spanner runs two-phase commit over Paxos groups with clock-uncertainty bounds, and CockroachDB and TiDB reproduce the design over Raft. And the newest index type, the approximate-nearest-neighbor graph, turned out to be just that, an index type: pgvector inside PostgreSQL is architecturally the same move as the B+tree inside System R. The durable skill is not any one engine but the cost accounting that lets one predict, before running anything, which layout, index, join order, and isolation level a workload needs; this page is organized around exactly that accounting, with every major claim either derived or measured.

Core theory

The relational model and its algebra

Codd's model has three parts. First, the data structure: a relation is a finite subset of a cartesian product of domains, that is, a set of tuples over a fixed list of named, typed attributes; a database is a set of relations. Second, integrity: keys (a minimal set of attributes whose values identify a tuple) and foreign keys (an attribute set in one relation whose values must appear as a key of another). Third, the manipulation language: an algebra of operators that consume relations and produce relations, so operators compose. Closure under composition is the whole trick; it is what makes a query plan a tree and an optimizer a tree-rewriting system. The base operators are:

OperatorNotationMeaningOutput size
Selection\( \sigma_p(R) \)rows of \(R\) satisfying predicate \(p\)\( \le |R| \)
Projection\( \pi_A(R) \)columns \(A\), duplicates removed (set semantics)\( \le |R| \)
Cartesian product\( R \times S \)every pairing of tuples\( |R|\,|S| \)
Union / difference\( R \cup S \), \( R - S \)set operations on union-compatible relationsvaries
Rename\( \rho_{a \to b}(R) \)relabel attributes\( |R| \)
Join (derived)\( R \bowtie_\theta S = \sigma_\theta(R \times S) \)product then filter\( \le |R|\,|S| \)
Division (derived)\( R \div S \)tuples of \(R\) paired with every tuple of \(S\)\( \le |\pi_{R-S}(R)| \)

The first five are independent: none can be expressed by the others. Everything else, natural join, theta join, semijoin, intersection, division, is definable from them. Codd proved the algebra equivalent in expressive power to a relational calculus (first-order logic over tuples), and that equivalence, relational completeness, is the formal statement that a declarative language can be compiled to a procedural plan without loss. Two consequences matter daily. The algebra cannot express recursion (transitive closure needs a fixpoint, which is why SQL grew WITH RECURSIVE), and it cannot count (aggregation is a separate extension, the grouping operator \( \gamma \)). SQL differs from the pure algebra in one more load-bearing way: SQL relations are bags, not sets. SELECT deptno FROM emp keeps duplicates unless DISTINCT is written, and the optimizer must track bag semantics because \( \pi \) over bags is free while set-projection costs a dedup. Throughout this page the running schema is: 

emp(eid, name, deptno, salary)         dept(dno, dname, budget)
works(eid, pid, hours)                 proj(pid, dno, cost)
keys underlined by position: emp.eid, dept.dno, works.(eid,pid), proj.pid
foreign keys: emp.deptno -> dept.dno,  works.eid -> emp.eid,
              works.pid -> proj.pid,   proj.dno -> dept.dno

SQL to algebra, worked

Take the query: names of employees earning above 90,000 who work in a department named 'widgets'. 

SELECT e.name
FROM   emp e JOIN dept d ON e.deptno = d.dno
WHERE  e.salary > 90000 AND d.dname = 'widgets';

The mechanical translation is product, then selection, then projection:

$$ \pi_{\text{name}}\Big( \sigma_{\text{salary} \gt 90000 \,\wedge\, \text{dname} = \text{'widgets'} \,\wedge\, \text{deptno} = \text{dno}} \big( \text{emp} \times \text{dept} \big) \Big) $$

which is correct and terrible: it materializes \( |{\text{emp}}| \cdot |{\text{dept}}| \) tuples before discarding almost all of them. The algebraic identities that license a better plan are the ones every optimizer applies. Selection splits over conjunctions, \( \sigma_{p \wedge q}(R) = \sigma_p(\sigma_q(R)) \); a selection whose predicate mentions only \(R\)'s attributes commutes through a join, \( \sigma_p(R \bowtie S) = \sigma_p(R) \bowtie S \); and a selection whose predicate equates attributes of the two sides converts a product into a join. Applying all three:

$$ \pi_{\text{name}}\Big( \sigma_{\text{salary} \gt 90000}(\text{emp}) \;\bowtie_{\text{deptno}=\text{dno}}\; \sigma_{\text{dname}=\text{'widgets'}}(\text{dept}) \Big) $$

Each rewrite is an equality of relations, provable from the definitions; the optimizer's search space is the set of plans reachable through such equalities, and its job (taken up below) is to pick the cheapest member. Projection pushdown adds one more step, \( \pi \) can be pushed below the join as long as join attributes are retained, so the scan of emp needs to carry only (name, deptno), which is the identity that column stores exploit structurally.

Algebra to SQL: division, the hard direction

Division is the algebra's universal quantifier. The query "employees who work on every project of department 7" is, in algebra,

$$ \text{answer} \;=\; \pi_{\text{eid},\text{pid}}(\text{works}) \;\div\; \pi_{\text{pid}}\big( \sigma_{\text{dno}=7}(\text{proj}) \big) $$

Division is not primitive; it unfolds by double negation, the same move as writing \( \forall x\, P(x) \) as \( \neg \exists x\, \neg P(x) \). Let \( W = \pi_{\text{eid},\text{pid}}(\text{works}) \) and \( T = \pi_{\text{pid}}( \sigma_{\text{dno}=7}(\text{proj})) \). Then

$$ W \div T \;=\; \pi_{\text{eid}}(W) \;-\; \pi_{\text{eid}}\Big( \big( \pi_{\text{eid}}(W) \times T \big) - W \Big) $$

Read it inside out: \( \pi_{\text{eid}}(W) \times T \) is every (employee, required project) pair that ought to exist; subtracting \(W\) leaves the pairs that are missing; projecting to eid names the employees missing at least one required project; and subtracting those from all employees leaves the ones missing none. SQL has no division operator, so the same double negation is written with nested NOT EXISTS: 

-- employees for whom no project of dept 7 exists that they do not work on
SELECT e.eid, e.name
FROM   emp e
WHERE  NOT EXISTS (
         SELECT 1 FROM proj p
         WHERE  p.dno = 7
         AND    NOT EXISTS (
                  SELECT 1 FROM works w
                  WHERE  w.eid = e.eid AND w.pid = p.pid));

-- equivalent counting form: often faster, and easier to read
SELECT w.eid
FROM   works w JOIN proj p ON w.pid = p.pid AND p.dno = 7
GROUP  BY w.eid
HAVING COUNT(DISTINCT w.pid) =
       (SELECT COUNT(*) FROM proj WHERE dno = 7);

The counting form works because an employee's distinct dept-7 projects can never exceed the department's total, so equality holds exactly at "all of them". The edge case that separates the two: if department 7 has zero projects, the double-negation form returns every employee (vacuous truth), while the counting form returns employees whose join produced at least one group, which is none. Deciding which semantics is wanted is a specification question, not a SQL question, and interviews probe exactly this seam.

The calculus, and Codd's completeness theorem

The algebra is procedural: it names operations in an order. The tuple relational calculus is declarative: a query is a set comprehension \( \{ t \mid \varphi(t) \} \) where \( \varphi \) is a first-order formula over tuple variables, relation membership atoms \( R(t) \), and comparisons on attributes. The same query as above reads

$$ \{\, t^{(1)} \mid \exists e \in \text{emp}\; \exists d \in \text{dept}\;\big( e.\text{deptno} = d.\text{dno} \wedge e.\text{salary} \gt 90000 \wedge d.\text{dname} = \text{'widgets'} \wedge t.\text{name} = e.\text{name}\big) \,\} $$

Not every calculus expression denotes a finite relation: \( \{ t \mid \neg \text{emp}(t) \} \) is the set of all tuples not in emp, which is infinite over an infinite domain. The fix is safety: restrict to formulas whose free variables are bounded by positive membership atoms, so every answer tuple's values come from the finite active domain of the database. Codd's completeness theorem is the pair of translations between safe calculus and algebra. Algebra to calculus is a structural induction: \( \sigma_p(R) \) becomes \( \{t \mid R(t) \wedge p(t)\} \), \( R \times S \) becomes a conjunction of two membership atoms, \( \pi_A \) an existential quantifier over the dropped attributes, and \( R - S \) a negated atom guarded by \( R(t) \). Calculus to algebra is the interesting direction: put \( \varphi \) in prenex form, replace each \( \exists \) by a projection, each \( \wedge \) by a join, each \( \vee \) by a union, and each \( \neg \psi \) by a difference against the active domain, which the safety condition guarantees is a finite algebra expression. The practical content of the theorem is that a declarative language costs nothing in expressive power, so an optimizer is free to pick any procedural realization; the practical content of the failure of the theorem beyond first order is that transitive closure, "all employees who report to Kim transitively", is provably not expressible, which is why SQL needed a fixpoint construct (WITH RECURSIVE) rather than a new function.

SQL's three deviations: bags, nulls, and three-valued logic

SQL is not the relational algebra, and the three gaps are where correctness bugs live. The first is bags. SELECT deptno FROM emp returns one row per employee, not one per department, so \( \pi \) in SQL is duplicate-preserving projection and \(\texttt{DISTINCT}\) is a separate, expensive operator (a sort or a hash aggregation). Bag semantics changes which algebraic identities hold: \( R \cup R = R \) is true for sets and false for bags (UNION deduplicates, UNION ALL does not), which is exactly why UNION ALL is the faster one and why an optimizer must know whether duplicates are observable before it pushes a projection down.

The second is nulls, and the third is the logic they force. A null is not a value; it is a marker for "no value here", and any comparison with it evaluates to unknown rather than true or false. SQL therefore evaluates predicates in Kleene's three-valued logic, with the truth tables

\( p \wedge q \)TUF \( p \vee q \)TUF \( \neg p \)
TTUFTTTT TF
UUUFUTUU UU
FFFFFTUF FT

Every trap follows mechanically from those tables plus one asymmetric convention: WHERE and ON keep a row only when the predicate is true, while CHECK constraints reject a row only when the predicate is false. Unknown passes a check and fails a filter. The consequences, each derivable rather than memorized: 

-- 1. NULL is not equal to itself, so this returns zero rows even if nulls exist
SELECT * FROM emp WHERE deptno = NULL;          -- always empty; use IS NULL

-- 2. NOT IN against a set containing NULL returns nothing at all.
--    x NOT IN (a, NULL) = NOT (x=a OR x=NULL) = NOT (F OR U) = NOT U = U.
SELECT name FROM emp
WHERE  deptno NOT IN (SELECT dno FROM dept);    -- empty if any dept.dno IS NULL
-- NOT EXISTS has no such hole, because EXISTS is two-valued:
SELECT name FROM emp e
WHERE  NOT EXISTS (SELECT 1 FROM dept d WHERE d.dno = e.deptno);

-- 3. Aggregates skip nulls, and the empty aggregate is NULL rather than zero.
SELECT COUNT(*),        -- counts rows
       COUNT(salary),   -- counts non-null salaries: can be smaller
       SUM(salary),     -- NULL, not 0, when every input row is filtered away
       COALESCE(SUM(salary), 0) AS safe_total
FROM   emp WHERE deptno = -1;

-- 4. GROUP BY and DISTINCT treat nulls as equal to each other; UNIQUE indexes
--    in PostgreSQL treat them as distinct, so both of these succeed:
INSERT INTO t(u) VALUES (NULL);
INSERT INTO t(u) VALUES (NULL);   -- allowed under a plain UNIQUE(u)
--    while SELECT u, COUNT(*) FROM t GROUP BY u collapses them into one group.

-- 5. A CHECK constraint accepts unknown; the same predicate in WHERE rejects it.
ALTER TABLE emp ADD CONSTRAINT positive_salary CHECK (salary > 0);
INSERT INTO emp(eid, name, deptno, salary) VALUES (9, 'kim', 7, NULL); -- accepted

The design argument against nulls, made forcefully by Date and never fully answered, is that a marker inside the value domain corrupts the algebra: with nulls present, \( \sigma_{p}(R) \cup \sigma_{\neg p}(R) \ne R \), so the law that lets an optimizer split and recombine a predicate no longer holds unconditionally. The engineering answer is that outer joins manufacture nulls whether or not the schema allows them, so the logic is unavoidable and the discipline is to declare NOT NULL aggressively and to reach for NOT EXISTS over NOT IN by default.

Functional dependencies and normalization

A functional dependency (FD) \( X \to Y \) on relation schema \(R\) asserts that any two tuples agreeing on the attribute set \(X\) also agree on \(Y\). Keys are the special case \( X \to R \). FDs are facts about the application, not the data at hand; they come from the domain ("an order has one customer") and the schema must be judged against them. Reasoning about FDs uses Armstrong's axioms, which are sound and complete: reflexivity (\( Y \subseteq X \Rightarrow X \to Y \)), augmentation (\( X \to Y \Rightarrow XZ \to YZ \)), and transitivity (\( X \to Y, Y \to Z \Rightarrow X \to Z \)). Completeness means every FD that logically follows from a set \(F\) is derivable by the axioms; the practical tool is not derivation but the attribute closure \( X^+ \): the set of all attributes determined by \(X\), computed by the obvious fixpoint loop (start with \(X\), repeatedly add the right side of any FD whose left side is contained so far). \( X \to Y \) follows from \(F\) if and only if \( Y \subseteq X^+ \), and \(X\) is a superkey iff \( X^+ = R \).

The normal forms grade how far a schema is from storing each fact once. Boyce-Codd normal form (BCNF) is the clean one: for every nontrivial FD \( X \to Y \) in \(F^+\), \(X\) is a superkey. Violations are exactly the redundancies: if \( X \to Y \) holds and \(X\) is not a superkey, then the \(Y\)-value for a given \(X\)-value is repeated across every tuple sharing that \(X\), and with repetition come update anomalies (change it in one row and not another), insertion anomalies (cannot record an \(X,Y\) fact without a full tuple), and deletion anomalies (deleting the last tuple loses the fact). Third normal form (3NF) relaxes BCNF by additionally allowing \( X \to Y \) when every attribute of \(Y\) belongs to some key; the relaxation exists because BCNF decomposition can destroy the ability to check an FD within a single table, while 3NF synthesis never does.

BCNF decomposition, traced on a concrete schema

Schema: a flat order-lines table R(order_id, cust, city, item, qty, price), abbreviated \( R(O, C, Y, I, Q, P) \), with dependencies

$$ F \;=\; \{\; O \to C, \quad C \to Y, \quad I \to P, \quad OI \to Q \;\} $$

("an order belongs to one customer, a customer has one city, an item has one list price, an order line has one quantity.") First find the key. Compute \( (OI)^+ \): start \( \{O, I\} \); apply \( O \to C \) to get \(C\); apply \( C \to Y \) to get \(Y\); apply \( I \to P \) to get \(P\); apply \( OI \to Q \) to get \(Q\). So \( (OI)^+ = \{O, C, Y, I, Q, P\} = R \), and \(OI\) is a superkey; neither \(O^+ = \{O, C, Y\}\) nor \(I^+ = \{I, P\}\) is all of \(R\), so \(OI\) is the unique candidate key. Now the decomposition algorithm: while some relation \(S\) in the current set has a violating FD \( X \to Y \) (nontrivial, \(X\) not a superkey of \(S\)), replace \(S\) by \( S_1 = X^+ \cap S \) and \( S_2 = (S - X^+) \cup X \). Each split is lossless because the shared attributes \( S_1 \cap S_2 = X \) form a key of \(S_1\) (this is the binary lossless-join test: a two-way split on \(X\) is lossless iff \( X \to S_1 \) or \( X \to S_2 \)).

Step 1. \( O \to C \) violates BCNF in \(R\) (\(O\) is not a superkey). \( O^+ = \{O, C, Y\} \). Split \(R\) into \( R_1(O, C, Y) \) and \( R_2(O, I, Q, P) \) (all of \(R\) minus \( \{C, Y\} \)).

Step 2. In \( R_1(O, C, Y) \) the projected FDs are \( O \to C \), \( C \to Y \) (and transitively \( O \to Y \)); the key is \(O\). The FD \( C \to Y \) violates BCNF (\(C\) is not a superkey of \(R_1\)). \( C^+ \cap R_1 = \{C, Y\} \). Split into \( R_{11}(C, Y) \) with key \(C\), and \( R_{12}(O, C) \) with key \(O\). Both are now in BCNF: the only nontrivial FDs are their key dependencies.

Step 3. In \( R_2(O, I, Q, P) \) the projected FDs are \( I \to P \) and \( OI \to Q \); the key is \(OI\). The FD \( I \to P \) violates BCNF. \( I^+ \cap R_2 = \{I, P\} \). Split into \( R_{21}(I, P) \) with key \(I\), and \( R_{22}(O, I, Q) \) with key \(OI\). Both in BCNF.

Final schema: customer_of_order(O, C), city_of_customer(C, Y), price_of_item(I, P), order_line(O, I, Q). Every FD in \(F\) lands intact inside one table, so this decomposition is also dependency-preserving, which is not guaranteed in general: the standard counterexample is \( R(A, B, C) \) with \( AB \to C \) and \( C \to B \) (say, street-and-city determine postal code, postal code determines city). \(C \to B\) forces a split that separates \(A\) from \(B\), after which \( AB \to C \) can only be checked by joining, and no BCNF decomposition of that schema preserves it. 3NF synthesis (minimal cover, one table per FD, add a key table if needed) accepts the small redundancy of \( C \to B \) inside \( R(A,B,C) \) to keep every check local. The engineering translation: BCNF when the FDs decompose cleanly, 3NF when they fight, and deliberate denormalization only downstream of a measured join cost, never as a default.

Problem 1

Let \( R(A, B, C, D, E) \) with \( F = \{ AB \to C,\; C \to D,\; D \to B,\; E \to A \} \). (a) Compute \( (AE)^+ \) and \( (CE)^+ \). (b) Find all candidate keys of \(R\). (c) Is \(R\) in BCNF? In 3NF? Justify each FD's status.

Solution. (a) \( (AE)^+ \): start \( \{A, E\} \); \( E \to A \) adds nothing new; no left side is contained; closure is \( \{A, E\} \), so \(AE\) is not a superkey. \( (CE)^+ \): start \( \{C, E\} \); \( C \to D \) gives \( \{C, D, E\} \); \( D \to B \) gives \( \{B, C, D, E\} \); \( E \to A \) gives \( \{A, B, C, D, E\} \) which is all of \(R\). So \(CE\) is a superkey, and since neither \(C^+ = \{C, D, B\}\) nor \(E^+ = \{E, A\}\) is \(R\), \(CE\) is a candidate key.

(b) \(E\) is in no right side, so every key contains \(E\). \(E^+ = \{A, E\}\), so \(E\) alone is not a key; a key is \(E\) plus attributes reaching the rest. Test each single addition: \( (BE)^+ \): \( \{B, E\} \to \{A, B, E\} \) via \(E \to A\); then \( AB \to C \) gives \(C\); \( C \to D \) gives \(D\): all of \(R\), so \(BE\) is a key. \( (CE)^+ = R \) from (a): key. \( (DE)^+ \): \( D \to B \) gives \(B\), \(E \to A\) gives \(A\), \( AB \to C \) gives \(C\): all of \(R\), so \(DE\) is a key. \( (AE)^+ \ne R \) from (a). Adding \(A\) never helps beyond what \(E\) gives. Candidate keys: \( \{BE, CE, DE\} \).

(c) BCNF fails: \( C \to D \) has \( C^+ = \{C, D, B\} \ne R \), so \(C\) is not a superkey; same for \( D \to B \), \( E \to A \), \( AB \to C \) (\((AB)^+ = \{A, B, C, D\} \ne R\)). No FD in \(F\) has a superkey left side, so \(R\) is nowhere near BCNF. 3NF asks instead whether each right side is contained in some key. Prime attributes (members of some candidate key): \( B, C, D, E \). Check: \( AB \to C \): \(C\) prime, allowed. \( C \to D \): \(D\) prime, allowed. \( D \to B \): \(B\) prime, allowed. \( E \to A \): \(A\) is not prime and \(E\) is not a superkey, so this FD violates 3NF. \(R\) is in neither form; the minimal fix is to split out \( R_1(E, A) \), after which the remainder \( R_2(B, C, D, E) \) has keys \(BE, CE, DE\) and every FD right side prime, hence 3NF but still not BCNF (\(C \to D\) remains a violation with a non-superkey left side).

Testing a decomposition: lossless join and dependency preservation

A decomposition is only a decomposition if the original relation can be recovered. For a general \(n\)-way split the test is the chase, but the binary case covers almost every practical decision and has a one-line criterion: \( R = R_1 \bowtie R_2 \) for every instance satisfying \(F\) if and only if \( (R_1 \cap R_2) \to R_1 \) or \( (R_1 \cap R_2) \to R_2 \) is in \( F^+ \). The reason is direct. The join always contains \(R\), since every tuple of \(R\) projects into both halves and rejoins with itself; the only way to fail is a spurious tuple, formed by pairing a \(R_1\)-tuple with a \(R_2\)-tuple that share their common attributes but came from different original rows. If the common attributes functionally determine one of the halves, that pairing is impossible, because agreement on \( R_1 \cap R_2 \) forces agreement on the whole of that half. Applying the test to Step 1 of the trace above: \( R_1(O,C,Y) \) and \( R_2(O,I,Q,P) \) intersect in \( \{O\} \), and \( O \to CY \) holds, so \( O \to R_1 \) and the split is lossless. Applying it to a bad split, \( R_1(O, C) \) and \( R_2(Y, I, Q) \), which intersect in nothing: the join is a cartesian product, every city pairs with every order, and the original is unrecoverable.

Dependency preservation is the second, independent property: is \( (F_1 \cup \dots \cup F_n)^+ = F^+ \), where \( F_i \) is \(F\) projected onto \( R_i \)? If it fails, enforcing the original constraints requires a join at every insert, which no engine will do. The trace above preserved everything. The standard counterexample is R(street, city, zip) with \( \{ \text{street},\text{city} \} \to \text{zip} \) and \( \text{zip} \to \text{city} \). The candidate keys are \( \{\text{street},\text{city}\} \) and \( \{\text{street},\text{zip}\} \), so zip is not a superkey and \( \text{zip} \to \text{city} \) violates BCNF. The only BCNF decomposition splits off \( (\text{zip}, \text{city}) \), leaving \( (\text{street}, \text{zip}) \); now \( \{\text{street},\text{city}\} \to \text{zip} \) mentions attributes living in two different tables and can be checked only by joining. Every attribute of both FDs' right sides is prime, so the undecomposed relation is already in 3NF, and 3NF is exactly the normal form that trades this one redundancy for locality of constraint checking. The rule that follows: decompose to BCNF when it costs nothing, stop at 3NF when BCNF would strand a constraint across tables.

Multivalued dependencies and fourth normal form

BCNF removes every redundancy explainable by functional dependencies, and there are redundancies it cannot see. Consider course(cno, instructor, text), where a course has a set of instructors and, independently, a set of required texts. There is no nontrivial FD at all: cno does not determine instructor, and the only key is the whole relation, so the schema is trivially in BCNF. Yet with 3 instructors and 4 texts the relation must store 12 rows to express 7 facts, and inserting one text requires 3 new rows. The dependency at work is a multivalued dependency (MVD) \( X \twoheadrightarrow Y \), read "the \(Y\)-values associated with an \(X\)-value are independent of the remaining attributes \( Z = R - X - Y \)". Formally, \( X \twoheadrightarrow Y \) holds if for any two tuples \( t_1, t_2 \) agreeing on \(X\), the tuple that takes \(Y\) from \( t_1 \) and \(Z\) from \( t_2 \) is also present:

$$ t_1[X] = t_2[X] \;\Longrightarrow\; \exists\, t_3 :\; t_3[X] = t_1[X],\;\; t_3[Y] = t_1[Y],\;\; t_3[Z] = t_2[Z]. $$

Here \( \text{cno} \twoheadrightarrow \text{instructor} \) and, by the complementation rule, \( \text{cno} \twoheadrightarrow \text{text} \); MVDs always come in complementary pairs, and every FD is an MVD (if \(X\) determines a single \(Y\)-value, the swap is vacuous). Fourth normal form asks that for every nontrivial MVD \( X \twoheadrightarrow Y \), \(X\) is a superkey. The decomposition rule mirrors BCNF: split \(R\) into \( XY \) and \( XZ \), a split that is lossless precisely because the MVD holds (Fagin's theorem: \( R = \pi_{XY}(R) \bowtie \pi_{XZ}(R) \) iff \( X \twoheadrightarrow Y \)). So course becomes teaches(cno, instructor) and requires(cno, text), 3 rows plus 4 rows instead of 12, and the cartesian blowup is gone. The hierarchy is strict: 4NF implies BCNF implies 3NF implies 2NF implies 1NF, and the further form, project-join normal form (5NF), handles relations that decompose losslessly only into three or more pieces, which is rare enough in practice that recognizing an MVD is where the useful boundary sits.

The honest case for denormalization

Normalization minimizes redundancy, and redundancy is sometimes the cheapest thing in the system. The case for keeping a denormalized copy has three legitimate forms. First, read-path arithmetic: if a page render needs five joins at 2 ms each and the denormalized row costs one 0.2 ms lookup, and the underlying facts change a thousand times less often than they are read, the write-side cost of maintaining the copy is smaller than the read-side cost of recomputing it; this is the same calculation an incremental view maintenance system does, and writing it down is the difference between engineering and taste. Second, immutability: an order line that stores the price at the time of sale is not a denormalized copy of the current price, it is a different fact, and normalizing it away is a correctness bug, not a space saving. Third, analytic layouts: star and snowflake schemas deliberately widen dimension tables because the scan cost of a wide dimension is negligible next to the fact table, and because column stores charge only for the columns actually read.

The costs are equally concrete and are the reason denormalization should follow a measurement rather than precede one. Every duplicated fact needs a maintenance path, and that path is a distributed-consistency problem in miniature: a trigger, a materialized view refresh, or application code that must be transactional with the base write or else the copies drift. Update anomalies return exactly as the normalization theory predicts. A practical middle position is to keep the normalized schema authoritative and to derive denormalized structures explicitly (materialized views, a search index, a column-store replica), so that the redundancy is named, refreshable, and rebuildable from the source of truth rather than being a second source of truth.

Storage: pages, heaps, rows, and columns

Below the algebra sits a file of fixed-size pages, because the unit of transfer between disk and memory is a block and every cost model counts blocks. PostgreSQL uses 8 KB pages, SQLite defaults to 4 KB, InnoDB to 16 KB. A heap-file page holds a header, an array of line pointers growing forward, and tuple bodies growing backward from the end, so a tuple is addressed by (page number, slot number), the tuple identifier or TID, and can be moved within its page (compaction) without changing its address. This is the N-ary storage model, NSM: all attributes of a tuple stored contiguously.

Open formats: Parquet, ORC, Arrow

Three formats dominate, and the distinction that matters is on-disk versus in-memory. Parquet and ORC are on-disk formats built on PAX: the file is a sequence of row groups (Parquet: typically 128 MB or 1 million rows; ORC: stripes of 64 MB), each row group is split into one column chunk per column, and each chunk into pages that carry their own encoding and statistics. The footer holds the schema and, crucially, per-chunk and per-page min/max/null-count statistics, so a reader with a predicate can skip whole row groups without touching their bytes, and can skip whole columns for free because the projection is a set of byte ranges. Nested data is handled by Dremel-style repetition and definition levels: for every leaf value, two small integers record how deep the repetition began and how deep the record was defined, which is enough to reconstruct arbitrary nested structures column by column.

The practical rule: Parquet or ORC for what sits in object storage, Arrow for what crosses a process or language boundary, and the engine's own compressed segments for what sits in its buffer pool.

B+trees: fanout arithmetic

The B+tree is the default ordered index of every row store since the 1970s. All data entries live in the leaves, which are linked left to right for range scans; internal nodes hold only separator keys and child pointers; every path from root to leaf has the same length; and nodes stay at least half full under the standard split/merge rules, so the tree's height is logarithmic in the number of entries with the fanout as the base. The fanout is where the engineering lives, so compute it. Take an 8 KB page with a 24-byte header. An internal entry is an 8-byte key plus an 8-byte child pointer, 16 bytes, so an internal node holds

$$ f \;=\; \left\lfloor \frac{8192 - 24}{16} \right\rfloor \;=\; 510 \ \text{children}, $$

and a leaf holds about 510 entries of (key, TID) at the same 16 bytes each. For \( N = 10^9 \) indexed rows:

$$ \text{leaves} = \left\lceil \frac{10^9}{510} \right\rceil = 1{,}960{,}785, \qquad \text{level 2} = \left\lceil \frac{1{,}960{,}785}{510} \right\rceil = 3{,}845, \qquad \text{level 1} = \left\lceil \frac{3{,}845}{510} \right\rceil = 8, \qquad \text{root} = 1. $$

Four levels: a point lookup touches 4 pages. And the top three levels total \( 1 + 8 + 3{,}845 = 3{,}854 \) pages, 31.6 MB, which fits in memory on anything, so a billion-row index costs one disk I/O per cold lookup and zero warm. This is the calculation to internalize: height grows as \( \log_f N \), and with \( f \approx 500 \) each additional level multiplies capacity by 500, so real B+trees are 3 or 4 levels essentially forever. A range scan of \(k\) matching entries costs the descent plus \( \lceil k / 510 \rceil \) linked leaf pages, which is why clustered ranges are nearly sequential I/O. Insertion splits a full leaf in two and pushes a separator up, splitting upward recursively; the root splitting is the only way height grows, which keeps the tree balanced without rebalancing passes. The price of the B+tree is random writes: updating one 100-byte row dirties an entire 8 KB page, a write amplification of 80 on a write-heavy workload, and that number is the LSM tree's opening argument.

Problem 2

An engine uses 4 KB pages with a 96-byte header. Keys are 16 bytes, child pointers 8 bytes, and leaf entries are 16-byte keys plus 8-byte TIDs. (a) Compute internal fanout and leaf capacity. (b) How many levels does the index need for \( 2 \times 10^8 \) rows, and how many page reads is a cold point lookup? (c) How much memory pins every level except the leaves? (d) A range predicate matches 100,000 consecutive keys; count page reads.

Next: \( \lceil 44/166 \rceil = 1 \), the root. That is 4 levels (root, 44, 7,258, then leaves is level 4); a cold lookup reads 4 pages.

(c) Non-leaf pages: \( 1 + 44 + 7{,}258 = 7{,}303 \) pages \( \times 4 \) KB \( = 29.2 \) MB. Trivially cacheable, so steady-state lookups cost one leaf I/O.

(d) Descent 4 pages (or 3 if the upper levels are cached, but count cold), then \( \lceil 100{,}000 / 166 \rceil = 603 \) leaf pages via sibling links: \( 4 + 602 = 606 \) total (the first leaf was reached by the descent). About 2.4 MB of sequential reads for 100k rows, versus 100,000 random heap fetches if the index were unclustered and every match required a heap visit, which is why the clustered/unclustered distinction can dominate every other constant in this subject.

Clustered, secondary, and covering indexes

A clustered index determines the physical order of the rows themselves; there can be at most one, and in InnoDB and SQL Server the leaf level of the clustered index is the table. A secondary index stores (key, row locator) pairs, where the locator is a physical address (PostgreSQL's TID) or the clustered key (InnoDB). The difference decides a query's cost model entirely. A range scan on a clustered index reads matching rows in leaf order, essentially sequentially. The same range on a secondary index produces a stream of locators in key order but in random heap order, so the heap fetches are random I/Os and the plan degrades linearly with the number of matches: this is exactly the crossover derived in the optimization problem below and measured in the implementation section, where the index plan beat a full scan up to 2.9955% selectivity and lost from 3.4895% onward on a 2,000,000-row table.

A covering index removes the second half of that cost by including every column the query needs, so the heap is never visited (PostgreSQL's index-only scans; INCLUDE columns in SQL Server and PostgreSQL 11+). The plan text says so directly: in the measurements below, SQLite reports SEARCH t USING COVERING INDEX t_k when the query reads only the indexed column and SEARCH t USING INDEX t_k when it also needs pad, the same index, two different cost regimes. The price is write amplification and index size, which is why covering indexes are added deliberately for known queries rather than speculatively. PostgreSQL adds one subtlety: an index-only scan still consults the visibility map to check that a page's tuples are all-visible, because the index carries no MVCC information; a table that is never vacuumed loses index-only scans without any plan change being visible.

Hash indexes and zone maps

A hash index answers equality in \(O(1)\) expected probes and answers nothing else: no ranges, no ordering, no prefix matching. That makes it a narrow tool, and modern engines use hashing mostly inside operators (hash join, hash aggregation) rather than as a persistent index, where the B+tree's extra \( \log_f N \) buys range scans, sorted output, and the interesting orders an optimizer can reuse. The exception is main-memory and key-value systems, where the constant matters and ranges do not.

Zone maps invert the idea: rather than an auxiliary structure that finds rows, they are per-block summaries that let a scan skip rows. Each block (Parquet row group, DuckDB row group, Snowflake micro-partition, Netezza zone) stores the min and max of each column, and a scan with predicate \( a \in [l, u] \) skips any block whose \([\min,\max]\) is disjoint from \([l,u]\). Cost is negligible (two values per column per block) and the benefit is entirely a function of how clustered the data is: on sorted data the min/max intervals are disjoint and almost every block is skipped, on shuffled data every block's interval spans the whole domain and nothing is skipped. Measured on this machine, the same narrow-range aggregate over 5,000,000 rows in DuckDB took 10.6115 ms against an unsorted table and 1.2824 ms against the same data stored sorted by the predicate column, a 8.3x difference from nothing but block ordering, while a full-table aggregate that no zone map can help took 8.239 ms and 8.3569 ms respectively, confirming the mechanism is skipping rather than layout. The design consequence is that in a column store, sort order is the closest thing to an index, and choosing it is the physical-design decision with the largest single effect.

LSM trees: write amplification, derived

The log-structured merge tree (O'Neil, Cheng, Gawlick, O'Neil, 1996) refuses random writes entirely. Writes go to an in-memory sorted structure (the memtable) and, for durability, a sequential write-ahead log. When the memtable fills (say 64 MB) it is written to disk in one sequential burst as an immutable sorted run (an SSTable). Runs accumulate and are merged in the background into a hierarchy of levels of geometrically increasing capacity, ratio \( T \) (commonly 10): level \(L_1\) holds, say, 256 MB, \( L_2 \) 2.56 GB, \( L_3 \) 25.6 GB, \( L_4 \) 256 GB. Reads must consult the memtable and potentially every run (Bloom filters spare most point reads the disk touches). The design space is how eagerly to merge, and the two poles have exactly computable costs. Write amplification (WA) is total bytes written to disk per byte of user data ingested.

Leveled compaction (RocksDB default for lower levels): each level is a single sorted run. When \( L_{i-1} \) overflows, a file from it is merged into \( L_i \). Because \( L_i \) is \( T \) times larger and covers the same key space, a file's key range in \( L_{i-1} \) overlaps about \( T \) files' worth of data in \( L_i \), so merging 1 byte down rewrites roughly \( T \) resident bytes alongside it. Amortized over a byte's life: it is written once by the WAL, once by the flush to \( L_0 \), and then approximately \( T \) times at each of the \( L \) levels it descends through:

$$ \text{WA}_{\text{leveled}} \;\approx\; 2 + T \cdot L. $$

With \( T = 10 \) and \( L = 4 \) levels (the 256 GB configuration above): \( \approx 42 \). The payoff is on the read side: one run per level means a point read consults at most \( L + \) (number of \(L_0\) files) runs, and a range scan merges \( \approx L \) iterators. Space amplification is small: levels below the last hold \( 1/T + 1/T^2 + \dots \approx 11\% \) extra, plus obsolete versions awaiting compaction, so \( \approx 1.11 \times \) data size in steady state.

Tiered compaction (Cassandra's size-tiered, RocksDB universal): each level accumulates up to \( T \) independent runs; when full, all \( T \) are merged into a single run deposited at the next level. Now a byte is written once per level, because merges only combine runs, never rewrite a resident lower level:

$$ \text{WA}_{\text{tiered}} \;\approx\; 2 + L, $$

which is 6 for the same configuration, seven times less than leveled. The bill arrives on reads and space: up to \( T \) runs per level means \( T \cdot L = 40 \) runs to consult in the worst case (Bloom filters mitigate point reads, range scans pay in full), and space amplification can approach \( 2\times \) or worse transiently because a full merge at the last level needs its inputs and output resident together, and obsolete versions survive until their level's merge. Bloom filters in the read path

The LSM read cost above is a count of runs consulted, and most of those consultations find nothing: a point read of a key that lives in \( L_4 \) still has to prove the key is absent from the memtable, from every \( L_0 \) run, and from \( L_1 \) through \( L_3 \). A Bloom filter makes that proof cheap. Allocate \(m\) bits and \(k\) hash functions per run; insert a key by setting the \(k\) bits it hashes to; test a key by checking those bits. False negatives are impossible, so a negative answer skips the run with certainty. False positives cost one wasted lookup. After inserting \(n\) keys, a given bit is still zero with probability \( (1 - 1/m)^{kn} \approx e^{-kn/m} \), so the probability that all \(k\) bits of an absent key are set is

$$ \varepsilon \;\approx\; \Big( 1 - e^{-kn/m} \Big)^{k}, \qquad \frac{\partial \ln \varepsilon}{\partial k} = 0 \;\Longrightarrow\; k^{*} = \frac{m}{n}\ln 2, \qquad \varepsilon^{*} = 2^{-k^{*}} = 0.6185^{\,m/n}. $$

The optimum sets exactly half the bits, which is the intuition behind \( \ln 2 \): more hashes buy more evidence per query but fill the array faster, and the two effects balance at 50% occupancy. Query execution: iterator, vectorized, compiled

A plan is a tree of operators, and the question of how control flows through that tree is worth an order of magnitude. The Volcano or iterator model (Graefe, 1994) gives every operator the interface open(), next(), close(), with next() returning one tuple. Composition is trivial, pipelining is automatic, and any operator can be dropped into any plan, which is why the model survived; the cost is that producing one tuple costs one virtual call per operator in the pipeline, and the work done inside that call is a few nanoseconds of actual computation. Boncz, Zukowski, and Manegold measured the consequence in MonetDB/X100 (CIDR 2005): on TPC-H the interpretation overhead dominated, and the fix was to change the granularity rather than the interface. In the vectorized model, next() returns a block of values, typically 1,024 or 2,048 per column, chosen to fit comfortably in L1. Now the virtual call is amortized over a thousand values, the inner loop over a primitive type is a tight loop the compiler can unroll and vectorize, branches become selection vectors, and the pipeline stays in cache.

The third design, compiled execution, was argued by Neumann (VLDB 2011) for HyPer: rather than pushing tuples through operators, invert control so that data stays in registers as long as possible. The plan is split at pipeline breakers (the points where a tuple must be materialized: the build side of a hash join, a sort, an aggregation) and each pipeline is compiled into a single loop of machine code via LLVM, so a tuple entering a pipeline is loaded once into registers and does not touch memory again until the next breaker. The two camps are still both alive and correct: DuckDB and Velox are vectorized, Umbra and Spark's whole-stage codegen are compiled, and Kersten and colleagues' careful head-to-head ("Everything you always wanted to know about compiled and vectorized queries but were afraid to ask", VLDB 2018) found the gap between them smaller than the gap from either to tuple-at-a-time, with compilation ahead on compute-heavy queries and vectorization ahead on memory-bound scans.

The same effect is reproducible in miniature. Running one filtered aggregate over 5,000,000 values on this machine, tuple-at-a-time evaluation in Python cost 645.7 ms (extrapolated from a 1,000,000-value run), the same computation in 2,048-value blocks cost 45.5539 ms, and one whole-column pass cost 37.2262 ms, a 14.2x gap from block size alone with the algorithm and the data held fixed. An interpreted language exaggerates the per-tuple constant relative to a C++ engine, so the multiplier is not transferable; the shape of the curve is, and so is the reason the curve flattens after a couple of thousand values, which is that the amortized dispatch cost has already fallen below the memory traffic.

External sorting and the pass-count formula

Sorting more data than fits in memory is the oldest cost model in the field and still the one that prices ORDER BY, GROUP BY without hashing, index builds, and sort-merge join. With \(M\) data pages and \(B\) buffer pages, phase one reads \(B\) pages at a time, sorts in memory, and writes \( \lceil M/B \rceil \) initial sorted runs. Phase two merges \( B - 1 \) runs at a time (one buffer is reserved for output), reducing the run count by a factor of \( B-1 \) per pass. Total passes and I/O:

$$ \text{passes} \;=\; 1 + \left\lceil \log_{B-1} \left\lceil \frac{M}{B} \right\rceil \right\rceil, \qquad \text{page I/O} \;=\; 2M \cdot \text{passes}. $$

The logarithm has base \( B - 1 \), not 2, and that is the whole practical content: memory buys fan-in, and fan-in enters logarithmically, so the second pass is almost always the last one. Concretely, sorting \( M = 10^6 \) pages (8 GB at 8 KB pages) with \( B = 1000 \) buffers gives \( \lceil 10^6/1000 \rceil = 1000 \) initial runs, and one merge pass at fan-in 999 finishes them: 2 passes, \( 4 \times 10^6 \) page I/Os. Doubling memory to \( B = 2000 \) still gives 2 passes and halves nothing; halving it to \( B = 500 \) gives 2000 runs merged 499-at-a-time, still 2 passes. The regime where memory matters is precisely \( B \lt \sqrt{M} \), which is why the rule of thumb "two passes whenever \( B \gtrsim \sqrt{M} \)" is the one to remember. The implementation below verifies the formula exactly: across buffer sizes from 4 to 256 pages on 977 pages of data, the measured pass count matched \( 1 + \lceil \log_{B-1} \lceil M/B \rceil \rceil \) in every case, and measured page I/O matched \( 2M \cdot \text{passes} \) to the page.

Two refinements are standard. Replacement selection produces initial runs of expected length \( 2B \) rather than \(B\) by maintaining a heap and emitting keys that still fit the current run, halving the run count at the cost of random-ish access; modern engines often skip it because it fights the prefetcher. And "sorting" in a column store usually means sorting a vector of (key, row-id) pairs and permuting the payload columns afterwards, so the bytes moved per pass are far smaller than the tuple width would suggest.

Aggregation and parallel execution

Aggregation has the same two implementations as join, for the same reason. Sort-based aggregation sorts on the grouping key and folds equal-key runs, costing an external sort and producing sorted output. Hash-based aggregation keeps a hash table keyed by the group, costing one pass when the table fits and a partitioning pass when it does not, and is the default in every analytic engine because the number of groups is usually far smaller than the number of rows. The distinction that matters at scale is between algebraic aggregates, which have a fixed-size partial state that merges associatively (count, sum, min, max, and average as the pair (sum, count)), and holistic ones, which do not (exact median, exact distinct count). Algebraic aggregates parallelize by the standard two-phase plan: each worker aggregates its partition locally, then a single merge combines partial states, so the data crossing the exchange is proportional to the number of groups rather than the number of rows. Holistic aggregates either shuffle by group first or accept an approximation, which is why APPROX_COUNT_DISTINCT exists: HyperLogLog keeps a fixed \( 2^p \)-register sketch with standard error \( 1.04/\sqrt{2^p} \), so 16 KB of registers gives roughly 1% error on any cardinality, and the sketches merge by taking register-wise maxima, which is exactly the algebraic property the exact version lacks.

Parallelism itself comes in three forms, and a production engine uses all three. Inter-query parallelism is independent queries on separate cores, which needs only concurrency control. Inter-operator parallelism runs different operators of one plan concurrently, connected by queues; this is what the exchange operator (Graefe's Volcano parallelism) encapsulates, and it is how classic parallel databases scaled. Intra-operator parallelism splits one operator's input across workers, which is where nearly all of the speedup lives on modern hardware: a scan over row groups, a hash build partitioned by hash bits, an aggregation with per-worker tables merged at the end. The measured effect is visible in the row-versus-column experiment below, where the same grouped aggregate over five million rows ran in 37.8 ms on one thread and 3.22 ms on 52, a 11.7x speedup from intra-operator parallelism alone. The sublinearity is the interesting part: the query reads a fixed number of bytes, and past a certain thread count the bottleneck moves from instructions to memory bandwidth and to the final merge, which is Amdahl's law arriving on schedule. Morsel-driven parallelism (Leis and colleagues, SIGMOD 2014) is the design that handles this well: break the input into small morsels, keep a work queue, let each worker pull the next morsel and prefer NUMA-local ones, so skew and stragglers are absorbed by the queue rather than by a static partitioning decided before the query ran.

Join algorithms and their costs

Every cost below counts page I/Os, the System R convention; \( M \) and \( N \) are the page counts of the outer relation \(R\) and inner relation \(S\), \( m \) the tuple count of \(R\), and \( B \) the buffer pages available. The three families:

Nested loops. Naive tuple-at-a-time: for each tuple of \(R\), scan \(S\): \( M + m N \), catastrophic (\(m\) is tuples, not pages). Block nested loops fixes the units: read \( B - 2 \) pages of \(R\), scan \(S\) once per chunk:

$$ \text{cost}_{\text{BNL}} \;=\; M + \left\lceil \frac{M}{B-2} \right\rceil \cdot N. $$

If \(R\) fits in memory this is \( M + N \), optimal. Index nested loops replaces the inner scan with an index probe: \( M + m \cdot (\text{probe cost}) \), where a probe is 1-4 I/Os as computed in the B+tree section plus one heap fetch per match if unclustered. Unbeatable when \(m\) is small, hopeless when \(m\) is millions, exactly the two regimes the optimizer must distinguish, which is why cardinality estimation errors flip plans between these extremes.

Sort-merge. Sort both inputs on the join key, then merge. External sort of \(M\) pages with \(B\) buffers makes runs of length \( \approx 2B \) (replacement selection) or \(B\), then merges \( B - 1 \) at a time, so the pass count is \( 1 + \lceil \log_{B-1} \lceil M/B \rceil \rceil \) and each pass reads and writes everything: cost \( 2M \cdot \text{passes} \). With realistic memory (\( B \ge \sqrt{M} \)) sorting is two passes, so

$$ \text{cost}_{\text{SMJ}} \;=\; 4M + 4N + (M + N) \;=\; 5(M+N) \ \text{worst},\quad 3(M+N)\ \text{if the merge consumes runs directly}. $$

Sort-merge additionally emits output sorted on the join key, an "interesting order" (Selinger's term) that a later ORDER BY, GROUP BY, or merge join upstream can consume for free; a plan that loses on this join can win on the query.

Hash join. Build an in-memory hash table on the smaller input, probe with the larger: \( M + N \) I/Os when \( \min(M, N) \lesssim B \). Otherwise Grace hash join partitions both inputs by a hash of the key into \( \approx B - 1 \) partitions (write everything once), then joins matching partitions pairwise (read everything once):

$$ \text{cost}_{\text{Grace}} \;=\; 3(M + N) $$

provided \( B \gtrsim \sqrt{\min(M,N)} \) so each build partition fits; skewed keys break the provision and recursive repartitioning or hybrid variants handle the residue. Hash join is the workhorse of every analytic engine, but it only computes equijoins; inequality joins fall back to sort-merge or nested loops. The one-line summary the optimizer lives by: index nested loops for tiny outers, hash for large equijoins, merge when order is useful or memory is tight.

Cost-based optimization: the Selinger dynamic program, traced

Selinger, Astrahan, Chamberlin, Lorie, and Price (1979) contributed three ideas that define query optimization to this day: estimate result sizes from catalog statistics, price plans in I/O and CPU, and search join orders with dynamic programming over subsets, keeping only the cheapest plan per subset (plus per interesting order). The estimation rules: store per relation the tuple count \( |R| \) and per column the distinct-value count \( V(R, a) \). Then

$$ |\sigma_{a = c}(R)| \approx \frac{|R|}{V(R,a)}, \qquad |R \bowtie_{a=b} S| \approx \frac{|R| \cdot |S|}{\max\big(V(R,a),\, V(S,b)\big)}, $$

the first from assuming uniform values, the second from assuming the smaller value set is contained in the larger (so each tuple of the side with more distinct values matches \( |S| / V \) tuples of the other). Modern engines refine with histograms and samples but the skeleton is unchanged; PostgreSQL's versions live in src/backend/utils/adt/selfuncs.c.

Cardinality estimation, and how badly it fails

A cost model is a function of estimated cardinalities, and the estimates are where the errors are. Leis and colleagues made this precise in "How Good Are Query Optimizers, Really?" (VLDB 2015): running a join-order benchmark over real, skewed IMDB data, they found that cost models are approximately fine and cardinality estimates are not, with errors that grow by orders of magnitude as joins compose, and that a plan chosen from true cardinalities beats a plan chosen from estimated ones by a wide margin even with a crude cost model. The right mental model is that query optimization is a well-solved search problem sitting on top of an unsolved statistics problem.

Uniformity, and histograms. The Selinger estimate \( |\sigma_{a=c}(R)| = |R|/V(R,a) \) assumes every value of \(a\) is equally frequent. Real columns are Zipfian, so the estimate is right on average and wrong on every individual value, which is worse: the frequent values are the ones queries ask about. Histograms fix the marginal distribution. An equi-width histogram splits the domain into equal ranges and is defeated by skew; an equi-depth histogram splits into buckets of equal count, so each bucket carries \( |R|/B \) rows and a narrow, dense region gets many narrow buckets. PostgreSQL stores both an equi-depth histogram (default_statistics_target buckets, 100 by default) and, separately, a most-common-values list with exact frequencies, which is the practical answer to skew: the MCVs handle the heavy head exactly and the histogram handles the tail approximately. For a range predicate the estimate sums whole buckets and interpolates the partial bucket at each end, which is where the residual error lives.

Independence, and why it is worse. For a conjunction the textbook rule multiplies selectivities, \( s(p \wedge q) = s(p)\,s(q) \), which is exactly right if the predicates are independent and arbitrarily wrong otherwise. Correlation in real schemas is the norm rather than the exception: city and postal code, manufacturer and model, region and customer id. Measured on the five-million-row table built for this page, where cust is by construction determined by region: the predicate region = 'north' has selectivity 12.519%, the predicate cust BETWEEN 0 AND 99999 has selectivity 12.519%, so the independence assumption predicts \( 0.125 \times 0.125 \times 5{,}000{,}000 \approx \) 78,362 rows, and the true answer is 625,947 rows, an underestimate by 8x. Flip the correlation and it fails in the other direction: region = 'north' AND cust BETWEEN 300000 AND 399999 is estimated at 78,220 rows and returns zero, a q-error of 78,220x. Both directions are dangerous, and they are dangerous asymmetrically. Underestimation makes the optimizer choose an index nested loop or a hash build that does not fit, and the plan degrades by orders of magnitude; overestimation makes it choose a scan where an index would have done, and the plan degrades by the ratio of the scan to the lookup, which is bounded. This asymmetry is why production optimizers are tuned to be conservative about small estimates.

Error propagation through joins. The multi-join case is where the errors become fatal, and the reason is compositional. If each join's cardinality estimate carries a multiplicative error \( \varepsilon_i = \hat{n}_i / n_i \), and estimates are computed from the estimated inputs rather than the true ones, then the error of a chain of \(k\) joins is the product \( \prod_{i \le k} \varepsilon_i \). Errors do not cancel, because the selectivity assumptions are biased in a consistent direction (independence almost always understates correlated conjunctions), so the expected log-error grows linearly in the number of joins and the error itself grows geometrically:

$$ \log \varepsilon_{\text{total}} \;=\; \sum_{i=1}^{k} \log \varepsilon_i \;\approx\; k \cdot \E[\log \varepsilon], \qquad \varepsilon_{\text{total}} \approx \varepsilon^{\,k}. $$

Put numbers on it. A modest per-join error of 2.4x, well below the 8x measured above for a single correlated conjunction, compounds to \( 2.4^6 \approx 191 \)x across a six-way join, which is more than enough to invert the ranking of any two plans; at the measured 8x the same chain reaches \( 8^6 \approx 2.6 \times 10^{5} \). This is the quantitative content of the Leis result, and it is also why the best practical mitigation is not a better estimator but a plan that is robust to the error: adaptive join operators that switch strategy at runtime when the build side overflows, and mid-query re-optimization at pipeline breakers where the true intermediate cardinality has just become known.

Sketches and multi-column statistics. Two directions attack the problem at its source. Sketches give small, mergeable, bounded-error summaries: HyperLogLog for distinct counts, Count-Min for heavy hitters, KMV for set cardinality, all with the property that they merge associatively so a distributed system can compute them per partition. Multi-column statistics attack correlation directly: PostgreSQL's CREATE STATISTICS ... (dependencies, ndistinct, mcv) stores functional- dependency degrees and joint MCV lists, so the planner can replace the independence product with a measured joint frequency for the column groups a user names. Both are opt-in, which is the honest summary of the state of the art: the system will not discover which correlations matter without being told.

Learned cardinality estimation, honestly. Since 2019 a large body of work has trained models to replace the estimator. Three families exist. Query-driven models (MSCN, Kipf and colleagues at TU Munich, 2019) featurize a query and regress its cardinality from a workload of executed queries; they are fast and only as good as the workload's coverage, degrading sharply on query shapes they were not trained on. Data-driven models (Naru, Yang and colleagues at Berkeley, 2019; DeepDB, Hilprecht and colleagues at TU Darmstadt, 2020) learn the joint distribution of the data itself with autoregressive models or sum-product networks and answer any predicate by inference, which fixes the coverage problem but pays inference latency inside the optimizer and needs retraining as data drifts. Hybrid and Bayesian approaches sit between. The honest ledger from independent evaluations (Wang and colleagues' "Are we ready for learned cardinality estimation?", VLDB 2021, and follow-ups) is that on static data and in-distribution queries the learned estimators do reduce q-error substantially, often by an order of magnitude at the tail; and that the engineering costs, training time, model size, inference latency inside a millisecond-budget planner, updates under data drift, lack of worst-case guarantees, and the difficulty of debugging a regression, have kept them out of the default path of every major production system. The idea that has landed instead is narrower and more robust: use learning for the parts where a bad answer is cheap, keep exact statistics where they are affordable, and prefer runtime adaptivity to better prediction. It is a good example of a research direction whose main contribution so far has been to sharpen what the real problem is.

Transactions: ACID made precise

A transaction is a sequence of reads and writes that the application wants treated as one unit. ACID names four separable guarantees. Atomicity: all of the transaction's writes become visible or none do; the mechanism is undo (rollback via the log). Consistency: if each transaction individually preserves the application's invariants, the system never exposes a state violating them; this is a contract about composition, discharged by isolation, not a mechanism of its own. Isolation: concurrent execution is equivalent to some serial execution, in a sense made exact below, or explicitly weakened to a named level. Durability: once commit is acknowledged, the writes survive crashes; the mechanism is redo (write-ahead logging, the ARIES section). Isolation and durability carry the intellectual weight, so each gets its own treatment.

Conflict serializability and the precedence graph

Model a schedule as the interleaved sequence of operations \( r_i(x) \), \( w_i(x) \) of transactions \( T_1, \dots, T_n \). Two operations conflict when they belong to different transactions, touch the same object, and at least one is a write: read-write, write-read, and write-write pairs. A schedule is conflict-serializable if it can be transformed into a serial schedule by swapping adjacent non-conflicting operations; equivalently, if some serial schedule orders every conflicting pair the same way. The test is the precedence graph: one node per committed transaction, an edge \( T_i \to T_j \) whenever some operation of \( T_i \) conflicts with and precedes some operation of \( T_j \).

Theorem. A schedule is conflict-serializable iff its precedence graph is acyclic. Proof sketch, both directions. If the graph is acyclic, take a topological order of it and claim the serial schedule in that order is conflict-equivalent: every conflicting pair's order is an edge, and topological order respects every edge; the swaps needed to reach that serial schedule only ever exchange non-conflicting neighbors, since any blocked exchange would be a conflicting pair ordered against its edge. Conversely, a cycle \( T_a \to T_b \to \dots \to T_a \) means any candidate serial order must place \( T_a \) both before and after itself, contradiction. \( \square \)

Worked. Schedule \( S_1: \; r_1(A)\; w_2(A)\; r_3(B)\; w_1(B)\; w_3(C)\; r_2(C) \). Enumerate conflicting pairs in temporal order: \( r_1(A) \) before \( w_2(A) \): edge \( T_1 \to T_2 \). \( r_3(B) \) before \( w_1(B) \): edge \( T_3 \to T_1 \). \( w_3(C) \) before \( r_2(C) \): edge \( T_3 \to T_2 \). No other pairs share an object with a write. Graph: \( T_3 \to T_1 \to T_2 \) and \( T_3 \to T_2 \): acyclic. Topological order \( T_3, T_1, T_2 \), so \( S_1 \) is conflict-equivalent to running \( T_3 \) then \( T_1 \) then \( T_2 \) serially. Now swap two operations: \( S_2: \; r_1(A)\; w_2(A)\; r_2(B)\; w_1(B) \). Edges: \( T_1 \to T_2 \) (on \(A\)) and \( T_2 \to T_1 \) (on \(B\)): a 2-cycle, so no serial order agrees with both conflicts, and \( S_2 \) is not conflict-serializable; it is the classic inconsistent-analysis interleaving. Conflict serializability is sufficient but not necessary for correctness (view serializability is the weaker exact notion, but its test is NP-complete, per Papadimitriou 1979), so every practical scheduler enforces the conflict version.

Two-phase locking, with the proof

Two-phase locking (2PL) is the classical scheduler: a transaction acquires a shared lock before reading and an exclusive lock before writing, and once it releases any lock it may acquire no more. Each transaction thus has a lock point, the moment it holds its maximal lock set.

Claim. Every 2PL schedule is conflict-serializable. Proof. Suppose \( T_i \to T_j \) in the precedence graph: some operation \( o_i \) precedes and conflicts with \( o_j \). Conflicting operations need incompatible locks on the same object, so \( T_i \) must release that lock before \( T_j \) acquires it. By the two-phase rule, \( T_i \)'s release happens at or after \( T_i \)'s lock point, and \( T_j \)'s acquisition happens at or before \( T_j \)'s lock point. Therefore \( \text{lockpoint}(T_i) \lt \text{lockpoint}(T_j) \). If the graph had a cycle, lock points would decrease around it and return to the start, impossible for real numbers; hence the graph is acyclic and the theorem above finishes the argument. \( \square \)

Plain 2PL still allows cascading aborts (\( T_j \) reads \( T_i \)'s write, then \( T_i \) aborts) and does not prevent a transaction from exposing uncommitted data. Strict 2PL, holding all exclusive locks until commit, buys recoverability and cascadelessness, and is what real lock managers implement. 2PL trades this correctness for two costs: deadlocks, handled by waits-for-graph detection or timeouts (a deadlock is precisely a cycle of lock waits, the dynamic shadow of the precedence cycle 2PL prevents), and blocking, quantified by Gray, Lorie, Putzolu, and Traiger's 1976 granularity paper, which also introduced the intention-lock hierarchy (IS, IX, SIX) that lets a table scan take one table lock while row updates take row locks, with compatibility checked at every level of the hierarchy. Phantoms require one more idea: a predicate read ("all employees with salary above 90,000") cannot lock rows that do not exist yet, so serializable locking must lock the gap: index-range locks (next-key locks in InnoDB) lock the leaf interval the predicate scanned, blocking inserts into it.

Deadlock: detection versus prevention

Two-phase locking buys serializability and pays in deadlock, which is unavoidable because the lock acquisition order is chosen by the application, not the scheduler. A deadlock is a cycle in the waits-for graph, one node per transaction and an edge \( T_i \to T_j \) when \( T_i \) waits for a lock \( T_j \) holds. Three responses exist and each is deployed somewhere. Detection builds the waits-for graph periodically or on demand, finds a cycle, and aborts a victim chosen by some cost heuristic (fewest locks held, least work done, youngest); InnoDB and PostgreSQL both detect, PostgreSQL after deadlock_timeout (1 second by default) so that the common case, a lock wait that resolves quickly, never pays for graph construction. Prevention makes cycles impossible by ordering: give every transaction a timestamp, and on conflict apply either wait-die (an older transaction waits for a younger one; a younger one requesting from an older one dies and restarts) or wound-wait (an older transaction wounds, that is aborts, the younger holder; a younger requester waits). Both schemes only ever let waits point in one direction along the timestamp order, so the waits-for graph is acyclic by construction, and both are starvation-free because a restarted transaction keeps its original timestamp and thus grows older. Timeout is the degenerate prevention scheme: abort anything that waits too long, which is cheap, never wrong, and aborts innocent transactions in proportion to how badly the timeout is tuned.

Choosing between them is a contention calculation. Detection costs graph maintenance proportional to the number of waiting transactions and aborts only genuine deadlocks; prevention costs nothing to detect and aborts transactions that would have completed fine. At low contention, detection wins because the false-abort rate of prevention is pure waste. At high contention, and especially in a distributed system where building a global waits-for graph means a round of messages, prevention wins because the detection latency becomes the time locks sit held, and that time multiplies the contention that caused the problem. This is why single-node engines detect while distributed ones (Spanner, TiDB) lean on wound-wait or on timestamp ordering that avoids the question.

Optimistic concurrency control

Kung and Robinson (1981) inverted the assumption. Rather than acquiring locks defensively, let every transaction run against a private workspace and check for conflicts only at the end. Three phases: read, during which the transaction reads freely and buffers its writes, recording a read set \( \text{RS}(T) \) and a write set \( \text{WS}(T) \); validate, during which the system checks that \(T\) can be serialized after every transaction that committed while it ran; and write, during which the buffered writes are installed. Backward validation, the common form, checks for each concurrent committed \( T_j \):

$$ \text{RS}(T_i) \cap \text{WS}(T_j) = \emptyset \quad \text{for every } T_j \text{ that committed after } T_i \text{ began}, $$

which says \( T_i \) did not read anything \( T_j \) overwrote; if the intersection is nonempty, \( T_i \) read a value that a later-serialized transaction changed, and serializing \( T_i \) after \( T_j \) is impossible, so \( T_i \) aborts and retries. The validation phase must be atomic, which in practice means a short critical section or a per-record version check, and the write phase must be atomic with it, which is where OCC implementations spend their engineering.

The trade against locking is exactly quantifiable and is the same trade as optimistic versus pessimistic anything. Let \(p\) be the probability a transaction conflicts. Locking pays a per-operation lock-manager cost on all transactions plus blocking on the \(p\) fraction; OCC pays nothing on the \( 1-p \) fraction and pays a full re-execution on the \(p\) fraction. OCC therefore wins when \(p\) is small and loses catastrophically when \(p\) is large, because aborts consume the very resources that caused the contention and the system can enter a livelock where throughput falls as load rises. The measured version of this appears in the isolation problem worked below: at 500 conflicting pairs per day against a high-throughput system, the optimistic protocol's total added cost was under a second of work, while the pessimistic protocol's deadlock-detection latency stalled bystanders for a hundred times longer. The modern main-memory systems (Silo at MIT, 2013; Hekaton at Microsoft Research, 2013) are OCC-derived precisely because their workloads are short and mostly non-conflicting, and their central innovation, epoch-based group commit and avoiding a shared counter for timestamps, was about removing the last centralized bottleneck from validation rather than about the validation rule itself.

Snapshot isolation (SI) is the interesting row, because it is what "REPEATABLE READ" means in PostgreSQL and what "SERIALIZABLE" meant in Oracle for years. SI gives every transaction a consistent snapshot as of its start, plus the first-committer-wins rule: two concurrent transactions writing the same object cannot both commit. This kills dirty reads, fuzzy reads, phantoms (the snapshot is a frozen row set), and lost updates (first-committer-wins). It permits exactly the anomaly whose two writes touch different objects: write skew. The canonical instance: a clinic requires at least one doctor on call; Alice and Bob are both on call; two concurrent transactions each check the invariant (each reads both rows in its snapshot, sees 2 on call) and each takes its own doctor off call. The writes are disjoint (Alice's row, Bob's row), so first-committer-wins never fires, both commit, and zero doctors are on call: each transaction was individually correct, the pair violated the invariant, the exact failure the C in ACID delegates to isolation. Fekete, Liarokapis, O'Neil, O'Neil, and Shasha (2005) characterized when SI is safe: anomalies require a dangerous cycle with two consecutive read-write antidependency edges between concurrent transactions, and Cahill, Röhm, and Fekete's serializable snapshot isolation (SSI, 2008) turned that theorem into a runtime: track rw-antidependencies, abort a transaction when two consecutive edges appear. Ports and Grittner shipped SSI as PostgreSQL's SERIALIZABLE level (VLDB 2012); it is optimistic, so its failure mode is serialization-failure retries under contention rather than blocking. The isolation experiment in the implementation section reproduces the on-call scenario against a real engine and shows all three outcomes: the nonrepeatable read without a transaction, the stable snapshot within one, and the write on a stale snapshot rejected.

MVCC as PostgreSQL implements it

Multiversion concurrency control implements snapshots by never updating in place. Every PostgreSQL heap tuple carries system columns xmin (the transaction ID that created it) and xmax (the transaction that deleted or superseded it, or 0). An UPDATE is a delete-plus-insert: the old version gets its xmax set, a new version is inserted (in the same page when possible, chained as a heap-only tuple so indexes need no new entry), and both versions coexist. A snapshot is the triple (xmin horizon, xmax horizon, in-progress list) captured at statement or transaction start; a tuple is visible to a snapshot iff its creator committed before the snapshot and its deleter (if any) had not. Readers therefore never block writers nor writers readers; write-write conflicts still lock. The costs are structural, not incidental: dead versions accumulate until VACUUM removes those older than the oldest live snapshot (a long-running transaction pins the horizon and bloats every hot table), each page stores its versions inline so hot-update workloads inflate heap and WAL, and the 32-bit transaction counter wraps, so anti-wraparound freezing must relabel old tuples as "frozen" before 2 billion transactions pass; the failure to keep up is a production-outage genre of its own. The contrasting design, used by MySQL/InnoDB, Oracle, and SQL Server's version store: keep only the newest version in place and reconstruct old versions from undo segments on demand: reads of hot rows pay reconstruction, but there is no VACUUM debt and space reclaims itself. Neither dominates; the choice surfaces as PostgreSQL's bloat versus InnoDB's long-transaction undo growth.

The buffer pool, and the steal/force matrix

Every page a query touches passes through the buffer pool: a fixed array of frames, a hash table from page id to frame, per-frame pin counts and dirty bits, and a replacement policy. The policy is not LRU in any serious engine, because a single large sequential scan under LRU evicts the entire working set to cache pages it will never look at again. The standard fixes are LRU-K (O'Neil, O'Neil, and Weikum, 1993), which tracks the last \(K\) references and so distinguishes a page touched twice from a page touched once, and the clock/second-chance approximations that most engines actually ship because they need no per-access bookkeeping. PostgreSQL uses a clock sweep with usage counts plus a small ring buffer for large sequential scans so that a table scan cannot evict the shared working set; this is the buffer-pool version of the same insight.

The recovery-relevant question is not replacement but flushing policy, and it factors into two independent binary choices. Steal: may the buffer manager write an uncommitted transaction's dirty page to disk? Force: must all of a transaction's dirty pages reach disk before its commit returns?

Force (flush data pages at commit)No-force (flush lazily)
No-stealNo undo, no redo. Simplest recovery, and unusable: every transaction must pin all its dirty pages until commit, so a transaction larger than the buffer pool cannot run. No undo, redo needed. Still requires pinning; bounded transaction size.
StealUndo needed, no redo. Commit latency includes random writes of every touched page; throughput collapses under scattered updates. Undo and redo both needed. The universal choice: unlimited transaction size, commit costs one sequential log write, data pages flush in the background.

The bottom-right cell is what ARIES implements and what nearly every disk-based engine runs, and the two other cells explain why the log has to carry both undo and redo information. The two write-ahead logging rules are exactly the obligations that cell creates. The undo rule (write-ahead proper): before a dirty page is written to disk, the log records describing its changes must already be durable, so that a crash leaving an uncommitted change on disk can still be reversed. The redo rule (commit rule): before a commit returns, all of that transaction's log records, up to and including the commit record, must be durable, so that a crash losing the un-flushed data pages can still replay them. Together they mean the log is the database and the data pages are a cache of it, which is the sentence that makes cloud-native designs like Aurora ("the log is the database", Verbitski and colleagues, SIGMOD 2017) an engineering conclusion rather than a slogan: if the log is authoritative, the only thing that must cross the network is the log.

What fsync actually guarantees

Durability bottoms out in one system call, and its contract is narrower than it looks. write() copies bytes into the kernel page cache and returns; nothing is on stable media. fsync(fd) asks the kernel to push that file's dirty pages and its metadata to the device and, on a correctly configured stack, to issue a cache-flush command so the drive's volatile write cache is emptied. What it does not guarantee: that the containing directory entry is durable (a newly created file needs an fsync on the directory too); that a failed fsync can be retried meaningfully (Linux historically dropped the error state after reporting it once, the "fsyncgate" problem that forced PostgreSQL to panic-and-recover on fsync failure rather than retry); that ordering is preserved between two files without explicit barriers; or that a drive lying about its cache flush will be caught by anything short of pulling the power cord.

The cost is what shapes every commit path. Measured on this machine, a 4 KB write followed by fsync took 0.5808 ms against 0.0023 ms for the same write without it, a factor of 252.2, implying a ceiling near 1,722 durable commits per second if every commit forces its own flush. SQLite committing one row per transaction reached 2,431 commits/s with synchronous=FULL and 105,570 commits/s with synchronous=OFF, a 43x difference that is entirely the durability guarantee and not the database work. This gap is the reason group commit exists: batch the log records of many concurrent transactions into one flush, so \(n\) transactions share one fsync and per-transaction latency rises slightly while throughput rises by nearly \(n\). It is also the reason a benchmark that reports a million writes per second is quoting either a batched commit, a replicated acknowledgment that never touched a platter, or a database configured to lose data on power failure, and asking which is the correct response to the number.

ARIES recovery: write-ahead logging and the three passes

Durability and atomicity reduce to one discipline and one algorithm. The discipline is write-ahead logging: every change writes a log record before the changed page can reach disk (the undo rule), and all of a transaction's log records reach disk before commit is acknowledged (the redo rule). The log is a sequential file of records, each with a log sequence number (LSN); every data page stores the pageLSN of the last record applied to it, which is the hinge of the whole algorithm, because comparing a page's LSN with a record's LSN says exactly whether that update is already on the page. Each record carries the transaction's previous record (prevLSN), chaining a transaction's history for undo. ARIES (Mohan, Haderle, Lindsay, Pirahesh, Schwarz, 1992) commits to two policies that maximize normal-case performance: steal (dirty pages may be flushed before their transaction commits, so undo must be possible) and no-force (commit does not flush data pages, only the log, so redo must be possible). Recovery then runs three passes.

Undo rolls back losers, newest record first, following prevLSN chains: the to-undo set starts at \( \{50\} \). Undo LSN 50: restore D to 1, and write a compensation log record (CLR) LSN 60 recording "undid 50", whose undoNextLSN points at 10, the next record still to undo. Undo LSN 10: restore A to 100, write CLR LSN 70 with undoNextLSN null, then an end record for \(T_1\). CLRs are redo-only: they are never themselves undone. If the system crashes again mid-undo, the next recovery redoes the CLRs already written (their effects have LSNs like any update) and resumes undo exactly at the surviving undoNextLSN, so progress is monotone: each crash strictly shrinks the remaining undo work, and recovery is idempotent without ever undoing an undo. Checkpoints in ARIES are fuzzy: they write the two tables, not the data pages, so a checkpoint costs milliseconds and bounds analysis time by how often it runs; redo time is bounded by the age of the oldest recLSN, which is why background page flushing, not checkpointing, is what really bounds recovery. PostgreSQL implements physical redo very much in this mold in src/backend/access/transam/xlog.c, with full-page images after each checkpoint to defeat torn pages; SQLite's rollback-journal mode is the opposite corner of the design space (undo-only, force-at-commit), and its WAL mode moves it halfway toward ARIES.

Distributed databases: commit, replication, and placement

Two-phase commit

When one transaction spans machines, atomicity needs agreement: either every participant commits or none. Two-phase commit (2PC) is the minimal protocol. Phase one: the coordinator sends prepare; each participant forces a prepared record (redo and undo information durable) and votes yes, or votes no. Phase two: if all voted yes, the coordinator forces a commit record, the decision, and tells everyone; any no, and the decision is abort. The cost accounting: two network round trips and two forced log writes on the critical path per participant, so a transaction that commits locally in microseconds pays milliseconds distributed. The deeper cost is blocking: a participant that voted yes has surrendered its right to decide, and if the coordinator dies after prepare but before broadcasting, the participant must hold its locks until the coordinator recovers; it cannot commit (maybe someone voted no) and cannot abort (maybe the decision was commit). This window is not an implementation flaw: no non-blocking atomic commit exists under crash faults with unbounded message delay, a corollary of the Fischer-Lynch-Paterson impossibility result. Production systems shrink the window rather than close it, by making the coordinator's decision itself highly available: Spanner (Corbett and colleagues, 2012) runs each participant group and the coordinator's state over Paxos replication, so "coordinator recovers" means "the replica group elects a new leader in seconds", and 2PC over consensus groups became the standard recipe (CockroachDB and TiDB do the same over Raft).

Replication and consensus, briefly

Replication choices form a small matrix: synchronous (commit waits for replicas: zero data loss, latency of the slowest acked replica) versus asynchronous (commit is local: fast, and a failover loses the unshipped tail); leader-based (all writes through one node: simple, total order for free) versus multi-leader or leaderless (Dynamo-style quorums with \( R + W \gt N \), DeCandia and colleagues 2007: available under partition, but concurrent writes need version vectors and application-level merge). Consensus protocols make leader-based replication safe through failover by making the replica set agree on one log: Raft (Ongaro and Ousterhout, 2014) decomposes the problem into leader election (randomized timeouts, majority votes, terms), log replication (leader appends, followers acknowledge, an entry commits when a majority holds it), and the safety argument (a candidate cannot win without a log at least as up to date as any majority, so committed entries survive elections). A committed write therefore costs one round trip to a majority, and reads either go through the leader with a lease or pay a quorum check. The pointer for the full argument is the Raft paper's Figure 2 and its safety proof section; the practical content for a database engineer is the cost model: majority round trips on the write path, and the throughput ceiling of a single leader, which is what pushes systems to shard first and replicate per shard.

Sharding and consistent hashing, with the arithmetic

Hash partitioning by \( h(k) \bmod N \) distributes perfectly until \(N\) changes. Adding one node to eight: a key stays put only when \( h \bmod 8 = h \bmod 9 \); by the Chinese remainder theorem \(h \bmod 72\) is uniform over 72 residues, and exactly the 8 residues \( r \in \{0, \dots, 7\} \) with \( r \bmod 8 = r \bmod 9 = r \) qualify, so \( 8/72 = 1/9 \) of keys stay and \( 8/9 \) of the entire dataset migrates for a 12.5% capacity increase. Consistent hashing (Karger and colleagues, 1997) hashes nodes and keys onto the same ring; each key belongs to the first node clockwise. Adding a node claims one arc, moving only the keys in it: expected fraction \( 1/(N+1) \). With 1.2 billion keys: mod-hashing moves \( 1.2 \times 10^9 \times 8/9 = 1.067 \times 10^9 \) keys; consistent hashing moves \( 1.2 \times 10^9 / 9 = 1.33 \times 10^8 \), eight times fewer, and only to the new node. One refinement is mandatory: a single point per node makes arc lengths exponentially distributed (coefficient of variation 1), so loads vary wildly; giving each node \(v\) virtual points makes its share a sum of \(v\) independent arcs with relative standard deviation \( \approx 1/\sqrt{v} \): \( v = 100 \) brings imbalance to about 10%, \( v = 1000 \) to about 3%. Dynamo popularized exactly this construction; Cassandra, Riak, and many caches inherit it. The alternative that most SQL-on-Raft systems chose instead is range partitioning with explicit split/move (Bigtable tablets, CockroachDB ranges, TiKV regions), which preserves order for scans and lets a placement service move hot ranges deliberately, at the price of running that placement service.

CAP, stated precisely, and the misstatements corrected

Brewer's conjecture (1998), proved by Gilbert and Lynch (2002) at MIT, is a theorem about one specific model, and almost every popular statement of it is wrong. The precise claim: in an asynchronous network where messages between two groups of nodes can be dropped indefinitely, no implementation of a read/write register can be simultaneously available (every request to a non-failing node returns a non-error response in finite time) and atomically consistent (linearizable). The proof is three lines and worth carrying. Partition the nodes into \( G_1 \) and \( G_2 \) with all messages between them lost. A client writes \( x = 1 \) at a node in \( G_1 \); by availability that write returns. A client then reads \(x\) at a node in \( G_2 \); by availability that read returns, and by the partition it has no information about the write, so it returns the old value. That execution is indistinguishable to \( G_2 \) from one in which the write never happened, and returning a stale value after a completed write violates linearizability. Hence no algorithm has all three. \( \square \)

The corrections, in order of how often they are needed. "Pick two of three" is wrong: partitions are not a property one chooses, they are a fault the network imposes, so the real statement is that when a partition occurs, a system must choose between consistency and availability, and when there is no partition it can have both. "CA systems exist" is wrong for any system spanning a network: a single-node database is trivially CA because it has no partitions to survive, and calling a distributed system CA usually means it becomes unavailable during a partition, which is CP. "Consistency means ACID's C" is wrong: the C in CAP is linearizability, a statement about the visible order of operations, while the C in ACID is about application invariants and is delivered by isolation. "Availability means low latency" is wrong: CAP availability is an eventual, unbounded-time guarantee, so a system that answers every request after ten minutes is available in CAP's sense and useless in practice. Brewer said as much in his 2012 retrospective ("CAP twelve years later"), whose main point is that the theorem's practical content is a design question about partition recovery, detecting the partition, entering an explicit degraded mode, and reconciling afterwards, rather than a taxonomy to sort databases into.

Abadi's PACELC (2012) is the extension that closes the gap the theorem leaves open: if there is a Partition, choose between Availability and Consistency; Else, in normal operation, choose between Latency and Consistency. The second clause is the one that describes daily engineering, because partitions are rare and the latency cost of consensus is paid on every write. Classifying real systems under it is informative: Dynamo-style stores are PA/EL (available under partition, low latency by default), traditional single-leader replication with synchronous replicas is PC/EC, Spanner is PC/EC and buys back latency with hardware clocks, and PostgreSQL with asynchronous replicas is PC/EL at the primary and eventually consistent at the replicas.

Linearizability and serializability are different guarantees

These two words are routinely conflated and they constrain different things. Serializability is a property of transactions: the concurrent execution of a set of multi-operation transactions is equivalent to some serial order of them. It says nothing about which serial order, and in particular a serializable system may order a transaction that started later before one that finished earlier, so a read can return a value that is stale in real time. Linearizability is a property of single objects (or a whole store treated as one object): every operation appears to take effect at a single instant between its invocation and its response, and that instant respects real time, so an operation that begins after another completed must see its effect. It says nothing about grouping operations into transactions.

The conjunction is strict serializability: a serial order exists and it is consistent with real time. That is what Spanner provides and what "external consistency" names in its paper. The distinctions matter operationally. A read replica serving snapshot reads can be serializable and not linearizable, which is exactly the "read your own write" bug: a user posts a comment, the write commits on the leader, the subsequent read hits a lagging replica, and no transaction was ever executed non-serializably. Conversely, a linearizable key-value store with no multi-key transactions gives no serializability guarantee across keys at all, which is the write-skew failure again in a distributed costume. Herlihy and Wing (1990) at Carnegie Mellon defined linearizability with the composability property that makes it valuable: a system whose objects are each linearizable is linearizable as a whole, which is emphatically not true of serializability across independently serializable stores.

Distributed transactions in practice: Spanner, Calvin, Percolator

Three production designs solve the same problem in three different places, and knowing which is which is the useful form of this knowledge.

Spanner (Corbett and colleagues, 2012) puts the cost in hardware. Each datacenter has GPS receivers and atomic clocks, and the TrueTime API returns not a timestamp but an interval \( [\text{earliest}, \text{latest}] \) guaranteed to contain true absolute time, with uncertainty \( \epsilon \) typically a few milliseconds. A read-write transaction takes a commit timestamp \(s\) and then executes a commit wait: it does not release locks or acknowledge the client until TT.after(s) is true, meaning absolute time has certainly passed \(s\). Since every transaction waits out its own uncertainty, timestamp order is guaranteed to agree with real time, which delivers external consistency; read-only transactions then need no locks at all, because a snapshot at timestamp \(s\) is well defined globally. The price is roughly \( 2\epsilon \) of added commit latency, which is why the clock infrastructure is load-bearing: shrinking \( \epsilon \) shrinks the tax. CockroachDB reproduces the design without atomic clocks by using a larger uncertainty window and restarting transactions that observe a value inside it, trading hardware for occasional retries.

Calvin (Thomson and colleagues at Yale, 2012) puts the cost in determinism. A sequencing layer, itself replicated by consensus, decides a global order of transactions before they execute; every replica then executes that same batch in that same order with a deterministic locking scheme, so all replicas reach the same state without agreeing on anything afterward. Two-phase commit disappears entirely, because a participant cannot unilaterally abort: the decision was made in the sequence. The cost is the constraint that made it work: a transaction's read and write sets must be known before execution, so interactive transactions and dependent reads require a reconnaissance step that may have to be redone if the data moved. FaunaDB is the clearest commercial descendant.

Percolator (Peng and Dabek at Google, 2010) puts the cost in latency, and asks nothing special of the infrastructure. It layers snapshot isolation over Bigtable using client-driven two-phase commit and per-row metadata columns: a lock column, a write column, and the data itself. Prewrite writes the value plus a lock record pointing at a designated primary row; commit atomically flips the primary's lock into a write record, which is the single point that decides the transaction; secondary rows are cleaned up lazily, and any reader encountering a stale lock can resolve it by checking the primary. The design is attractive because it needs no dedicated transaction manager and inherits the underlying store's fault tolerance, and it is slow, with several round trips per transaction, which is fine for Google's incremental indexing workload and is why TiDB, which adopted the protocol, has spent years optimizing around it with one-phase commit and async commit paths.

Problem 6

A cache cluster of \( N = 12 \) nodes uses consistent hashing with \( v = 200 \) virtual nodes each, holding \( K = 3.6 \times 10^9 \) keys. (a) One node fails permanently; how many keys move, and where do they land? (b) Compare with \( h \bmod 12 \) to \( h \bmod 11 \) rehashing. (c) Approximately what relative load imbalance should be expected across nodes before the failure? (d) A hot key receives 2 million requests/s regardless of placement; does consistent hashing help, and what does?

Solution. (a) The failed node owned an expected \( K/N = 3.6 \times 10^9 / 12 = 3.0 \times 10^8 \) keys. Exactly those keys move (nothing else changes ownership), and because its 200 virtual points are scattered around the ring, the orphaned arcs are absorbed by many distinct successors, spreading the 300M keys across the surviving 11 nodes at roughly \( 2.7 \times 10^7 \) each, a 9% load bump per survivor rather than a doubling of one neighbor.

(b) Keys stay under mod-rehash when \( h \bmod 12 = h \bmod 11 \): over the CRT modulus 132 the residues \( r \in \{0, \dots, 10\} \) qualify, 11 of 132, so \( 11/132 = 1/12 \) stay and \( 11/12 \times 3.6 \times 10^9 = 3.3 \times 10^9 \) keys move: eleven times more traffic than the \(3.0 \times 10^8\) minimum, hitting every node's cache hit rate at once.

(c) Each node's share is approximately a sum of 200 exponential arc lengths; relative standard deviation \( \approx 1/\sqrt{200} = 0.071 \), so about \( \pm 7\% \) typical imbalance (and roughly \( \pm 14\% \) at two sigma). Doubling \(v\) to 400 improves this only to 5%: the square root is the whole story.

(d) No: consistent hashing places a key on exactly one node (plus replicas), so a single hot key still lands its 2M req/s on one server. Placement algorithms cannot split a point; the remedies are replication of that key with client-side load spreading, a small front cache at the routers (the "few hot keys fit in L1 of the fleet" observation), or key splitting at the application. Solution. Analysis: \(T_2\) committed (LSN 40). \(T_1\) is in-rollback but unfinished: still a loser, lastLSN 60, and LSN 60 is a CLR with undoNextLSN = 10. Dirty page table: P5 recLSN 10, P3 recLSN 20, P9 recLSN 50 (LSN 60 also touches P9 but P9 entered the table at 50).

Redo from LSN 10: LSN 10 (P5): pageLSN 30 \( \ge \) 10, skip. LSN 20 (P3): pageLSN 20 \( \ge \) 20, skip (the flush already captured it). LSN 30 (P5): 30 \( \ge \) 30, skip. LSN 50 (P9): pageLSN 60 \( \ge \) 50, skip. LSN 60 (CLR on P9): 60 \( \ge \) 60, skip. Redo does no work at all: every effect provably reached disk, and the pageLSN comparisons are what prove it.

Undo: losers \( \{T_1\} \); start from lastLSN 60. LSN 60 is a CLR: never undo it, jump directly to its undoNextLSN = 10. Undo LSN 10: restore A to 100 on P5, write CLR LSN 70 (undoNextLSN null), write end for \(T_1\). Total new log: two records. The already-performed undo of LSN 50 is not repeated, which is precisely what undoNextLSN buys: bounded, monotone undo across any number of crashes.

Problem 8

Derive the on-call write-skew outcome under three regimes, for the initial state (alice on call, bob on call) and the invariant "at least one on call". \(T_A\): read count of on-call doctors; if 2, set alice off. \(T_B\): read count; if 2, set bob off. Both start together. State, for (i) strict 2PL with predicate/range locking, (ii) plain snapshot isolation, (iii) SSI, exactly which locks or conflicts arise, which transactions commit, and the final state. Then compute, for a clinic running 500 such pair-conflicts per day under SSI with a 1 ms retry, the added latency budget.

Solution. (i) Strict 2PL: each transaction's count is a predicate read over on_call = true, requiring shared locks covering both rows (or the index range). \(T_A\) takes S locks on both rows; \(T_B\) likewise (shared locks are compatible). Then \(T_A\) requests X on alice's row: it must wait for \(T_B\)'s S lock; \(T_B\) requests X on bob's row: waits for \(T_A\)'s S lock. Deadlock; the detector aborts one, say \(T_B\); \(T_A\) commits (alice off, count 1); \(T_B\) retries, reads count 1, and declines. Final: one on call. Invariant held, at the cost of a deadlock-abort cycle.

(ii) SI: both snapshots show count 2. Writes are disjoint (alice's row versus bob's row), so first-committer-wins sees no overlap; both commit. Final: zero on call, invariant broken. This is write skew, and no reads-writes over single objects detect it, because the cycle is \( T_A \xrightarrow{rw} T_B \xrightarrow{rw} T_A \): each read something the other wrote afterward.

(iii) SSI: the engine tracks rw-antidependencies. \(T_A\) read bob's row which \(T_B\) then wrote: edge \( T_A \xrightarrow{rw} T_B \). \(T_B\) read alice's row which \(T_A\) then wrote: edge \( T_B \xrightarrow{rw} T_A \). Two consecutive rw edges with both endpoints concurrent is the dangerous structure; at the second commit the engine aborts one transaction with a serialization failure. The retry reads count 1 and declines. Final: one on call.

Budget: 500 conflicts/day each costing one abort + 1 ms retry: 0.5 s of added work per day, invisible. The comparison to price instead: under 2PL the same 500 conflicts are 500 deadlock detections (typically resolved on a 10 ms to 1 s detector cycle); at a 100 ms average detection delay that is 50 s of daily lock-held stall time affecting bystander transactions queued behind the held locks. The optimistic protocol wins precisely because the conflict rate (500/day) is tiny relative to throughput; at high contention the inequality flips, which is the general OCC-versus-locking trade stated quantitatively.

The measured SQLite outcome is the interesting one: SQLite is not vulnerable to write skew, but not because it detects the rw-cycle; WAL mode permits exactly one writer at a time and refuses a write from a connection whose snapshot is stale (SQLITE_BUSY_SNAPSHOT, surfacing in Python as "database is locked"). That is a stronger and blunter rule than first-committer-wins: it aborts on any concurrent committed write, overlapping or not, which makes SQLite serializable by construction and unscalable for concurrent writers by the same construction. The three engines line up as a spectrum of the same design variable: SQLite serializes writers entirely, PostgreSQL SI orders only overlapping writes and admits write skew, PostgreSQL SSI adds the dependency tracking that recovers serializability optimistically.

A Selinger optimizer in fifty lines, and a hash join in C++

The dynamic program from the optimization section, executable: it reproduces the cost table exactly (1,051 / 3,000 / 2,050 for the pairs, 1,552 for the winning three-way plan), and extending the catalog dictionary is the fastest way to build intuition for how estimates steer plans. The C++ tab implements the two phases of an in-memory hash join over the same schema shapes, with the build/probe asymmetry and the reason the smaller input builds spelled out in code. 

from itertools import combinations

# catalog: pages, tuples; join predicates carry max(V,V) per Selinger
CAT = {
    "E": dict(pages=1000, tuples=10_000),
    "D": dict(pages=50,   tuples=500),
    "W": dict(pages=500,  tuples=30_000),
}
PREDS = {frozenset("ED"): 500, frozenset("EW"): 10_000}
FILTER = {"D": 1 / 500}          # dname = 'widgets'
TPP = 20                          # tuples per materialized page

best = {}                         # frozenset -> (cost, rows, plan string)
for r, c in CAT.items():
    best[frozenset(r)] = (c["pages"], c["tuples"] * FILTER.get(r, 1.0), r)

names = sorted(CAT)
for size in (2, 3):
    for combo in combinations(names, size):
        subset = frozenset(combo)
        for right in sorted(subset):          # left-deep: extend by one relation
            left = subset - {right}
            lcost, lrows, lplan = best[left]
            sel = 1.0                          # selectivity of applicable predicates
            for pair, maxv in PREDS.items():
                if right in pair and (pair - {right}) <= left:
                    sel *= 1.0 / maxv          # else: cartesian product, sel stays 1
            out_rows = lrows * CAT[right]["tuples"] * FILTER.get(right, 1.0) * sel
            if size < len(names):              # intermediate: pay to materialize it
                cost = lcost + CAT[right]["pages"] + max(1, round(out_rows / TPP))
            else:                              # top: reread left intermediate instead
                cost = lcost + max(1, round(lrows / TPP)) + CAT[right]["pages"]
            plan = f"({lplan} JOIN {right})"
            if subset not in best or cost < best[subset][0]:
                best[subset] = (cost, out_rows, plan)

for s in sorted(best, key=len):
    c, r, p = best[s]
    print(f"{''.join(sorted(s)):3s}  cost={c:6.0f}  rows={r:8.0f}  {p}")

#include <cstdint>
#include <string>
#include <unordered_map>
#include <vector>

struct Emp  { int64_t eid; int64_t deptno; std::string name; };
struct Work { int64_t eid; int64_t pid; double hours; };
struct Out  { int64_t eid; int64_t pid; std::string name; };

std::vector<Out> hash_join(const std::vector<Emp>& emp,     // M pages
                           const std::vector<Work>& works)  // N pages, larger
{
    // Phase 1: BUILD. One pass over the smaller input. The table maps
    // key -> row indexes; multimap semantics because keys repeat.
    std::unordered_map<int64_t, std::vector<uint32_t>> ht;
    ht.reserve(emp.size());
    for (uint32_t i = 0; i < emp.size(); ++i)
        ht[emp[i].eid].push_back(i);           // O(M) build, O(M) memory

    // Phase 2: PROBE. One pass over the larger input; each probe is O(1)
    // expected. Total: O(M + N) work and M + N page reads, the formula
    // from the cost section. Build side chosen small so 'ht' fits: if it
    // does not, partition both inputs by hash(eid) % k first (Grace).
    std::vector<Out> out;
    for (const Work& w : works) {
        auto hit = ht.find(w.eid);
        if (hit == ht.end()) continue;         // no match: most probes, ideally
        for (uint32_t i : hit->second)         // usually 1 employee per eid
            out.push_back({w.eid, w.pid, emp[i].name});
    }
    return out;
    // What a real engine adds: a Bloom filter on the build keys pushed into
    // the probe scan (skips misses before hashing), vectorized probes that
    // process 2048 keys per call, and NUMA-partitioned tables. The
    // asymmetry survives every refinement: small side owns memory, large
    // side streams.

How it is done in practice

Vectorized execution and query compilation

The classical executor is the Volcano iterator model (Graefe, 1994): every operator exposes next() returning one tuple, plans compose as iterator trees. Its elegance hides a cost invisible in the I/O model: per-tuple virtual calls, branch mispredictions, and attribute interpretation, hundreds of instructions of overhead per row for operators whose useful work is one comparison or one addition. Two research lines fixed it, and both now run the industry. The CWI line, Boncz, Zukowski, and Nes's MonetDB/X100 (CIDR 2005), keeps the iterator tree but passes vectors of roughly a thousand values per call: interpretation overhead amortizes across the vector, primitives become tight loops the compiler auto-vectorizes with SIMD, and the vector size is tuned to keep the working set inside the CPU cache (the same group's earlier MonetDB had gone all the way to full-column materialization and found it thrashes memory; X100 is the deliberate midpoint). This architecture became VectorWise, then the design template for Snowflake's executor, Meta's Velox, and DuckDB, whose authors Raasveldt and Mühleisen (CWI, SIGMOD 2019) describe it as "vectorized processing in an embedded package"; the 49.4 ms single-thread aggregate measured above is this architecture doing its job. The competing line is query compilation: Neumann's HyPer (TU Munich, VLDB 2011) generates LLVM IR per query, fusing each pipeline (scan to hash-build, scan-probe to aggregate) into one loop that keeps the current tuple in registers, erasing operator boundaries entirely. Amazon Redshift compiles to C++; SingleStore and Umbra (HyPer's successor) compile; SQLite's bytecode VM is compilation's humble ancestor. Kersten and colleagues' head-to-head study (VLDB 2018, TU Munich and CWI together) found the honest verdict: compilation wins calculation-heavy queries and tight OLTP loops, vectorization wins memory-bound scans and offers better profiling, adaptivity, and compile-time; both beat tuple-at-a-time by one to two orders of magnitude, and the industry keeps both.

HTAP and the storage-compute split

Two architectural moves define the current production landscape. The first is hybrid transactional/analytical processing: serving the analytic query from the operational data without an overnight ETL. Kemper and Neumann's HyPer (ICDE 2011) proposed one machine doing both via MVCC snapshots; the shipping designs are mostly dual-format instead: TiDB replicates each Raft group into TiFlash, a columnar learner replica, and routes analytics there with freshness guaranteed by the Raft index (VLDB 2020); SingleStore mixes an in-memory rowstore with columnar segments; Oracle Database In-Memory and SQL Server columnstore indexes bolt the column format onto the row engine; AlloyDB attaches a columnar cache to a PostgreSQL primary. The second move is disaggregation. Aurora (Verbitski and colleagues, SIGMOD 2017) observed that a replicated MySQL ships full pages five ways and made one change with system-wide consequences: the storage tier understands the redo log, so the database ships only log records to six storage replicas across three availability zones (write quorum 4/6, read quorum 3/6), and storage nodes materialize pages themselves; crash recovery becomes the storage tier's steady-state behavior, and the paper reports 35x throughput over MySQL on the same hardware class. Snowflake (Dageville and colleagues, SIGMOD 2016) split along a different axis: tables live in object storage as immutable compressed micro-partitions with min-max metadata (PAX at datacenter scale), and stateless "virtual warehouses" of compute spin up per workload, scale independently, and share nothing but the data; time travel and zero-copy cloning fall out of immutability for free. Every subsequent cloud warehouse and the lakehouse formats (Parquet plus Iceberg/Delta tables) are variations on this separation, and the classical cost model survives translated: page I/O became object-store GET requests, and the buffer pool became the local NVMe cache.

Vector search as an index type

Retrieval-augmented systems made "nearest neighbor under cosine distance over \( 10^8 \) embeddings" a database workload, and the right frame is the one this page has used throughout: it is an index type with a cost model, not a new kind of database. Exact search is a scan: \( 10^8 \) vectors at 768 fp32 dimensions is 307 GB and \( 7.7 \times 10^{10} \) multiply-adds per query. The two approximate index families trade recall for that cost. IVF (inverted file) clusters vectors into \( n_{\text{list}} \) cells by k-means and searches only the \( n_{\text{probe}} \) cells nearest the query: with \( n_{\text{list}} = 10^4 \) and \( n_{\text{probe}} = 32 \), the scan shrinks to \( 32 \times 10^8 / 10^4 = 3.2 \times 10^5 \) candidates, a 312x reduction, and product quantization (Jégou, Douze, Schmid, 2011) compresses each candidate to 8-64 bytes so the scan stays in cache; recall is recovered by re-ranking survivors at full precision. HNSW (Malkov and Yashunin, 2018) builds a multi-layer navigable small-world graph: greedy descent through sparse upper layers lands near the target in \( O(\log N) \) hops, then a beam search of width efSearch over the bottom layer (max degree \(M\), typically 16-64) refines; queries touch roughly \( \text{efSearch} \times M \) vectors, thousands rather than hundreds of thousands, with 95-99% recall, at the price of memory-resident graph plus vectors and slow, hard-to-parallelize builds. IVF rebuilds cheaply and pairs with disk; HNSW answers fastest from RAM. The architectural point: pgvector implements both as PostgreSQL index access methods, so the planner weighs an HNSW probe against a B+tree scan with the same machinery Selinger built, and the dedicated vector databases (Faiss-backed services, Milvus, and peers) compete on exactly the terms this page prices: index build amplification, memory residency, and filtered-search selectivity, the vector world's rediscovery of predicate pushdown.

The current research frontier

Four active fronts, each with several groups contending. Learned components: Kraska and colleagues at MIT proposed replacing B+tree internals with learned models of the key distribution (2018) and followed with SageDB; the measured wins are real for static sorted data (a model is a smaller, faster router than three internal levels) and contested for updates, where the Wisconsin and TU Munich groups showed tuned classical structures close most of the gap; learned cardinality estimation, attacking the optimizer's weakest input, is further along, with Neumann's group at TU Munich and the DSAIL group at MIT publishing competing estimators, and industry deployment still cautious because a bad estimate is a regressed plan in production. Cloud-native transactions: after Aurora, the question is multi-writer disaggregation: PolarDB-MT (Alibaba), Aurora Limitless, and Neon's copy-on-write branching of the WAL each pick a different point between shared-disk and sharded-nothing; academically the FoundationDB paper (SIGMOD 2021) rehabilitated deterministic and optimistic designs, and Calvin-style deterministic execution (Yale, Abadi's group) returned in production as Fauna. HTAP freshness: the open trade is how stale the analytic replica may be; TiFlash pins freshness to Raft, Google's F1 Lightning and Napa choose lag with SLAs, and the academic line (Berkeley, ETH Zurich) measures the throughput cost of true freshness. Embedded and composable engines: DuckDB made the single-node column engine a library, Velox (Meta) and DataFusion (Apache Arrow, Rust) make executors embeddable components, and Arrow is becoming the lingua franca that lets optimizers, executors, and storage from different projects compose, a quiet unbundling of the monolithic DBMS that Hellerstein and Stonebraker's Red Book essays anticipated. Vector indexes are converging into general engines from both directions, with pgvector and DiskANN-in-SQL-Server (Microsoft Research's graph index engineered for SSDs) as the leading edge.

Common misconceptions

"SQL is the relational model." SQL diverges from Codd's model in ways that bite: relations are bags, not sets (duplicates change aggregate results and equivalence rules); NULL introduces three-valued logic in which x = NULL is never true, so NOT IN against a list containing NULL silently returns nothing, and two NULLs are equal for GROUP BY but unequal for =. Knowing where the model ends and the standard begins is what debugging those queries requires.

"Normalization is obsolete because analytics denormalizes anyway." The star schema's wide fact table is a deliberate, priced denormalization at the read replica, downstream of a normalized system of record. Running the business's writes on denormalized tables reintroduces exactly the update anomalies BCNF names, and the modern twist runs opposite to the folklore: columnar compression makes the normalized form cheaper to scan than the folklore expects, because repeated dimension values dictionary-encode to almost nothing.

"B-trees are deep; that is why lookups are slow." The fanout arithmetic gives 510 children per 8 KB node: a billion rows in four levels, three of which (31.6 MB) sit permanently in cache. A cold lookup is one I/O. When an indexed lookup is slow the cause is almost never depth: it is an unclustered index fetching scattered heap pages, the regime Problem 3 priced at one I/O per matching row.

"REPEATABLE READ means my transactions are serializable." In PostgreSQL, REPEATABLE READ is snapshot isolation, and Oracle's SERIALIZABLE is SI too; both permit write skew, demonstrated concretely above with two committed transactions breaking an invariant each one checked. Serializability requires either range-locking 2PL or SSI, and both cost something: blocking or retries. Choosing a level is choosing which anomalies the application tolerates, from the Berenson catalog, not picking a speed dial.

"NoSQL replaced relational databases." The systems that defined the movement converged back: Bigtable grew Megastore and then Spanner, which speaks SQL and runs 2PC; DynamoDB added transactions; MongoDB added multi-document ACID and a WiredTiger B+tree/LSM storage engine. What survived of the movement is valuable and narrower: explicit partition keys, quorum tuning, and the discipline of designing for the access path, all of which are relational-era ideas (the granularity paper, range partitioning) wearing new clothes.

"Column stores are simply faster." The same data, same machine, same queries measured above: the column store wins the scan aggregate 48x single-threaded and loses the point lookup 100x. Neither number is an implementation accident; both follow from bytes touched per operation. An engine is fast for a workload, and the workload term is not optional.

"The optimizer finds the best plan." It finds the cheapest plan under a cost model fed by cardinality estimates, and those estimates rest on uniformity and independence assumptions that correlated real data violates; errors compound multiplicatively across joins, so six tables can put the estimate off by 10^4 (the VLDB 2015 measurement study's central result). This is why hints, plan pinning, and "it was fast yesterday" incidents exist, and why cardinality estimation, not search, is the optimizer research frontier.

"WAL is an implementation detail of one database." Write-ahead logging is the load-bearing idea of durable systems generally: every filesystem journal, every message queue with acknowledged delivery, Raft's replicated log, and Aurora's log-is-the-database design are the same two rules (log before data, log before ack) re-instantiated. Recognizing ARIES's three passes inside a filesystem checker or a stream processor's checkpoint recovery is the transfer test for this material.

Key takeaway

A database is the systematic exploitation of three gaps: between what a query says and how it can be executed (the algebra and its optimizer), between memory and durable storage (pages, indexes, and the log), and between one transaction's view and many transactions' truth (isolation and its anomalies). Every design in this page is an arithmetic answer to one of those gaps: the fanout calculation says why B+trees are four levels deep, the write-amplification sum says when an LSM tree earns its read penalty, the Selinger estimates say which join order survives, and the anomaly catalog says exactly what each isolation level costs in correctness. The measured experiments make the same point empirically: identical data was 48x faster to aggregate in a column store and 100x faster to look up in a row store, and the same snapshot machinery that made reads repeatable is what let write skew through. The modern systems, vectorized engines, disaggregated storage, Raft-replicated shards, vector indexes, are recombinations of these invariants under new cost ratios, not departures from them. Learn the accounting, and each new engine becomes an afternoon of reading rather than a new field.
