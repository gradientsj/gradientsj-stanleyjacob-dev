Big-O and complexity analysis

Algorithms · foundations · Jun 2026

Big-O describes how an algorithm's running time or memory grows as the input grows, ignoring constant factors and lower-order terms. It is the shared language for comparing approaches before you write a line of code.

What matters is the shape of the growth, not the exact operation count. An O(n) algorithm eventually beats an O(n squared) one regardless of constants, so naming the class tells you whether a solution will scale to the input you expect.

When to use it

  • Choosing between two approaches before implementing either.
  • Explaining why a brute-force solution times out while a smarter one does not.
  • Reasoning about extra memory, not just time.

Step by step

  1. Count the dominant operation as a function of the input size n.
  2. Drop constant factors and lower-order terms.
  3. Take the worst case unless told otherwise, but also know average and amortized cost.
  4. Repeat the count for extra memory to get the space complexity.
# O(n) time, O(n) space: one pass with a hash set
def has_duplicate_seen(a):
    seen = set()
    for x in a:
        if x in seen:
            return True
        seen.add(x)
    return False

# O(n^2) time, O(1) extra space: check every pair
def has_duplicate_pairs(a):
    for i in range(len(a)):
        for j in range(i + 1, len(a)):
            if a[i] == a[j]:
                return True
    return False

Complexity

Common classes, fastest to slowest: O(1), O(log n), O(n), O(n log n), O(n squared), O(2^n), O(n!). The two functions above solve the same problem and trade time for space: the first is O(n) time and O(n) memory, the second is O(n squared) time and O(1) memory.
