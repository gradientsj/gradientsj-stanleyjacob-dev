CAP theorem and consistency

Software & architecture · architecture · Jun 2026

The CAP theorem says that during a network partition a distributed system must choose between consistency (every read sees the latest write) and availability (every request gets a response). You cannot have both while partitioned.

Partitions are rare but inevitable, so the real question is what the system does during one: refuse to serve possibly-stale data (CP) or stay up and reconcile later (AP).

How to read it

  • CP systems (a single-leader SQL database, ZooKeeper) reject or block during a partition to stay consistent.
  • AP systems (Cassandra, Dynamo) keep serving and resolve conflicts afterward.
  • With no partition you can have both; CAP only forces the choice during a failure.

Trade-offs

Strong consistency simplifies application logic but reduces availability and adds coordination latency. Eventual consistency maximizes availability and speed but forces you to handle stale reads and conflicts. Most systems choose per operation: strong for money, eventual for likes.

Worked example

Picture two data centers and the link between them drops. A client writes to each side.

  • AP store: both sides accept the write and stay available; when the link heals, the versions are merged (last-write-wins, version vectors, or CRDTs). A reader might briefly see a stale value.
  • CP store: only the side with a majority quorum accepts writes; the minority side returns errors until the partition heals, so no one ever reads stale data.

Output: the same incident yields availability on one design and consistency on the other; you pick based on what a stale read costs.

Follow-up questions

  • Is CAP a strict pick-two? More nuanced: you only trade C vs A during a partition. PACELC extends it: else (no partition) you trade latency vs consistency.
  • What is eventual consistency? Replicas converge to the same value if writes stop; reads may be stale in the meantime.
  • How do AP systems resolve conflicts? Last-write-wins by timestamp, version vectors, or conflict-free replicated data types (CRDTs).
  • Where do you need strong consistency? Anywhere a stale read causes real harm: account balances, inventory, unique constraints.
