Building a bytecode virtual machine

Notes · compilers & interpreters · Jun 2026

Writing a small language is the best way to demystify how code runs. The pipeline is short and every stage is satisfying to build: source to tokens to syntax tree to bytecode to a virtual machine that executes it. This is the design behind my small language & bytecode VM project.

Lexing and parsing

The lexer turns characters into tokens, numbers, identifiers, operators. The parser turns tokens into structure. A Pratt parser (top-down operator precedence) is the sweet spot here: it handles precedence and associativity cleanly by associating a parsing function and a binding power with each token, and it is far less code than a full grammar.

Bytecode vs tree-walking

The simplest interpreter just walks the syntax tree and evaluates nodes recursively. It works, but it is slow: every step chases pointers and re-dispatches on node type. Compiling to a flat array of bytecode instead gives a compact, cache-friendly instruction stream the VM can rip through in a tight loop.

The virtual machine

A stack-based VM keeps a value stack and an instruction pointer, and runs one big dispatch loop: 

loop:
  op = code[ip++]
  switch op:
    OP_CONST:  push(constants[code[ip++]])
    OP_ADD:    b = pop(); a = pop(); push(a + b)
    OP_JUMP_IF_FALSE: ...
    OP_RETURN: return pop()

Arithmetic becomes push, push, add. Variables become loads and stores against the stack. Control flow, if and loops, becomes conditional and unconditional jumps that move the instruction pointer.

What you learn

  • How precedence and associativity actually get resolved.
  • Why a flat instruction stream is faster than walking a tree.
  • How high-level control flow compiles down to nothing but jumps.

From here the natural next steps are functions and call frames, a garbage collector, and constant folding, each one a small, self-contained project on top of the VM.
