DynamoDB

Software &amp; architecture · databases · Jul 2026

Amazon DynamoDB is AWS's fully managed key-value and document database. You never see a node, a disk, or a replica. You create a table, declare a primary key, and issue reads and writes over HTTPS while AWS operates the fleet, splits the data, and keeps three copies across availability zones. During Prime Day 2021 the service peaked at 89.2 million requests per second while holding single-digit millisecond latency, which is the whole pitch in one number.

The name causes persistent confusion. The 2007 Dynamo paper describes a leaderless, eventually consistent store Amazon built internally for its shopping cart, and Cassandra is that design's direct open-source descendant. DynamoDB, launched in 2012, kept Dynamo's goals of incremental scale and predictable performance but replaced the architecture wholesale with single-leader partitions, Paxos replication, and a central control plane. The 2022 USENIX paper on DynamoDB is the authoritative account of what runs today.

The productive mental model is an enormous hash table whose buckets are small replicated B-trees. Indexes, capacity units, adaptive capacity, and single-table design all follow from that picture.

The one-line version. DynamoDB gives up joins and ad-hoc queries to buy one guarantee, that any access you can phrase as a key lookup plus a range scan stays single-digit milliseconds no matter how large the table grows.

What it is and the mental model

A table in DynamoDB is a keyspace split into partitions. The partition key of every item is hashed, the hash space is divided into contiguous ranges, and each range is owned by one partition. A partition is a replicated B-tree with a throughput budget, three copies spread across availability zones. When a table outgrows a partition in size or traffic, the partition splits and the ranges are reassigned. None of this is visible to you, which is precisely the product.

Items hold up to 400 KB of attributes and carry no fixed schema beyond the declared key attributes, so two items in the same table can have entirely different shapes and attribute validation moves into application code. Attributes can be scalars, sets, lists, or nested maps. JSON is only the transport format, the storage format is proprietary.

The 2007 Dynamo system used consistent hashing on a peer-to-peer ring, sloppy quorums with hinted handoff, and vector clocks that pushed conflict resolution onto the application. It was always writable and operationally demanding, and every internal team ran its own installation. DynamoDB inverted that into one multi-tenant service, a stateless request router fleet in front, a single elected leader per partition replicating through Multi-Paxos, and a metadata system mapping keys to partitions. The USENIX paper describes the service as combining the best parts of Dynamo with the best parts of SimpleDB, Amazon's earlier managed database.

Predictable performance ranks above raw speed in the design. A GetItem touches one partition's B-tree whether the table stores a gigabyte or a petabyte, so latency stays flat as data grows. Marc Brooker calls this the paper's central theme, that consistent performance at any scale matters more to users than a lower mean.

Mental model. A giant hash table whose buckets are replicated B-trees. If your access pattern is a hash lookup plus an optional range scan, DynamoDB scales it indefinitely. Anything else, the database pushes back.

How you use it

The primary key is either a partition key alone or a partition key plus a sort key. The partition key picks the physical partition through the hash. The sort key orders items inside the partition in a B-tree, which enables Query conditions like begins_with and between and descending reads via ScanIndexForward=False. Choose the partition key for even traffic spread and the sort key for the range dimension your queries need, chronological order for a message feed, a date prefix for orders.

There are two read paths. Query retrieves items matching a partition key and an optional sort key condition, and it reads only those items. Scan walks the entire table, costs capacity in proportion to everything stored, and belongs in migrations and backfills rather than request paths. PartiQL adds SQL-shaped syntax over the same operations without adding capability, and a PartiQL statement without a key condition is still a scan.

Billing and throughput both flow through capacity units. One read capacity unit buys one strongly consistent read per second of an item up to 4 KB, or two eventually consistent reads. One write capacity unit buys one write per second up to 1 KB, rounded up, so a 100-byte write still bills a full unit. ProjectionExpression trims what crosses the network but not what you pay, because capacity is charged on the full item size read from storage.

For time-ordered sort keys, prefer identifiers that sort chronologically and are unique, such as ULIDs, UUIDv7, or Snowflake IDs. Raw timestamps collide when two writes land in the same millisecond, and a colliding composite key silently overwrites the earlier item.

import boto3
from ulid import ULID

ddb = boto3.resource("dynamodb")

table = ddb.create_table(
    TableName="Messages",
    KeySchema=[
        {"AttributeName": "chat_id", "KeyType": "HASH"},      # partition key
        {"AttributeName": "message_id", "KeyType": "RANGE"},  # sort key
    ],
    AttributeDefinitions=[
        {"AttributeName": "chat_id", "AttributeType": "S"},
        {"AttributeName": "message_id", "AttributeType": "S"},
        {"AttributeName": "user_id", "AttributeType": "S"},
    ],
    GlobalSecondaryIndexes=[{
        "IndexName": "by_user",
        "KeySchema": [
            {"AttributeName": "user_id", "KeyType": "HASH"},
            {"AttributeName": "message_id", "KeyType": "RANGE"},
        ],
        "Projection": {"ProjectionType": "KEYS_ONLY"},
    }],
    BillingMode="PAY_PER_REQUEST",   # on-demand mode
)
table.wait_until_exists()  # creation is asynchronous, wait for ACTIVE

# write: a ULID sort key gives chronological order plus uniqueness
table.put_item(Item={
    "chat_id": "chat#4821",
    "message_id": str(ULID()),
    "user_id": "user#77",
    "body": "shipping the fix now",
})

# read: newest 50 messages in one chat. One partition, already sorted.
resp = table.query(
    KeyConditionExpression="chat_id = :c",
    ExpressionAttributeValues={":c": "chat#4821"},
    ScanIndexForward=False,
    Limit=50,
)

# read: everything one user wrote across all chats, via the GSI.
# Eventually consistent by definition, ConsistentRead is not allowed here.
resp = table.query(
    IndexName="by_user",
    KeyConditionExpression="user_id = :u",
    ExpressionAttributeValues={":u": "user#77"},
)

Single-table design

DynamoDB has no joins, deliberately, because a join's cost grows with data volume and would break the flat-latency promise. Single-table design is the standard answer. Instead of joining at read time you pre-join at write time, storing several entity types in one table under generic key attributes so that one Query returns everything a screen needs. All items sharing a partition key form an item collection, and the item collection is the unit you fetch.

The convention uses opaque attribute names, PK and SK, with entity-type prefixes packed into the values. A user's profile and all of their orders share PK = USER#77 and differ by sort key prefix, so begins_with(SK, 'ORDER#') selects the orders while a bare partition key query returns profile and orders together in one round trip. GSIs get overloaded the same way, with one index serving several lookup patterns across different entity types.

Alex DeBrie, who wrote the standard book on this, is explicit about the costs. The modeling is rigid, an access pattern you did not anticipate can require an ETL rewrite of the whole table, and analytics tooling cannot read the overloaded layout without re-normalizing it first, usually by streaming changes into a warehouse. He also names the cases where you should skip it, early-stage products where flexibility is worth more than a few milliseconds saved, and GraphQL backends whose resolvers issue serial fetches anyway.

PK           SK                    attributes
USER#77      PROFILE               name, email, plan
USER#77      ORDER#2026-07-18#901  total, status
USER#77      ORDER#2026-07-20#944  total, status
ORDER#901    ITEM#1                sku, qty
ORDER#901    ITEM#2                sku, qty

Query PK = USER#77                              -> profile + orders, one request
Query PK = USER#77 AND begins_with(SK, ORDER#)  -> orders only
The trade. Single-table design buys one-request reads for known access patterns and pays for it with rigidity. Model the queries first, then the table.

Partitions, hashing, and replication

A request enters through a stateless request router fleet. The router authenticates the call, hashes the partition key, and looks up which partition owns that hash range and where its replicas live. That mapping lives in MemDS, a distributed in-memory metadata store built on a structure the paper calls a Perkle tree, a Patricia trie merged with a Merkle tree. Routers keep a local cache of the map and forward the operation to the correct storage node.

Each partition holds roughly 10 GB and is served by a replication group of three replicas in three availability zones. The group elects a leader with Multi-Paxos. Writes go to the leader, which appends a write-ahead log record and ships it to its peers, acknowledging once two of the three have persisted the log entry. Strongly consistent reads are served only by the leader. Eventually consistent reads can hit any replica, which is why they cost half as much and can trail the leader briefly. When a replica fails, the group adds a log replica, a member storing only recent WAL entries and no B-tree, restoring the durability quorum in seconds instead of waiting for a full data copy. The replication protocol was specified and model-checked in TLA+.

The storage node keeps each partition as a B-tree indexed by the sort key, which makes DynamoDB read-optimized where Cassandra's LSM engine is write-optimized. A hard ceiling of 3,000 read units and 1,000 write units per second applies to every physical partition, so one partition can serve about 12 MB of reads and 1 MB of writes per second, and those figures anchor any capacity estimate you sketch.

The metadata path carries one of the paper's best lessons. The original per-router partition map caches hit 99.75 percent, which sounds excellent until a cold restart threatens the backing store with up to 400 times its steady-state traffic, a classic metastable failure. MemDS fixed it by absorbing the full request rate by design. Routers send an asynchronous refresh to MemDS even on a cache hit, so the metadata tier sees constant load whether caches are warm or empty, and a mass cache flush changes nothing.

The router hashes chat_id, consults the MemDS-backed partition map, and forwards the write to the owning partition's Paxos leader. Committed writes then flow asynchronously to GSI partitions, re-hashed under user_id, which is why a GSI read can only ever be eventually consistent.

Why it works. Hashing spreads keys, Paxos-replicated B-trees keep each key range durable and fast, and a metadata tier engineered for constant load keeps routing from becoming the failure mode.

Adaptive capacity and paying for throughput

Provisioned throughput was originally divided statically and equally among partitions. A table with 10,000 write units across ten partitions gave each partition a thousand, and the design assumed uniform access. Real traffic is skewed, so hot partitions throttled while the table as a whole sat under its limit, and the failure compounded, because splitting a hot partition also split its allocation and each half could end up with less headroom than the whole had before.

The fixes arrived in layers. Bursting lets a partition retain up to 300 seconds of unused capacity and spend it during spikes. Adaptive capacity then shifted allocation toward hot partitions, but reactively, after throttling had already started. The current mechanism is global admission control, a token bucket service tracking consumption at the table level. Each request router holds a local token bucket and refills it from GAC every few seconds, so any partition can immediately draw on the table's unused headroom, and per-partition ceilings survive only to protect the physical node underneath.

Partitions also split for consumption, not just for size, and the split point follows the observed key access distribution rather than the midpoint of the range. DynamoDB deliberately refuses splits that cannot help, since a single hot item or a monotonically increasing sort key concentrates heat on one edge no matter where you cut. For those workloads the fix is application-level write sharding, appending a small random suffix to the partition key and fanning reads back in.

Two billing modes sit on top of this machinery. On-demand charges per request, sixty-two and a half cents per million write request units and twelve and a half cents per million reads in us-east-1 since AWS halved on-demand prices in November 2024, and it is now the recommended default. Provisioned mode bills reserved throughput by the hour, about sixty-five cents per hour for a thousand write units and thirteen cents for a thousand read units, and wins when utilization is high and steady. Storage runs twenty-five cents per gigabyte-month in the standard class. The math sets real architectural limits. Ingesting ten million small writes per second costs about six and a half thousand dollars per hour, roughly one hundred fifty-six thousand dollars a day at provisioned rates, which is why nobody stores raw clickstream events in DynamoDB.

Capacity rule. The table-level budget is fluid thanks to GAC, but 1,000 writes and 3,000 reads per second per partition, and one write unit per rounded-up kilobyte, are the physics your design must respect.

Indexes, consistency, and DAX

A local secondary index keeps the table's partition key and swaps in a different sort key. It lives inside the same physical partitions as the base items and is updated synchronously as part of the write, which is why it supports strongly consistent reads. The constraints are real. LSIs exist only if declared at table creation, cannot be dropped, cap at five per table, and bound each item collection to 10 GB.

A global secondary index is effectively a second table DynamoDB maintains for you, with its own partition key, its own partitions, and its own capacity. Committed base-table writes propagate to GSI partitions asynchronously, re-hashed under the index key, which is why GSI reads are eventually consistent with no strong option. A table carries up to twenty GSIs, each one bills its own write for every indexed change, and an under-provisioned GSI backpressures the base table, throttling writes until the index catches up.

Consistency is chosen per read, not per table. The default eventual read costs half a unit and may be served by a follower replica. Setting ConsistentRead=true routes to the partition leader and costs a full unit. Transactions via TransactWriteItems and TransactGetItems give serializable, all-or-nothing semantics across up to 100 items spanning multiple tables at double the capacity cost. Global Tables replicate across regions with last-writer-wins conflict resolution by default, and AWS added an opt-in multi-region strong consistency mode for tables that cannot tolerate cross-region conflicts.

DAX is the built-in cache, a cluster of nodes in your VPC that speaks the DynamoDB API through a swapped client library and serves eventually consistent GetItem and Query results from memory in microseconds. It keeps separate item and query caches with a default five-minute TTL and operates write-through, so a write must land in both DynamoDB and the cache to succeed. Strongly consistent reads pass straight through uncached, and writes that bypass DAX leave stale entries in place until TTL expiry, so mixing DAX with direct table access is a known trap.

DynamoDB Streams provides change data capture, an ordered record per item modification retained for twenty-four hours, and it is the standard bridge to Lambda triggers, OpenSearch indexing, and warehouse sync. Stream records are ordered per item, and consumers should be built for at-least-once processing.

Consistency rule of thumb. Base table and LSI reads can be strong. A GSI read never can, because a GSI is an asynchronously maintained copy living under a different hash.

Trade-offs and when not to use it

The constraints follow directly from the architecture. No joins and no ad-hoc aggregations, items cap at 400 KB, transactions cap at 100 items, scans cost in proportion to table size, and every new access pattern needs an index or a remodel rather than a new WHERE clause. Sustained high-volume writes become a large and predictable line item, and the service is proprietary to AWS, so leaving means a data migration project.

Choose Postgres when query patterns are still evolving or genuinely relational. A single well-tuned Postgres node handles tens of thousands of transactions per second with joins, constraints, secondary indexes that are free to add, and ad-hoc SQL for anything you did not plan. DynamoDB earns its place when you can enumerate the access patterns up front, need horizontal write scale or active-active multi-region replication, and want zero database operations. If you find yourself layering GSIs to simulate relational flexibility, you wanted Postgres.

Cassandra shares the modeling philosophy, partition key plus clustering key, query-first design, denormalization, because it descends from the same 2007 paper by way of Bigtable's storage engine. Underneath it differs on every axis. Cassandra runs an LSM engine optimized for sustained write throughput where DynamoDB's B-tree favors reads, offers leaderless replication with per-query quorum tuning where DynamoDB fixes a leader-based scheme, and runs as self-managed clusters on any cloud or on-premises with no per-request billing. Pick Cassandra for vendor neutrality or write volumes where capacity-unit pricing is prohibitive, and DynamoDB when you would rather buy the operations, the SLA, and the ecosystem of DAX, Streams, IAM, and point-in-time recovery than staff them.

In production

The scale numbers come straight from the USENIX paper. Across the sixty-six hours of Prime Day 2021, Amazon systems including Alexa, the Amazon.com sites, and the fulfillment network drove trillions of DynamoDB API calls, peaking at 89.2 million requests per second, with single-digit millisecond latency and no availability loss. The published availability targets are 99.99 percent for single-region tables and 99.999 percent for Global Tables.

DynamoDB powers those Amazon properties alongside hundreds of thousands of external customers on the same multi-tenant fleet, which is why admission control gets so much space in the paper. One tenant's hot key cannot be allowed to degrade a neighbor, so isolation is enforced with token buckets at the router layer rather than by handing anyone dedicated hardware.

The operational lessons transfer beyond AWS. Design caches for constant backend load rather than for hit rate, since the 99.75 percent cache that threatened a 400x cold-start surge is now the canonical metastability example. Watch write amplification, because every GSI bills an extra write per indexed change and the rounded-up kilobyte makes small hot writes expensive. Watch item collection growth against the LSI 10 GB cap, and shard writes for monotonic or single-key hotspots before adaptive capacity has to rescue you, because for those shapes it cannot.

The production story. 89.2 million requests per second at single-digit milliseconds is not magic, it is hashing, Paxos-replicated B-trees, token buckets, and a metadata tier that cannot be surprised.

Follow-up questions

  • Why can a GSI never serve a strongly consistent read? A GSI is a physically separate copy of the data, partitioned by a different hash and updated asynchronously from the base partition's committed writes. No single node holds both key orders, so no node can promise it has seen every acknowledged base-table write. An LSI shares the base partition and is updated as part of the same write, so it can read from the partition leader and be strong.
  • A single partition key is getting hammered. What does DynamoDB do, and where does it give up? Burst capacity spends up to 300 seconds of banked unused throughput, global admission control lets the hot partition draw on table-level headroom, and split-for-consumption divides the partition at the observed hot spot rather than the range midpoint. It gives up when the heat is one item or a monotonically increasing sort key, because no split point separates that load. At that point you write-shard in the application, spreading the key across a set of suffixes and merging on read.
  • How does DynamoDB relate to the 2007 Dynamo paper? Name and goals only. Dynamo was a leaderless consistent-hashing ring with sloppy quorums, hinted handoff, vector clocks, and application-side conflict resolution. DynamoDB is single-leader per partition with Multi-Paxos replication, a central request-routing and metadata tier, and per-request consistency choice. Cassandra is the real architectural descendant of the 2007 paper.
  • When does provisioned capacity beat on-demand? When utilization is high and steady. On-demand costs several times more per request than a fully utilized provisioned unit but wastes nothing at idle. Since AWS halved on-demand prices in November 2024 it positions on-demand as the default, with provisioned mode plus auto scaling reserved for stable high-throughput tables where the utilization math clearly wins.
  • What does one write actually cost, mechanically? One write capacity unit per 1 KB, rounded up, so a 100-byte item still bills a full unit. Every GSI whose keys or projected attributes are touched bills its own index write out of the index's capacity, and a transactional write doubles the units. A 3 KB transactional write to a table with two affected GSIs bills six units on the base table plus index writes on both GSIs.
  • Postgres, Cassandra, or DynamoDB in one sentence each? Postgres for evolving or genuinely relational query patterns at moderate scale, with joins, constraints, and ad-hoc SQL. Cassandra for self-managed, vendor-neutral clusters absorbing enormous sustained write volume on an LSM engine. DynamoDB for enumerable key-value access patterns that need flat latency at any scale, zero database operations, and native AWS integration.
