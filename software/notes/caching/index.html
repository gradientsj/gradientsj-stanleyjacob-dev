Caching and cache invalidation

Software & architecture · architecture · Jun 2026

A cache stores the result of expensive work close to where it is needed so repeats are cheap. It is the highest-leverage performance tool in most systems, because real workloads are dominated by a small hot set.

The hard parts are not speed but correctness: when to populate the cache, when to invalidate it, and what to drop when it is full.

When it matters

  • Read-heavy workloads where the same data is requested repeatedly.
  • Expensive computations or slow downstream calls.
  • Smoothing load spikes on a database.

Patterns and policies

  1. Cache-aside (lazy): the app checks the cache and, on a miss, loads from the source and fills the cache.
  2. Write-through: writes update the cache and the store together, keeping them in sync.
  3. Write-back: writes hit the cache and flush to the store later; faster but riskier.
  4. Eviction: LRU (least recently used) is the common default; LFU and FIFO are alternatives.
  5. Invalidation: TTL expiry, or explicit deletes on write.

Implementation

from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.store = OrderedDict()

    def get(self, key):
        if key not in self.store:
            return -1
        self.store.move_to_end(key)        # mark most-recently used
        return self.store[key]

    def put(self, key, value):
        if key in self.store:
            self.store.move_to_end(key)
        self.store[key] = value
        if len(self.store) > self.cap:
            self.store.popitem(last=False)  # evict least-recently used

Complexity (time and space)

get and put are both O(1): a hash map for lookups plus a doubly linked list (OrderedDict) to move the touched key and pop the oldest. Space is O(capacity).

Worked example

c = LRUCache(2)
c.put(1, 1)
c.put(2, 2)
print(c.get(1))    # 1   (and marks key 1 as recently used)
c.put(3, 3)        # over capacity -> evict key 2 (least recent)
print(c.get(2))    # -1  (evicted)
print(c.get(3))    # 3

Follow-up questions

  • What is the hardest part of caching? Invalidation: keeping cached data consistent with its source as the source changes, usually via TTL expiry or explicit deletes on write.
  • How does LRU achieve O(1)? A hash map for lookups plus a doubly linked list to move the touched node to the front and pop the tail on eviction.
  • Cache-aside vs write-through? Cache-aside fills on read misses and can serve stale data; write-through keeps cache and store in sync on every write, at the cost of write latency.
  • How do you stop a thundering herd on a cold key? Single-flight or a short lock so only one request recomputes while others wait, plus jittered TTLs.
