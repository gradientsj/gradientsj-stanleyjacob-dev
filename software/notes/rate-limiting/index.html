Rate limiting

Software & architecture · architecture · Jun 2026

Rate limiting caps how many requests a client can make in a window, protecting a service from overload, runaway clients, and abuse. The token bucket is the most common algorithm because it allows short bursts while bounding the average rate.

A bucket refills with tokens at a fixed rate up to a capacity; each request spends a token, and requests are rejected when the bucket is empty.

When it matters

  • Public APIs that need per-client quotas.
  • Protecting a downstream dependency from being overwhelmed.
  • Smoothing bursty traffic and preventing abuse.

Common algorithms

  1. Token bucket: refill over time, allow a request if a token is available; permits bursts up to capacity.
  2. Leaky bucket: drain at a constant rate, queueing or dropping overflow; strictly smooths output.
  3. Fixed window: count per calendar window; simple but allows a 2x burst at the boundary.
  4. Sliding window: smoother and fairer, at the cost of more memory.

Implementation

import time

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate            # tokens added per second
        self.capacity = capacity
        self.tokens = capacity
        self.last = time.monotonic()

    def allow(self, cost=1):
        now = time.monotonic()
        self.tokens = min(self.capacity, self.tokens + (now - self.last) * self.rate)
        self.last = now
        if self.tokens >= cost:
            self.tokens -= cost
            return True
        return False

Complexity (time and space)

Each check is O(1) time and O(1) space per client (a token count and a timestamp). Distributed limiting keeps that state in Redis keyed by client.

Worked example

tb = TokenBucket(rate=1, capacity=2)   # 1 token/sec, holds at most 2
print(tb.allow())   # True   (2 -> 1 token)
print(tb.allow())   # True   (1 -> 0 tokens)
print(tb.allow())   # False  (empty; refills over time)

Follow-up questions

  • Token bucket vs leaky bucket? Token bucket allows bursts up to capacity; leaky bucket enforces a strictly smooth output rate.
  • How do you rate limit across many servers? Keep counters in a shared store like Redis (atomic increment with expiry) so the limit is global, not per instance.
  • What should a limited request return? HTTP 429 Too Many Requests, ideally with a Retry-After header.
  • Why does a fixed window allow a 2x burst? A client can send a full window at the end of one and another full window at the start of the next; sliding windows fix this.
