Google Zanzibar

Software & architecture · authorization · Jul 2026

Zanzibar is the authorization system that answers one question for most of Google, from Drive to Maps to Photos, and answers it the same way everywhere: can this user do this thing to this object right now? It was described in a 2019 paper, and it matters because it solves a problem every growing company eventually hits, which is that permissions start as a column in a table and end as a tangle of copy-pasted access checks scattered across dozens of services, each with its own bugs and its own idea of who can see what.

The core move is to pull authorization out of every application and into one service with one data model. Instead of each team writing its own can_user_access logic against its own tables, they all write and query a shared store of relationships. That model is called relationship-based access control, or ReBAC, and it is what lets Zanzibar express not just roles but the messy real rules like this document is shared with that group, that group is nested inside another group, and this folder's viewers can see everything inside it.

The one-line version. Zanzibar stores permissions as relationship tuples like doc:readme#viewer@user:bob, resolves a permission check by walking those relationships (through groups and folders), and uses tokens called zookies plus Spanner's global clock to guarantee a check never reflects an ACL older than the content the user is looking at. That last guarantee is the hard part and the reason the system exists.

Relation tuples, the whole data model

Everything in Zanzibar is a relation tuple, and the entire model fits in one line: object#relation@user. The object is a namespaced id like doc:roadmap, the relation is a named role on that object like viewer or owner, and the user is either a specific account like user:bob or, and this is the important part, another userset like group:eng#member. A tuple is a single fact: this user has this relation to this object.

doc:roadmap#owner@user:alice          # alice owns the roadmap doc
doc:roadmap#parent@folder:eng         # the roadmap doc lives in the eng folder
folder:eng#viewer@group:eng#member    # anyone in group eng can view the eng folder
group:eng#member@user:bob             # bob is a member of group eng

Because the user side of a tuple can itself be a userset, relationships chain. Bob is not written anywhere near the roadmap document, yet he can view it, because the tuples connect doc:roadmap to folder:eng to group:eng to user:bob. This is why the model beats a flat permissions table. Sharing with a group, nesting groups, and inheriting a folder's access all become the same operation, which is adding one tuple.

Userset rewrites, where the rules live

Raw tuples say who is directly related to what. The interesting rules, like every editor is also a viewer, or a document's viewers include the viewers of its parent folder, live in the namespace configuration as userset rewrites. A relation's effective members are computed by combining three building blocks with union, intersection, and exclusion.

  • _this is the set of users written directly in tuples for this relation.
  • computed_userset pulls in another relation on the same object, which is how every editor is a viewer is expressed.
  • tuple_to_userset is the powerful one. It follows a relation to another object and then reads a relation there, which is how a document inherits its folder's viewers works.
name: "doc"
relation { name: "owner" }
relation { name: "editor"
  userset_rewrite { union {
    child { _this {} }
    child { computed_userset { relation: "owner" } }   # owners are editors
  }}}
relation { name: "viewer"
  userset_rewrite { union {
    child { _this {} }
    child { computed_userset { relation: "editor" } }   # editors are viewers
    child { tuple_to_userset {
      tupleset { relation: "parent" }                   # follow doc -> its folder
      computed_userset { relation: "viewer" }           # ... to that folder's viewers
    }}
  }}}

A permission check is then a recursive walk over this rewrite. To answer check(doc:roadmap#viewer@user:bob), Zanzibar unions the direct viewers, the editors (and their owners), and the viewers of the parent folder, and the folder branch expands group:eng#member until it finds Bob. The Check API, and the two harder ones

The read side is small. Check answers a single yes or no. Expand returns the full effective userset tree for a relation, which powers UIs that show everyone who can see a thing. On the write side, Read and Write manage tuples, Write can carry a precondition so you can safely do compare-and-set on permissions, and Watch streams tuple changes so downstream systems, including Zanzibar's own indexes, can stay current.

# Check: may bob view the roadmap, consistent with the content he is looking at?
resp = zanzibar.check(
    object="doc:roadmap",
    relation="viewer",
    subject="user:bob",
    consistency=at_least_as_fresh(zookie),   # the zookie from his last content load
)
assert resp.allowed is True   # resolved via folder -> group -> bob

The new-enemy problem, and why zookies exist

Authorization data changes, content changes, and the two must not be reordered relative to each other, or you leak. That danger is what separates Zanzibar from a permissions table with a cache in front of it, and the paper names the failure the new-enemy problem. It has two shapes.

First shape: Alice removes Bob from a document, then adds sensitive new content she only wants the remaining people to see. If a permission check is allowed to run against a stale ACL snapshot from before the removal, Bob still passes the check and reads content that was added specifically to keep him out. Second shape: Alice removes Bob from a folder's ACL, then a document is moved into that folder, and because documents inherit their folder's access, the safety of that move depends on the removal being applied first. If the two ACL updates land out of order, a check sees the document sitting in the folder but misses Bob's removal, and he reads it. In both, the bug is not a wrong tuple, it is reading the right tuples at the wrong point in time.

The fix is a token called a zookie. When content is written or read, the application gets back an opaque zookie that encodes a timestamp. When it later asks Zanzibar to check a permission for that content, it passes the zookie, and Zanzibar evaluates the check against an ACL snapshot at least as fresh as that timestamp. Bob's removal, which happened before the content he is now trying to read, is therefore always visible to the check. This works because tuples are stored in Spanner, whose TrueTime clock gives every write a globally meaningful timestamp and lets any server take a consistent snapshot read at a chosen time. The consistency guarantee that sounds like magic is really just Spanner's external consistency, exposed through the zookie.

The zookie pins the check to an ACL snapshot at least as fresh as the content the user is viewing, so a removal that happened before that content is always seen. Most checks do not need this and run against a slightly stale but faster default snapshot.

Zanzibar does not force strong consistency on every call. The default is a recent, cached snapshot that is fast and almost always fine, and the zookie is the opt-in for the specific checks where a stale answer would leak. Paying for consistency only where it matters is what keeps the median check cheap.

Scaling checks, and the Leopard index

At Google's scale this store holds trillions of relation tuples and serves millions of checks per second, with a 95th-percentile check latency under 10 milliseconds and availability above 99.999 percent across three years of production, according to the paper. Most of that speed comes from aggressive caching of subproblems and from spreading tuples across many Spanner shards, but one case breaks the simple recursive walk, which is deeply nested or very large groups. Checking whether a user is in a group that contains groups that contain groups can fan out into a huge traversal on the hot path.

That is what the Leopard indexing system is for. Leopard consumes the Watch stream of tuple changes and maintains a flattened, denormalized index that answers group membership as fast set operations over sorted id lists, so a nested-group check becomes a union and intersection of precomputed sets rather than a live tree walk. It trades a background pipeline and some index staleness, bounded and tracked, for turning the worst-case check into a cheap one. The pattern is the same one behind secondary indexes everywhere, which is precompute the expensive traversal so the read is a lookup.

aclservers evaluate checks against the tuple store in Spanner and, for large or nested groups, the Leopard index, which the Watch stream keeps current. Databases are drawn as cylinders.

When you would use it, and when you would not

Reach for a Zanzibar-style system when authorization is relationship-shaped and shared across services. The signals are concrete. You have per-object sharing rather than just global roles, meaning access to this specific document, not just admin versus user. You have groups, nested groups, or resource hierarchies like folders and projects. Several services need to agree on the same permission answers, so a single source of truth beats each team reimplementing checks. And you care about correctness under concurrent permission and content changes, which is exactly the new-enemy guarantee. Collaboration and sharing products, think Docs, Drive, GitHub-style repo access, or any B2B app with org, team, and resource nesting, land here naturally.

Do not reach for it when the shape is simpler. If your authorization is a handful of roles inside one application, a roles-and-permissions table with normal RBAC is less to run and less to reason about. Zanzibar earns its complexity at fine granularity, cross-service scope, and scale. Below that bar it is a globally-distributed answer to a local question.

You do not have to build it. The ideas are available as open source, so you can adopt the model without Google's infrastructure. SpiceDB from Authzed and OpenFGA, a CNCF project seeded by Auth0, are the two most active Zanzibar-inspired engines, and Ory Keto is another. They give you relation tuples, userset rewrites, and a check API on infrastructure you can actually run, and they are where most teams should start rather than porting the paper by hand.

Questions and answers

The ideas that matter most, as questions with the answers given outright. Each wrong option is marked with why it is wrong, since the wrong turns are where the intuition forms.

1What is a Zanzibar relation tuple, at its most basic?
  • ✓A single fact of the form object#relation@user, where the user may itself be another userset like group:eng#member
  • ✗A row that lists every permission a user holds across the whole system. Each tuple is one relationship, not a per-user permission list. The full set of a user's access is derived by walking many tuples.
  • ✗A cryptographic token the client sends to prove its identity. That is authentication. Zanzibar handles authorization and assumes the caller is already authenticated.
  • ✗A cached copy of a SQL row used to speed up permission reads. Tuples are the source of truth for relationships, not a cache of some other table.
Why: The whole model is relationships expressed as object#relation@user tuples. Letting the user side be another userset is what makes groups, nesting, and hierarchies all reduce to adding a tuple.
2How does ReBAC differ from plain role-based access control (RBAC)?
  • ✓RBAC grants access by global role, while ReBAC grants it by relationships to specific objects, so it can express per-document sharing, nested groups, and folder inheritance
  • ✗ReBAC is just RBAC with more roles. Adding roles does not capture object-specific sharing or hierarchy. ReBAC models relationships between users and individual objects.
  • ✗ReBAC removes the need to authenticate users. Authentication is a separate concern. Both models still need to know who the user is.
  • ✗ReBAC only works for a single application, unlike RBAC. It is the reverse. A shared relationship store is precisely what lets many services agree on one answer.
Why: Roles answer "what kind of user is this." Relationships answer "how is this user connected to this exact object," which is what real sharing and hierarchy need.
3The new-enemy problem is fundamentally a bug about what?
  • ✓Ordering in time: reading correct ACL tuples at the wrong point relative to a content change, so a stale permission snapshot leaks newer content
  • ✗A malformed or incorrect tuple being written. The tuples are correct. The failure is evaluating a check against a snapshot older than the content it protects.
  • ✗A user guessing another user's credentials. That is an authentication attack, unrelated to snapshot ordering of ACLs.
  • ✗Running out of space to store trillions of tuples. It is a correctness-under-concurrency problem, not a capacity one.
Why: If a check can run against ACLs from before a revocation, a removed user still passes for content added after the removal. The bug is time ordering, not a wrong tuple.
4What does a zookie actually guarantee?
  • ✓That the check is evaluated against an ACL snapshot at least as fresh as the content the zookie was minted for
  • ✗That every check always reads the single latest global state. Default checks run on a slightly stale, faster snapshot. The zookie asks for "at least as fresh as this," not "the newest possible."
  • ✗That the user is who they claim to be. That is authentication again. A zookie is about snapshot freshness, not identity.
  • ✗That the tuple write has been replicated to every region. It encodes a timestamp for consistent snapshot reads, built on Spanner, not a replication receipt.
Why: The zookie carries a timestamp so the check reads ACLs no older than the content, which closes the new-enemy gap while still allowing fast default reads elsewhere.
5Why does Zanzibar need the Leopard index instead of always walking tuples live?
  • ✓Deeply nested or very large groups make a live traversal expensive, so Leopard precomputes flattened group membership as fast set operations
  • ✗Spanner cannot store relation tuples, so Leopard is the real database. Spanner is the tuple store. Leopard is a derived index for the group-expansion hot path.
  • ✗Leopard replaces zookies for consistency. Consistency comes from zookies and Spanner. Leopard is about check latency for nested groups.
  • ✗It encrypts the tuples at rest. Leopard is a performance index, not an encryption layer.
Why: Nested-group checks can fan out badly. Leopard consumes the Watch stream and keeps a denormalized membership index so those checks become set unions and intersections instead of deep tree walks.
6When is a Zanzibar-style system the wrong choice?
  • ✓When authorization is a few global roles inside one application, where a normal RBAC table is simpler to run and reason about
  • ✗Whenever you have more than one user. Scale of users is not the signal. Relationship shape, cross-service scope, and hierarchy are.
  • ✗Any time you need per-document sharing across services. That is exactly where it fits, not where it fails.
  • ✗When you want to use open source instead of building it. SpiceDB, OpenFGA, and Ory Keto exist precisely so you can adopt the model without building it.
Why: The model pays for itself at fine granularity, cross-service scope, and scale. For a handful of roles in one app it is a globally-distributed answer to a local question.
