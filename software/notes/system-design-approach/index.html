System design

Software & architecture · architecture · Apr 2023

System design is deciding how the pieces of a system fit together: clients, services, storage, and the paths between them. Most designs come down to a handful of recurring patterns applied to a given set of requirements and scale.

A consistent order helps: settle the requirements, do rough capacity math, sketch the API and data model, draw the high-level components, then look at where load concentrates and how to relieve it.

When it matters

  • Designing a new service or feature before building it.
  • Writing a design doc or RFC.
  • Reasoning about how an existing system will handle more load.

The framework

  1. Clarify requirements: functional (what it does) and non-functional (scale, latency, availability, consistency).
  2. Estimate: users, requests per second, storage, and bandwidth, with round numbers.
  3. Define the API: a few endpoints with their inputs and outputs.
  4. Design the data model: entities, access patterns, and SQL or NoSQL.
  5. Draw the high-level design: clients, load balancer, services, caches, databases, queues.
  6. Find the bottlenecks and scale them: caching, replication, sharding, async work.

Trade-offs

Every choice trades something: caching adds staleness, sharding adds operational complexity, strong consistency costs availability. The useful skill is naming which trade-off a design is making, and why.

Worked example

A small, classic example is a URL shortener. The redirect read path dominates, so a short code is generated from a unique ID with base62 (62 characters pack far more IDs per character than base10):

import string
ALPHABET = string.ascii_letters + string.digits   # 62 symbols

def encode(n):
    if n == 0:
        return ALPHABET[0]
    s = []
    while n:
        n, r = divmod(n, 62)
        s.append(ALPHABET[r])
    return ''.join(reversed(s))

print(encode(125))   # 'cb'   (a 7-char code covers 62**7 > 3 trillion ids)

Follow-up questions

  • How do you estimate QPS? Daily active users times actions per user per day, divided by 86400, times a peak factor of 2 to 5.
  • Why base62 instead of hashing the URL? Encoding a unique counter guarantees no collisions; hashing the URL can collide and needs extra handling.
  • Where does a cache help here? In front of the redirect lookup: reads vastly outnumber writes and hot links are read repeatedly.
  • How would you scale writes to the ID generator? Hand out ID ranges to each app server, or use a dedicated key-generation service so servers never coordinate per request.
