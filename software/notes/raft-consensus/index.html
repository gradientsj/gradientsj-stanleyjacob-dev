Consensus with Raft

Notes · distributed agreement · Mar 2025

Consensus is how a cluster of machines agrees on a single sequence of operations even though any of them can crash or fall behind. It is the foundation under replicated databases, configuration stores, and the kind of durable job queue that has to survive a node dying mid-task.

The problem

Each node runs a copy of the same state machine. To stay consistent they must apply the same commands in the same order. Consensus is the protocol that gets them to agree on that order despite crashes, delays, and lost messages. Paxos solved this first but is notoriously hard to follow; Raft was designed to be understandable while providing the same guarantees.

Leader election

Raft elects one leader that handles all client requests. Time is divided into terms; the leader sends periodic heartbeats, and if a follower stops hearing them it starts an election for a new term. Randomized timeouts make it unlikely that two nodes campaign at once, so a leader emerges quickly.

Log replication

Clients send commands to the leader, which appends them to its log and replicates them to followers. Once a majority have stored an entry, the leader marks it committed and applies it to the state machine. A majority quorum is what lets the system tolerate a minority of failures and still make progress.

Safety

The subtle part is guaranteeing nothing committed is ever lost. Raft restricts elections so only a node with an up-to-date log can win, and it is careful about when entries from older terms count as committed. Those rules are what make the replicated log provably consistent. Real systems like etcd and Consul are built on exactly this.

A quorum-replicated log is the standard way to make stateful infrastructure fault-tolerant, and Raft is the most approachable way to understand how it works.

Step by step: the two core rules

  1. A follower grants a vote only for a term at least as new as its own.
  2. It votes at most once per term.
  3. It refuses candidates whose log is less up to date than its own (the election restriction).
  4. A leader advances the commit index to the highest entry replicated on a majority, from the current term.
  5. Those two rules are what keep the replicated log consistent.
def should_grant_vote(state, cand_term, cand_id, cand_last_idx, cand_last_term):
    if cand_term < state.current_term:           # reject stale terms
        return False
    if state.voted_for not in (None, cand_id):   # already voted this term
        return False
    last_term = state.log[-1].term if state.log else 0
    last_idx = len(state.log)
    # candidate log must be at least as up to date as ours
    return (cand_last_term > last_term or
            (cand_last_term == last_term and cand_last_idx >= last_idx))

def advance_commit_index(match_index, current_term, log, num_servers):
    """Leader commits entry n once a majority have replicated it."""
    for n in range(len(log), 0, -1):
        replicated = 1 + sum(m >= n for m in match_index.values())
        if replicated > num_servers // 2 and log[n - 1].term == current_term:
            return n
    return 0
