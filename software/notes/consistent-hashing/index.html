Consistent hashing

Software & architecture · architecture · Jun 2026

When you shard data across N servers, the naive hash(key) % N remaps almost every key the moment N changes. Consistent hashing fixes that: adding or removing a node moves only about 1/N of the keys.

Keys and nodes are placed on a hash ring, and a key belongs to the next node clockwise. Virtual nodes scatter each physical node around the ring so load stays even.

When it matters

  • Sharding a cache or database across many nodes.
  • Distributed hash tables and sticky load balancers.
  • Any time nodes scale up and down and you want minimal data movement.

How it works

  1. Hash each node, plus several virtual copies, onto a ring of integers.
  2. Hash each key onto the same ring.
  3. A key is served by the first node clockwise (a binary search over the sorted hashes).
  4. Adding a node steals only the keys between it and its predecessor; removing one hands its keys to the next.

Implementation

import hashlib, bisect

class HashRing:
    def __init__(self, nodes=(), vnodes=100):
        self.vnodes = vnodes
        self.ring = {}          # ring position -> node
        self.keys = []          # sorted ring positions
        for n in nodes:
            self.add(n)

    def _hash(self, s):
        return int(hashlib.md5(s.encode()).hexdigest(), 16)

    def add(self, node):
        for i in range(self.vnodes):
            h = self._hash(f"{node}#{i}")
            self.ring[h] = node
            bisect.insort(self.keys, h)

    def get(self, key):
        if not self.ring:
            return None
        h = self._hash(key)
        i = bisect.bisect(self.keys, h) % len(self.keys)   # next node clockwise
        return self.ring[self.keys[i]]

Complexity (time and space)

Lookup is O(log V) over V = vnodes times nodes ring positions; adding a node is O(vnodes log V). Space is O(V). More virtual nodes means more even load but more memory.

Worked example

ring = HashRing(["A", "B", "C"], vnodes=100)
print(ring.get("apple"))    # 'A'
print(ring.get("banana"))   # 'B'

# Remap test: hash 1000 keys, drop node C, count how many move.
# Result: about 336 / 1000 keys move (~1/N); the other two thirds stay put.

Follow-up questions

  • Why not hash(key) % N? Changing N remaps nearly every key, causing a cache stampede or a massive data shuffle; consistent hashing moves only ~1/N.
  • What do virtual nodes solve? Few nodes create uneven arcs and skewed load; many virtual copies per node smooth the distribution.
  • Who uses it? Amazon Dynamo, Cassandra, and many distributed caches and load balancers.
  • How do you handle a hot key one node cannot serve? Replicate it to the next few nodes on the ring, or apply bounded-load constraints.
