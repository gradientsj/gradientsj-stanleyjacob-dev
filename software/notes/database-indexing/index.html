Database indexing and B-trees

Software & architecture · architecture · Feb 2024

An index is a secondary structure that lets the database find rows without scanning the whole table. It is the difference between an O(n) full scan and an O(log n) lookup.

Most relational databases use B-trees (B+ trees): wide, shallow, balanced trees tuned for reading whole pages from disk.

When it matters

  • Columns you filter, join, or sort on frequently.
  • Enforcing uniqueness (a unique index).
  • Range queries (between, greater-than) on ordered data.

Why B-trees

  1. A B-tree node holds many keys and many children, so the tree is wide and only a few levels deep for millions of rows.
  2. Each node maps to a disk page, so a lookup touches only a handful of page reads.
  3. Keys stay sorted, so equality and range queries are both fast.
  4. Writes keep the tree balanced by splitting and merging nodes, which is why every index slows inserts.

Implementation

import bisect

class Index:                       # illustrates index behavior (a real DB uses a B-tree)
    def __init__(self):
        self.keys = []
        self.rows = {}

    def insert(self, key, row):
        bisect.insort(self.keys, key)
        self.rows[key] = row

    def get(self, key):
        i = bisect.bisect_left(self.keys, key)
        if i < len(self.keys) and self.keys[i] == key:
            return self.rows[key]
        return None

    def range(self, lo, hi):
        i = bisect.bisect_left(self.keys, lo)
        j = bisect.bisect_right(self.keys, hi)
        return [self.rows[k] for k in self.keys[i:j]]

Complexity (time and space)

A B-tree gives O(log n) search, insert, and delete, plus efficient range scans. The sorted-array index above has O(log n) search but O(n) insert (shifting), which is exactly what a B-tree avoids. Every index also costs extra storage and slows writes.

Worked example

idx = Index()
for k, v in [(5,"e"), (2,"b"), (8,"h"), (1,"a"), (6,"f")]:
    idx.insert(k, v)
print(idx.get(6))        # 'f'              (point lookup)
print(idx.get(3))        # None             (missing key)
print(idx.range(2, 6))   # ['b', 'e', 'f']  (range scan)

Follow-up questions

  • Why a B-tree instead of a binary search tree? High fan-out keeps the tree shallow and aligns nodes with disk pages, minimizing slow I/O.
  • What is a covering index? An index holding every column a query needs, so the query is answered from the index alone without touching the table.
  • When does an index hurt? On write-heavy tables: every insert must update each index, and unused indexes waste space and slow writes.
  • B-tree vs LSM-tree? B-trees are read-optimized and update in place; LSM-trees (Cassandra, RocksDB) are write-optimized via append-only sorted runs and compaction.
