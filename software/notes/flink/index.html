Flink

Software &amp; architecture · stream processing · Jul 2026

Apache Flink is a distributed engine for stateful stream processing. It grew out of the Stratosphere research project at TU Berlin and became a top-level Apache project in December 2014. You hand it a directed graph of operators, it runs each operator as parallel subtasks across a cluster, and it keeps the state those operators accumulate correct through machine failures, out-of-order events, and rescaling.

The mental model that makes everything else click is the dataflow graph. Sources pull records from systems like Kafka, streams carry records between operators, operators transform records one at a time and may hold state, and sinks push results out. Every hard guarantee Flink offers is implemented as metadata flowing through that same graph alongside the data. Watermarks are messages in the stream. Checkpoint barriers are messages in the stream. Once you see that, the internals stop being magic.

Flink earns its keep when the processing is stateful. A plain Kafka consumer can reformat messages as they pass. The moment you need counts per user over the last five minutes, joins between two streams, or fraud patterns spanning several events, you need partitioned, durable, recoverable state, and that is the problem Flink actually solves.

The one-line version. Flink in one line: a dataflow engine that keeps per-key state next to the compute and threads watermarks and checkpoint barriers through the stream itself, so unbounded data yields windowed, event-time-correct results with exactly-once state guarantees.

What it is and the mental model

A Flink program describes a directed graph. The nodes are operators, transformations like map, filter, windowed aggregations, and joins. The edges are streams, unbounded sequences of records flowing between operators. The graph begins with sources that read from external systems, Kafka most commonly, and ends with sinks that write results to databases, warehouses, object stores, or other topics. You define the graph and Flink plans, places, and runs it.

When you submit a job, the client compiles your program into a JobGraph and hands it to the JobManager, the cluster coordinator that schedules tasks, tracks progress, and drives checkpoints. TaskManagers are the workers. Each one offers a number of task slots, and each operator runs as parallel subtasks spread across those slots, so an operator with parallelism 8 is eight subtask instances each owning a share of the data. JobManager high availability comes from standby instances with leader election through ZooKeeper or Kubernetes.

Execution is pipelined and record at a time. A record enters a subtask, gets processed, and is forwarded immediately, so latency through a healthy job is milliseconds. This is the deep difference from Spark Structured Streaming, which in its default mode collects records into micro-batches and schedules a small batch job per interval. Forwarding between operators is a local handoff when operators are chained into the same slot and a network shuffle when the partitioning changes. keyBy is the shuffle that matters. It hashes each record's key and routes the record to the subtask that owns that key, which is exactly what makes per-key state possible.

Everything in Flink is a message moving through a dataflow graph. Data, watermarks, and checkpoint barriers all travel the same edges, which is why the guarantees compose cleanly.

How to use it

The DataStream API is available in Java and Python, the Scala API having been deprecated in Flink 1.17 and removed in Flink 2.0. The job below is the canonical shape, clicks in from Kafka, per-user counts over 5-minute event-time windows, results out to Kafka with exactly-once delivery.

Three lines carry most of the semantics. enable_checkpointing(60_000) makes the JobManager inject a checkpoint barrier at the sources every 60 seconds, which becomes the unit of fault tolerance for the whole pipeline. The watermark strategy declares that events may arrive up to 10 seconds out of order, so the window covering 12:00 to 12:05 fires once Flink has seen event timestamps past 12:05:10. And key_by partitions the stream so that all clicks for one user land on one subtask, where that user's window state lives.

Running it means submitting to a cluster, where the JobManager turns the logical graph into physical tasks, allocates slots on TaskManagers, and starts a pipeline that runs indefinitely. Upgrades happen by taking a savepoint, a manually triggered snapshot of all operator state, stopping the job, and restarting the new version from that savepoint. The same pipeline can also be written in Flink SQL, which compiles onto the identical runtime, and large deployments such as Alibaba's now run most jobs that way.

from pyflink.common import Duration, Time, Types, WatermarkStrategy
from pyflink.common.serialization import SimpleStringSchema
from pyflink.datastream import StreamExecutionEnvironment
from pyflink.datastream.connectors import DeliveryGuarantee
from pyflink.datastream.connectors.kafka import (
    KafkaOffsetsInitializer, KafkaRecordSerializationSchema,
    KafkaSink, KafkaSource)
from pyflink.datastream.window import TumblingEventTimeWindows

env = StreamExecutionEnvironment.get_execution_environment()
env.enable_checkpointing(60_000)          # barrier every 60 s = unit of fault tolerance

source = (KafkaSource.builder()
          .set_bootstrap_servers("kafka:9092")
          .set_topics("clicks")
          .set_group_id("click-agg")
          .set_starting_offsets(KafkaOffsetsInitializer.committed_offsets())
          .set_value_only_deserializer(SimpleStringSchema())
          .build())

watermarks = (WatermarkStrategy
              .for_bounded_out_of_orderness(Duration.of_seconds(10))
              .with_timestamp_assigner(ClickTimestampAssigner()))  # event time from payload

clicks = env.from_source(source, watermarks, "clicks")

counts = (clicks
          .map(parse_click,                        # json -> (user_id, 1)
               output_type=Types.TUPLE([Types.STRING(), Types.LONG()]))
          .key_by(lambda c: c[0])                  # all events for a user -> one subtask
          .window(TumblingEventTimeWindows.of(Time.minutes(5)))
          .reduce(lambda a, b: (a[0], a[1] + b[1])))

sink = (KafkaSink.builder()
        .set_bootstrap_servers("kafka:9092")
        .set_record_serializer(
            KafkaRecordSerializationSchema.builder()
            .set_topic("click-counts")
            .set_value_serialization_schema(SimpleStringSchema())
            .build())
        .set_delivery_guarantee(DeliveryGuarantee.EXACTLY_ONCE)  # two-phase commit
        .set_transactional_id_prefix("click-agg")  # required for exactly-once
        .build())

counts.map(lambda c: f"{c[0]},{c[1]}").sink_to(sink)
env.execute("clicks-per-user-5m")

Event time, watermarks, and windows

Flink separates two clocks. Processing time is the wall clock of the machine executing an operator. It is cheap and always available, but results depend on when events happen to arrive, so reruns, recoveries, and backfills produce different answers. Event time is the timestamp embedded in the record when it was created. Windows defined on event time produce the same answer no matter how delayed, reordered, or replayed the input is, which is what you want for anything billing-adjacent or user-facing (Flink docs on timely stream processing).

The cost of event time is deciding when a window is complete. A watermark is a special record carrying a timestamp t that flows through the stream and asserts that no more events with timestamps at or below t should be expected. With the common bounded out-of-orderness strategy, each source subtask emits watermarks equal to the largest event timestamp it has seen minus the configured bound. When an operator has multiple inputs, its event-time clock is the minimum of the watermarks across all inputs, so the slowest input holds everything back. That rule keeps parallel event time consistent, and it also produces the classic operational trap. An idle Kafka partition emits no watermarks, its input stays pinned at the minimum, and every downstream window stalls. The fix is declaring idleness on the watermark strategy so silent inputs are temporarily excluded from the minimum.

Events with timestamps below the current watermark are late. By default a window that already fired drops them, but you can configure allowed lateness, which keeps window state alive for a grace period and re-fires with corrected results, or route stragglers to a side output for offline reconciliation. Window choice sets the cost profile. Tumbling windows are fixed and non-overlapping, one bucket of state per key per window. Sliding windows overlap, and a 30-minute window sliding every 5 minutes puts each event in six windows with state multiplied to match. Session windows grow until an inactivity gap closes them and fit user-behavior analysis. Mission-critical aggregation pipelines usually pair all of this with a periodic batch true-up, because some data arrives later than any bound you are willing to wait for online.

A watermark is a promise flowing through the stream that time t is over. Windows fire when the promise passes their end, and the promise for any operator is only as fresh as its slowest input.

Keyed state and the RocksDB backend

After keyBy, operator state becomes keyed state, a key-value store embedded inside the operator and scoped to the key of the record being processed. When a subtask handles a click for user 123, calls like ValueState.value() and update() touch only that user's slice. Because the shuffle guarantees all records for a key arrive at exactly one subtask, every state access is local. There is no remote database on the hot path, no locking, and no cross-node coordination, which is where Flink's throughput comes from. Flink offers value, list, map, reducing, and aggregating state types, and everything registered through the state API is included in checkpoints automatically.

Rescaling is solved with key groups. The key space is hashed into a fixed number of groups equal to the job's maximum parallelism, and each subtask owns a contiguous range of groups. Scaling from 4 to 8 subtasks reassigns group ranges rather than splitting individual keys, so state moves in coarse, well-defined units (Flink docs).

Where working state physically lives is the state backend choice. The HashMapStateBackend keeps state as live objects on the JVM heap, fastest to access but bounded by memory and exposed to garbage-collection pauses. The EmbeddedRocksDBStateBackend keeps state as serialized bytes in RocksDB, an LSM-tree store embedded in each TaskManager and writing to local disk. Every access pays serialization plus a JNI hop, which makes it a few times slower per operation than heap state, but jobs can hold many terabytes of state across a cluster, memory stays off-heap and GC-quiet, and it is the only backend supporting incremental checkpoints, which upload just the SST files created since the last snapshot instead of the full state (Flink blog). Keep the layering straight, because it is misread constantly. The state backend is the fast local working copy. Durability comes from checkpoints shipped to remote checkpoint storage such as S3 or HDFS, a separate setting, and checkpoints are immutable write-once blobs so object stores suit them perfectly.

State lives next to the compute, partitioned by key. Heap backend for small hot state, RocksDB for big state, and durable truth always lives in the checkpoint store, not the backend.

Checkpoint barriers, from Chandy-Lamport to ABS

Flink's fault tolerance descends from the Chandy-Lamport distributed snapshot algorithm from 1985, which captures a consistent global snapshot of a running message-passing system without stopping it by sending marker messages along channels. Each process records its own state when it first sees a marker, then records every message arriving on other channels until the marker shows up there too, because those in-flight messages are part of the global state. Carbone and the Flink authors specialized this in the 2015 Asynchronous Barrier Snapshotting paper. In an acyclic dataflow graph you can skip recording channel contents entirely. If each operator waits until the marker, called a barrier, has arrived on all of its inputs before snapshotting, then by construction no in-flight records are unaccounted for, and the snapshot holds operator state only. Cyclic graphs need a small log of records on back edges, and nothing ever stalls the whole computation.

In operation, the JobManager triggers checkpoint n, and each source records its position, the Kafka offsets, and injects barrier n into its output. Barriers flow with the data and never overtake records. A single-input operator snapshots when the barrier arrives. A multi-input operator performs alignment first, buffering records from inputs whose barrier already arrived until the remaining barriers catch up, so records from checkpoint n and n+1 never mix in one snapshot. The snapshot itself is asynchronous. RocksDB takes a cheap local snapshot and uploads to durable storage in the background while processing continues. Once the sinks acknowledge barrier n, the JobManager marks checkpoint n complete. Recovery is the mirror image. Restore every operator from the last complete checkpoint, rewind sources to the recorded offsets, and replay, which means Kafka retention must cover at least the checkpoint interval plus your worst recovery time.

Alignment has a known failure mode. Under backpressure, barriers move at the speed of the slowest congested channel, checkpoints start timing out, and your recovery window grows exactly when the system is least healthy. Unaligned checkpoints, added in Flink 1.11, let a barrier jump over queued buffers, with the skipped in-flight records persisted as part of the snapshot, trading larger checkpoints for barriers that cross a backpressured job in roughly constant time. Dropping alignment while discarding nothing gives Flink's at-least-once mode, which is cheaper but can double-count records that were in flight during a failure. Stateless one-to-one operators like map and filter keep exactly-once behavior either way because they have nothing to align.

Checkpoint n in flight. The source records its Kafka offsets and injects barrier n, the keyed operator aligns, snapshots its RocksDB state, and uploads SST files to S3 asynchronously, and the sink pre-commits its Kafka transaction. When the JobManager sees every acknowledgment for n, it tells the sink to commit, making the results visible to read_committed consumers.

A checkpoint is a barrier flood through the graph. Alignment removes the need to snapshot channels, asynchronous upload removes the pause, and recovery is state restore plus source rewind to the barrier's offsets.

Exactly-once sinks with two-phase commit

Checkpoints give exactly-once semantics for Flink's internal state, meaning every record affects state exactly once even across failures. They do not, by themselves, protect the outside world. Between two checkpoints a sink has already emitted results, and after recovery those records replay, so a naive sink writes duplicates. Two fixes exist. Idempotent writes, upserts keyed on a deterministic ID, let replays overwrite themselves harmlessly. Transactional writes make output invisible until the checkpoint that produced it completes.

The transactional route is a two-phase commit driven by the checkpoint protocol itself (Flink blog). The sink opens a transaction in the external system and writes into it as data flows. When barrier n arrives, the pre-commit happens, which flushes the transaction, snapshots its handle into checkpointed state, and opens a fresh transaction for records after the barrier. When the JobManager later notifies the sink that checkpoint n is complete, the sink issues the commit. The contract is strict, in that a successful pre-commit must guarantee the commit can eventually succeed, so commits are idempotent and Flink preemptively re-issues them when restoring from a checkpoint. For Kafka this maps directly onto Kafka transactions, available since broker version 0.11, and Flink's exactly-once Kafka sink sets a long producer transaction timeout, one hour by default, so open transactions survive a recovery, with the broker's transaction.max.timeout.ms raised to match.

The price is latency and coupling. Downstream consumers must read with isolation.level=read_committed, and committed data only appears at checkpoint boundaries, so end-to-end latency is at least the checkpoint interval no matter how fast the pipeline runs internally. Checkpoint every 60 seconds and exactly-once consumers see results roughly a minute stale. That is the standard reason teams either shorten checkpoint intervals or fall back to at-least-once delivery with idempotent sinks, which gives fresh output and pushes deduplication to the reader.

End-to-end exactly-once is checkpointing plus a transaction protocol. Pre-commit at the barrier, commit on checkpoint completion, and accept that output freshness is quantized to the checkpoint interval.

Trade-offs and when not to use it

What you get is event-time correctness, per-key state at terabyte scale, and exactly-once guarantees inside one framework. What you pay is operations. A Flink deployment is a cluster with its own scheduler, memory model, RocksDB tuning, checkpoint storage, savepoint lifecycle, and upgrade choreography. State is both the superpower and the standing liability, because unbounded key spaces or missing state TTLs grow checkpoints until recovery times and job restarts start to hurt.

Skip Flink when the job is stateless. A consumer service that reformats messages from one topic to another needs none of this machinery. Skip it when you do not actually need streaming latency, since a Spark batch job every 15 minutes is far easier to operate, and Spark Structured Streaming's micro-batch mode is a reasonable middle ground when hundreds of milliseconds to seconds of latency are acceptable and the team already lives in the Spark ecosystem. Flink is also not a serving layer. It computes results, and something else, Redis, Cassandra, or a warehouse, has to serve them to users.

The closest call is Kafka Streams, which covers a large share of real workloads. It is a library rather than a cluster, so your stream processor is a normal JVM service deployed like any microservice, with state in embedded RocksDB made durable through changelog topics back in Kafka and exactly-once semantics built on Kafka transactions (Confluent's comparison). Choose it when the pipeline is Kafka to Kafka, state is moderate, and you want your existing deployment story. Choose Flink when you need sources and sinks beyond Kafka, very large state, fine-grained event-time and windowing control, CEP-style pattern matching, or streaming SQL shared across many teams on a managed platform.

Flink for stateful, event-time, large-state streaming across heterogeneous systems. Kafka Streams for Kafka-to-Kafka microservices. Spark when micro-batch latency is fine and the ecosystem is already Spark. A plain consumer when there is no state at all.

Flink in production

Alibaba runs the largest known deployment and produced much of modern Flink, since its internal Blink fork was merged back upstream starting in 2019. During the Double 11 shopping festival in 2020, its Flink-based real-time compute platform peaked at four billion records per second, moving about seven terabytes of data per second, on a cluster of more than one and a half million CPU cores running over thirty-five thousand jobs (Alibaba Cloud). The flagship consumer was the live GMV dashboard tracking roughly seventy-four billion dollars of orders across the event, alongside real-time recommendation, anti-fraud, and monitoring pipelines, and the same cluster was reused at night for Flink batch jobs instead of provisioning separate capacity.

Netflix runs Flink inside Keystone, its stream processing platform, which moves trillions of events and petabytes of data per day from device and service telemetry through Kafka into Flink jobs, both a managed routing tier and custom analytics jobs deployed by product teams, feeding sinks like S3, Elasticsearch, and downstream topics (Netflix TechBlog). The design worth stealing is the platform shape, a paved road where teams submit a Flink job or SQL and the platform owns checkpointing, scaling, deployment, and recovery.

Underneath both deployments sits the same production pattern, Kafka in front as the replayable log, Flink in the middle as the stateful compute, and a serving store behind it, the pattern also used for fraud scoring, dynamic pricing, and real-time personalization across the industry. When you propose Flink in a system design, that trio plus an offline true-up path for events later than any watermark bound is the credible, production-shaped answer.

Follow-up questions

  • Walk me through what happens when a TaskManager dies mid-job. The JobManager stops receiving heartbeats and marks the TaskManager failed, cancels the job's tasks, and reschedules them onto healthy slots. Every operator restores its state from the last completed checkpoint in durable storage, sources rewind to the Kafka offsets recorded in that checkpoint, and processing replays forward. No duplicates leak through transactional sinks because output past the last completed checkpoint was never committed.
  • What does Flink's exactly-once guarantee actually cover? Checkpointing guarantees each record affects internal operator state exactly once, even across failures. It says nothing about external systems by itself. End-to-end exactly-once additionally requires either idempotent sinks or a two-phase commit sink, where writes are pre-committed at the checkpoint barrier and committed only when the JobManager confirms the checkpoint, with downstream Kafka consumers reading at read_committed.
  • How is a watermark computed across parallel Kafka partitions? Each source subtask emits watermarks independently, typically its maximum seen event timestamp minus the out-of-orderness bound. Every downstream operator sets its event-time clock to the minimum watermark across all of its inputs, so the slowest partition gates progress. An idle partition therefore freezes downstream windows unless the watermark strategy declares idleness so silent inputs are excluded.
  • Why do checkpoints time out under backpressure, and what is the fix? Aligned barriers travel with the data and cannot overtake it, so on a congested channel the barrier queues behind the backlog and alignment at multi-input operators stalls. Unaligned checkpoints let the barrier jump over buffered records and persist those in-flight buffers as part of the snapshot, making barrier transit roughly constant time at the cost of larger checkpoints.
  • When would you pick Kafka Streams or Spark over Flink? Kafka Streams when the pipeline is Kafka in and Kafka out with moderate state, because it is a library embedded in a normal service with no separate cluster. Spark Structured Streaming when latency of hundreds of milliseconds to seconds is fine, workloads are batch-heavy, and the team already runs Spark. Flink wins on very large state, strict event-time semantics, low per-record latency, non-Kafka sources and sinks, and CEP.
  • Heap state backend or RocksDB? Heap (HashMapStateBackend) for small state and the lowest access latency, since values are live JVM objects. RocksDB for state beyond memory, up to terabytes, since it stores serialized bytes on local disk off-heap, avoids GC pauses, and uniquely supports incremental checkpoints that upload only new SST files. The cost is serialization plus a JNI hop, a few times slower per access.
