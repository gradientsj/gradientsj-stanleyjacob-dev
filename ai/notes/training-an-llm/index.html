What it takes to train an LLM

Notes · training large language models · Jun 2026

Training a language model is conceptually simple, predict the next token, but the engineering around that objective is where most of the work lives. The pipeline is roughly: data to tokenizer to model to optimizer to evaluation.

Tokenization

Text is split into subword tokens, usually with byte-pair encoding, which starts from bytes and greedily merges the most frequent pairs into a fixed vocabulary. A larger vocabulary means shorter sequences but a bigger embedding table; byte-level fallback guarantees nothing is ever out of vocabulary.

The objective

The model outputs a probability distribution over the vocabulary at every position, and we minimize the cross-entropy of the true next token. Because of causal masking, every position in the sequence contributes a prediction, so one forward pass yields many training signals.

Optimization

  • AdamW with decoupled weight decay is the default optimizer.
  • Learning-rate schedule: a short linear warmup, then cosine decay to a small final value.
  • Gradient clipping by global norm tames the loss spikes that show up at scale.
  • bf16 mixed precision keeps memory down while staying numerically stable.

Scaling

Bigger models need proportionally more data. The Chinchilla result is the rule of thumb: for a compute budget, train a smaller model on more tokens, roughly 20 tokens per parameter, rather than a huge model on too little data. Fitting the model across many GPUs uses data, tensor, and pipeline parallelism, plus gradient accumulation to reach a large effective batch size.

After pretraining

A pretrained model predicts text but does not yet follow instructions. Supervised fine-tuning on instruction/response pairs teaches the format, and preference methods (RLHF or the simpler DPO) align it with what people actually prefer.

None of these pieces is exotic on its own. What makes LLM training hard is doing all of them at once, reproducibly, while watching a loss curve that takes hours or days to react.

Step by step: byte-pair encoding

  1. Start with the text as a list of individual characters.
  2. Count every adjacent pair of symbols.
  3. Merge the single most frequent pair everywhere into one new symbol.
  4. Record the merge and repeat until you hit the target vocabulary size.
  5. To tokenize new text later, apply the learned merges in order.
from collections import Counter

def get_pairs(tokens):
    return Counter(zip(tokens, tokens[1:]))      # adjacent symbol pairs

def merge(tokens, pair):
    out, i = [], 0
    while i < len(tokens):
        if i < len(tokens) - 1 and (tokens[i], tokens[i + 1]) == pair:
            out.append(tokens[i] + tokens[i + 1])   # join the merged pair
            i += 2
        else:
            out.append(tokens[i])
            i += 1
    return out

def train_bpe(text, num_merges):
    tokens, merges = list(text), []
    for _ in range(num_merges):
        pairs = get_pairs(tokens)
        if not pairs:
            break
        best = max(pairs, key=pairs.get)          # most frequent pair
        tokens = merge(tokens, best)
        merges.append(best)
    return tokens, merges

# train_bpe("low lower lowest", 3) learns ('l','o'), ('lo','w'), (' ','low')
