Benchmarking vector search, from the embedding model to the index

Project write-up · retrieval & vector search · Jul 2026

Every retrieval-augmented system rests on one operation repeated billions of times: turn a piece of text into a vector, then find the handful of stored vectors nearest to a query vector. The quality of an answer is capped by whether that search surfaced the right passage, and the cost of the product is set by how fast and how cheaply the search runs at scale, so the two numbers that matter most in a RAG system live at opposite ends of this one step and are usually measured by two different teams, if they are measured at all.

The purpose of this project is to measure both ends honestly, with one codebase, on real data. It embeds a text corpus, builds five different quantized approximate- nearest-neighbour indexes over it, and reports recall, latency, and memory for each, alongside a separate head-to-head of six embedding models scored against human relevance judgments. A second purpose shaped the design more than the first: to keep the two questions strictly apart. Retrieval quality — did the embedding model put the right document near the top — is graded against official qrels. ANN fidelity — did the approximate index reproduce what exact search would have returned — is graded against a brute-forced ground truth the project computes itself. They are different questions, they fail for different reasons, and a benchmark that blends them into one number is worse than no benchmark at all.

The same code runs in three tiers selected purely by a config file: a CPU tier over a small synthetic corpus that finishes in seventy-five seconds so the whole pipeline can run anywhere, a GPU tier on my desktop's RTX 4090 that embeds a million-plus real MS MARCO passages through vLLM, and a cloud tier designed for a Lambda instance and hundreds of millions to a billion chunks. The reusable half is a library; the benchmark is an application on top of it, so the next retrieval project reuses the platform unchanged. The code is at github.com/gradientsj/rag-platform. This page walks through the decisions first and the measured results after, because the decisions are what make the numbers mean something.

The pipeline at a glance

Six stages compose, each a command that takes only the config and the output of the stage before it, so any one can be re-run in isolation and the whole thing resumes after a crash rather than starting over.

The manifest is the spine. It records every shard, its content hash, and whether its vectors have been written, so a preempted run re-reads it and skips the work already done. Ray gives fault tolerance inside a run; the manifest gives it across runs, and at the scale this is built for, both matter.

Step 1: three tiers, one config, no forked code

The temptation in a project like this is to write a quick local script and a separate "real" pipeline for the cloud, and then to watch them drift until only one of them works. The whole thing is instead a single typed configuration — corpus sources, chunking, embedder, index specifications, sweep grids — with three profiles that differ only in their values. The CPU profile points at a deterministic synthetic corpus and a 384-dimension MiniLM model; the GPU profile points at MS MARCO and a 1024-dimension Qwen3 embedder served by vLLM; the cloud profile stacks Wikipedia and web text and writes its artifacts to S3. Nothing in the code hardcodes a path, a model, a dimension, or a GPU count, which is what lets the seventy-five-second CPU run and the multi-hour GPU run exercise the same lines.

A small piece of this earns its place quietly: before the GPU tier loads a model, it reads the card's free memory and fails with a clear message and a smaller-model suggestion if the embedder will not fit, rather than dying halfway through a run with an out-of-memory stack trace. The check is torch-free on purpose, so the config layer stays cheap to import and testable on a machine with no GPU at all.

Step 2: streaming ingestion and a manifest you can resume from

Corpora at this scale do not fit in memory, and the disk is not much better once billion-scale vectors enter the picture, so ingestion streams: each source yields documents one at a time, a chunker cuts them into overlapping token windows, and fixed- row parquet shards are written as they fill. Every source — a deterministic synthetic generator, MS MARCO through ir_datasets, Wikipedia and FineWeb streamed from the Hugging Face hub — sits behind one interface, so the profiles compose them without the ingestion code knowing which is which. Document ids are made globally unique as {source}:{native_id}:{chunk}, and the per-source counts and licenses are recorded in the manifest, because at a billion chunks "which of these am I even allowed to ship" is a real question with a real answer.

The resumability is unit-tested by killing a run mid-way and asserting that the next run embeds exactly the shards the first one missed. That test caught more than one ordering bug, and it is the difference between a preemption on a rented GPU costing a few minutes and costing the whole run.

Step 3: the embedding model is the ceiling, so measure it first

No index can retrieve a document the embedding model failed to place near the query; the model sets the ceiling and the index decides how cheaply you approach it. So before comparing indexes, six embedding models were run head-to-head on two small benchmarks from BEIR — SciFact and NFCorpus — small enough that every document is embedded and the retrieval-quality numbers are exact, with no sampling. Each model was given its own documented retrieval prompt (BGE's "Represent this sentence…", E5's "query:"/"passage:", Qwen3's instruction prefix), because scoring them with a shared prompt would quietly handicap the ones that expect their own.

Three findings in those tables were worth the whole exercise. The first is that bigger is not better: Qwen3-Embedding at 0.6 billion parameters and 1024 dimensions, the largest and by far the slowest model in the sweep at 114 documents per second, lands in the middle of the pack on both datasets and is beaten on SciFact by gte-small, a 384-dimension model that runs fifteen times faster. The second is that the sweet spot is small: bge-base, a 110-million-parameter model, wins both benchmarks while embedding at a rate that makes a billion-chunk corpus tractable, and the 384-dimension models trail it by only two to four points at a quarter of the vector storage. The third is a check on the harness itself — these scores land within a point or two of the published BEIR numbers for the same models, which is the evidence that the measurement is trustworthy before any of it is used to make a decision. A pooled "big models are better" intuition would have picked Qwen3 and paid for it in both quality and latency; the per-model rows are what turned that into "measure on your own data, and the 768-dimension model is the buy."

One caveat belongs next to that table: these are two small English benchmarks, chosen because they are fully embeddable and carry trustworthy judgments, not because they settle the question for every domain. The ranking is a starting point and a reusable harness, not a universal verdict.

Step 4: exact ground truth, and why recall needs two definitions

An approximate index is only worth measuring against the truth, so the project computes that truth itself: for a held-out set of queries it does an exact brute-force top-100 search over the full corpus — sharded across the GPU with faiss, each shard's partial result merged on the driver with a heap — and persists it as qrels. Every recall number for an ANN index is scored against this, never against another approximate index, which would only measure how two approximations agree.

This is where the two eval tracks earn their separation, because the word "recall" means two different things depending on which track you are in. In the retrieval-quality track, recall@100 is the fraction of the documents a human judged relevant that appear in the top hundred — a statement about the model. In the ANN-fidelity track, recall@10 is the fraction of the exact top ten that the approximate index also returned — a statement about the index. Same word, different denominator, different question. The run database keeps them in separate tables and the report keeps them in separate sections, and that discipline is the difference between a benchmark you can reason about and a pile of numbers.

Step 5: five FAISS indexes and what each one trades away

Every index type is a different point on the recall-speed-memory triangle, and you only ever get to pick two. The project builds five, from exact to aggressively compressed, so the trade-off is measured rather than assumed.

  • Flat stores every vector uncompressed and scans all of them per query. It is exact — it is the ground truth — but at a million vectors of 1024 dimensions that scan is a quarter of a second per query. It is the honest baseline the others are measured against.
  • SQ8 is still an exhaustive scan, but each float is compressed from four bytes to one by mapping every dimension onto 256 levels. Four times smaller, a fraction of a point of recall lost.
  • HNSW builds a navigable small-world graph: each vector links to its nearest neighbours, sparse upper layers act as highways, and a query greedily hops toward its target while keeping a candidate frontier whose width, efSearch, trades recall for speed. It is the fastest index here by three orders of magnitude, at the cost of storing the full vectors plus the graph.
  • IVF-PQ partitions the space into cells with k-means, scans only the nprobe nearest cells, and inside them replaces each vector with a product-quantized code — the 1024-dimension vector split into 64 sub-vectors, each stored as a single byte, for a sixty-four-fold compression. It fit a million vectors into a hundred megabytes, and it is where a real bug lived (below).
  • RaBitQ is a newer one-bit-per-dimension quantizer with a provable error bound; it produced the smallest index of all, at a recall the write-up reports rather than hides.

Every build path trains its coarse quantizer and codebooks on the GPU when a GPU is present, moves the finished index to the CPU for serving — the realistic deployment for these quantization schemes at this scale — and writes a build record: parameters, wall time, on-disk size, in-memory size, peak memory. The report reads those records so the memory column in the results is measured, not estimated.

The index benchmark, over 1.2 million real vectors

On the GPU tier, 1.2 million MS MARCO passages were embedded with Qwen3 into 1024- dimension vectors, and all five index types were built and searched against the exact ground truth. Recall is the best over the search-parameter sweep; latency is single-query, closed-loop:

That table is the entire argument for approximate search in one row. Flat is exact and useless in production: a quarter of a second per query is four queries a second, and no serving budget survives it. HNSW returns the same neighbours 99.1% of the time in 0.23 milliseconds — a thousand times faster than exact search, at four thousand queries a second — which is why it is the default answer for latency-bound retrieval when memory is available. IVF-SQ8 gives up a point of recall to fit in a quarter of the memory, and IVF-PQ gives up a great deal more to fit the whole index into a hundred megabytes, which is the trade you take at a billion vectors when a GPU-resident index is the only affordable option. There is no best index in that table, only a curve, and the point of the project is to draw the curve on your data rather than repeat someone else's.

How to read these numbers

Every metric here scores a ranked list against a set of relevance judgments and then averages over queries; they differ only in what they reward. Take one query whose relevant documents are D3 (highly relevant, grade 3) and D7 (relevant, grade 1), with a system that returned [D5, D3, D9, D7, D1].

Recall@k asks whether the relevant documents were found at all: the fraction of the relevant set that appears in the top k, ignoring order. Here recall@5 is 2/2 = 1.0 — both relevant documents made the top five. It is a coverage measure, which is why recall@100 is the generous "did the model surface the answer anywhere near the top" number in the model sweep.

MRR@k, mean reciprocal rank, asks how high the first relevant document sits: one over its rank, or zero if none is in the top k. Here the first relevant document is D3 at rank two, so the reciprocal rank is 1/2 = 0.5. It only looks at the first hit and ignores everything after, which makes it the right metric when there is essentially one correct answer and it needs to be at the top.

nDCG@k, normalized discounted cumulative gain, is the complete one, and the one worth being able to derive on a whiteboard. It uses graded relevance and rewards putting more-relevant documents higher. Walk down the ranking and sum each document's grade discounted by its position, grade / log2(rank + 1), so a great document at rank one is worth more than at rank ten. For this list the discounted gain is 3/log2(3) + 1/log2(5) = 1.893 + 0.431 = 2.324. Then compute the same sum for the ideal ranking, D3 then D7, which is 3/log2(2) + 1/log2(3) = 3.0 + 0.631 = 3.631, and divide: nDCG@5 = 2.324 / 3.631 = 0.640. Normalizing by the ideal puts every query on a [0, 1] scale regardless of how many relevant documents it has, so the average across queries is meaningful. nDCG rewards finding relevant documents, ranking the most relevant ones first, and putting them near the top, all at once, which is why it is the number the BEIR and MTEB leaderboards report and the number the model sweep is ranked on.

The bugs only real hardware and real data will show you

The unit tests were green throughout and the type checker was clean, and none of that prevented six genuine bugs that appeared only when the code met a GPU and a real corpus. Cataloguing them is the point, because the fixes are the part of the project a reader learns the most from.

  • Ray workers that could not import Ray. Under uv, Ray 2.56 detected the launcher and started its workers with a copy of the project that had no dependencies installed, so the pipeline hung with the workers unable to import ray. The fix was to disable that auto-detection and let workers use the installed environment directly.
  • A CUDA library load-order conflict. Importing faiss before torch loaded faiss's older bundled cuBLAS, which was missing a symbol; importing torch first, so its newer CUDA libraries win, resolved it. A one-line helper now guarantees the order.
  • IVF-PQ recall that fell as you searched harder. Product quantization encodes the residual from a cell's centroid, which is only metric-correct under L2; built under inner product, its recall dropped as nprobe rose — exactly backwards. Because the embeddings are unit-normalized, nearest by inner product equals nearest by L2, so building IVF-PQ under L2 restored monotonic recall.
  • A passage vLLM refused to embed. A real MS MARCO passage tokenized to 513 tokens against a 512 limit, and the engine rejected it rather than truncating; passing a truncation flag fixed it. Real corpora contain the inputs your tests never generate.
  • A single-GPU deadlock in the distributed layer. Ray's actor pool restarted the vLLM worker after its engine stalled, but the dead engine never released its twenty gigabytes of VRAM, so the restart waited forever. On one GPU the distributed machinery was pure overhead and this failure mode; the fix routes the single-actor case through a plain sequential loop and keeps Ray for the multi-GPU case it is actually for.
  • A latent bug an adversarial review caught before any run. A panel of review agents reading the code flagged that the GPU ground-truth path under an L2 metric negated its distances and would have selected the farthest neighbours; it was unreachable by the inner-product profiles but real, and it is fixed.

Limitations

The billion-scale tier is designed, documented, and costed against Lambda pricing, but it has not been run; the largest run here is 1.2 million vectors on a single desktop GPU. The model sweep is two small English benchmarks, chosen for exact evaluability, not a broad multilingual or long-document study. The retrieval-quality MRR on MS MARCO itself is not reported: computing it requires embedding the full corpus through vLLM, and a stability issue in that engine under sustained embedding — the same one behind the deadlock above — made the reliable path the sentence-transformers backend, which is what the model sweep runs on. Every GPU code path is exercised on real data, but the full-corpus single-engine embed is gated on that mitigation. And everything is one run per configuration; the numbers come with their configs in the repository, so anyone can rebuild the identical benchmark and shake them.
