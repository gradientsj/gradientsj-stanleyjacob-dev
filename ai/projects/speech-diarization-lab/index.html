Transcribing conversations and labeling who said what

Project write-up · speech recognition & serving · Dec 2025

A transcript that does not say who is speaking is close to unusable for meetings, interviews, and calls, because the thing people actually want is who said what, with timestamps. This project builds that end to end: Whisper, run through CTranslate2, produces the words; a diarization pipeline assembled from open parts decides the speakers; and a small, fully tested alignment joins the two. As with my other projects, the part I cared most about is knowing whether the thing actually works, so the scoring metrics are implemented from scratch against hand-computed values, and every number on this page comes from a reproducible benchmark with its exact configuration stated. The code is at github.com/gradientsj/speech-diarization-lab.

The pipeline at a glance

Two paths leave the audio file and meet at the join: the ASR path produces words with timestamps, the diarization path produces speaker turns, and the alignment assigns each word to the turn it overlaps most.

Step 1: transcription, and why CTranslate2

The words come from Whisper running on faster-whisper, which executes the same weights on the CTranslate2 inference engine instead of the reference PyTorch code. The practical difference is serving: quantized int8 inference makes CPU transcription usable and GPU transcription very fast, and the benchmark below measures that as a real-time factor, processing seconds per second of audio, where 0.1 means ten times faster than real time. Word-level timestamps are switched on because the alignment step downstream consumes words, not whole segments.

Step 2: diarization built from parts

Most products call a pretrained end-to-end diarization pipeline and treat it as a black box. I built mine from parts instead: silero VAD finds the speech, 1.5 second windows of it are embedded by a pretrained ECAPA-TDNN speaker model, the windows are clustered by cosine distance with average linkage, and consecutive same-speaker windows merge into turns, with overlapping disagreements cut at the midpoint. Building it this way keeps every join visible, and the joins are where these systems quietly fail, so each stage boundary is a pure function with unit tests rather than a line buried inside a framework.

The pretrained pyannote 3.1 pipeline is wired in as a second backend behind the same interface, because a thing built from parts only means something when it is judged next to a strong reference under identical metrics. That is the same shape as my eval-harness and robot imitation projects, and it is deliberate.

Step 3: joining words to speakers

The ASR and the diarizer run independently and disagree at the edges, so the join is where output quality is decided. The rules are simple and fully tested: a word goes to the turn it overlaps most, ties break to the earlier turn so the output is deterministic, and a word that overlaps no turn is given to the nearest one only if it is within a second, otherwise it stays unattributed, because in a transcript a visible gap is more useful than a confident mistake.

Step 4: scoring you can audit

Two metrics decide everything, so both are implemented from scratch in plain Python and tested against hand-computed values. WER, word error rate, counts substitutions, deletions, and insertions against the reference transcript. DER, diarization error rate, integrates missed speech, false alarm speech, and speaker confusion over the reference timeline, with a no-score collar around turn boundaries and the speaker mapping chosen by the Hungarian algorithm, the same semantics as the standard NIST scorer. Where a metric is undefined it returns NaN, and NaN is treated as failure, never as a pass.

The benchmark is built rather than downloaded: single-speaker LibriSpeech utterances are interleaved into 12 seeded conversations of two to four speakers, 10.9 minutes of audio in total, so the turn boundaries and transcripts are exact by construction and anyone can rebuild the identical benchmark with no credentials. The trade-off is stated plainly: there is no overlapped speech, no channel mismatch, and the acoustics are read speech, so these scores are an upper bound on conversational performance, useful for comparing systems under identical conditions rather than for quoting as real-world accuracy.

Two findings in that table were worth the whole exercise. The first is that small beats large-v3 here, 2.84% against 4.23% at float16, at a quarter of the runtime cost; on clean read speech the largest model buys nothing, which is why serving decisions should come from a measured table rather than from model reputation. The second is hiding in the last row: int8 quantization is accuracy-neutral at every other size, but small at int8 on CPU matches float16 on 11 of the 12 mixtures and then silently drops whole utterances on the twelfth, deleting 59 of its 157 reference words and inflating that mixture's WER to 40.1% against 3.8% at float16. A pooled average would have read as "int8 is a bit worse"; the per-mixture rows are what turned it into "int8 can silently skip speech", which is the kind of tail failure a serving system needs to know exists.

The diarization number is the same in every ASR configuration and, more interestingly, identical between the Windows CPU and the Linux A10 runs per mixture to three decimals, which says the pipeline is fully decoupled from the ASR and deterministic across platforms. The oracle row is the diagnosis: told the true number of speakers, the clusterer attributes nothing to the wrong person, so the entire 1.28% confusion component is speaker counting (a few mixtures came back 3 found where 2 exist, or 5 where 4), and the remaining 4.36% floor is missed speech at VAD boundaries. Each error component points at exactly one stage to improve, which is what a decomposition is for.

Limitations

This is 12 synthetic conversations and 10.9 minutes of clean read speech, with no overlapped talking, one run per configuration, and the clustering threshold left at its default rather than calibrated. The pretrained pyannote reference is wired in but its score on these mixtures has not landed yet, and that comparison is the one the project exists to make, so it gets measured on the same benchmark with the same DER implementation or not quoted at all. What the project does demonstrate end to end is a working speaker-attributed transcription pipeline, scoring math that can be audited line by line, and a benchmark anyone can rebuild exactly.
