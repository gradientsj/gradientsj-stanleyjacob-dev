nginx

nginx is the web server, reverse proxy, and load balancer that Igor Sysoev wrote to handle enormous connection counts on early-2000s hardware and that now fronts a huge share of the internet's traffic; the canonical source lives at nginx.org with an official mirror at github.com/nginx/nginx. This walkthrough covers how to use it in five minutes, the three ideas that make a codebase from 2004 still the tool of choice, and a guided path for reading the C source without getting lost.

Using it in five minutes

Every package manager carries it, so install is one line and the three commands you will actually use daily are check, reload, and stop: 

sudo apt install nginx     # Debian/Ubuntu
brew install nginx         # macOS
nginx -t          # parse and validate the config, without touching the running server
nginx -s reload   # apply new config with zero dropped connections
nginx -s quit     # graceful shutdown

The canonical minimal use is a reverse proxy in front of an application server. Drop this into a file in conf.d/ or sites-enabled/, depending on how your distro lays things out, then nginx -t and nginx -s reload: 

server {
    listen 80;
    server_name example.com;

    location /static/ {
        root /var/www/myapp;
    }

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

Static files are served directly, everything else is forwarded to the app listening on port 8000, and that one block is the shape of most nginx deployments in the wild, with TLS termination, caching, compression, and rate limiting layered on as extra directives rather than extra software.

Idea one: the event-driven master/worker architecture

nginx was designed against the C10K problem, the observation that the thread-per-connection and process-per-connection servers of its era collapsed under ten thousand concurrent connections because each connection carried a stack, a scheduler entry, and context-switch overhead. nginx's answer is a small fixed set of processes: one master that reads the configuration, binds the listening sockets, and supervises, plus a handful of workers, typically one per CPU core, that do all the actual work. Each worker is a single-threaded event loop that multiplexes tens of thousands of connections over non-blocking sockets, asking the kernel via epoll on Linux or kqueue on BSD which connections are ready and running a short callback for each, so concurrency scales with memory per connection rather than with threads. The same process model is what makes operations graceful: on nginx -s reload the master parses the new config, starts fresh workers with it, and tells the old workers to finish their in-flight requests and exit, so configuration changes and even binary upgrades happen without dropping a connection. The discipline this imposes on the code is the part worth internalizing: nothing in a worker may block, so every module is written as a state machine that can be suspended whenever it would wait.

Idea two: request processing as a pipeline of phases

Inside a worker, an HTTP request does not run through one big function; it advances through a fixed pipeline of eleven phases, from post-read after the headers are parsed, through the rewrite phases, location lookup, pre-access and access checks, a pre-content phase for things like try_files, the content phase that produces the response, and finally the log phase. Modules never call each other; they register handlers at phases, and the phase engine calls them in order, with each handler able to pass control onward, finalize the request, or park it and resume later when an event arrives. Response bodies then flow out through a separate chain of filters, which is how gzip, chunking, byte ranges, and SSI compose without knowing about one another. This is why nginx modules feel like configuration rather than plugins: access control runs in the access phase whether it is IP-based, password-based, or a subrequest to an auth service, and rate limiting is just a pre-access phase handler, the same leaky-bucket idea I work through in my rate limiter design write-up.

Idea three: the config is a declarative language

The nginx config file looks like settings but behaves like a small declarative language. Directives live in nested contexts, main, http, server, location, and values set at an outer level are inherited by inner ones unless overridden, so a config reads as a tree of increasingly specific answers to "what should happen to a request that looks like this." Crucially there is no general control flow: you declare which server names and location patterns exist, and nginx selects among them with fixed, documented matching rules at request time. The implementation mirrors the surface: every module declares the directives it understands along with which contexts they may appear in, the core config parser reads tokens and dispatches to the owning module, and the whole file is compiled at startup into per-module configuration structs that request processing then consults without ever re-parsing anything. That is why nginx -t can prove a config valid before it goes live, and why the sharp edges people hit, such as if inside location behaving strangely, are really places where an imperative-looking directive leaks the declarative machinery underneath.

Reading the repository

The source is compact C under a single src/ tree, and the layout maps cleanly onto the ideas above: src/core/ holds the primitive types and infrastructure, strings, memory pools, the red-black tree, and the cycle object that represents one lifetime of a loaded configuration; src/event/ is the event loop with the platform pollers under src/event/modules/; src/http/ is the HTTP machinery with the optional modules in src/http/modules/; and src/os/unix/ is the platform layer. Start at src/core/nginx.c, where main() lives, and follow it into src/os/unix/ngx_process_cycle.c to read the master process loop and the worker loop, which is the master/worker story in perhaps a thousand lines. Then read the event core in src/event/ngx_event.c alongside the epoll module to see what "process events and timers" concretely means. For HTTP, src/http/ngx_http_request.c takes a connection from accepted to parsed request, and src/http/ngx_http_core_module.c contains the phase engine that walks the eleven phases; with those two files understood, any module in src/http/modules/ becomes readable as "handlers hooked into known places," and ngx_http_upstream.c is the biggest and most rewarding of them, the shared machinery behind proxying. The official development guide on nginx.org is unusually good and worth keeping open beside the code.

Key takeaway: nginx endures because three ideas reinforce each other: a few single-threaded workers that never block, a fixed phase pipeline that lets independent modules compose, and a declarative config compiled once at startup. Read the process cycle, the event loop, and the phase engine, and the rest of the codebase turns from folklore into pattern.
