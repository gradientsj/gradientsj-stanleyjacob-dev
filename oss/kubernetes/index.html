Kubernetes

Kubernetes is the container orchestrator that came out of Google's decade of running Borg, was open sourced in 2014, and became the founding project of the Cloud Native Computing Foundation and the de facto operating system of the cloud. This walkthrough covers how to get a cluster running in five minutes, the small set of ideas that the entire enormous codebase repeats, and an honest reading path that starts at the readable cores instead of the front door.

Using it in five minutes

You do not install the repository; you run a distribution of it. The fastest local option is kind, which boots a conformant cluster inside a Docker container, with minikube as the equally standard alternative: 

go install sigs.k8s.io/kind@latest   # or: brew install kind
kind create cluster
kubectl create deployment web --image=nginx --replicas=3
kubectl get pods

The five-minute experiment that teaches the most is to fight the system and lose. Delete a pod and watch the deployment's replacement appear within seconds; scale the deployment and watch the pod count follow: 

kubectl delete pod <one-of-the-pod-names>
kubectl get pods          # three pods again, one very young
kubectl scale deployment web --replicas=5

Nothing you did was a command to start or stop containers; every command edited a record of desired state, and something else made reality match. That "something else" is the entire subject of this walkthrough.

The one design pattern: the reconcile loop

Kubernetes has essentially one design pattern, applied at every scale. A controller watches two things, the desired state written in an object's spec and the observed state of the world, and repeatedly acts to converge one toward the other, recording what it sees in the object's status. The deployment controller reconciles deployments into replica sets, the replica set controller reconciles those into pods, the scheduler is a controller that reconciles "pods with no node" into "pods bound to a node," and the kubelet on each machine is a controller that reconciles "pods bound to my node" into running containers. The loops are level-triggered rather than edge-triggered: each iteration compares full desired state against full observed state, so a controller that crashes, restarts, or misses an event simply converges on the next pass. That is the property that makes the system self-healing without any component being responsible for healing, and it is why deleting a pod above was so uneventful: no alarm fired, the replica set controller just noticed two is less than three. Once you see this pattern you see it everywhere, and the codebase stops being ten systems and becomes one idea with many instances.

The API server and etcd as the source of truth

The pattern only works because there is exactly one place state lives. Every object, pods, deployments, nodes, secrets, is a record persisted in etcd, the Raft-based consistent key-value store, and the API server is the sole component allowed to touch it. No Kubernetes component talks to another directly; controllers, the scheduler, the kubelet, and kubectl all read and write objects through the API server's REST interface, and coordination happens entirely through this shared state. This hub-and-spoke design buys enormous simplifications: there is one place to enforce authentication, authorization, admission policy, and validation; components can be restarted or upgraded independently because none of them holds state that matters; and extending the system means registering a new object type, which is what custom resources are, rather than teaching processes new protocols. Optimistic concurrency via resource versions replaces distributed locking, and the API server's watch mechanism streams changes to anyone interested, which turns the "poll the world" picture of the reconcile loop into something that reacts in milliseconds. If you have read my systems design write-ups, this is the rare production system that is genuinely architected the way the whiteboard version says.

Informers: making a thousand watchers cheap

A naive controller polling the API server would melt it, and a naive watcher still pays a network round trip per read. The client-go library's informer machinery is the load-bearing optimization: an informer opens one watch per resource type, maintains a local in-memory cache that the watch stream keeps current, and hands the controller events plus a read-only lens over the cache, so a controller's reads cost nothing and the API server serves one stream instead of a thousand queries. Shared informer factories let every controller in a process share the same caches, and the standard shape of a controller, informer events feeding keys into a rate-limited work queue drained by reconcile workers that read from the cache, is repeated so uniformly across the ecosystem that reading one controller teaches you all of them. The design also explains a famous Kubernetes behavior: because caches are eventually consistent, controllers are written to tolerate stale reads, which is the level-triggered philosophy again, pushed down into the client library.

Reading the repository

Honesty first: kubernetes/kubernetes is millions of lines of Go, and trying to read it front to back, or starting from cmd/kube-apiserver/, is how most attempts die. The repo's shape is worth knowing, cmd/ for the binaries' entry points, pkg/ for their implementations, api/ for API specifications, and staging/src/k8s.io/ for the code published as separate libraries, including client-go, but the right strategy is to read the small readable cores that the ideas above live in. Start outside the main repo entirely with kubernetes/sample-controller, the official minimal controller, a few hundred lines that wire an informer to a work queue to a reconcile function with nothing else in the way. Then read one real controller, pkg/controller/replicaset/ being the classic choice because its job fits in one sentence, and compare it with its siblings under pkg/controller/ to watch the same skeleton repeat. For the scheduler, the readable core is the plugins under pkg/scheduler/framework/plugins/, where what were historically called the fit predicates live: noderesources, nodeaffinity, tainttoleration, and interpodaffinity are each a self-contained answer to "can this pod go on this node, and how good a fit is it." The informer machinery itself is in client-go under staging/src/k8s.io/client-go/tools/cache/, worth reading once to demystify the reflector, the store, and the event distribution. The kubelet under pkg/kubelet/ is the hardest of these, but grepping for its sync loop shows the reconcile pattern one more time, now against a container runtime instead of etcd.

Key takeaway: Kubernetes is one pattern and one place: level-triggered reconcile loops converging the world toward desired state, and a single API server over etcd where that state lives. The repository is enormous, but it is enormous the way a crystal is large, the same structure repeated, so read one controller, one scheduler plugin, and the informer machinery, and you have read the whole design.
