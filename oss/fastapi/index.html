FastAPI

FastAPI is the Python web framework created by Sebastián Ramírez that turned "declare your types and get everything else for free" into the default way to build APIs in Python, and it did so as a remarkably thin layer over two other libraries, Starlette for the web machinery and Pydantic for the data machinery. This walkthrough covers how to use it in five minutes, the three ideas that carry the framework, and a guided path for reading a codebase that is much smaller than its popularity suggests.

Using it in five minutes

The recommended install pulls in the standard extras, which include Uvicorn as the server and the fastapi command-line tool: 

pip install "fastapi[standard]"

A complete application is a file with typed functions. The types in the signature are not decoration; they are the API contract: 

from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.get("/items/{item_id}")
def read_item(item_id: int, q: str | None = None):
    return {"item_id": item_id, "q": q}

@app.post("/items/")
def create_item(item: Item):
    return item
fastapi dev main.py

That starts Uvicorn with auto-reload, and the payoff is immediate: open http://127.0.0.1:8000/docs and there is a full interactive Swagger UI describing every route, parameter, and body schema, generated from nothing but the code above. Send /items/abc and you get a structured 422 explaining that item_id is not a valid integer. For production the same app runs under fastapi run or directly with uvicorn main:app.

Idea one: type hints as the single source of truth

Most frameworks make you say things three times: once in the handler, once in a validation schema, and once in the API documentation, and the three drift apart the moment a deadline appears. FastAPI's founding observation is that a Python type hint already contains everything those three artifacts need. Declare each parameter once, with a standard type annotation, and the framework derives validation, serialization, and documentation from that single declaration. A path parameter annotated int is coerced and range-checked by Pydantic; a parameter whose type is a BaseModel is read from the JSON body and validated recursively; the same metadata is walked a second time to emit a JSON Schema, which is what the OpenAPI document and the two bundled doc UIs render. Errors come back as structured JSON pointing at the exact offending field. The deeper consequence is social rather than technical: the docs cannot lie, because they are computed from the code that actually runs, and your editor's autocomplete and type checker are working with the same information the server enforces.

Idea two: a thin layer over Starlette and Pydantic

FastAPI's README says it stands on the shoulders of giants, and reading the source shows how literally that is meant. The FastAPI application class is a subclass of Starlette's application class, so routing, middleware, background tasks, WebSockets, the request and response objects, and the test client are all Starlette, an async-first toolkit speaking ASGI, the standard interface between Python web apps and servers. Being ASGI end to end is what lets handlers be either async def or plain def, with sync handlers transparently pushed to a threadpool so they never block the event loop. Everything about data, meanwhile, is delegated to Pydantic, which does its validation in Rust-backed core code in v2. FastAPI itself is mostly the glue that inspects your function signatures and wires these two libraries together, which is why the whole framework fits in one modestly sized Python package and why understanding it transfers directly to understanding the layers beneath it.

Idea three: dependency injection as function parameters

The third idea is the dependency system, and it reuses the same trick as the first: the function signature is the declaration. A parameter annotated with Depends(get_db) tells FastAPI to call get_db first and pass its result in, and because dependencies are themselves plain functions with their own typed parameters, they can declare query parameters, headers, or further dependencies of their own, so the framework resolves a whole tree of them per request, caching each dependency so shared ones run once. Dependencies written as generators with yield get teardown for free, which is how database sessions are opened and reliably closed, and the security utilities, OAuth2 flows, API keys, HTTP auth, are just dependencies that additionally register themselves in the OpenAPI schema so the docs UI grows an Authorize button. There is no container, no registration step, and no framework-specific interface to implement, which keeps everything trivially testable by calling functions with fake arguments, a property I care about a lot in my own software projects.

Reading the repository

The runnable framework lives in the single fastapi/ package at the repo root, and most files are short enough to read in one sitting; the bulk of the repository is actually tests and translated documentation, which tells you where the project spends its effort. Start with applications.py, where the FastAPI class extends Starlette and you can see exactly which behavior is inherited and which is added. Then read routing.py, the true heart: when a route is registered it inspects the endpoint's signature, and the request handler it builds is where parameters are extracted, validated, and the return value serialized, so one file shows the whole request lifecycle. The signature analysis and per-request resolution machinery lives in the dependencies/ package, with params.py and param_functions.py defining the small marker classes behind Query, Path, Body, and Depends. After that, encoders.py is the pleasantly readable jsonable_encoder that turns arbitrary objects into JSON-safe structures, the openapi/ package walks the routes to build the schema document, security/ shows dependencies and documentation composing, and _compat.py is the instructive shim that let one codebase span Pydantic v1 and v2. A good first exercise is to put a breakpoint in the handler that routing.py builds and watch a single request get picked apart, validated, and reassembled.

Key takeaway: FastAPI is one idea applied everywhere: the typed function signature is the single source of truth, from which validation, serialization, documentation, and dependency wiring are all derived. The repository is worth reading precisely because the framework adds so little machinery of its own; it is a masterclass in getting leverage by composing Starlette and Pydantic instead of rebuilding them.
