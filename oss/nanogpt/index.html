nanoGPT

nanoGPT is Andrej Karpathy's minimal GPT training repository, the successor to his earlier minGPT, and it makes a striking claim good on arrival: a training loop of roughly three hundred lines and a model definition of roughly three hundred lines that together reproduce GPT-2 (124M) on OpenWebText. It has become the default codebase people fork to try architecture ideas and the default codebase people read to learn how transformers are actually trained. This walkthrough covers how to train something in five minutes, what each of the two files teaches, and the order I would read the repository in.

Using it in five minutes

There is no package to install; you clone the repo and run scripts from its root. The dependencies are ordinary: 

git clone https://github.com/karpathy/nanoGPT
cd nanoGPT
pip install torch numpy transformers datasets tiktoken wandb tqdm

The canonical first run trains a character-level model on the complete works of Shakespeare, which takes a few minutes on one GPU. The prepare script downloads the text and writes tokenized train.bin and val.bin files, training writes checkpoints to an output directory, and sampling generates from the latest checkpoint: 

python data/shakespeare_char/prepare.py
python train.py config/train_shakespeare_char.py
python sample.py --out_dir=out-shakespeare-char

On a machine without a GPU the same run works with --device=cpu --compile=False plus a smaller model, and the README gives an exact flag set for that. The serious reproduction is the same two steps at larger scale: tokenize OpenWebText with python data/openwebtext/prepare.py, then launch data-parallel training across a node with 

torchrun --standalone --nproc_per_node=8 train.py config/train_gpt2.py

which on 8 A100s takes about four days to reach the GPT-2 quality regime, per the README. That the toy run and the reproduction are the same script with different configs is the whole point of the project.

The model fits in your head

model.py is a complete GPT-2 in plain PyTorch: a causal self-attention module, an MLP, a block that stacks them with pre-layer normalization and residual connections, and a GPT class that adds token and learned position embeddings, ties the embedding matrix to the output head, and computes the loss. There is no configuration framework, no abstraction layer for future architectures, no registry of attention variants. Every tensor shape in a GPT forward pass is visible on one screen or the next, which is what makes this the best first transformer codebase to read. When papers talk about heads, you can point at the reshape that creates them.

Two details reward attention. The attention module dispatches to PyTorch's scaled_dot_product_attention when available, so you can see exactly what the fused flash kernel replaces, a manual matmul, causal mask, and softmax written a few lines above it; my softmax page covers why that fusion is possible at all. And the from_pretrained method loads real GPT-2 weights from HuggingFace into this reimplementation, which is both a correctness proof for the architecture code and a template for the useful habit of validating your own model against a reference checkpoint.

The training loop is the curriculum

train.py is the more quietly educational file, because it contains, in linear order and without indirection, essentially every technique a real pretraining run uses. Distributed data parallelism activates when the script is launched under torchrun and degenerates gracefully to a single process otherwise. Gradient accumulation simulates batch sizes larger than memory allows. Mixed precision runs the forward pass in bfloat16 or float16 under autocast with a gradient scaler, gradients are clipped, the AdamW optimizer applies weight decay only to the parameters that should receive it, and the learning rate follows linear warmup into cosine decay. A periodic evaluation loop estimates train and validation loss and decides when to checkpoint, and torch.compile is a single opt-in line that the README credits with a large drop in iteration time. Most frameworks hide this list behind a Trainer class; nanoGPT is valuable precisely because it refuses to.

Configuration is handled by configurator.py, a deliberately crude few dozen lines that exec a config file over the globals of train.py and then apply any --key=value overrides from the command line. It is not a pattern to copy into production, and Karpathy says as much in the comments, but it keeps every default visible as an ordinary variable at the top of the training script, which is the right trade for a codebase meant to be read and hacked.

Reading the repository

Read model.py first, top to bottom, with the GPT-2 paper's architecture section beside it; it is the rare model file where nothing needs to be skipped on a first pass. Then read train.py the same way, and when you hit the DDP setup, the autocast context, or the learning rate function, treat each as a checklist item to understand rather than boilerplate to skim, because each one exists to solve a failure mode you will eventually meet in your own training runs.

After the two main files, the rest is short. The data/ directory holds one prepare.py per dataset, worth reading to see how corpora become the flat binary token files that the training loop memory-maps and samples from. sample.py shows autoregressive decoding with temperature and top-k in a page, bench.py isolates the model forward and backward for timing, and the two notebooks, transformer_sizing.ipynb and scaling_laws.ipynb, show how to reason about parameter counts, FLOPs, and scale. A good exercise after reading is to reimplement a piece from scratch and check it against this code, which is roughly how the exercises in my ML implementations section are built.

Key takeaway: nanoGPT demonstrates that a real GPT-2 training run needs only two readable files, a faithful model definition and an honest training loop, and that almost everything larger frameworks add is convenience rather than necessity. Read it before any other transformer codebase, because every other transformer codebase is this one plus abstraction.
