TensorRT-LLM

TensorRT-LLM is NVIDIA's open source LLM inference library, the successor to FasterTransformer and the vendor's own answer to how far its GPUs can be pushed, which makes it the engine to study when the question is peak performance on NVIDIA hardware rather than portability. This walkthrough covers how to serve a model in five minutes, the compile-ahead-of-time philosophy and its recent PyTorch-shaped revision, in-flight batching, quantization co-designed with the silicon, and a guided path for reading a repository that spans Python and a large C++ core.

Using it in five minutes

TensorRT-LLM is Linux-plus-NVIDIA-GPU software with real system prerequisites (matching CUDA toolkit, a specific PyTorch build, and OpenMPI), which is why the docs steer many users toward the prebuilt NGC container; on a machine that meets them, installation is a pip command: 

pip3 install tensorrt_llm

The front door is the LLM API, which loads a Hugging Face checkpoint directly: 

from tensorrt_llm import LLM, SamplingParams

llm = LLM(model="TinyLlama/TinyLlama-1.1B-Chat-v1.0")
params = SamplingParams(temperature=0.8, top_p=0.95)

for output in llm.generate(["The capital of France is"], params):
    print(output.outputs[0].text)

The online equivalent is an OpenAI-compatible server, with trtllm-bench and trtllm-eval beside it for benchmarking and accuracy checks: 

trtllm-serve "TinyLlama/TinyLlama-1.1B-Chat-v1.0"
curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model": "TinyLlama/TinyLlama-1.1B-Chat-v1.0",
       "messages": [{"role": "user", "content": "Hello"}]}'

Idea one: ahead-of-time engine building, and its revision

TensorRT-LLM inherits its worldview from TensorRT, NVIDIA's general inference compiler: instead of interpreting a model graph at runtime, you compile it once into an engine, a serialized plan in which layers have been fused, precisions decided, and kernels selected and tuned for one specific GPU architecture. The classic workflow made this explicit, convert the checkpoint, run trtllm-build, deploy the resulting engine file, and the payoff is real: work that other engines do on every forward pass, choosing kernels, fusing operations, planning memory, is done once at build time, so the runtime simply replays an optimized plan. The cost is rigidity, since every new model architecture and every attention variant had to be expressible in the compiler before it could run at all. That cost is why the 1.0 release rebuilt the project around a PyTorch-native backend, now the default, in which models are ordinary PyTorch modules calling into the same hand-tuned kernels, and the engine-building path survives as a legacy workflow. The lesson generalizes beyond this repo: in a field where model architectures change monthly, flexibility won, and the compile-time philosophy retreated into the kernels, where it belongs.

Idea two: in-flight batching in a C++ runtime

In-flight batching is NVIDIA's name for continuous batching, the iteration-level scheduling that vLLM popularized: finished sequences leave the batch after each step, waiting requests join immediately, and prefill and decode for different requests share the same iterations, so the GPU never idles while work is queued. TensorRT-LLM pairs it with a paged KV cache and chunked prefill, the same trio found in every serious engine, but what distinguishes the implementation is where it lives: the batch manager and executor are C++, sitting under cpp/tensorrt_llm/, with Python bindings above them. The scheduling loop runs in C++ precisely because at high request rates the scheduler itself becomes the bottleneck, and keeping it out of Python is part of how the engine sustains throughput at production request rates. The same executor API is what Triton Inference Server drives when TensorRT-LLM is deployed behind it in a datacenter.

Idea three: quantization co-designed with the hardware

Every inference engine quantizes; TensorRT-LLM is where quantization meets the silicon first, because the formats are NVIDIA's own. FP8 arrived with Hopper's transformer engine and FP4 with Blackwell, each backed by dedicated tensor-core datapaths, and this library carries the kernels that exploit them, from fused attention through the CUTLASS-based GEMM extensions in the C++ tree. The checkpoints themselves typically come from the TensorRT Model Optimizer toolkit, and NVIDIA publishes pre-quantized FP8 and FP4 versions of popular open models on Hugging Face so that serving them is a model-name change rather than a calibration project. The practical takeaway is that on recent NVIDIA hardware, low-precision floating point is not an accuracy compromise bolted on afterwards but the intended operating point the chips were designed around, and this repository is where that intent becomes runnable code.

Reading the repository

The repo is two codebases sharing a roof: the Python package in tensorrt_llm/ and the C++ core in cpp/tensorrt_llm/. Start on the Python side with the LLM API under llmapi/, the class from the quickstart, and the CLI entry points under commands/, which show how trtllm-serve assembles the same pieces. The PyTorch backend lives under _torch/, and it is the most instructive read in the project: model definitions, attention backends, and the glue that swaps standard modules for optimized kernels, with the model zoo in models/ and the quantization interfaces in quantization/ beside it. The executor/ package is the bridge downward into C++, where batch_manager/ owns request scheduling and the KV cache, runtime/ owns engine and buffer management, and kernels/ with cutlass_extensions/ holds the attention and GEMM code, bound back to Python through the torch ops in thop/. The examples/ tree keeps per-model scripts including the legacy engine-build flow. A good first session is to run the LLM API example, then trace one generate call from llmapi/ through the executor bindings until you cross the C++ boundary; my LLM serving design write-up is a useful companion for why each layer you pass through exists.

Key takeaway: TensorRT-LLM shows what inference looks like when the engine and the silicon come from the same company: compile-time optimization pushed as far as it can go, then deliberately pulled back into the kernels when model velocity made whole-graph compilation a liability, wrapped around a C++ runtime and quantization formats the GPUs themselves were designed to run.
