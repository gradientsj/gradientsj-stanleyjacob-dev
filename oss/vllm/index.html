vLLM

vLLM is the reference open source engine for serving large language models, born from the PagedAttention paper at UC Berkeley (Kwon et al., SOSP 2023) and now the default answer to "how do I serve this checkpoint fast." This walkthrough covers how to use it in five minutes, the two ideas that make it fast, and a guided path for reading the repository. It pairs with my LLM serving platform design write-up, which derives the same ideas from requirements instead of from code.

Using it in five minutes

vLLM has two front doors. The offline one is a Python class, good for batch jobs and evaluation sweeps: 

pip install vllm
from vllm import LLM, SamplingParams

llm = LLM(model="Qwen/Qwen2.5-7B-Instruct")
params = SamplingParams(temperature=0.7, max_tokens=256)
outputs = llm.generate(["Explain KV caching in one paragraph."], params)
print(outputs[0].outputs[0].text)

The online one is an OpenAI-compatible server, which is why swapping a proprietary API for a self-hosted model is usually a one-line base URL change in the client: 

vllm serve Qwen/Qwen2.5-7B-Instruct
from openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1", api_key="unused")
resp = client.chat.completions.create(
    model="Qwen/Qwen2.5-7B-Instruct",
    messages=[{"role": "user", "content": "Hello"}],
)

The flags that matter most in practice: --tensor-parallel-size to split a model across GPUs, --gpu-memory-utilization to set how much VRAM the KV cache may claim, --max-model-len to cap context length, and --enable-prefix-caching to reuse computation across requests that share a prompt prefix (on by default in recent versions).

Idea one: PagedAttention

Decoding is dominated by the KV cache, the stored keys and values of every token seen so far. Before vLLM, engines allocated each request's cache as one contiguous region sized for the maximum possible length, and the PagedAttention paper measured the consequence: most of that memory was wasted, through internal fragmentation (reserved but never reached) and external fragmentation (free but not contiguous). Since batch size is limited by KV memory, wasted cache is wasted throughput.

vLLM's answer is the operating-systems move: treat the KV cache like virtual memory and page it. The cache is carved into fixed-size blocks, 16 tokens by default, a sequence's blocks need not be contiguous, and a per-sequence block table maps logical positions to physical blocks, so waste collapses to a fraction of the final block. Paging also makes sharing natural: parallel samples from one prompt share prefix blocks copy-on-write, and prefix caching is just looking up blocks by content hash before computing them. The attention kernel pays a small indirection cost to gather from the block table, which is the trade the whole design happily accepts.

Idea two: continuous batching

The second idea governs time rather than memory. Static batching runs a batch until every sequence finishes, so one long generation holds hostages. Continuous batching, which vLLM inherits from the Orca paper, reschedules every iteration: after each model step, finished sequences leave, waiting requests join, and the batch is rebuilt. Together with paging this is why vLLM sustains high GPU utilization under ragged real-world traffic, and the combination is worth internalizing: iteration-level scheduling creates constant churn in batch membership, and paged memory is what makes that churn cheap. When memory does run out, the scheduler preempts a sequence and either recomputes its cache later or swaps it out, a policy you can read directly in the scheduler code.

Reading the repository

The repo is large, but one request's life touches everything worth reading, so follow a single generate call. Note that vLLM went through a V1 engine rewrite (the vllm/v1/ tree), which is the default in current releases; the module names below are stable concepts even where exact paths have moved between versions.

Start at the entry points in vllm/entrypoints/: the LLM class and the OpenAI server both funnel into the engine, so the API layer is thin and readable. The engine loop is the heartbeat: each step asks the scheduler what to run, executes one forward pass, samples, and streams results back. From there read the scheduler, which owns admission, preemption, and chunked prefill (splitting long prompt processing across steps so decode latency stays flat), and the KV cache manager beside it, which is PagedAttention's bookkeeping half: block allocation, reference counts, copy-on-write, and the prefix-cache hash table. The compute half lives in the model runner and the attention backends, where vLLM dispatches to FlashAttention, FlashInfer, or its own kernels, and the custom CUDA lives under csrc/, paged attention and the fused ops. Model definitions in vllm/model_executor/models/ are surprisingly plain PyTorch, which is the point: architecture code stays legible because the cleverness is concentrated in memory and scheduling.

A good first session is ninety minutes: run the offline example under a debugger, break inside the engine step, and watch one iteration of schedule, forward, sample. The block table indirection and the batch churn stop being abstract the first time you inspect them live. If you want the kernel-level view of why attention can stream through memory it never holds at once, the softmax page derives the online-softmax trick that paged and flash attention kernels are built on.

Key takeaway: vLLM is two ideas executed thoroughly. Page the KV cache so memory stops being wasted, and reschedule every iteration so time stops being wasted; everything else in the repository is the engineering that lets those two ideas survive contact with real models, real GPUs, and real traffic.
