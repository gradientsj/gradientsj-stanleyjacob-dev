torchtitan

torchtitan is the PyTorch team's own answer to the question of how large language models should be pretrained in native PyTorch, without a Megatron or DeepSpeed layered on top: a platform that trains models like Llama 3.1 from 8B to 405B by composing FSDP2, tensor, pipeline, and context parallelism over DTensor. It matters because it is simultaneously a usable trainer and the reference showcase for PyTorch's distributed stack, so reading it is the shortest path to understanding how modern N-dimensional parallelism actually fits together. This walkthrough covers launching a run in five minutes, the two ideas that carry the design, and a reading path through the repository.

Using it in five minutes

There is a pip install torchtitan release, but the project moves with PyTorch nightlies and is really meant to be run from source: 

git clone https://github.com/pytorch/torchtitan
cd torchtitan
pip install -r requirements.txt

Training a Llama model needs its tokenizer, which a helper script downloads from HuggingFace once you have accepted the license and have a token: 

python scripts/download_hf_assets.py --repo_id meta-llama/Llama-3.1-8B \
  --assets tokenizer --hf_token=...

A run is then one script. run_train.sh wraps torchrun, takes the GPU count from NGPU (default 8), and selects a model family and a registered configuration by name: 

MODULE=llama3 CONFIG=llama3_8b ./run_train.sh

Every knob, from batch size to the degree of each parallelism dimension, lives in the named config and can be overridden as a command line flag, so scaling an experiment from FSDP-only on one node to FSDP plus tensor parallel plus pipeline parallel on many is a config change rather than a code change. The default config is a tiny debug model that runs in seconds, which is also the right thing to step through with a debugger on a first read.

DTensor makes parallelism a property of tensors

Classic megamodel trainers achieve tensor or pipeline parallelism by rewriting the model: column-parallel linear layers, manually placed collectives, forked architecture files per strategy. The bet torchtitan demonstrates is that PyTorch's DTensor abstraction dissolves most of that. A DTensor is an ordinary tensor plus a placement: it knows which devices it lives on, via a device mesh, a logical N-dimensional grid over the GPUs, and how it is laid out across them, sharded along some dimension or replicated. Once layout is metadata carried by the tensor itself, the framework can insert the required communication automatically, and parallelizing a model becomes annotating it rather than rewriting it.

FSDP2, the data parallel workhorse here, shows why this is more than aesthetics. The original FSDP flattened each wrapped module's parameters into one opaque buffer; FSDP2 instead represents every parameter as a DTensor sharded per parameter along its first dimension. Because the sharding is visible and typed, it composes with tensor parallelism (the same weight can be sharded one way by TP across one mesh dimension and another way by FSDP across another), with fine-grained control over what stays in which precision, and with distributed checkpointing that can resave and reshard a run onto a different world size.

One model definition, N dimensions of parallelism

The second idea is a discipline about where code lives. The model definition in torchtitan is deliberately plain, single-device PyTorch, close in spirit to a scaled-up nanoGPT, and all distribution logic is applied to it from outside. A ParallelDims object turns the configured degrees into a device mesh, and a per-model parallelize function then walks the model and applies each technique in a careful order: tensor parallel sharding of attention and MLP weights (with an async variant that overlaps the TP collectives with compute), activation checkpointing, torch.compile, and finally FSDP2 across the data parallel mesh dimension. Pipeline parallelism splits the stack of blocks into stages scheduled across ranks, and context parallelism shards the sequence dimension itself so million-token contexts fit, with the attention exchange handled under the hood.

The payoff of keeping these techniques orthogonal is that they multiply instead of interfere: the same Llama definition trains with FSDP alone on eight GPUs or with four parallelism dimensions composed on thousands, plus meta-device initialization so the 405B model never has to materialize on one host, Float8 training, and fault-tolerant checkpointing. torchtitan's real product is the demonstration that composability, not a monolithic trainer, is how PyTorch wants large-scale training to be expressed. The same properties that make checkpoints reshardable here are what make them loadable by inference engines later.

Reading the repository

Everything of interest is under the torchtitan/ package. Start with train.py and trainer.py at its top level: the Trainer owns the loop, and one pass through it shows the full lifecycle of a step, batch loading, forward, loss, backward, clipping, optimizer, and metrics, with every distributed concern delegated outward. Then read one model family end to end, models/llama3/ being the canonical one: model.py is the plain definition, parallelize.py is where the ordered application of TP, activation checkpointing, compile, and FSDP2 happens, and config_registry.py is where named configurations like llama3_8b are assembled as ordinary Python objects.

The conceptual core is the distributed/ directory, one focused file per technique: parallel_dims.py for the device mesh construction, fsdp.py, tensor_parallel.py, pipeline_parallel.py, context_parallel.py, and activation_checkpoint.py. Reading these against the corresponding PyTorch APIs they orchestrate is the fastest way I know to learn the current distributed stack. The components/ directory holds the swappable parts, checkpointing, optimizers, learning rate schedules, quantization, metrics, and validation, and the experiments/ tree is where newer work like reinforcement learning and torchft-based fault tolerance incubates; the docs/ folder has a readable note per subsystem. If the model side is the part you want to internalize first, the exercises in my ML implementations section build the single-device pieces this repository assumes.

Key takeaway: torchtitan shows that large-scale pretraining does not require a framework that owns your model. Represent layout as DTensor placements on a device mesh, keep the model definition single-device and plain, and apply each parallelism as a composable transformation, and the same few hundred lines of training loop carry you from one GPU to thousands.
