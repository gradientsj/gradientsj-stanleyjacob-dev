Faiss

Faiss is Meta's library for similarity search and clustering of dense vectors, a C++ core with Python bindings that grew out of FAIR's billion-scale GPU search work and now sits under a large fraction of the world's embedding retrieval, either directly or inside the vector databases built on top of it. It matters because it is where the canonical approximate nearest neighbor index structures live as production code rather than as papers. This walkthrough covers using it in five minutes, the index families and the factory string that composes them, how to choose an index, and a reading path through the repository.

Using it in five minutes

The official binaries ship through conda, as faiss-cpu and faiss-gpu (the pip packages of the same names are community builds): 

conda install -c pytorch -c conda-forge faiss-cpu

Every index speaks the same tiny API: construct for a dimensionality, add a matrix of vectors, search a matrix of queries for the k nearest, getting back distances and integer ids. Exact search is two lines: 

import numpy as np
import faiss

d = 128
xb = np.random.random((100_000, d)).astype("float32")
xq = np.random.random((5, d)).astype("float32")

index = faiss.IndexFlatL2(d)
index.add(xb)
D, I = index.search(xq, 10)

Approximate indexes add one step, train, which learns the index's internal structure from your data before anything is added, and a speed-versus-recall knob queried at search time: 

index = faiss.index_factory(d, "IVF1024,Flat")
index.train(xb)
index.add(xb)
index.nprobe = 16
D, I = index.search(xq, 10)

Moving to a GPU is a wrapper around an existing index, faiss.index_cpu_to_gpu(faiss.StandardGpuResources(), 0, index), and multi-GPU variants exist; the GPU indexes are designed as drop-in replacements for the CPU ones.

Four families and one triangle

Everything in Faiss is a point in a triangle whose corners are recall, speed, and memory, and the index families are named strategies for trading among them. Flat indexes are the exact corner: store every vector as-is and brute-force every query, perfectly accurate, memory-hungry, and linear in dataset size, but so well optimized that exact search is the right answer far later than intuition suggests. IVF, the inverted file, buys speed by pruning: k-means clusters the vectors into nlist cells at train time, each vector is stored in the list of its nearest centroid, and a query visits only the nprobe closest cells, so recall becomes a dial you turn at search time. HNSW takes the graph route instead, linking each vector to neighbors in a hierarchy of layers that a search greedily descends; it needs no training, answers in logarithmic hops with excellent recall, and pays for it in memory for the links and in slower construction.

The fourth family attacks memory rather than time. Product quantization splits each vector into m subvectors and replaces each with the id of its nearest centroid in a small learned codebook, compressing a vector to m bytes at typical settings, so a 512-byte float vector might become 32 bytes that still support fast approximate distance computation directly on the codes. The families compose, and the composition is the point: an IVF structure to prune the search, PQ codes inside the lists to shrink memory, and optionally a flat re-ranking pass over exact vectors for the final few candidates. IVF plus PQ is what makes billion-vector search fit in RAM at all, and it is the combination the GPU implementation accelerates hardest.

The factory string is the library's grammar

With families that compose, constructing indexes by hand gets verbose, so Faiss exposes a small language instead. index_factory takes a comma-separated string with up to three parts, an optional preprocessing transform, the coarse structure, and the encoding, and builds the whole pipeline: "Flat" is exact search, "HNSW32" is a graph with 32 links per node, "IVF4096,PQ32" is 4096 inverted lists over 32-byte PQ codes, and "OPQ16_64,IVF4096,PQ16" first learns a rotation that makes the data friendlier to quantization, then prunes, then compresses. Learning to read factory strings is learning Faiss, because every deployed index is one of these one-liners and every benchmark in the literature can be named in the same grammar. The knobs that remain at search time, such as nprobe for IVF and efSearch for HNSW, are exactly the ones that trade latency for recall on a live system, and the ParameterSpace and AutoTune machinery can sweep them against a ground truth to map the frontier for you.

Choosing an index

The wiki's guidelines page is the reference, but the shape of the decision is simple. If the dataset is small, up to the low millions, and latency is not desperate, use flat and be exact, since every approximate method is measured against that baseline anyway. If you need speed and memory is plentiful, HNSW is the usual answer. At larger scales IVF variants take over, with nlist commonly set around a small multiple of the square root of the dataset size, flat codes inside the lists while memory lasts and PQ codes when it does not, and beyond RAM entirely there are on-disk inverted lists and sharding across machines. The honest part of the decision is that recall targets come from the product, not the library, so the workflow is to pick a candidate factory string, measure recall at your latency budget on your own vectors, and adjust. I walk through that end-to-end sizing exercise in the visual search design write-up, and the retrieval half of the RAG system design is largely about what happens around an index like this.

Reading the repository

Faiss is a C++ library first, with the Python API generated from it, so read the C++ headers even if you only ever call Python; the names map one to one. Start with faiss/Index.h, the abstract base class whose train, add, and search methods define the contract everything implements, then read IndexFlat as the smallest real implementation. IndexIVF is the heart of the library, the coarse quantizer plus inverted lists structure that most large-scale deployments use, and its subclasses like IndexIVFFlat and IndexIVFPQ show how the storage inside lists varies independently of the pruning around them. The algorithmic cores live in faiss/impl/, where HNSW.cpp and ProductQuantizer.cpp are self-contained enough to read against their papers, and index_factory.cpp is the parser that turns the grammar above into object graphs, a genuinely useful map of what exists.

After the core, faiss/gpu/ mirrors the index classes for CUDA and ROCm, contrib/ holds the higher-level Python utilities, and the tutorial/ and demos/ directories contain short runnable examples that make good first sessions; the benchs/ directory reproduces published results. A productive exercise is to take one dataset, build "Flat", "HNSW32", and "IVF1024,PQ32" over it, and plot recall against queries per second while sweeping each index's search-time knob, which teaches more about the triangle in an hour than any amount of reading.

Key takeaway: Faiss is a small contract, train, add, search, over a composable set of trade-offs: prune with IVF, link with HNSW, compress with PQ, verify with flat. Learn the factory string grammar and the recall-speed-memory triangle behind it, and both the library and the vector databases built on it stop being magic.
