SGLang

SGLang is a serving framework for large language and vision-language models that began as a research project from Stanford and Berkeley (Zheng et al., NeurIPS 2024) and now lives under the LMSYS non-profit, with production deployments at companies like xAI running it at enormous GPU scale. Its founding claim is that the runtime and the programming interface should be co-designed, and this walkthrough covers how to serve a model in five minutes, the three ideas that follow from that claim, and a guided path for reading the repository.

Using it in five minutes

SGLang installs from PyPI (the docs recommend the uv installer and pin matching PyTorch builds, so check the install guide if you hit version friction) and launches as a server process: 

pip install sglang
python3 -m sglang.launch_server \
  --model-path meta-llama/Llama-3.1-8B-Instruct \
  --host 0.0.0.0 --port 30000

The server is OpenAI-compatible, so the usual client works with a base URL change: 

from openai import OpenAI

client = OpenAI(base_url="http://localhost:30000/v1", api_key="unused")
resp = client.chat.completions.create(
    model="meta-llama/Llama-3.1-8B-Instruct",
    messages=[{"role": "user", "content": "Hello"}],
)

The same package also contains the frontend language the project is named for, a Python-embedded DSL that runs against the server you just launched: 

from sglang import function, gen, system, user, assistant, RuntimeEndpoint
from sglang.lang.api import set_default_backend

set_default_backend(RuntimeEndpoint("http://localhost:30000"))

@function
def basic_qa(s, question):
    s += system("You are a helpful assistant.")
    s += user(question)
    s += assistant(gen("answer", max_tokens=512))

state = basic_qa("List 3 countries and their capitals.")
print(state["answer"])

Idea one: RadixAttention

Real workloads repeat themselves. Multi-turn chats resend the whole conversation each turn, few-shot prompts share their examples across thousands of requests, and agent loops replay long histories with small additions, so the same prompt prefixes get re-processed constantly. Prefix caching, reusing the KV cache of a shared prefix instead of recomputing it, is the fix, and vLLM offers it as a lookup layered over its paged cache. SGLang's contribution was to promote it from a feature to the organizing principle of the runtime. RadixAttention keeps the entire KV cache pool indexed by a radix tree over token sequences, so every cached prefix of every past request is a candidate for reuse, with cold branches evicted LRU when memory runs short. Because the tree makes reuse visible ahead of time, the scheduler can be cache-aware, ordering waiting requests so that ones sharing prefixes run near each other and hit rates stay high, which is why SGLang shines on exactly the workloads above.

Idea two: structured generation as a fast path

Constrained decoding usually means compiling a JSON schema, regex, or grammar into an automaton and masking invalid tokens at each step, which guarantees valid output but pays the full per-token decoding cost even for characters the grammar has already decided. SGLang's insight, introduced as jump-forward decoding with a compressed finite state machine, is that whenever the grammar permits exactly one continuation, the server can append those tokens directly and skip the forward passes entirely, so the model only ever decodes where it has a real choice. In a JSON object that is most of the punctuation and every fixed key. The grammar engine itself is pluggable, with XGrammar as the default backend, and it is all reachable through the OpenAI API's structured-output parameters, so clients get schema-guaranteed JSON without knowing any of this is happening.

Idea three: the frontend DSL

The name stands for Structured Generation Language, and the frontend is the co-design half of the paper: if callers describe whole programs instead of issuing isolated completion calls, the runtime can see structure it could never infer from raw requests. A function decorated with @function builds up a prompt state with primitives like gen, select, and fork, using ordinary Python control flow in between. gen is non-blocking, select asks the model to choose among fixed options by comparing their probabilities, and fork branches one state into parallel continuations, which the runtime executes concurrently while the radix tree shares the common prefix's KV cache among all branches. In practice most production users drive SGLang through the plain OpenAI API and the DSL is the less-traveled half of the project, but it explains the architecture: the runtime is built to exploit exactly the sharing and parallelism the language expresses.

Reading the repository

Almost everything lives under python/sglang/, split between lang/, the frontend language, and srt/, the SGLang Runtime, which is the serving engine and the part worth most of your time. launch_server.py is the entry point; from there the HTTP layer in srt/entrypoints/ hands requests to a pipeline of processes you can read in srt/managers/: a tokenizer manager feeding a scheduler feeding a detokenizer, connected by queues so tokenization never blocks the GPU. The scheduler is the heart, it owns continuous batching and the cache-aware policy, and it leans on srt/mem_cache/, where the radix cache and the KV allocators implement RadixAttention's bookkeeping. Model execution is srt/model_executor/ plus the attention backends and layers under srt/layers/, with grammar-constrained decoding wired in under srt/constrained/. Custom CUDA kernels live in the separate sgl-kernel/ package at the repo root, and the Rust model gateway beside it routes traffic across replicas with the same cache-awareness applied one level up. A good first session is to launch the server with one small model, send the same long prompt twice, and watch the logs report the second request's prefix hit; then read the radix cache with that observation in hand. For how these serving-layer choices compose into a whole platform, my LLM serving design write-up covers the same ground from the requirements side.

Key takeaway: SGLang is a bet on structure: index the whole KV cache so any shared prefix is reusable, let grammars skip the tokens they have already determined, and give callers a language whose programs expose that structure on purpose; the runtime is what those three commitments look like when engineered to production quality.
