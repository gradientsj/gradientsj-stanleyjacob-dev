RocksDB

RocksDB is an embeddable persistent key-value store from Meta's database engineering team, built on the earlier LevelDB work of Sanjay Ghemawat and Jeff Dean, and it has quietly become the storage engine that other systems are made of: MyRocks puts it under MySQL, TiKV and Kafka Streams and Flink's state backend all keep their bytes in it. It is the LSM tree in production form, and this walkthrough covers how to use it in five minutes, the ideas that define it, and a guided path for reading the repository. It pairs with my key-value store design write-up, which derives the same LSM structure from requirements instead of from code.

Using it in five minutes

RocksDB is a library, not a server. There is no daemon to start and no port to bind; you link it into your process and it manages a directory of files on disk. The canonical way in is to build from source, though prebuilt packages exist on most platforms (librocksdb-dev on Debian and Ubuntu, a rocksdb formula on Homebrew): 

git clone https://github.com/facebook/rocksdb.git
cd rocksdb
make static_lib
cd examples && make all

The entire public API lives behind a handful of headers, and the minimal program is short enough to read in one breath: 

#include "rocksdb/db.h"
#include "rocksdb/options.h"

rocksdb::Options options;
options.create_if_missing = true;
options.IncreaseParallelism();
options.OptimizeLevelStyleCompaction();

rocksdb::DB* db;
rocksdb::Status s = rocksdb::DB::Open(options, "/tmp/testdb", &db);
assert(s.ok());

s = db->Put(rocksdb::WriteOptions(), "key1", "value");
std::string value;
s = db->Get(rocksdb::ReadOptions(), "key1", &value);

This is a lightly trimmed version of examples/simple_example.cc in the repository, which goes on to show WriteBatch for atomic multi-key writes and PinnableSlice for copy-free reads. Every call returns a Status rather than throwing, iterators give you ordered range scans with Seek and Next, and the repository ships two inspection tools worth knowing early: ldb for poking at a live database from the command line and sst_dump for reading the files it produces.

The write path: memtables, WAL, and SST files

RocksDB describes itself in its own README as a log-structured-merge database, and the write path is the cleanest place to see what that means. A write goes to two places: it is appended to a write-ahead log for durability, and it is inserted into the memtable, an in-memory sorted structure that is a skiplist by default. Nothing else happens at write time, which is why writes are fast and why they never depend on how much data is already on disk. When the memtable fills (64 MB by default, the write_buffer_size option), it becomes immutable, a fresh memtable takes over, and a background thread flushes the frozen one to disk as an SST file, a sorted string table holding keys in order alongside an index block and optional bloom filter.

The LSM tree turns random writes into sequential I/O by absorbing them in memory and only ever writing sorted, immutable files. The cost is paid on the read side: a Get must consult the memtables first, then the SST files from newest to oldest, because a newer file may shadow an older value for the same key. Bloom filters let most files answer "definitely not here" without a disk read, and the block cache keeps hot data blocks (about 4 KB uncompressed by default) in memory, but the fundamental tension remains: the more files accumulate, the more expensive reads become. Managing that accumulation is the job of compaction.

Compaction is where the trade-offs live

Compaction rewrites SST files in the background, merging overlapping files, dropping overwritten and deleted entries, and keeping read cost bounded. The README frames the whole engine as offering flexible trade-offs between write amplification, read amplification, and space amplification, and the compaction style is the main lever. You cannot minimize all three at once, and choosing a compaction strategy is choosing which one you are willing to pay.

Leveled compaction, the default, organizes files into levels of exponentially increasing size (roughly ten times per level by default) where each level except L0 is a single sorted run; freshly flushed files land in L0, and once enough of them pile up (four by default) they are merged down into L1 and onward. This keeps space and read amplification low at the price of rewriting data as it migrates down, which is write amplification. Universal compaction is the tiered alternative: it merges files of similar size into larger sorted runs without rewriting the destination, trading much lower write amplification for temporarily higher space use and more runs to check on reads, which suits ingest-heavy workloads. FIFO compaction barely compacts at all; it simply drops the oldest files once a size budget is exceeded, which makes RocksDB behave like a persistent cache for time-ordered data.

Column families and the knobs that matter

A column family is an independent keyspace inside one database: each has its own memtable, its own SST files, and its own options, so one family can run universal compaction with heavy compression while its neighbor runs leveled with none. What they share is the write-ahead log, and that sharing is the point, because it gives you atomic writes across families and a consistent view when reading from several at once. The pattern shows up everywhere RocksDB is embedded: a system stores its data in one family and its metadata or secondary index in another, tuned differently but committed together.

The options surface is famously large, but a small set carries most of the weight in practice: write_buffer_size and max_write_buffer_number control how much RAM absorbs writes before flushes and stalls, the L0 file triggers control when compaction starts and when writes are throttled, the block cache size controls read hit rates, bloom filter bits per key trade memory for skipped reads, and per-level compression lets you keep upper levels fast and the bottom level small. The defaults are deliberately conservative, and the wiki's tuning guide is honest that most users should change only a few of these; my databases note covers where an LSM engine fits relative to B-tree storage when you are choosing rather than tuning.

Reading the repository

Start where users start, in include/rocksdb/: the headers db.h and options.h are the contract everything else implements, and the options comments are some of the best documentation in the project. Then read examples/simple_example.cc and its siblings, which demonstrate column families, transactions, and compaction filters in a page each.

The implementation of the DB interface lives under db/, with the DBImpl class as the hub; follow a Put through the write path to see the WAL append and memtable insert, including the group commit logic that batches concurrent writers. Memtable implementations live in memtable/, and the SST file format, block layout, bloom filters, and table reader and builder live under table/, with the block-based table as the default format. Compaction has its own corner of db/, split between the pickers, which decide what to compact under each style, and the job that does the merging; beside it, the version set code is worth real attention because it holds the metadata heart of the engine, the MANIFEST log that records which SST files constitute each level at each point in time, which is how the database recovers its shape after a crash. The block cache lives in cache/, and utilities/ holds the optional layers, transactions, backup, and TTL among them, that show how far the core primitives stretch.

A good first session is to run the simple example, then look at what it left behind: list the database directory, run sst_dump on an SST file to see blocks and properties, open the LOG file and read the compaction statistics RocksDB prints. The LSM tree stops being a diagram the first time you watch level sizes shift under load.

Key takeaway: RocksDB is the LSM tree taken seriously as an engineering artifact. Writes are absorbed in memory and logged, files on disk are sorted and immutable, and compaction continuously renegotiates the balance between write, read, and space amplification; column families and the options surface exist so that each embedded use can strike that balance differently without forking the engine.
