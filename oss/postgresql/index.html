PostgreSQL

PostgreSQL is the open source relational database, descended from Michael Stonebraker's POSTGRES project at Berkeley and developed since 1996 by the PostgreSQL Global Development Group, a community that takes patches by mailing list rather than pull request; the canonical repository lives at git.postgresql.org with a read-only mirror on GitHub. It has quietly become the default backing store for new applications, and its source tree is one of the best-commented large C codebases in existence. This walkthrough covers how to use it in five minutes, the ideas that carry the system, MVCC and vacuum, the write-ahead log, the process model and its extension points, and a reading path straight down the query pipeline.

Using it in five minutes

The quickest zero-install route is the official container, which gives you a full server and the psql client: 

docker run -d -e POSTGRES_PASSWORD=secret -p 5432:5432 postgres:17
docker exec -it $(docker ps -lq) psql -U postgres

On a real machine the package manager sets up a cluster and a postgres system user for you: 

sudo apt install postgresql      # or: brew install postgresql@17
sudo -u postgres createdb demo
sudo -u postgres psql demo
demo=# CREATE TABLE events(id bigserial PRIMARY KEY, kind text, at timestamptz DEFAULT now());
demo=# INSERT INTO events(kind) SELECT 'signup' FROM generate_series(1, 100000);
demo=# CREATE INDEX ON events(kind);
demo=# EXPLAIN ANALYZE SELECT count(*) FROM events WHERE kind = 'signup';

That last command is the one to remember, because EXPLAIN ANALYZE prints the plan the optimizer chose with estimated and actual costs side by side, and reading its output is the daily interface to everything the rest of this page describes.

Idea one: MVCC, or never update in place

PostgreSQL's answer to concurrency is multi-version concurrency control, and its particular flavor is radical: an UPDATE never modifies a row in place but writes a whole new row version, and every version carries the transaction IDs that created and deleted it, so each transaction simply ignores versions that are not visible to its snapshot. Readers therefore never block writers and writers never block readers, which is a large part of why Postgres behaves so well under mixed workloads. The bill arrives later: superseded and deleted row versions, dead tuples, accumulate in table files as bloat, and reclaiming them is the job of vacuum, run continuously by the autovacuum daemon. Vacuum also has a second, non-negotiable duty, because transaction IDs are 32-bit counters that wrap around, so old row versions must periodically be frozen, marked visible to everyone, before the counter laps them. Most operational Postgres lore, from tuning autovacuum on hot tables to fearing wraparound on neglected ones, is a direct corollary of this one storage decision, and it is the central trade-off I lean on in my databases note.

Idea two: everything flows through the WAL

Durability rests on a single mechanism: before any change touches a table or index page, a record describing it is appended to the write-ahead log, and a commit means the WAL, not the data files, has reached disk. Data pages are modified in a shared buffer cache and written back lazily, with periodic checkpoints bounding how much log must be replayed after a crash. What makes the WAL the most load-bearing component in the system is how much else turned out to be the same problem: crash recovery, streaming replication, and point-in-time recovery are all one mechanism, because a replica is just a server replaying the primary's WAL and a restore is just a base backup replayed forward to a chosen moment. Logical replication and change-data-capture tools decode the same stream into row-level events, which is why the WAL is also the integration point for the wider data ecosystem.

Idea three: a process per connection, extensible at every joint

Architecturally the server is a family of processes rather than threads: the postmaster listens for connections and forks a dedicated backend process for each one, cooperating through shared memory that holds the buffer cache and lock tables, while background processes, the checkpointer, the WAL writer, the autovacuum launcher and its workers, do the housekeeping. The model buys isolation, since a crashing backend cannot silently corrupt its siblings, at the price of connections being expensive, which is why serious deployments put a pooler such as PgBouncer in front. The other defining trait is extensibility, inherited from the original Berkeley design: types, operators, functions, and even index access methods are rows in system catalogs rather than hardcoded lists, so an extension loaded with CREATE EXTENSION can add all of them. That is how PostGIS turns Postgres into a geospatial database and how pg_stat_statements, foreign data wrappers, and the vector-search extensions work, and it is the main reason Postgres keeps absorbing workloads that once required specialized systems.

Reading the repository

Clone the mirror and live in src/backend/, whose subdirectories map almost one-to-one onto the pipeline, and read the README files scattered through the tree, which are genuinely excellent, with src/backend/optimizer/README the standout. Begin at postmaster/postmaster.c to see the fork-per-connection model, then move to tcop/postgres.c, the traffic cop, where exec_simple_query() is the spine of the whole system: a query string goes to the parser in parser/ with the grammar in gram.y, through the rewriter in rewrite/ where views and rules are expanded, into the planner via optimizer/plan/planner.c, which builds candidate paths in optimizer/path/allpaths.c and prices them with the cost model in optimizer/path/costsize.c using statistics gathered by ANALYZE, and finally to the executor in executor/, starting at execMain.c, where plan nodes pull tuples from each other one at a time. Below that, row storage and visibility live in access/heap/, vacuum's hard work in access/heap/vacuumlazy.c with its entry point in commands/vacuum.c, the buffer cache in storage/buffer/bufmgr.c, and the WAL machinery in access/transam/xlog.c. A good first exercise is to take one query, read its EXPLAIN output, and find the executor node file that implements each plan node, since the names match almost exactly.

Key takeaway: PostgreSQL is a small number of old, sound decisions compounding for decades: version rows instead of locking them and pay for it with vacuum, log every change once and derive recovery and replication from that log, isolate connections in processes, and keep the system's own behavior in catalogs so extensions can teach an ancient codebase new tricks.
