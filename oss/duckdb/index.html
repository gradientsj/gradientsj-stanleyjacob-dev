DuckDB

DuckDB is an in-process analytical database, a single MIT-licensed library that runs full SQL with no server and no dependencies, started at the CWI research institute in Amsterdam by Mark Raasveldt and Hannes Mühleisen and often summarized as SQLite for analytics. It matters because it moved serious columnar query execution into the places data scientists actually work, a Python process, a notebook, a laptop, and made "just query the Parquet file" a normal thing to type. This walkthrough covers how to use it in five minutes, the execution ideas that make it fast, and a guided path for reading the repository.

Using it in five minutes

The command-line shell installs with one line and starts in an in-memory database, and because DuckDB treats file paths as table names, the first query can run against a raw Parquet file with no import step: 

curl https://install.duckdb.org | sh
duckdb
SELECT city, avg(price) FROM 'listings.parquet' GROUP BY city;
SELECT count(*) FROM 'logs/*.csv.gz';

The Python client is the same engine as a module, and it is where most people meet DuckDB: 

pip install duckdb
import duckdb

duckdb.sql("SELECT 42").show()
duckdb.sql("SELECT city, avg(price) FROM 'listings.parquet' GROUP BY city").show()

con = duckdb.connect("analytics.db")   # persistent single-file database
con.sql("CREATE TABLE test AS SELECT * FROM 'listings.parquet'")

With no argument, connect gives an in-memory database; with a path, everything persists to one file. Query results convert to and from pandas DataFrames and Arrow tables directly, and you can query a DataFrame sitting in your Python process by simply naming the variable in SQL, which is the feature that tends to convert people on first contact.

In-process, columnar, vectorized

The first design decision is where the database lives: inside your process, like SQLite, rather than across a socket, like Postgres. There is no server to operate, and result sets do not have to be serialized across a connection, since the engine can hand columns of memory to the host language, which is what makes the Arrow and pandas integration close to free. The whole engine compiles into an amalgamation needing only a C++ compiler, which is why the same code runs on a laptop, in a lambda, and in the browser via WebAssembly.

The second decision is how queries execute. Row-at-a-time interpreters like SQLite and classic Postgres pay function call and dispatch overhead per value, and compiling queries to machine code avoids that at the cost of enormous complexity. DuckDB takes the middle path worked out in the vectorized execution literature: the interpreter still walks an operator tree, but each call processes a vector of values, 2048 by default, so the overhead of interpretation is amortized across thousands of values while the inner loops stay tight, cache-friendly, and amenable to SIMD. Data moves between operators as chunks of column vectors, and the operators themselves are push-based: sources push chunks through pipelines of operators into sinks, which is the shape that makes the parallelism story below work.

Morsel-driven parallelism

DuckDB parallelizes queries using the approach from the morsel-driven parallelism paper by Leis, Boncz, Kemper, and Neumann, and the idea is worth having in your head independent of DuckDB. Instead of planning up front which thread handles which partition of the data, the input is divided into many small fragments, morsels, and a pool of worker threads repeatedly grabs the next available morsel and pushes it through the current pipeline. Scheduling becomes self-balancing: a thread that gets slow data or a slow core simply takes fewer morsels, and no thread ever waits on a partitioning decision made before the data was seen. A query plan is broken at its pipeline breakers, the operators like hash join builds and aggregations that must consume all input before producing output, and each pipeline is then executed morsel by morsel across however many threads the machine has.

The storage format is shaped to serve this. Tables in DuckDB's own single-file format are stored column by column within row groups of 122,880 rows, and the row group is the natural unit of parallel scanning, so a large table presents itself as a long queue of independently scannable chunks. The same chunking is why checkpointing, compression choices, and scan parallelism all line up on the same boundaries.

Querying data where it lives

The feature that made DuckDB a habit is that it treats external files as first-class tables. A replacement scan turns 'listings.parquet' in a FROM clause into a table function call, the Parquet reader scans with projection and filter pushdown so only the referenced columns and the row groups whose statistics might match are ever read, and the same works for CSV with a sniffer that infers delimiters and types, and for glob patterns over many files. With the httpfs extension the path can be an HTTP URL or an S3 object, read with range requests rather than downloaded whole, and Arrow data in the host process is queried zero-copy. The practical effect is that a great deal of work that used to require loading data into a warehouse becomes a single process reading columns straight off files; my databases note places this pattern among the other analytical options.

Reading the repository

The C++ source under src/ is organized as the textbook pipeline, and you can read it in query order: parser, planner, optimizer, execution, with storage, catalog, transaction, and function supporting them and main holding the connection and client context that tie it together. The parser is derived from Postgres's grammar, so DuckDB speaks a thoroughly debugged SQL dialect; the planner's Binder resolves names against the catalog and produces logical operators, the optimizer rewrites them, and the execution layer turns them into the physical operators that do the work.

The two most instructive corners are common/types/, where the Vector and DataChunk types define the unit of data every operator consumes and produces, including the compressed vector representations that let constant and dictionary data flow through without expansion, and parallel/, where pipelines, events, and the task scheduler implement the morsel-driven design from the previous section. The Parquet reader lives in the extension/ tree along with JSON, httpfs, and the other bundled extensions, which is a deliberate statement that even core features go through the public extension interface. A good first session is to build the shell, run a GROUP BY over a Parquet file under EXPLAIN and then EXPLAIN ANALYZE, and match the printed pipelines and operator timings to the operator classes in src/execution/operator/; the push-based pipeline model becomes concrete as soon as you see a plan split at its hash aggregate.

Key takeaway: DuckDB is what happens when the last two decades of analytical database research, columnar storage, vectorized execution, morsel-driven parallelism, are packaged as a dependency-free library instead of a server: the hard ideas live in the engine, and the user-facing result is that SQL over a Parquet file on your laptop is one install command and one query away.
