llama.cpp

llama.cpp is the C/C++ inference engine that made running large language models on your own hardware normal. Georgi Gerganov started it in early 2023 as a weekend port of LLaMA to plain C, and it has since grown, under the ggml-org organization, into the substrate for most local AI tooling, from Ollama to the GGUF ecosystem on Hugging Face. This walkthrough covers how to run a model in five minutes, the three ideas that make local inference practical, and a guided path for reading the repository.

Using it in five minutes

You can install prebuilt binaries from a package manager, or build from source, which takes about a minute on a laptop because the project has essentially no dependencies: 

brew install llama.cpp
git clone https://github.com/ggml-org/llama.cpp
cd llama.cpp
cmake -B build
cmake --build build --config Release

Add -DGGML_CUDA=ON (or the Vulkan, SYCL, or HIP equivalent) to the first cmake call to build a GPU backend; on Apple Silicon the Metal backend is on by default. The fastest way to try a model is the -hf flag, which downloads a GGUF file straight from Hugging Face and drops you into an interactive chat: 

llama-cli -hf ggml-org/gemma-3-1b-it-GGUF

The same flag works for the server, which speaks the OpenAI API and ships a built-in web UI: 

llama-server -hf ggml-org/gemma-3-1b-it-GGUF --port 8080
from openai import OpenAI

client = OpenAI(base_url="http://localhost:8080/v1", api_key="unused")
resp = client.chat.completions.create(
    model="gemma-3-1b-it",
    messages=[{"role": "user", "content": "Hello"}],
)

The flags worth knowing first: -m points at a local GGUF file instead of a Hugging Face repo, -ngl sets how many layers to offload to the GPU (a large value like 99 means all of them), -c sets the context size, and -t sets CPU threads. Partial offload is the distinctive one: a model that does not fit in VRAM can put some layers on the GPU and run the rest on the CPU.

Idea one: ggml, the tensor library underneath

Everything in llama.cpp sits on ggml, a C tensor library that lives in-tree under ggml/ and is developed alongside it. ggml has no third-party dependencies, builds anywhere a C compiler runs, and represents a model's forward pass as an explicit static computation graph that is constructed first and executed afterwards. That sounds austere next to PyTorch, but it is the design decision the whole project flows from: because the graph is explicit and the runtime is tiny, backends for CUDA, Metal, Vulkan, SYCL, and plain CPU with SIMD can each implement the same small set of operations, and the scheduler can split one graph across several of them at once. The point of ggml is not to be the fastest tensor library on any one chip but to make the same model file run on nearly every chip, which is why the -ngl flag can slide a model between CPU and GPU layer by layer instead of demanding all-or-nothing.

Idea two: GGUF and block-wise quantization

A model for llama.cpp is a single GGUF file, a binary format that packs the architecture metadata, the tokenizer, and every tensor into one memory-mappable blob. Because the file is mmapped, loading is nearly instant and the weights live in page cache rather than being copied, and because the metadata is self-describing key-value pairs, one file works across machines and versions with no sidecar configs. You produce one with the convert_hf_to_gguf.py script at the repo root, then shrink it with llama-quantize.

Quantization is where most of the project's accumulated cleverness sits. Weights are quantized block-wise, small groups of values sharing scale factors, at rates from 8-bit down past 2-bit. The K-quants (names like Q4_K_M) use super-blocks with quantized scales and mix precision across tensor types, spending more bits where errors hurt most, and the community consensus that Q4_K_M is the size-to-quality sweet spot has held for a long time. The IQ series pushes below 3 bits and leans on an importance matrix, computed from calibration text and passed to the quantizer, to protect the weights that matter. The bet behind all of it is that memory bandwidth, not arithmetic, bounds local inference, so every bit shaved off the weights is speed as well as space.

Idea three: the server

llama-server turns the library into a self-hosted OpenAI-compatible endpoint with a web UI, and it is more capable than its size suggests: it handles concurrent requests with slot-based continuous batching, supports speculative decoding, and serves embedding and reranking models. It is worth being clear about positioning. An engine like vLLM assumes a datacenter GPU and optimizes aggregate throughput across many users, while llama-server assumes your hardware and optimizes for a handful of users at the lowest possible footprint; the right choice follows directly from which assumption matches your deployment.

Reading the repository

The layout separates cleanly into the tensor library, the model library, and the tools, and reading in that order top-down works well. Start in tools/, where llama-cli and the server live along with quantize and perplexity; the CLI's main loop is a readable tour of the public API, tokenize, decode, sample, repeat, with the shared argument and chat plumbing in common/.

The core library is the public header llama.h with its implementation under src/, split into files whose names are a map of the domain: llama-model.cpp for loading and architecture definitions, llama-vocab.cpp for tokenizers, llama-kv-cache.cpp for cache management, llama-graph.cpp for building the forward graph, and llama-sampler.cpp for the sampling chain. Below that, ggml/ holds the tensor library, with the core API in one header and each backend in its own subdirectory of ggml/src/, which is where the CUDA, Metal, and SIMD-heavy CPU kernels live; the attention kernels are a good place to apply the online softmax ideas if you want to read them closely. The GGUF side is gguf-py/ plus the conversion scripts at the root. A good first session is to convert and quantize a tiny model yourself, then run llama-cli under a debugger and step one token through graph build, backend dispatch, and sampling.

Key takeaway: llama.cpp is what you get when you optimize for reach instead of peak throughput: a dependency-free tensor library that runs the same graph on nearly any chip, a single-file model format built for mmap, and quantization schemes that trade bits for bandwidth, which together turned local LLM inference from a research stunt into infrastructure.
