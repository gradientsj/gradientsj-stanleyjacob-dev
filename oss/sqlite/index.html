SQLite

SQLite is an embedded SQL database engine, written by D. Richard Hipp and a small team, dedicated to the public domain, and very likely the most widely deployed database in the world: it ships inside every phone, every browser, and most operating systems. Development happens in Fossil at sqlite.org rather than on GitHub, with a read-only mirror at github.com/sqlite/sqlite that does not accept pull requests. This walkthrough covers how to use it in five minutes, the layered design that makes the code readable (bytecode engine, B-tree, pager and WAL), and the testing culture that makes the project trustworthy.

Using it in five minutes

You almost certainly have it already, and if not it is one package away. There is no server to start and no configuration to write; a database is a single ordinary file: 

sudo apt install sqlite3     # or: brew install sqlite
sqlite3 app.db
sqlite> CREATE TABLE notes(id INTEGER PRIMARY KEY, body TEXT);
sqlite> INSERT INTO notes(body) VALUES ('hello');
sqlite> SELECT * FROM notes;
1|hello
sqlite> .quit

From application code the story is the same because the engine is a library linked into your process, which Python ships in its standard library: 

import sqlite3

con = sqlite3.connect("app.db")
con.execute("INSERT INTO notes(body) VALUES (?)", ("from python",))
con.commit()
for row in con.execute("SELECT id, body FROM notes"):
    print(row)

And if you want to feel how self-contained the project is, download the amalgamation from sqlite.org and build the whole engine plus its command-line shell with one compiler invocation: 

gcc shell.c sqlite3.c -lpthread -ldl -lm -o sqlite3

Idea one: SQL compiles to bytecode

When you call sqlite3_prepare, SQLite does not build a tree of iterators the way most databases do. The tokenizer in tokenize.c and a parser generated by Lemon, SQLite's own parser generator, from the grammar in parse.y feed a code generator, and the output is a small program for a register machine called the VDBE, the Virtual Database Engine. A prepared statement in SQLite is literally a compiled bytecode program, and executing a query means running that program in the interpreter loop in vdbe.c. You can see this directly: prefix any statement with EXPLAIN and the shell prints the opcodes, with instructions like OpenRead, Column, Next, and ResultRow that read like a story of the query. The design buys a clean separation between the front half of the system that decides what to do and the back half that touches storage, and it makes query behavior inspectable in a way few engines match.

Idea two: a B-tree on a pager, with a WAL underneath

Below the bytecode engine the layering continues. The VDBE never touches the file; it talks to cursors provided by btree.c, which implements the one on-disk format SQLite has: tables are B-trees keyed by rowid with the row data in the leaves, and indexes are B-trees keyed by the indexed columns. The B-tree in turn never calls the operating system; it requests fixed-size pages from the pager in pager.c, and the pager is where the ACID story lives. In the classic mode it copies each page about to be modified into a rollback journal so a crash mid-transaction can be undone, and in the newer mode implemented in wal.c it does the opposite: committed changes are appended to a write-ahead log while the main database file stays untouched, which is what lets readers proceed concurrently with a writer, with a checkpoint step later folding the log back into the database. At the very bottom sits the VFS abstraction in os_unix.c and os_win.c, the porting layer that is a large part of why SQLite runs everywhere from phones to aircraft. If you want the comparison view of when an embedded single-file engine is the right tool against a server database, my databases note covers that trade-off.

Idea three: the amalgamation and the testing culture

Two habits of the project matter as much as its architecture. The first is the amalgamation: for release, over a hundred separate source files are concatenated into one big sqlite3.c of roughly a quarter million lines, which sounds like a joke until you see the effect, since a whole database engine becomes two files you drop into any build system, and the project measures the single-translation-unit build as five to ten percent faster because the compiler can inline across module boundaries. The second is the testing culture, which is the real moat. The project reports hundreds of times more test code than library code, and the proprietary TH3 harness achieves one hundred percent branch coverage and one hundred percent MC/DC coverage of the deployed configuration, the standard used for avionics software. On top of that sit the public TCL test suite, the SQL Logic Test runs that compare results against other databases across millions of queries, and continuous fuzzing on the order of a billion mutations per day. The lesson generalizes: SQLite can afford aggressive internal change, and users can afford blind upgrades, because the tests are the specification.

Reading the repository

Read the tree, not the amalgamation: clone the GitHub mirror and work in src/, which holds around a hundred C files with the architecture document at sqlite.org/arch.html as your map. The layering above is also the reading order. Start with main.c and prepare.c to see the public API and how a statement becomes a VDBE program, glance at tokenize.c and the grammar in parse.y, then look at the code generators, build.c for DDL, select.c and where.c for queries, where the query planner's index selection logic lives. Then read the heart of the machine, the giant opcode switch in vdbe.c, with the opcode documentation generated straight from its comments. Descend into btree.c next, which rewards reading alongside the file format documentation, and finish with pager.c and wal.c, where commit, rollback, and checkpointing actually happen. The command-line shell is shell.c.in, a surprisingly good single-file example of using the API.

A good first session is an evening with the shell and EXPLAIN: write a join, read its bytecode, add an index, and watch the program change shape. Because every layer has a stable interface and a matching document on sqlite.org, SQLite is arguably the best codebase in the world for learning how a database actually works.

Key takeaway: SQLite is a compiler stacked on a storage engine: SQL becomes bytecode, bytecode drives B-tree cursors, the B-tree rides on a transactional pager, and a once-in-an-industry testing regime is what lets that little stack run unattended on billions of devices.
