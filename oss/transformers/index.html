Transformers

Transformers is Hugging Face's library of pretrained model definitions, the de facto standard interface between the research world that publishes checkpoints and everyone who wants to run them, covering hundreds of architectures across text, vision, audio, and multimodal models and serving as the reference implementation that inference engines and training stacks build against. This walkthrough covers how to use it in five minutes, how the Auto classes turn a name on the Hub into a working model, the deliberately repetitive modeling-file philosophy that makes one repository able to hold all those architectures, and a guided path for reading the code.

Using it in five minutes

Install the library together with a backend, which for almost everyone means PyTorch: 

pip install "transformers[torch]"

The highest-level door is pipeline, which bundles preprocessing, the model, and postprocessing behind a task name: 

from transformers import pipeline

generator = pipeline(task="text-generation", model="Qwen/Qwen2.5-1.5B-Instruct")
print(generator("The secret to a good walkthrough is ")[0]["generated_text"])

One level down is the pairing you will actually use in real code, a tokenizer and a model loaded by name, with generation invoked explicitly so you control the knobs: 

from transformers import AutoModelForCausalLM, AutoTokenizer

tok = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-1.5B-Instruct")
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-1.5B-Instruct", dtype="auto", device_map="auto"
)

messages = [{"role": "user", "content": "Explain attention in two sentences."}]
inputs = tok.apply_chat_template(
    messages, add_generation_prompt=True, return_tensors="pt"
).to(model.device)
out = model.generate(inputs, max_new_tokens=128)
print(tok.decode(out[0][inputs.shape[-1]:], skip_special_tokens=True))

The same from_pretrained call works for every architecture in the library, downloads and caches weights from the Hub on first use, and accepts a local path just as happily, which is the entire user-facing contract in one method.

Idea one: the Auto classes and config-driven loading

The library's central trick is that a checkpoint carries its own identity. Every model repo on the Hub contains a config.json whose model_type field names the architecture, so when you call AutoModelForCausalLM.from_pretrained the library first fetches the config, reads that field, looks it up in a registry mapping model types to concrete classes, and only then instantiates the right one, say LlamaForCausalLM, and loads the weights into it. The Auto classes are a dictionary lookup wearing a trench coat, and that is precisely what makes them durable: adding a new architecture means adding a registry entry, not changing any caller's code.

The same pattern repeats across every artifact type, tokenizers, image processors, feature extractors, and across every task head, so AutoModelForSequenceClassification and friends are just different registries over the same config resolution. The consequence users feel is that model choice becomes a string you can put in a config file: swapping Qwen for Llama in an application is a one-line change, and code written before an architecture existed can load it the day it lands.

Idea two: repeat yourself, on purpose

Open two modeling files, say Llama's and Mistral's, and you will find long stretches of near-identical code, attention classes, MLP blocks, rotary embedding helpers, repeated in each file rather than imported from a shared base. In most codebases this would be a defect; here it is the stated design. The philosophy, sometimes summarized as a single-model-file policy, holds that a researcher should be able to understand and hack on one model by reading one file top to bottom, without spelunking through a tower of abstractions, because machine learning moves too fast for shared abstractions to survive: yesterday's elegant base class becomes today's obstacle the moment a new architecture breaks its assumptions. Repetition contains the blast radius, since modifying one model cannot break three hundred others.

The maintenance cost is managed mechanically rather than by abstraction. Duplicated blocks carry # Copied from comments naming their source, and CI checks that the copies have not drifted, so a fix to the original either propagates or fails the build. More recently the library added modular model definitions, small files that declare a new model as deltas against an existing one and are then expanded by a tool into the full self-contained modeling file, which keeps the authoring convenient while preserving the readable flattened artifact. It is worth sitting with how contrarian this is: the library chose reviewability and hackability over DRY, at the scale of hundreds of architectures, and it has held up for years.

Idea three: pipelines and Trainer, the layers above

Model definitions alone would leave every user rewriting the same glue, so the library ships the two layers most people need. Pipelines wrap a preprocessing step, a model forward, and a postprocessing step behind a task name, and their value is less the convenience than the standardization: every task has a defined input and output shape, so tools can compose them without knowing which architecture sits inside. Trainer is the training counterpart, a full loop with mixed precision, gradient accumulation, checkpointing, evaluation, logging, and distributed execution handled for you, configured through a TrainingArguments dataclass and extended by overriding methods like compute_loss rather than by rewriting the loop. Around it has grown an ecosystem division of labor worth knowing before you read any code: datasets live in the datasets library, parameter-efficient fine-tuning in peft, RLHF-style training in trl, and multi-device orchestration in accelerate, with Transformers itself staying the home of model definitions. That is also why serving engines treat it as the source of truth; vLLM, covered in its own walkthrough, loads Hub checkpoints and configs in Transformers' formats while replacing the execution stack underneath.

Reading the repository

Nearly everything that matters is under src/transformers/, and the layout mirrors the ideas above closely enough that you can navigate by concept. Each architecture owns a directory under src/transformers/models/ holding its configuration file, its modeling file, and where needed its tokenizer and processor files, all following the same naming pattern, so models/llama/modeling_llama.py is the shape of every other model in the library and reading one modern decoder-only model carefully means you have effectively read dozens.

Start there, with one modeling file read top to bottom: the attention class, the decoder layer, the bare model that stacks layers, and the task-headed class that wraps it. Then read the machinery it inherits. PreTrainedModel in src/transformers/modeling_utils.py owns from_pretrained, weight loading, dtype and device placement, and is the file to read when loading behaves surprisingly. The Auto registries live in src/transformers/models/auto/, where the mapping from model_type to class is plainly visible. Text generation, the loop that model.generate runs with its sampling strategies and stopping criteria, lives in src/transformers/generation/, and it is where to look when generation parameters do not do what you expect. Pipelines sit in src/transformers/pipelines/, one file per task over a shared base, and the training loop is src/transformers/trainer.py, a long file best read by following train into its inner loop and ignoring the callback plumbing on a first pass. A good first session is to pick one small checkpoint, set a breakpoint in its modeling file's forward, and step through a single generate call from tokenizer to sampled token; the library is unusually rewarding to read precisely because the philosophy above means the code in front of you is the whole story.

Key takeaway: Transformers is a registry of hundreds of deliberately self-contained model files behind one loading interface. The Auto classes make every checkpoint loadable by name, and the repeat-yourself philosophy makes every architecture readable in one sitting, which together is why a single repository can be both the community's model zoo and its reference implementation.
