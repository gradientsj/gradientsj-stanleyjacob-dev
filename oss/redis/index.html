Redis

Redis is an in-memory data structure server, created in 2009 by Salvatore Sanfilippo and stewarded today by Redis Ltd, which after a period under source-available licenses is offered under the AGPLv3 again as of Redis 8. It is the default cache, queue, and leaderboard of the industry, and it is also one of the most readable C codebases you will ever open. This walkthrough covers how to use it in five minutes, the single-threaded event loop, the object system and its space-saving encodings, how persistence works from a process that never wants to block, and a reading path that follows one command from socket to reply.

Using it in five minutes

Install the server from your package manager or run the official container, start it, and talk to it with the bundled client: 

sudo apt install redis-server   # or: brew install redis
redis-server                    # or: docker run -p 6379:6379 redis
$ redis-cli
127.0.0.1:6379> SET user:1000:name "Ada"
OK
127.0.0.1:6379> GET user:1000:name
"Ada"
127.0.0.1:6379> EXPIRE user:1000:name 60
(integer) 1
127.0.0.1:6379> INCR pageviews
(integer) 1
127.0.0.1:6379> ZADD leaderboard 3172 ada 2891 lin
(integer) 2
127.0.0.1:6379> ZRANGE leaderboard 0 -1 WITHSCORES
1) "lin"
2) "2891"
3) "ada"
4) "3172"

That session already shows the shape of the product: strings with expiry give you a cache, atomic counters give you rate limiting, and sorted sets give you a leaderboard, each a data structure with commands rather than tables with queries. The protocol is plain text over TCP, so every language has a client, and redis-cli MONITOR in a second terminal is the best free tour of what an application actually asks its cache to do.

Idea one: one thread and an event loop

The core of Redis is a single thread running an event loop, written from scratch in ae.c in a few hundred lines over epoll, kqueue, or select depending on the platform. Every client socket is registered with the loop, and when data arrives the handler reads a command, executes it, and queues the reply, one command at a time for the whole server. This sounds like a limitation until you notice what it buys: because commands run one at a time against data that lives entirely in memory, every command is atomic by construction and the code has no locks, no data races, and no lost wakeups to reason about. The design works because the workload cooperates: typical operations are O(1) or O(log n) touches of RAM taking microseconds, so the bottleneck is the network rather than the CPU, and modern Redis lets extra I/O threads handle reading and writing sockets while command execution stays single-threaded. The famous corollary is that one slow command, a KEYS * against a big keyspace, stalls everyone behind it, which is why the documentation is so insistent about avoiding O(n) commands in production. When one box is not enough you shard across many, which is Redis Cluster's job and the subject of my distributed cache design write-up.

Idea two: the object system and its encodings

Every value in Redis is a robj, a small header defined in server.h and managed in object.c, carrying a type, strings, lists, sets, hashes, sorted sets and more, and separately an encoding, the concrete representation currently in use. That separation is the trick: a type keeps one command interface while swapping its underlying data structure based on size, so small collections are stored as a listpack, a compact serialized buffer scanned linearly, and converted to a real hash table or skiplist only when they grow past configured thresholds. A small hash is therefore a few contiguous bytes rather than a pointer-chasing table, a list is a quicklist, a linked list whose nodes are listpacks, and a large sorted set becomes the classic pairing of a skiplist for order plus a dict for O(1) score lookup, implemented in t_zset.c. The same economy shows up in strings, which come in an integer encoding, an embedded short-string encoding allocated inline with the header, and a general raw encoding built on sds.c, Redis's length-prefixed dynamic strings. You can watch all of this from the client with OBJECT ENCODING key, growing a hash element by element until it flips representation, which is the cheapest systems lesson available anywhere.

Idea three: persistence without blocking

An in-memory server still has to survive restarts, and Redis offers two answers with one shared trick. RDB, in rdb.c, is a point-in-time binary snapshot of the whole dataset, and AOF, in aof.c, is a log of every write command, replayable at startup and periodically rewritten into a compact form, with fsync policy configurable between every write, every second, or never. The shared trick is fork(): for a background save the process forks, and the child walks a frozen copy of the dataset provided for free by the kernel's copy-on-write pages while the parent keeps serving traffic, at the cost of extra memory as pages diverge under write-heavy load. The practical posture most deployments land on is AOF with everysec fsync for durability plus RDB snapshots for fast restarts and backups, and the same snapshot machinery doubles as the transfer format when a replica first synchronizes.

Reading the repository

Everything lives flat in src/, and the right way in is to follow one command. Start in server.c, where main() loads the config, calls the server initialization that creates the event loop and listening sockets, and enters aeMain(); read ae.c in full, it is short and self-contained with the platform backends in ae_epoll.c and friends. Then trace a SET: the readable event fires readQueryFromClient() in networking.c, the RESP protocol parser builds an argument vector, and processCommand() in server.c looks up the command in a table that recent versions generate from the JSON specs under src/commands/, checks ACLs and memory, and dispatches to setCommand() in t_string.c, which stores the value through the keyspace dict from dict.c and queues the reply back in networking.c. The type files t_list.c, t_hash.c, t_set.c, and t_zset.c all follow the same pattern over the structures in listpack.c, quicklist.c, and dict.c, whose incremental rehashing is a gem worth reading on its own. With that spine in place, rdb.c, aof.c, replication.c, and the background thread pool in bio.c each read as an extension of a loop you already understand.

Key takeaway: Redis is a demonstration that doing one thing at a time can be a performance strategy: a single-threaded loop over in-memory data structures makes every command atomic and lock-free, adaptive encodings make those structures cheap, and fork with copy-on-write lets durability happen off to the side without ever stopping the loop.
