PyTorch

PyTorch is the dominant open source deep learning framework, started at Meta AI and now governed by the PyTorch Foundation under the Linux Foundation, and it is the substrate that most of modern ML research and a large share of production training runs on. This walkthrough covers how to use it in five minutes, what actually happens when you call loss.backward(), the dispatcher that sits under every operator, a high-level view of torch.compile, and a guided path for reading a repository that can feel bottomless.

Using it in five minutes

Installation is one line, and the selector on pytorch.org will give you the right variant for your CUDA version if you need a specific one: 

pip install torch

The whole training idiom fits in a screen. You define a module, run a forward pass, compute a loss, call backward to populate gradients, and let an optimizer apply them: 

import torch
import torch.nn as nn

model = nn.Sequential(nn.Linear(784, 128), nn.ReLU(), nn.Linear(128, 10))
opt = torch.optim.AdamW(model.parameters(), lr=1e-3)
loss_fn = nn.CrossEntropyLoss()

x = torch.randn(64, 784)
y = torch.randint(0, 10, (64,))

logits = model(x)
loss = loss_fn(logits, y)
loss.backward()
opt.step()
opt.zero_grad()

Since the 2.0 release the same model can be handed to the compiler with one call, and the eager code above keeps working unchanged: 

compiled = torch.compile(model)
logits = compiled(x)

Everything else, moving to GPU with .to("cuda"), mixed precision with torch.autocast, distributed training with torch.distributed, is layered on top of this loop rather than replacing it, which is a large part of why the framework won.

Idea one: autograd is a tape recorded at runtime

PyTorch's original bet was define-by-run. There is no separate graph-construction phase; your model is ordinary Python, and the graph that backpropagation needs is recorded as a side effect of executing the forward pass. Every time an operator runs on a tensor that requires gradients, the output tensor gets a grad_fn pointing at a node that knows how to compute that operator's derivative, and each node holds edges to the nodes that produced its inputs. By the time you have a scalar loss, you are holding the root of a directed acyclic graph that describes, in reverse, everything the forward pass did.

So loss.backward() is not magic, it is a graph traversal. The autograd engine seeds the root with a gradient of one, then walks the graph from the loss backwards, calling each node's backward function to transform the incoming gradient and passing the results along the edges, using dependency counts so a node fires only after all gradients flowing into it have arrived. At the leaves, the tensors you created with requires_grad=True, special accumulator nodes add the arriving gradient into the .grad field, which is why gradients accumulate across calls and why training loops call zero_grad(). The forward pass writes the tape and backward replays it in reverse, so control flow, loops, and data-dependent branches differentiate for free because the tape only ever records what actually ran. The derivative formulas themselves are not written by hand at every call site; they are declared once per operator in a YAML file and code generation produces the graph-node classes, which is a pattern you will meet again reading the repo.

Idea two: the dispatcher

The second load-bearing idea is less visible but explains most of the repository's shape. A call like torch.add has to end up in very different code depending on whether the tensors are on CPU or CUDA, whether they are dense or sparse, whether autograd needs to record the call, whether autocast wants to change the dtype, and whether something like torch.vmap or a quantization wrapper is active. PyTorch resolves this with a dispatcher: every tensor carries a set of dispatch keys, every operator has a table of kernels registered per key, and each call finds the highest-priority relevant key and jumps to its kernel. Crucially the layers stack, so the autograd kernel does its recording and then re-dispatches to the device kernel below it, the way nested virtual method calls would.

Think of the dispatcher as an extensibility point rather than a switch statement: features like autograd, autocast, functorch transforms, and out-of-tree backends are all just dispatch keys, which is how they compose without knowing about each other. The operator inventory lives in a single schema file, native_functions.yaml, that declares every native operator's signature and which kernel implements it per backend, and code generation turns those declarations into the registration glue. Once you know this, the answer to "where is the code for operator X" stops being a mystery: find its entry in the YAML, and the entry names the C++ functions to read.

Idea three: torch.compile, at altitude

Eager execution pays a price: every operator is a separate kernel launch with Python overhead between launches, and no one gets to optimize across operator boundaries. torch.compile attacks this without giving up define-by-run, and it is best understood as a pipeline of three named components. TorchDynamo hooks Python frame evaluation and symbolically walks your function's bytecode, extracting the tensor operations into a graph and installing guards, cheap runtime checks on things like shapes and dtypes, so the captured graph is reused only when it is still valid; when it hits Python it cannot capture, it breaks the graph and falls back to eager for that piece. AOTAutograd then traces ahead-of-time through the autograd machinery so the backward pass becomes a graph too, not just the forward. Finally TorchInductor compiles the graphs, generating Triton kernels on GPU and C++ with OpenMP on CPU, fusing chains of pointwise operations into single kernels so intermediate tensors never touch memory. The design goal running through all three is graceful partiality: unsupported code degrades to eager instead of failing, which is what makes the one-line adoption story honest.

Reading the repository

The repo is one of the largest in open source, so do not read it breadth-first; follow one operator call and one backward call, and let the layers reveal themselves. A useful map first: pure Python lives under torch/, the C++ tensor and operator library is ATen under aten/src/ATen/, the small core that both depend on (TensorImpl, dispatch keys, device abstractions) is c10/, and the C++ that binds it all to Python, autograd engine included, is torch/csrc/.

Start where users start, in torch/nn/modules/, which is plain readable Python: Module with its parameter registration and hooks, then a layer like Linear that bottoms out in a call to a functional op. Chase that op downward into ATen: the schema entry in native_functions.yaml under aten/src/ATen/native/, then the CPU and CUDA kernels in the same directory tree. Next read the autograd half: tools/autograd/derivatives.yaml declares each operator's derivative, and the engine that executes the recorded graph is in torch/csrc/autograd/, where engine.cpp is long but rewards a careful read, since the ready queue and dependency counting are exactly the traversal described above. Then the compiler stack in Python: torch/_dynamo/ for capture and guards and torch/_inductor/ for code generation, with torchgen/ explaining how the YAML files become generated C++. A good exercise is to pick one tiny operator and trace it through all four layers, Python surface, dispatcher registration, kernel, derivative; after one full pass the repository stops feeling bottomless because every file you open slots into one of those layers. If you want the numerical side of what these frameworks compute, the ML implementations section builds several of the core pieces from scratch.

Key takeaway: PyTorch is a runtime-recorded autograd tape sitting on an extensible operator dispatcher, with a compiler bolted on top that captures graphs opportunistically instead of demanding them up front. Learn those three layers and the enormous repository resolves into a small number of ideas executed at industrial scale.
