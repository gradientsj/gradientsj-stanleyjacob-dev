llm-eval-harness

An LLM-as-judge evaluation harness with human calibration and two-tier CI quality gating, demonstrated on grounded QA. This article covers what the harness does, how to install and run it, and how to wire its gates into a CI pipeline. The code is at github.com/gradientsj/llm-eval-harness.

What it is and why it exists

Evaluating open-ended LLM output is an unsolved problem with immediate product consequences: you cannot ship what you cannot measure, and you cannot trust an auto-judge you have never calibrated against humans. The harness is a small, fully tested implementation of the loop that makes auto-judges trustworthy. A documented rubric drives an LLM judge; the judge's agreement with human labels is computed on a benchmark and enforced as a gate; and only a judge that passes calibration is allowed to compare a candidate release against a frozen baseline. Both gates run in CI on every push.

Three design commitments are worth stating up front. First, the judge is gated on its agreement with humans, not assumed correct: judge-versus-human agreement (Cohen's kappa, quadratic-weighted kappa, Spearman, MAE) is computed on a labeled benchmark, and a prompt or model change that degrades agreement below threshold fails the build. Second, a deterministic lexical judge is the control arm: it scores groundedness by content-word overlap, runs the entire pipeline in CI with zero secrets, and sets the floor an LLM judge has to beat to justify its cost. Third, the benchmark is adversarially composed: a third of the examples are constructed traps, negation flips, fact recombinations, and correct refusals, where shallow scoring is known to disagree with humans.

Install and quickstart

The only prerequisite is uv. No API key is needed for the default lexical backend, so the whole loop runs locally in a minute: 

uv sync --extra dev
uv run pytest                                        # 45 tests
uv run evalharness calibrate --backend lexical --check-gates
uv run evalharness baseline  --backend lexical       # freeze the baseline
uv run evalharness gate      --backend lexical       # regression-gate it

To run the real judge, set ANTHROPIC_API_KEY (copy .env.example to .env) and switch backends: 

uv run evalharness calibrate --backend anthropic --check-gates

This makes about 30 calls at temperature 0 against a pinned model snapshot, overridable via EVALHARNESS_JUDGE_MODEL. Pinning matters: an unpinned judge model is a moving measurement instrument, and the calibration gate exists precisely to catch it drifting.

The two-tier CI gate

The workflow enforces both gates on every push and pull request, after lint and unit tests. Tier 1, the calibration gate, fails the build if judge-versus-human agreement drops below thresholds (kappa at least 0.30 on the ship/no-ship decision, QWK at least 0.20 on groundedness). This catches rubric edits, judge-prompt regressions, and judge-model drift. Tier 2, the regression gate, re-scores a candidate answer set and compares aggregates against the frozen baseline with explicit tolerances (pass rate may drop at most 0.02, mean groundedness at most 0.15). Tier 2 only runs if Tier 1 passed, because a regression verdict from an uncalibrated judge is noise.

The repo ships a simulated bad release, the same 30 questions with eight answers replaced by fluent fabrications, to demonstrate the gate catching a real quality drop: 

$ uv run evalharness gate --backend lexical \
      --candidates data/grounded_qa_regressed.jsonl
  [FAIL] pass rate: baseline 0.600, current 0.333, allowed drop 0.020
  [FAIL] mean groundedness: baseline 3.733, current 2.767, allowed drop 0.150
Regression gate FAILED - candidate quality dropped below baseline tolerance.

Note that the candidate file carries no human labels. Gating fresh model output must not require annotation; only the separately calibrated judge is needed.

What the lexical baseline measures

Judge-versus-human agreement of the word-overlap baseline on the 30-example seed benchmark:

DimensionQWKSpearmanMAEExactWithin ±1
groundedness0.2560.3051.300.470.67
relevance0.5050.3800.830.430.80
coherence0.7250.6670.230.830.93

The gradient is the finding: coherence is nearly solvable with surface features, relevance is partially solvable, and groundedness is not. The failure analysis in the repo shows the three mechanisms. Negation flips score high overlap against a context that asserts the opposite, because overlap is blind to polarity. Fact recombination attaches real facts from the context to the wrong entity, so every word is "grounded" while the claim is false. And correct refusals, where the right answer is that the context does not say, are punished by lexical scoring for having near-zero overlap, which is exactly the behavior groundedness training tries to encourage. These numbers, kappa 0.40 on ship/no-ship and QWK 0.26 on groundedness, are the floor a model judge has to beat to be worth its latency and cost, and the harness is how you check that it does.

Where things live in the repo

The rubric is a single source of truth in src/evalharness/rubric.py, embedded in both the judge prompt and the annotator instructions (data/ANNOTATION_GUIDELINES.md). The benchmark and the frozen baseline live under data/, calibration reports under reports/, and the CI wiring in .github/workflows/eval-gate.yml. To adapt the harness to your own task, replace the benchmark examples and rubric dimensions, re-label, re-calibrate, and keep the gates.

Key takeaway: an LLM judge is a measurement instrument, and instruments get calibrated. Gate the judge on human agreement first, then let it gate your releases, and keep a deterministic baseline in the loop so you always know what the model judge is actually buying you.
