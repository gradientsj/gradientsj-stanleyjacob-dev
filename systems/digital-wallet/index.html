Design a digital wallet

Systems design · Reservations and money · Nov 2025

A digital wallet gives users a stored-value account: a balance held by the platform that they top up from a card or bank account and spend or send instantly, the way a PayPal balance, a Venmo account, or an Alipay wallet works. Transfers between two wallet users never touch a bank, since both balances live in the same system, which is exactly why they can be instant and why the system holding them must be incorruptible. The interview phrasing usually attaches an aggressive number, a million transfers per second, and the question is really asking two things at once: can you scale a money store past a single database, and do you understand that scaling it multiplies the ways money can go missing.

The hard parts follow from that pairing. A transfer that debits one account and credits another is trivial inside one database and genuinely subtle across two, so the design hinges on how accounts are sharded and what protocol moves money between shards. And because the data is money, every mechanism has to leave an audit trail good enough to reconstruct any balance at any moment, which is what pushes the design toward an event-sourced core rather than a mutable table of numbers.

Scope and requirements

Functionally the wallet supports balance accounts, peer-to-peer transfers, top-ups from external payment methods, withdrawals back out, and a complete transaction history per account. Know-your-customer and anti-money-laundering obligations, the identity verification and monitoring that regulators require, are acknowledged as gates on what an account may do rather than designed in depth here, and the card-charging side of top-ups is delegated to a payment service provider.

The invariants deserve to be stated plainly, because they are the actual requirements. No balance may go negative. Money is never created or destroyed by a transfer, only moved, so the sum across all accounts changes only at top-up and withdrawal boundaries. Every movement must be attributable to a command from an identified actor. And history must be replayable for audit: given the record, an examiner can re-derive any balance at any past moment. Latency should be sub-second and the headline throughput is 1 million transfers per second, which the sizing below treats seriously rather than waving at.

Sizing the problem

Start with what one machine can do. A well-tuned relational database executing small transactions with synchronous durability, meaning each commit is on disk before it is acknowledged, manages somewhere in the range of 5,000 to 10,000 such transactions per second. Taking the generous end, 1,000,000 divided by 10,000 is 100 databases at full saturation, so a realistic deployment needs 200 or more shards once headroom, replication, and uneven load are budgeted. That gap of two full orders of magnitude between the requirement and a single node is the whole reason this design exists.

Sharding creates the next problem by arithmetic alone. If accounts are spread across 200 shards and transfer pairs are roughly random, the chance both parties land on the same shard is about 1 in 200, which means 99.5 percent of transfers cross shards. The cross-shard path is therefore the common case, not the corner case, and it has to be fast and safe. The event volume is equally sobering: 1 million transfers per second is 86.4 billion events per day, and at around 120 bytes per event that is roughly 10 TB of log per day, so the log must tier into cheap object storage while staying replayable. Even a skeptic who reads the million as a burst peak with a tenth of that sustained still faces tens of shards and terabytes of history.

The API

Amounts are integers in minor units, and every mutating call carries a client-generated operation ID, the wallet's idempotency key, so that a retried request applies once. Transfers are accepted and then settle through the machinery below, which is why the response is a 202 with a pending status rather than an instant final answer.

POST /api/v1/transfers
Operation-Id: 0d9f3b71-2c44-4f8e-9a01-8be2f6c0d5a3
{ "from_account": "acc_211", "to_account": "acc_870",
  "amount": 2500, "currency": "usd" }
→ 202 { "transfer_id": "tr_5512", "status": "pending" }

GET  /api/v1/transfers/tr_5512        → 200 { "status": "completed" }
GET  /api/v1/accounts/acc_211/balance → 200 { "available": 1175, "reserved": 0 }
POST /api/v1/topups
Operation-Id: 77c1a9e4-5b02-4d33-8f1c-09d4e2b6a7f0
{ "account": "acc_211", "amount": 5000, "payment_method": "pm_card_tok" }

The data model, starting naive

The baseline everything else is measured against is one database with an accounts table and a transactions table. A transfer is a single ACID transaction that debits one row, credits another, and records what it did, and on one node this is simply correct: the balance check and the debit are atomic, the constraint stops overdrafts, and crash recovery is the database's problem.

CREATE TABLE accounts (
  account_id  BIGINT PRIMARY KEY,
  balance     BIGINT NOT NULL CHECK (balance >= 0),   -- minor units
  updated_at  TIMESTAMPTZ NOT NULL
);

BEGIN;
UPDATE accounts SET balance = balance - 2500
 WHERE account_id = 211 AND balance >= 2500;          -- 0 rows = insufficient funds
UPDATE accounts SET balance = balance + 2500
 WHERE account_id = 870;
INSERT INTO transactions (op_id, from_acct, to_acct, amount)
VALUES ('0d9f3b71...', 211, 870, 2500);               -- unique op_id dedupes retries
COMMIT;

Everything that follows is an attempt to keep these exact guarantees while the two UPDATE statements move to different machines. Naming that explicitly in the interview frames every later choice as a recovery of something the single node gave for free.

The high-level architecture

Accounts are partitioned by account ID across shards, and each shard is not one database but a small replicated group whose nodes agree on an ordered command log. Balances are derived from per-shard event logs, with read models built off those logs serving queries, and snapshots keeping recovery fast.

Commands route by account ID to a shard, each shard is a Raft-replicated state machine appending to its own event log, and projections off the logs build read models and snapshots. Dashed arrows are asynchronous.

Cross-shard transfers

A transfer between accounts on different shards is a distributed transaction, an operation that must appear atomic even though it spans machines that can fail independently, and the protocol choices are worth comparing through their failure stories rather than their happy paths.

Two-phase commit, or 2PC, is the database-level answer: a coordinator asks every participant to prepare, meaning each durably promises it can commit and holds locks while it waits, and once all vote yes the coordinator tells them to commit. The failure story is the famous hazard: if the coordinator crashes between the phases, participants are left in doubt, holding locks on account rows, unable to either commit or roll back until the coordinator returns. At hundreds of thousands of transfers per second, a blocked participant freezes a slice of all accounts on its shard, which is why high-volume money systems tend to avoid 2PC on the hot path.

Try-confirm-cancel, or TC/C, moves the transaction up into the application as two or three ordinary local transactions. The try step reserves funds on the debit side, moving 25.00 from available into a reserved bucket on shard A, a real committed row visible to audit. The confirm step applies the transfer, finalizing the debit on A and crediting the account on B. If anything fails after a successful try, the cancel step releases the reservation and the money never moved. Each step is idempotent and journaled, so a crashed coordinator is recovered by a scanner that finds in-flight transfers and drives them forward or backward; nothing blocks, because every intermediate state is a committed, legible business state rather than a held lock. The debit side always goes first, since the failure mode of credit-first ordering is a window where the recipient can spend money that was never successfully taken from the sender, which creates money out of a crash.

A saga generalizes the same idea: a sequence of local transactions where each step has a compensating action that undoes it if a later step fails. TC/C is essentially a two-step saga specialized for funds, and the specialization matters because reserving before spending is what preserves the no-negative-balance invariant mid-flight. The cost relative to 2PC is that the application owns the protocol, the recovery scanner, and the reserved-funds bookkeeping, and that users can briefly see a reserved amount on their balance; the gain is that no machine failure can leave locks pinned across the fleet.

Step 1: the client requests a transfer. Step 2, try: the coordinator reserves the amount on the debit shard. Step 3, confirm: it applies the credit on shard B and finalizes the debit. Step 4, the dashed cancel branch: any failure after the try releases the reservation, and no money moved. Step 5: the client sees the outcome.

Event sourcing as the backbone

Event sourcing means the balance is not a stored number but the fold of an append-only event log: the log records facts such as "transfer tr_5512 reserved 2500 from acc_211" and "tr_5512 confirmed," and the current state of any account is what you get by applying its events in order. Because replaying 86 billion events to answer a balance query would be absurd, each shard takes periodic snapshots, materialized states as of a log position, and a read is the latest snapshot plus the short tail of events after it.

What this buys is exactly what the invariants demanded. Audit comes free, because the log is the audit trail rather than a reconstruction of it. Recovery is deterministic replay: a node that crashes reloads its snapshot, replays the tail, and lands in provably the same state. Disputes get time travel, since "what was this balance last March 3rd at noon" is a fold up to a log position rather than a forensic project. What it costs is equally real: every queryable view, balances, histories, and statements, is a read model that must be built from the log and kept in sync, queries against those models are eventually consistent by some lag that must be monitored, and the log lives forever, which at 10 TB per day means aggressive tiering into object storage and a schema for events that can evolve without breaking replay of years-old records.

Consensus, idempotency, and risk controls

Each shard must itself survive machine failure without losing an acknowledged cent, and the standard tool is a replicated state machine: commands are sequenced through a consensus log using a protocol such as Raft, which gets a small cluster of replicas to agree on a single ordered log even when a minority of them fail. Every replica applies the same commands in the same order to the same deterministic logic, so every replica holds the same balances, and when a leader dies an election promotes a follower that already has the full sequence. The shard's event log and its Raft log are naturally the same ordered stream, which is a pleasing economy: consensus, durability, and audit share one structure.

Idempotency rides inside the state machine. Every command carries the client's operation ID, and the dedupe table mapping operation IDs to outcomes is itself part of the replicated state, so a retried transfer applies once even if the retry lands on a new leader after a failover; the new leader has the same dedupe table, because it has the same log. Risk controls sit on the command path before anything is appended: velocity checks, which are limits on rate and volume per account per time window, such as at most 50 transfers or 10,000.00 in a rolling hour, reject or hold commands synchronously, and KYC status gates capabilities, so an unverified account might receive but not send above a small threshold. Slow fraud models run asynchronously off the event stream and freeze accounts after the fact, which is a deliberate trade: the command path stays fast, and the freeze event is just another auditable command.

Scaling, failures, and operations

Shards scale by splitting account ranges: moving an account is replaying its events to the destination shard, pausing it briefly to drain in-flight operations, and cutting the routing table over, all of which event sourcing makes mechanical. Hot accounts are the asymmetry to watch, since a major merchant receiving thousands of credits per second can saturate its shard; the standard treatments are spreading a merchant across sub-accounts that aggregate in a read model, and batching micro-credits before they hit the state machine. The gateway and coordinators are stateless and scale flat; the recovery scanner that drives stuck TC/C transfers forward runs continuously, and its queue depth is a first-class health metric.

Failures map to short stories. A shard leader dies and Raft elects a new one in hundreds of milliseconds, with no acknowledged command lost because acknowledgment required replication. A coordinator dies mid-transfer and the scanner finds the journaled try without a confirm, and either confirms or cancels by policy and timeout. Read models lag and the balance endpoint serves slightly stale answers with the lag surfaced in monitoring, while commands keep validating against the authoritative shard state. The monitor that matters most is the conservation check, a continuous job folding per-shard sums plus in-flight reserves and comparing against the expected global total, because a nonzero drift there means an invariant broke somewhere, and finding it within minutes instead of at quarter close is the difference between a fix and an incident report.

Follow-up questions

  • Why not one big database on bigger hardware? The arithmetic gap is two orders of magnitude: 10,000 durable transactions per second per node versus 1,000,000 required. Vertical scaling buys perhaps one order, and a single node is also a single blast radius for all funds.
  • Why TC/C over two-phase commit? 2PC holds locks while prepared and blocks if the coordinator dies, which at this rate freezes slices of the fleet. TC/C's intermediate states are committed, auditable rows, and recovery is a scanner, not a lock-resolution outage.
  • What if the confirm on the credit shard keeps failing? Retry it, since a credit to a valid account cannot legitimately be refused once the try committed; if the account itself is gone, cancel and release the reservation. The transfer journal makes either path resumable.
  • How does a retried transfer apply exactly once across failover? The operation-ID dedupe table is part of the replicated state machine, so any leader that could process the retry necessarily has the record of the first application.
  • What does event sourcing actually cost? Read models to build and monitor, eventual consistency on queries, a log that grows about 10 TB per day at the headline rate, and event schema evolution that must keep years-old records replayable.
  • Where do fraud and compliance checks run? Velocity limits run synchronously on the command path before append; heavier scoring runs asynchronously off the event stream and issues freeze commands, keeping the hot path fast while every intervention stays auditable.
