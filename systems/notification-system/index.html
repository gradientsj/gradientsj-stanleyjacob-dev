Design a notification system

Systems design · Social and messaging · Jun 2025

A notification system is the platform service every other team calls when they need to reach a user outside the app: mobile push for the order that shipped, SMS for the login code, email for the weekly digest. It deserves to be its own system rather than a library each team embeds, because the hard parts are shared: integrations with external providers that rate limit and fail in their own ways, a registry of device tokens that rot constantly, user preferences and quiet hours that must be enforced everywhere, deduplication so retries never double-send, and analytics that tell product teams whether anyone actually opened the thing. Centralizing those concerns once is the whole argument for the design, and interviewers ask the question because it is a clean showcase of asynchronous pipelines, retries, and the discipline of treating third parties as unreliable.

Scope and requirements

Functionally, internal producer services submit notification requests referencing a template and a payload, and the platform delivers through three channels: mobile push, SMS, and email, with the channel chosen by the use case and the user's preferences. The platform owns user notification settings (per-category opt-outs, quiet hours, channel choices), device token registration for push, delivery tracking through sent, delivered, and opened states, and both kinds of traffic: transactional notifications triggered by a user's own action, like a password reset, and broadcast campaigns sent to large audiences, like a feature announcement.

Non-functionally, delivery should be at-least-once, meaning every accepted notification is eventually delivered or explicitly dead-lettered, never silently dropped, and the duplicate sends that at-least-once permits must be suppressed by idempotency. Transactional latency should be seconds at most, since a login code that arrives in two minutes is a failed product. Campaigns can take minutes to drain but must never delay transactional traffic. The system must also be polite by construction: rate caps and quiet hours sit in the platform, not in the goodwill of forty producer teams.

Sizing the problem

Assume a mid-size product sending 10 million pushes, 1 million SMS, and 5 million emails per day, 16 million notifications total. Against 86,400 seconds that averages about 185 per second, and transactional traffic peaking at five times the average is still under 1,000 per second, which is small. The number that actually sizes the system is the campaign burst: marketing launches a push to 5 million users at 9 a.m., and if the product wants that campaign delivered within 10 minutes, the pipeline must sustain 5 million over 600 seconds, more than 8,300 sends per second, an order of magnitude over the steady state. Queues absorb the difference between acceptance rate and provider throughput, and that is why the architecture is queues all the way through.

Storage stays manageable. A delivery log row of about 500 bytes per notification is 8 GB per day, and a 90-day retention window is around 720 GB, comfortably one partitioned table. The device token registry holds a few rows per user, say 300 million rows at 200 bytes for a 100-million-user product with multiple devices, which is 60 GB. SMS cost, not storage, is the constraint that surprises people: at half a cent per message, the 1 million daily SMS run about 5,000 dollars a day, which is why the design routes to SMS only when a cheaper channel cannot carry the message.

The API

Producers call one endpoint and never talk to providers. The idempotency key is supplied by the producer so that a producer-side retry of the same logical event cannot create a second notification, and the template ID keeps content rendering inside the platform where localization and formatting live.

POST /api/v1/notifications
{
  "idempotency_key": "order-58821-shipped",
  "user_id": "u-417",
  "template_id": "order_shipped_v3",
  "payload": { "order_id": "58821", "eta": "Friday" },
  "channels": ["push", "email"],          // preference-filtered server side
  "category": "transactional"
}
→ 202 { "notification_id": "n-77b2", "status": "accepted" }

GET /api/v1/notifications/n-77b2
→ 200 { "status": "delivered", "channel": "push",
        "events": ["accepted", "rendered", "sent", "delivered"] }

POST /api/v1/devices                      // from the mobile app
{ "user_id": "u-417", "platform": "ios", "token": "a91f..." }  → 204

The data model

Three tables anchor the platform: the token registry, user preferences, and the delivery log that doubles as the idempotency record, with a unique constraint on the producer's key making duplicate submissions visibly conflict instead of silently double-sending.

CREATE TABLE device_tokens (
  user_id    BIGINT NOT NULL,
  platform   TEXT   NOT NULL,             -- ios | android
  token      TEXT   NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (user_id, token)
);

CREATE TABLE preferences (
  user_id     BIGINT NOT NULL,
  category    TEXT   NOT NULL,            -- marketing, social, security ...
  channel     TEXT   NOT NULL,
  enabled     BOOLEAN NOT NULL DEFAULT true,
  quiet_start SMALLINT,                   -- local-time hour, e.g. 22
  quiet_end   SMALLINT,                   -- e.g. 8
  PRIMARY KEY (user_id, category, channel)
);

CREATE TABLE delivery_log (
  notification_id UUID PRIMARY KEY,
  idempotency_key TEXT NOT NULL UNIQUE,
  user_id         BIGINT NOT NULL,
  channel         TEXT,
  status          TEXT NOT NULL,          -- accepted ... delivered | dead
  attempts        SMALLINT NOT NULL DEFAULT 0,
  created_at      TIMESTAMPTZ NOT NULL
);

The high-level architecture

The flow is a pipeline with one validation brain and per-channel muscle. Producers emit events to the notification service, which checks the idempotency key, loads preferences and applies rate caps and quiet hours, renders the template into channel-specific content, resolves the user's device tokens or address, and enqueues a delivery task on the right channel queue. Channel workers then do nothing but talk to providers: APNs, the Apple Push Notification service that is the only road to iOS devices, and FCM, Firebase Cloud Messaging playing the same role for Android, for push; SMS gateway companies for texts; and an email sending provider or in-house MTA for email. Provider responses, including invalid-token feedback, flow back into the registry and the delivery log.

The notification service validates, applies preferences, renders, and enqueues one task per chosen channel, and channel workers deliver through external providers. Provider feedback flows back asynchronously to prune dead tokens and update the delivery log.

The enqueue label hides one essential split: each channel actually has at least two queues, one for transactional traffic and one for campaigns, with workers draining the transactional queue at strict priority. A campaign dropping 5 million tasks at 9 a.m. then fills the campaign queue and drains at whatever rate provider quotas allow, while a password reset submitted at 9:01 still ships in seconds because it never stands behind the marketing flood. Without that split, the platform's worst incident is self-inflicted: the business sends a campaign and locks its own users out of their accounts for half an hour.

The device token registry

Push delivery is only as good as the token registry, because APNs and FCM address devices by opaque tokens that change without warning. The app registers its current token at every launch and on every token-refresh callback, the registry upserts by user and token with a freshness timestamp, and a user with three devices simply holds three rows. Rot is constant: users reinstall the app, switch phones, or disable notifications, and the providers tell the platform after the fact, APNs through delivery failure responses and feedback, FCM through unregistered errors. The pruning rule is to treat a provider's invalid-token verdict as authoritative and delete that row immediately, and to age out tokens unseen for some long window, 60 or 90 days, since sending to dead tokens wastes quota and damages the sender reputation that providers track. The registry read sits on the hot path of every push, so it deserves a cache keyed by user ID, invalidated on registration and pruning events.

Delivery that survives retries

The reliability stance is at-least-once delivery with idempotency, because exactly-once across a network boundary to a third party is not achievable: a timeout after sending leaves the platform unsure whether the provider got the message, and the only safe move is to retry. Retries are made safe by keying every send with the notification ID, so the duplicate-suppression layer in front of each provider call (a check against a short-lived dedup store recording IDs already accepted by the provider) drops the second attempt of a send that actually succeeded. Backoff is exponential, meaning the wait doubles each attempt, for example 1, 2, 4, 8, and 16 seconds with jitter added so thousands of failures do not retry in lockstep, and transient provider errors get this treatment while permanent verdicts like invalid token or hard email bounce do not, because retrying a permanent failure is pure waste.

Step 1: a worker pulls a delivery task. Step 2: it sends to the provider under the notification's idempotency key. Step 3: the provider times out or returns a transient error. Step 4: the task is requeued with an exponentially growing delay. Step 5: it is redelivered after the delay for another attempt. Step 6: after the attempt limit, the task parks in the dead-letter queue, which alerts an operator and preserves the message for inspection or replay.

The dead-letter queue, a holding queue for messages that exhausted their retries, is what keeps at-least-once from quietly becoming at-most-once: nothing is dropped, failures become visible inventory, and an operator can replay the queue after the underlying fault is fixed. Provider failover covers the faults retries cannot: each channel should have a secondary route where the market allows it, such as a second SMS gateway receiving traffic when the primary's error rate trips a circuit breaker, which is a switch that stops calls to a failing dependency for a cooldown period so workers fail fast instead of stacking up timeouts. Push has no alternative to APNs and FCM, so its failover is buffering: queue depth grows during an outage and drains afterward, with quiet-hours and freshness checks re-applied at send time so a six-hour-old promotional push dies in the queue instead of waking anyone at 3 a.m.

Keeping the platform polite

Rate limiting at the user level is a product safeguard, not a capacity one: without it, forty independent producer teams can collectively send one user thirty messages a day, and the user's response is to disable notifications forever, which destroys the channel for everyone. The platform enforces per-user, per-category caps, for example at most two marketing messages a day with a counter in a fast store checked during validation, plus collapsing rules that merge bursts of similar events ("Ana and 12 others liked your post") before they reach a queue. Quiet hours are enforced in the user's local time zone, holding non-urgent messages until morning while letting security-critical categories through, and the security carve-out matters because a fraud alert delayed eight hours is a real harm.

Delivery receipts close the loop. Workers write sent status, providers report delivered where the channel supports it, and clients report opens through a lightweight event endpoint, with the events landing in the delivery log and flowing onward to an analytics pipeline. Those numbers operate the platform (a falling push delivery rate is how token rot or a provider problem announces itself) and they also feed product decisions, since a category with high sends and near-zero opens is a category that should send less. Campaign tooling reads the same data to throttle itself: a campaign that sees rising bounce rates mid-flight should slow down rather than burn sender reputation.

Scaling, failures, and operations

Every tier scales by a boring rule, which is the compliment a pipeline design wants. The notification service is stateless and scales behind a load balancer; queues partition by user ID so ordering per user is preserved where it matters; channel workers scale on queue depth, with the campaign worker pool capped at whatever the provider contracts allow (an SMS gateway contract might cap at 1,000 messages per second, and the worker pool must respect it or the gateway will enforce it less politely); and the delivery log partitions by time, with old partitions archived. The failure modes are mostly external: a provider outage becomes queue depth and a paused circuit breaker, a token database failover pauses push resolution for seconds, and a misconfigured template fails fast at render time during validation, before anything is queued, where the producer gets a synchronous error it can act on.

The operational dashboard tells the story of the system in four lines: end-to-end transactional latency p99, queue depth per channel and class, delivery success rate per provider, and dead-letter arrivals. The incidents worth rehearsing are the campaign that starves nothing (because the queues are split), the provider that goes dark (circuit breaker, failover or buffer, drain), and the duplicate storm from a retrying producer (absorbed by the idempotency key at the front door). When all three play out as designed, users notice nothing, which is the only good outcome a notification platform can have.

Follow-up questions

  • Why at-least-once instead of exactly-once delivery? A send that times out is ambiguous, since the provider may have delivered it, so the platform must retry to avoid loss, and exactly-once over that boundary cannot be guaranteed. Retries plus idempotency keys give the user-visible effect of exactly-once at achievable cost.
  • How exactly does the idempotency key prevent double sends? At intake it is a unique column in the delivery log, so a producer retry conflicts instead of creating a second notification, and at the provider edge a short-lived dedup record of completed sends drops re-attempts of work that already succeeded.
  • What stops a 5-million-user campaign from delaying password resets? Separate queues per channel for transactional and campaign traffic, drained at strict priority, so the campaign consumes leftover capacity rather than shared capacity. The split is structural, which beats any fairness tuning.
  • When do you delete a device token? Immediately when APNs or FCM declares it invalid in feedback or an error response, and lazily when it has not been re-registered within an aging window of 60 to 90 days. Continuing to send to dead tokens wastes quota and erodes provider trust scores.
  • How do you handle an APNs outage? There is no alternative provider, so the answer is buffering and patience: the circuit breaker stops futile sends, queue depth absorbs the backlog, and the drain re-checks quiet hours and message freshness so stale pushes are dropped rather than delivered at the wrong time.
  • Why do rate caps live in the platform instead of in producer services? Because the user's tolerance is a shared budget across all producers, and only the platform sees the whole picture. One team's reasonable volume plus another's reasonable volume is an unreasonable total, and the cap has to bind at the point where the streams meet.
