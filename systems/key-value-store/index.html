Design a distributed key-value store

Systems design · Distributed building blocks · Mar 2025

A key-value store offers the smallest useful database interface: put a value under a key, get it back later. On a single machine that is a hash table with persistence, and excellent single-node stores exist in exactly that shape. The interview question is what happens when the data outgrows one machine and the system must keep answering through node failures: shopping carts at Amazon, session data, user profiles, device state. The Dynamo paper, which described the store behind Amazon's cart, set the template that Cassandra, Riak, and Voldemort followed, and the question has become the canonical way to probe whether a candidate understands replication, consistency, and failure handling as one connected story rather than as vocabulary.

The hard parts are exactly the places where the single-node intuition fails: which node owns a key when nodes come and go, how many copies must confirm a write before it counts, what to do when two clients update the same key on opposite sides of a network partition, and how a node that was down for an hour catches up. The storage engine inside each node is its own rich problem, and a strong answer connects it to the distributed layer rather than treating it as a black box.

Scope and requirements

Functionally the store exposes get and put on binary keys and values, with values up to perhaps 100 KB, no schema, no secondary indexes, and no cross-key transactions; agreeing to that narrowness early is what makes the rest of the design achievable. The non-functional requirements drive everything: the data set is large, tens of terabytes and beyond what any single machine holds; availability is paramount, meaning the store keeps accepting reads and writes through node failures and even network partitions, the situations where some machines cannot reach others; latency targets are single-digit milliseconds at the median; and consistency is tunable per operation, so a shopping cart can choose to always accept writes while a different table chooses to always read its latest write.

That choice locates the design on the CAP triangle, the result stating that during a network partition a system must choose between staying consistent and staying available. This design chooses availability and partition tolerance, an AP system: it will accept writes during a partition and reconcile afterward, which is the right trade for carts and sessions and the wrong one for bank balances, and saying which workloads it suits is part of the answer.

Sizing the problem

Concrete numbers anchor the partitioning and replication choices. Assume 1 billion keys averaging 10 KB of value each, which is 10 TB of raw data, and with every item replicated to 3 nodes the cluster stores 30 TB. If each node comfortably serves 2 TB from fast storage, the cluster needs about 15 data nodes, and rounding up to 18 leaves headroom for growth and for the temporary imbalance that rebalancing creates. For traffic, assume 100,000 reads and 20,000 writes per second at peak; spread over 18 nodes that is under 7,000 operations per second per node, well within what an in-memory-indexed, log-structured engine sustains. Each write touches 3 replicas, so the internal write traffic is 60,000 replica writes per second, and at 10 KB per value that is about 600 MB per second of replication bandwidth across the cluster, which informs network provisioning more than any other number in the design.

The interface

The external API is small, and the only subtlety worth surfacing is the version token: because the store can hold conflicting versions of a key after a partition, reads return an opaque context that writes pass back, so the store knows which version a client thought it was updating.

GET /v1/keys/cart:8731
→ 200 { "value": {...}, "context": "vc:A2B1" }

PUT /v1/keys/cart:8731
X-Version-Context: vc:A2B1
{ "value": {...} }
→ 200

GET /v1/keys/cart:8731     (after a conflicting write elsewhere)
→ 300 { "siblings": [ {...}, {...} ], "context": "vc:A2B1C1" }

A per-request consistency hint (one acknowledgment or a quorum) rides the same API, which is what tunable consistency means in practice: the application chooses the trade per call rather than the database choosing once for everyone.

Partitioning: who owns a key

With 18 nodes, some function must map each key to its owners, and the function must not reshuffle everything when membership changes. Consistent hashing does this: nodes and keys hash onto the same circular space, a key belongs to the first node clockwise from its hash, and each node owns the arc between its predecessor and itself. Adding or removing a node then moves only about 1/N of the keys, the ones on the affected arc, rather than the near-total reshuffle a modulo scheme causes. Each physical node takes many virtual positions on the ring, a few hundred small arcs instead of one large one, which evens out the randomness of arc sizes and lets a dead node's load disperse across many survivors instead of doubling one neighbor's traffic. Every node knows the full ring, so any node can receive any request and route it, and the node that coordinates a given request is just the first replica for that key or whichever node the client's request landed on.

Replication and quorums

Durability and availability both come from writing each key to N nodes, conventionally the next N distinct physical nodes clockwise from the key's position on the ring, a list Dynamo calls the preference list. The interesting design lever is how many of those N must respond before the operation succeeds: W replicas must acknowledge a write, and R replicas must answer a read.

The arithmetic that makes this rigorous is overlap. With N = 3, W = 2, R = 2, any write is on at least 2 replicas and any read consults at least 2, and since 2 + 2 = 4 is greater than 3, the two sets must share at least one node, so every read sees at least one copy of the latest acknowledged write; version metadata then identifies it as the newest. Concretely: a write to key k is acknowledged by nodes A and B while C is slow; a later read contacts B and C, and B's copy carries the newer version, which the coordinator returns and, in passing, uses to repair C. Choosing W = 1 instead makes writes fast and maximally available, surviving even when only one replica is reachable, but a read at R = 2 can now consult two replicas that both missed the only copy, so W + R no greater than N means stale reads are possible and the application must tolerate them. The cart chooses W = 1 or 2 for write availability; a password-change table would choose W = 2, R = 2.

Step 1: the client's put reaches the coordinator. Step 2: the coordinator sends the write to all three replicas in the key's preference list. Step 3: as soon as two acknowledgments arrive (itself and Node B, say), W = 2 is met. Step 4: success returns to the client while the slow replica catches up in the background.

Versions and conflicts

An AP store accepts writes on both sides of a partition, so two clients can update the same key concurrently and the system must later decide what the key's value is. The simple policy is last-write-wins: each version carries a timestamp and the larger one survives. It is easy, it is what Cassandra does by default, and its cost should be stated plainly: one of the two concurrent updates is silently discarded, and clock skew between nodes can even discard the one that happened later in real time.

Vector clocks keep both updates instead. A vector clock is a small map from node to counter attached to each version, incremented by the coordinating node on each write, and it makes concurrency detectable: version X descends from version Y if X's counters are all greater than or equal to Y's, and if neither descends from the other the writes were concurrent. Worked through: a cart is written via node A, clock [A:1]. One client reads it and saves through B, producing [A:1, B:1]; another client who also read [A:1] saves through C, producing [A:1, C:1]. Comparing the two, B's entry is larger in one and C's in the other, so neither dominates, and the store keeps both as siblings and returns them together on the next read, where the application merges them (for a cart, the union of items) and writes back a version that descends from both. The trade is real: vector clocks grow with the number of coordinating nodes, clients must handle sibling lists, and the merge logic is application code, which is why Dynamo used them for the cart while many successors retreated to last-write-wins plus careful data modeling.

Surviving failures: sloppy quorum, handoff, and repair

A strict quorum refuses writes when too few of the key's home replicas are reachable, which sacrifices exactly the availability this design promised. The sloppy quorum keeps accepting: if a home replica is down, the coordinator writes to the next healthy node further around the ring, and that substitute stores the data with a hint naming the intended owner. When the owner returns, the substitute delivers the hinted writes and deletes its copy, a mechanism called hinted handoff. The effect is that a write succeeds whenever any W healthy nodes exist anywhere on the ring, at the cost that a read during the outage might miss data sitting on a substitute, another stale-read window the application has accepted by choosing this design.

Hints cover short outages, but a node that lost its disk needs a full resynchronization, and comparing terabytes key by key is too expensive. Merkle trees make the comparison cheap: each node builds a tree of hashes over its key range, where each leaf is the hash of a bucket of keys and each parent is the hash of its children, so two nodes can compare their trees from the root down and descend only into branches whose hashes differ, exchanging just the buckets that actually diverge. This anti-entropy process runs continuously in the background in Dynamo-style systems, bounding how long any divergence between replicas can persist.

Membership itself is maintained the same decentralized way. Rather than a master tracking who is alive, every node runs a gossip protocol: once per second it picks a random peer and exchanges its membership list with heartbeat counters, so news of a joined or failed node spreads epidemically and reaches the whole cluster in a few rounds, in time logarithmic in the node count. A node that stops updating its heartbeat is marked suspect locally and routed around, which ties failure detection back into the sloppy quorum machinery without any central authority that could itself fail.

Inside a node: the LSM storage engine

Each node still has to persist its share of data, and the engine of choice for write-heavy stores is the log-structured merge tree, the design behind LevelDB, RocksDB, and Cassandra's storage layer. The idea is to never update disk in place. A write first appends to a write-ahead log, a sequential file whose only job is crash recovery, and then inserts into the memtable, a sorted in-memory structure. When the memtable reaches a threshold, a few tens of megabytes, it is written to disk in one sequential pass as an SSTable, a sorted, immutable file. Sequential appends are the cheapest thing a disk can do, which is why this engine sustains write rates that update-in-place trees cannot.

Reads pay for that write speed. A get must check the memtable, then SSTables from newest to oldest, since any of them might hold the key's latest value, and two mechanisms keep that affordable: a Bloom filter per SSTable, a compact probabilistic structure that says definitely-not-here for most keys the file lacks, lets reads skip almost every irrelevant file, and background compaction merges overlapping SSTables into fewer, larger, non-overlapping ones, discarding overwritten and deleted versions as it goes. Deletes, counterintuitively, are writes: a tombstone marker is written and the data physically disappears only when compaction processes it.

Steps 1 and 2: a write appends to the write-ahead log and then inserts into the memtable. Step 3: a full memtable flushes to disk as an immutable SSTable, and compaction later merges files downward. Steps 4 to 6: a read checks the memtable first, then SSTables newest to oldest, with each file's Bloom filter letting most files be skipped.

Scaling, failures, and operations

Growth is the ring's job: a new node takes its virtual positions, streams its arcs from the current owners while they continue serving, and the cluster's capacity grows by one node's worth with about 1/N of the keys moving. Read scaling beyond that comes from lowering R for tolerant workloads or adding replicas for hot ranges, and write scaling is nearly linear in nodes because writes are sequential appends spread evenly by the hash. The coordination-free design means there is no master to outgrow, and the practical ceilings are network bandwidth for replication and compaction throughput inside each node, since a node that cannot compact as fast as it ingests will slowly accumulate read amplification until latencies degrade, making compaction backlog one of the two or three metrics worth alerting on.

The failure stories compose from the mechanisms above. A node down for minutes: sloppy quorum and hinted handoff carry its traffic, and hints drain back on recovery. A node dead for good: its virtual positions disperse to survivors, Merkle-tree repair rebuilds full replication in the background, and operators replace the hardware on no particular schedule. A network partition: both sides keep accepting writes for the keys they can reach, vector clocks record the divergence, and reads after healing surface siblings for merge; the system never returns an error it could have answered, which is precisely the AP promise and also precisely why a workload needing read-your-own-write semantics must set W and R accordingly or live elsewhere. Operationally, the recurring tasks are tuning N, W, and R per table against measured latency (quorum latency is the latency of the slowest required replica, so W = 2 of 3 also serves as a slow-node filter), watching disk headroom because compaction needs scratch space, and exercising the repair path regularly, since an anti-entropy mechanism that has never run in anger is a hypothesis rather than a safeguard.

Follow-up questions

  • Why choose AP here rather than strong consistency? The motivating workloads (carts, sessions, device state) lose more from refusing writes than from reconciling occasional conflicts. A workload where a stale or merged read is unacceptable should use a CP store or set W = N, accepting unavailability when replicas are down.
  • What do N = 3, W = 2, R = 2 actually guarantee? Every read overlaps every acknowledged write in at least one replica because 2 + 2 is greater than 3, so the newest acknowledged version is always seen by some contacted node. It does not guarantee that in-flight or hinted writes are visible, so it is overlap, not linearizability.
  • Vector clocks or last-write-wins? Vector clocks preserve concurrent updates at the cost of sibling handling and merge logic in the application; last-write-wins is operationally simple and silently drops one of two concurrent writes. Choose by asking what a lost update costs: for a cart item, too much; for a last-seen timestamp, nothing.
  • How does a read repair stale replicas? The coordinator collects versions from R replicas, returns the newest, and asynchronously writes that version back to any contacted replica that returned an older one, so frequently read keys converge without waiting for anti-entropy.
  • Why an LSM tree instead of a B-tree inside each node? The workload is write-heavy and the LSM turns random writes into sequential appends, which disks and SSDs reward; the read cost is contained by Bloom filters and compaction. A read-mostly, range-scan-heavy workload would tilt back toward B-trees.
  • What limits how big one value can be? Replication and compaction copy whole values repeatedly, so multi-megabyte values inflate write amplification and quorum latency; large blobs belong in an object store with the key-value store holding the pointer and metadata.
