Design a URL shortener

Systems design · Classic starters · Jan 2025

A URL shortener takes a long web address and returns a short one, so that tiny.url/2TX9a sends a visitor to the same page as a four-hundred-character link full of tracking parameters. Marketing teams need short links that fit in a text message, print cleanly on a poster, and report how many people clicked. The design question looks small, and that is exactly why interviewers like it: in thirty minutes it touches unique ID generation, the read-heavy caching story, database sharding, and an HTTP subtlety about redirects that has real product consequences. The walkthrough below is the way I would actually run it in an interview, starting from scope and working down into the parts where the hard decisions live.

Scope and requirements

The first few minutes belong to questions, because the design changes a lot depending on the answers. I would confirm that the core features are creating a short link from a long one, redirecting anyone who visits the short link, and optionally letting users pick a custom alias and an expiry time. Click analytics come up almost immediately, so I would agree to count clicks but keep rich analytics out of the core path. Deleting and updating links matters for abuse handling even if users rarely do it.

The non-functional requirements shape everything that follows. Redirects must be fast, ideally a few tens of milliseconds, because the shortener sits in front of every page view it serves. The service must be highly available, since a downed shortener breaks every link it ever issued, and for the same reason links must not be lost once created. Reads will vastly outnumber writes. A shortened link is created once and clicked many times, and a sensible assumption to state out loud is a 100 to 1 read-to-write ratio.

Sizing the problem

Back-of-the-envelope estimation is the habit of turning vague scale words into concrete numbers before designing, and it earns its keep here. Suppose the service creates 100 million links per month. A month is about 2.6 million seconds, so that is roughly 40 new links per second on average. With the 100 to 1 read ratio, redirects run at about 4,000 per second on average, and traffic is never flat, so planning for a peak of five times that gives 20,000 redirects per second.

Storage stays surprisingly small. A link record holds the short code, the long URL, an owner, and a couple of timestamps, which fits comfortably in 500 bytes. At 100 million links per month kept for five years, that is 6 billion records, or about 3 TB. A single modern server can hold 3 TB; the reason to shard, which means splitting the data across machines, will be read throughput and availability rather than raw size.

The cache sizing follows the usual skew argument. Link traffic obeys something like the 80/20 rule, where a small fraction of links receives most of the clicks. At 4,000 redirects per second the service answers about 350 million redirects a day, and caching the 20 percent of each day's distinct links that are hot, say 70 million entries at 500 bytes, costs around 35 GB of memory. That fits in one large cache node, though running a small cluster is wiser for availability.

The API

The external interface is small enough to write out completely. Creation is an authenticated JSON endpoint, and the redirect is a bare GET on the short path, because the whole point is that the short link works in any browser with no client logic.

POST /api/v1/links
{ "long_url": "https://example.com/some/very/long/path?utm_source=...",
  "custom_alias": "spring-sale",        // optional
  "expires_at": "2025-12-31T00:00:00Z"  // optional
}
→ 201 { "short_url": "https://tiny.url/2TX9a" }

GET /{code}
→ 301 Location: https://example.com/some/very/long/path?utm_source=...

DELETE /api/v1/links/{code}            → 204
GET    /api/v1/links/{code}/stats      → 200 { "clicks": 18234, ... }

Generating short codes

The heart of the question is how to mint a short, unique code for every link, and there are three families of answers worth comparing out loud.

The first idea most people reach for is hashing: run the long URL through a hash function such as MD5, which deterministically maps any input to a long fixed-size value, and keep the first seven characters. The appeal is statelessness, but truncating a hash invites collisions, where two different URLs land on the same code, so every write has to check the database and probe again on conflict. The same URL also always produces the same code, which sounds like a feature until two users shorten the same address and expect separate click counts.

The cleaner family is counter plus base62. Give every link a unique integer ID and encode the integer in base62, which is positional notation using the 62 characters 0 to 9, a to z, and A to Z. Seven base62 characters cover 627, about 3.5 trillion values, which at 100 million links a month lasts close to three thousand years. The encoding is worth doing on the whiteboard once: ID 11157 is 2 × 62² + 55 × 62 + 59, and with 55 mapping to T and 59 mapping to X, the code is 2TX.

ALPHABET = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ"

def encode(n: int) -> str:
    out = []
    while n:
        n, r = divmod(n, 62)
        out.append(ALPHABET[r])
    return "".join(reversed(out)) or "0"

encode(11157)   # '2TX'

The remaining question is where the unique integers come from once there is more than one server. A single auto-incrementing database column works until that database becomes the bottleneck and the single point of failure. The standard fix is range leasing: a small coordination store hands each application server a block of one million IDs at a time, the server burns through its block in memory, and it leases a new one when the block runs out. Losing a server wastes the rest of its block, which is harmless when the keyspace holds trillions. A third option, a key generation service that pre-computes random-looking codes into a pool, buys unguessable codes at the cost of another service to operate; sequential base62 codes are guessable, so if that matters, encrypt or permute the counter before encoding rather than running a whole extra service.

The data model

One table carries the service. The short code is the natural primary key, and at scale the store should be a key-value database such as DynamoDB or Cassandra, because the access pattern is a single-key lookup and those systems shard and replicate that pattern automatically. Postgres is the right starting point at small scale, and saying so in the interview shows judgment rather than weakness.

CREATE TABLE links (
  code        VARCHAR(10) PRIMARY KEY,   -- '2TX9a' or custom alias
  long_url    TEXT        NOT NULL,
  owner_id    BIGINT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at  TIMESTAMPTZ
);

Click counts deliberately live elsewhere. Incrementing a counter inside this table on every redirect would turn the hottest read path in the system into a write path, so clicks flow through an event queue into an analytics store, and the stats endpoint reads from there.

The high-level architecture

With those decisions made, the architecture nearly draws itself. A load balancer, which is the component that spreads incoming requests across identical servers, fronts two stateless services: a write service that leases ID blocks and inserts rows, and a redirect service that resolves codes. The redirect service leans on a cache, which is a fast in-memory store holding recently used code-to-URL mappings so most lookups never touch the database, and it emits click events asynchronously.

The write path leases ID blocks and inserts rows; the redirect path serves from the cache and falls back to the database, emitting click events off the critical path. Dashed arrows are taken only sometimes or asynchronously.

The redirect path in detail

Most of the system's work happens in one short sequence, so it deserves its own picture and its own scrutiny.

Step 1: the browser requests GET /2TX9a. Step 2: the service checks the cache. Step 3: on a miss it reads the database and refills the cache. Step 4: it answers with a redirect and the Location header. Step 5: a click event goes to the queue without blocking the response.

The response code is a real decision, not trivia. A 301 is a permanent redirect, and browsers cache it, so repeat visitors skip the shortener entirely. That lowers load but makes the service blind to repeat clicks and unable to change the destination later. A 302 is a temporary redirect that browsers re-request every time, which preserves analytics and editability at the cost of serving every click. A link shortener whose product is analytics should answer 302; one optimizing purely for cost can answer 301. I would choose 302 and say why.

Lookups for codes that do not exist deserve attention, because random probing and typos produce them constantly and each one would fall through the cache to the database. Caching the absence, storing a short-lived entry that says this code resolves to nothing, blunts most of it, and a Bloom filter, a compact probabilistic structure that can say definitely-not-present for most missing keys, can sit in front of the database if negative traffic is heavy.

Scaling, failures, and operations

Every tier in the diagram scales by a different rule, and walking through them in order closes the design. The two services are stateless, so they scale horizontally behind the load balancer, and at 20,000 redirects per second a handful of nodes is plenty. The cache scales by sharding keys across a small cluster, and because the data is tiny per entry, the working set fits with room to spare. The database shards by hashing the short code, which spreads both data and load evenly because codes themselves are effectively random; consistent hashing keeps a reshard from moving more than its fair share of keys. Each shard replicates to followers, and since a redirect read that is a few seconds stale is harmless, reads can come from replicas freely. Geo-distribution falls out naturally too, because links never change destination in the 301 design and rarely in the 302 design, so read replicas or an edge cache in each region cut latency for a worldwide audience.

Expiry is best handled lazily. The redirect path checks expires_at and answers 410 Gone for dead links, and a background job sweeps expired rows out in bulk, which avoids paying deletion costs on the hot path. Custom aliases ride the same table as ordinary codes with a uniqueness check at insert and a reserved-word list so nobody claims /api. Abuse is the operational tax of running any shortener: malicious actors hide phishing destinations behind short links, so creation should be rate limited per user, destinations should be checked against a service such as Google Safe Browsing both at creation and periodically afterward, and takedown needs to be a first-class admin operation.

The failure stories are reassuringly boring, which is the point of the architecture. A dead redirect node is replaced behind the load balancer, a dead cache node costs a temporary rise in database reads while its shard rewarms, and a dead database primary fails over to a replica. The one component that must not be casually centralized is the ID range store, and it is naturally protected by its own access pattern: servers contact it once per million links, so it can be a small strongly consistent store such as ZooKeeper or a single Postgres row updated transactionally, and brief unavailability only pauses link creation, never redirects.

Follow-up questions

  • Why not just hash the URL and skip the counter? Truncated hashes collide and need probing on write, and identical URLs collapse to one code, which breaks per-user analytics. The counter with base62 gives uniqueness by construction.
  • 301 or 302? 301 lets browsers cache the hop and cuts load; 302 keeps every click visible and the destination editable. The analytics product wants 302.
  • How long should codes be? Seven base62 characters give about 3.5 trillion values, roughly three thousand years at 100 million links a month. Six characters give 57 billion, which a large service exhausts in decades, so seven is the comfortable answer.
  • What breaks first at 10x traffic? Nothing structural: redirect nodes and cache shards multiply, and the database adds shards. The design point worth saying is that the data model never changes because the access pattern is already a single-key lookup.
  • How do you stop sequential codes from leaking creation volume? Permute or encrypt the counter with a keyed bijection before base62 encoding, so codes look random while staying collision-free.
  • Where do click counts live? In an analytics store fed by the event queue, never as a counter update on the links table, because that would put a write on the hottest read path.
