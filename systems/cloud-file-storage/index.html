Design Google Drive

Systems design · Media and storage · Aug 2025

Google Drive, like Dropbox and OneDrive, is the product where a folder on your laptop, the app on your phone, and a web page all show the same files, and a change made on any of them appears on the rest within seconds. The design question is really about a single decision made early and exploited everywhere: files are not stored as files. Content is split into fixed-size blocks named by their own hashes, metadata describes how blocks compose into files and versions, and once that split exists, deduplication, delta sync, versioning, and fast restores all fall out of it almost for free. Interviewers ask the question to see whether a candidate finds that move or spends the hour gold-plating an FTP server. The hard parts live in the sync protocol that keeps devices convergent over unreliable networks, and in the conflict story when two offline devices edit the same file.

Scope and requirements

In scope: upload and download of arbitrary files, automatic sync across a user's devices, folder sharing with permissions, link sharing, version history, and a trash with restore. Real-time collaborative editing of document content, the Google Docs problem, is a different system built on operational transforms or CRDTs, and I would scope it out explicitly. Non-functionally, sync should propagate a small edit in a few seconds, no acknowledged write may ever be lost, since this product is where people keep the only copy of their dissertation, and the client must behave well on bad networks: resume interrupted transfers, tolerate offline periods, and never saturate a home uplink without permission. Consistency can be relaxed between devices for seconds, but the server's record of a file version must be atomic, either a new version exists in full or it does not.

Sizing the problem

Take 500 million registered users averaging 10 GB stored each, which is 5 exabytes of logical data, the kind of number that makes raw replication tripling unaffordable before optimizations. Two levers control the bill. Deduplication exploits the fact that humanity stores the same bytes repeatedly, the same installers, the same forwarded PDFs, the same memes, and block-level content addressing collapses every identical block to one stored copy; measured dedup ratios on consumer corpora commonly save several tens of percent. Cold tiering exploits access skew, since most files are written once and rarely opened again, so blocks untouched for months migrate to cheaper erasure-coded storage where a later read costs a little extra latency. Traffic is gentler than storage: if 10 percent of users are active daily and each syncs 100 MB, that is 5 PB a day, an average near 60 GB per second across the fleet, comfortably within a horizontally scaled service tier; the byte volume problem lives in storage economics, not request throughput.

The block interface

The client and server speak in blocks, not files. Content is chunked into fixed 4 MB blocks, and each block is named by its hash, the fixed-length fingerprint a cryptographic hash function such as SHA-256 computes from the block's bytes, with the property that identical content always yields the identical name and different content effectively never collides. Content addressing makes the protocol idempotent and dedup automatic: the client never asks "store this again," it asks "which of these do you not have?"

POST /api/v1/blocks/diff
{ "hashes": ["9f31c2...", "a07be4...", "d2188a..."] }
→ 200 { "missing": ["d2188a..."] }       // server already holds the others

PUT /api/v1/blocks/d2188a...
Content-Length: 4194304
→ 201

POST /api/v1/files/f_8812/commit
{ "base_version": 6,
  "blocks": ["9f31c2...", "a07be4...", "d2188a..."], "size": 12582912 }
→ 201 { "version": 7 }
→ 409 { "error": "conflict", "head": 8 }  // someone committed version 7 first

The commit is the atomic step: it submits the ordered block list that constitutes the new version, conditional on the version the client last saw, so two racing commits cannot silently interleave, and the loser gets a 409 and handles the conflict deliberately.

The data model

Metadata is small, relational in shape, and strongly consistent, so it lives in a sharded SQL database partitioned by namespace, meaning a user's tree or a shared folder's tree. Blocks are immutable bytes in object storage keyed by hash, with reference counts so shared and historical versions can pin them.

CREATE TABLE files (
  file_id      BIGINT PRIMARY KEY,
  namespace_id BIGINT NOT NULL,        -- shard key: user or shared folder
  parent_id    BIGINT,                 -- folder tree
  name         TEXT   NOT NULL,
  head_version INT    NOT NULL,
  trashed_at   TIMESTAMPTZ
);

CREATE TABLE file_versions (
  file_id     BIGINT,
  version     INT,
  size_bytes  BIGINT,
  block_list  CHAR(64)[],              -- ordered block hashes
  created_at  TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (file_id, version)
);

CREATE TABLE acl_entries (
  namespace_id BIGINT,
  principal_id BIGINT,                 -- user or group
  role         TEXT,                   -- viewer / editor / owner
  PRIMARY KEY (namespace_id, principal_id)
);

Notice what a version is: a row with an ordered list of block hashes. Versions share every block they did not change, so version history costs only the deltas, and restoring an old version writes one metadata row pointing at blocks that never went away.

The high-level architecture

Four services carry the product. The block service fronts object storage, handling diff queries and block transfer. The metadata service owns trees, versions, and permissions, and is the only writer to the metadata database. The notification service keeps a lightweight channel to every online device, long polling, where the client parks a request that the server answers when something changes, or a platform push channel on mobile, used purely to say "something changed in your namespace," with the actual changes always pulled from the metadata service. Clients run a local watcher and a local database of synced state.

Device A uploads only blocks the server lacks and then commits a version through the metadata service; the notification service wakes the user's other devices along the dashed path, and they pull the delta and fetch just the new blocks.

Dedup and delta sync, with the arithmetic

Content addressing pays twice. Across users, when a popular 4 MB installer block sits in 10,000 accounts, naive storage holds 40 GB of it while the block store holds 4 MB and 10,000 references, and the saving applies to every commonly shared file on the service without anyone coordinating anything. Within a user, dedup becomes delta sync, defined as transferring only the parts of a file that changed rather than the whole file. Consider a 1 GB design file, which is 256 blocks of 4 MB. The user nudges one layer and saves; the client watcher notices the file changed, rehashes it, finds 255 hashes identical and one new, and the diff request comes back naming a single missing block, so the sync uploads 4 MB instead of 1 GB, a 256-fold reduction, and on a 10 Mbps home uplink that is the difference between roughly 3 seconds and 14 minutes. Every other device repeats the saving in the download direction, fetching one block and splicing it into the local copy.

The real limit of fixed-size blocks is insertion: adding ten bytes at the front of a file shifts every subsequent byte, so all 256 block hashes change and the delta degenerates to a full upload. Content-defined chunking, where block boundaries are chosen by a rolling hash of the content in the style of rsync so boundaries move with the data, fixes insertion shift at the cost of variable block sizes and a more complex protocol, and a consumer service can reasonably ship fixed blocks first because the dominant consumer workloads, photos and videos that never change and documents edited in place, rarely hit the bad case.

The sync protocol, step by step

Sync is a loop that converges the local database, the local files, and the server's metadata, and it must survive interruption at any step, which is why every step is idempotent: re-running it never makes things worse.

Step 1: the watcher on device A detects the edit, rehashes the file, and sends the block hashes to the diff endpoint. Step 2: it uploads only the blocks the server reported missing. Step 3: it commits version N+1 with the ordered block list, conditional on N. Step 4: the metadata service publishes the change. Step 5: the notification channel wakes device B. Step 6: device B fetches the new version's metadata. Step 7: it downloads just the changed blocks and splices them into its local copy.

The conditional commit in step 3 is where conflicts surface. Suppose devices A and B both edited the file while offline, both against version 6. A reconnects first and commits version 7; B's commit conditional on 6 is refused with a 409. Last-writer-wins, where B's commit simply replaces A's, is the wrong consumer answer because it silently destroys A's work, and silent data loss is the one unforgivable failure in a product holding people's only copies. The safe resolution is keep-both: the client renames its local version into a new file, the familiar "Report (conflicted copy, Device B)," commits it as a sibling, and lets the human merge, because the server cannot meaningfully merge a spreadsheet and should not pretend to. Both versions are preserved, both devices converge on both files, and the worst case is mild clutter instead of loss.

Sharing, versioning, and trash

Sharing is governed by ACLs, access control lists, the per-resource tables mapping each user or group to a role such as viewer or editor. ACLs attach at the folder level and apply to the subtree, and a shared folder becomes its own namespace, which matters for sharding because all its members' operations route to one partition and stay transactionally simple. The notification fanout generalizes naturally: a commit in a shared namespace wakes every member's devices, not just the committer's. Link sharing mints a scoped token, an unguessable random string embedded in a URL that grants exactly one capability, view or edit, on one subtree, revocable server-side at any time by deleting the token row.

Versioning and trash are nearly free given immutable blocks, and saying why is worth the breath: old versions are rows whose blocks still exist, so restoring a version or rescuing a file from trash writes metadata and moves no bytes. The cost shows up as retention policy, since pinned blocks cannot be reclaimed: a consumer tier might keep 30 days of versions and trash, after which a garbage collector decrements reference counts and deletes blocks that reach zero. That collector is the most dangerous process in the system, the only thing that destroys data on purpose, so it runs with delays, soft-delete grace periods, and rate limits, on the theory that reclaiming space a week late is invisible while reclaiming a live block once is catastrophic.

The client realities round out the design. Small files dominate counts while large files dominate bytes, so the client batches tiny files into grouped commits to amortize round trips, caps its bandwidth by default and backs off when the network is in use, verifies every block end to end by rehashing after download so a corrupt byte anywhere in transit or storage is detected and refetched, and keeps hashing and chunking off the foreground so a laptop on battery is not paying for a sync the user never notices.

Scaling, failures, and operations

The block service and notification tier are stateless and scale horizontally; object storage scales by hash space and is the easy exabyte. The metadata database is the tier that needs care, sharded by namespace so a user's operations are single-shard, with shared folders placed by their own namespace ID, and read replicas absorbing the listing-heavy read mix. The notification service holds millions of parked long-poll connections, which is a connection-count problem like a chat system's socket tier and scales the same way, by adding nodes behind a registry.

Failure stories: a client that dies mid-upload retries its idempotent block PUTs and recommits, and the orphaned blocks from abandoned uploads are swept by the garbage collector after a grace period. A metadata shard failover pauses commits for its namespaces for a few seconds while a replica promotes, and clients retry transparently. A lost notification wakes nobody, which is why clients also poll lazily every few minutes as a backstop, trading staleness for certainty. Object storage durability rests on replication or erasure coding underneath, and the end-to-end block hashes mean even a silently corrupted replica is caught at read time. The operations dashboard watches sync convergence time end to end, commit conflict rate, dedup ratio, GC backlog, and notification delivery lag, because those are the product's promises expressed as numbers.

Follow-up questions

  • Why split metadata from content at all? They have opposite shapes: metadata is small, hot, and needs transactions, while content is huge, immutable, and cold. The split lets a SQL database and object storage each do what they are good at, and block hashes are the join key.
  • Why 4 MB blocks instead of whole-file storage? Blocks make dedup and delta sync possible: identical blocks across files and users store once, and editing one block of a 1 GB file transfers 4 MB instead of the file. Whole-file storage re-uploads and re-stores everything on every save.
  • How does the system avoid losing data when two devices edit offline? Commits are conditional on the base version, the loser gets a conflict, and the client keeps both copies rather than letting either overwrite the other, surfacing a conflicted-copy file for the human to merge.
  • What makes restore from trash instant for a 50 GB folder? Nothing moves: blocks are immutable and were never deleted during the retention window, so restore is a metadata operation that re-points rows.
  • When do fixed-size blocks perform badly? Insertions near the start of a file shift every later boundary, changing all block hashes and forcing a full transfer. Content-defined chunking with a rolling hash fixes it at the cost of protocol complexity.
  • Why is the garbage collector treated so carefully? It is the only component that destroys bytes, and a reference-counting error there turns shared dedup into shared data loss, so it runs delayed, rate-limited, and with soft-delete grace periods that make mistakes recoverable.
