Design social graph search

Systems design · Search and discovery · May 2025

Type "jo" into the search box of a large social network and the first results are not the most famous people named Jo, they are your friends, then friends of your friends, then everyone else. That ordering is the product: people search where relevance is mostly relationship distance, computed live as the user types, with a keystroke-to-results budget of about 100 milliseconds. Interviewers like the question because it sounds like a search problem and turns out to be a graph problem, and the instructive moment is realizing that the textbook answer, breadth-first search over the friendship graph at query time, cannot work at social network scale, which forces the precomputation and caching design that real systems such as Facebook's actually use.

The hard parts are three: storing a graph with a trillion edges so that any user's neighbors are one cheap read away, computing relationship distance without traversing the graph per keystroke, and matching name prefixes against a billion names fast enough that distance ranking still has time to run, all while privacy rules filter the results before anything leaves the backend.

Scope and requirements

Functionally, the system serves typeahead people search: each keystroke sends the current prefix, and the response is a ranked list of perhaps eight people whose names match, ordered by graph distance first (friends, then friends of friends, then the global tier) with interaction signals breaking ties inside each tier. Results must respect privacy: blocked users never appear, and users whose visibility settings exclude the searcher are filtered server side. Friend and unfriend events must be reflected within seconds, since a brand-new friend who does not surface in search reads as a defect to the user. Full-text search over posts and pages is a different system and explicitly out of scope.

Non-functionally, the backend has roughly 50 milliseconds per query after the network takes its share of the 100-millisecond perceived budget, and every keystroke is a query, so the request rate is the typing rate of the user base. The system must degrade rather than fail: if graph distance cannot be computed in time, returning name-relevance results late beats returning nothing, and that degraded mode is a designed state, not an accident.

Sizing the problem

Assume 1 billion users with an average of 500 friends, which is the branching factor the rest of the article leans on. Friendship is mutual but stored in both directions for fast lookup, so the edge store holds 1 billion times 500, which is 500 billion directed entries twice over, about 1 trillion rows, and at roughly 12 bytes per entry (two 8-byte IDs compressed plus overhead) the raw adjacency data runs around 12 TB before indexes and replication, far beyond one machine and naturally sharded by user ID so that one user's friend list lives on one shard. The query side: 200 million daily searchers issuing 5 searches of 5 keystrokes averages 5 billion prefix queries per day, which is about 58,000 queries per second against 86,400 seconds, peaking past 150,000, and that rate is why per-query work must be a handful of cache reads rather than a graph walk.

The name index is small by comparison, which is good news. A billion names at a few dozen bytes is around 30 GB of raw strings, and even with prefix structures multiplying that several times it fits in memory across a modest shard set, so the design can afford to treat name matching as the cheap first stage and spend the latency budget on ranking.

The interface

One endpoint serves the box. The response carries the match tier so the client can render section headers, and a request token ties keystrokes from one session together for caching and logging.

GET /api/v1/typeahead?q=jo&limit=8&session=s-4471
→ 200 {
  "results": [
    { "user_id": "u-2241", "name": "Jordan Park",  "tier": "friend",
      "mutual_friends": 41 },
    { "user_id": "u-9077", "name": "Joanna Reyes", "tier": "friend_of_friend",
      "mutual_friends": 6 },
    { "user_id": "u-5550", "name": "Jo Martin",    "tier": "global" }
  ],
  "degraded": false
}

The data model

The graph lives as adjacency lists, which means each user's row holds the list of that user's direct neighbors, the natural layout when the dominant query is "give me this user's friends" rather than "does this exact edge exist". In a sharded key-value store partitioned by user ID, one read fetches one user's whole neighborhood, and Facebook's TAO is the canonical production example of this shape: a graph API over sharded MySQL with a write-through cache layer in front, serving association lists like friend edges at extremely high read rates. A sketch of the two stores:

-- Graph store, sharded by user_id (both directions materialized)
CREATE TABLE friend_edges (
  user_id    BIGINT NOT NULL,
  friend_id  BIGINT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  PRIMARY KEY (user_id, friend_id)
);

-- Name index, sharded by name token prefix
-- prefix "jo" → posting list of (user_id, name, popularity)
CREATE TABLE name_prefixes (
  prefix   VARCHAR(8) NOT NULL,
  user_id  BIGINT NOT NULL,
  name     TEXT   NOT NULL,
  rank_hint REAL,
  PRIMARY KEY (prefix, user_id)
);

The high-level architecture

A typeahead service orchestrates three stages within the budget. The prefix index, which is a precomputed map from each short name prefix to the list of user IDs whose names start with it, returns candidates in a couple of milliseconds; the distance ranker intersects those candidates with the searcher's cached first-degree and second-degree sets to assign a tier; and privacy filtering trims the list before ranking finishes. The friend-list cache underneath the ranker is the load-bearing component, rebuilt from the graph store whenever edges change.

Each keystroke fans the prefix out to name index shards, the returned candidates are tiered by the distance ranker against cached first and second degree sets, and ranked, privacy-filtered results flow back. The cache rebuilds from the graph store when friendships change.

Why query-time BFS fails

Breadth-first search explores a graph in rings: first the source's neighbors, then their neighbors, expanding one hop at a time, and on a whiteboard it is the obvious way to compute distance between the searcher and a candidate. At this scale it dies twice over. The first death is fan-out arithmetic: with branching factor 500, one hop reaches 500 people, and two hops reach about 500 squared, which is 250,000, so confirming that a candidate is a friend of a friend by forward search means materializing a quarter-million-node frontier, and three hops reach 125 million, an eighth of the network. The second death is the network: the graph is sharded, so each hop is a round trip of parallel reads across many shards, and even an excellent 5 milliseconds per fetch round turns a multi-hop walk with giant frontiers into hundreds of milliseconds of shard traffic, for one keystroke, multiplied by 150,000 queries per second platform-wide. The latency budget allows perhaps two or three cache reads, and BFS wants thousands of store reads, so the gap is structural rather than a tuning problem.

Bidirectional BFS, and why precomputation still wins

Bidirectional BFS is the classic remedy: expand a frontier from both endpoints simultaneously and stop when the frontiers intersect, which finds a shortest path of length d while exploring about 2 times bd/2 nodes instead of bd, where b is the branching factor. The worked numbers make the case: for distance 2 and b of 500, one-sided search touches about 250,000 nodes while two-sided search touches about 500 from each end, roughly 1,000 total, a 250-fold saving, and for distance 4 the gap widens to 62.5 billion versus about 500,000. The meeting test is cheap too, since checking whether two 500-element sets intersect is a hash set operation, not a graph operation.

Each side expands one hop and the mutual friends appear where the frontiers intersect, so a distance-2 check costs two friend-list reads and a set intersection, while one-sided BFS would materialize a frontier of about 250,000 nodes.

The production insight is that bidirectional BFS for distance 2 reduces to exactly the operation worth precomputing: intersect my friend list with the candidate's friend list. The system therefore keeps each user's first-degree list cached and, for active users, a compact second-degree structure such as a Bloom filter or a sorted ID array built by expanding the cached friend lists offline, so the per-candidate tier check at query time is a membership test against data already in memory: in my friends, in my second-degree set, or global. The expensive expansion happens once per friendship change rather than once per keystroke, which is the entire trick: the algorithm survives, but its work moves from query time to write time where there are eight orders of magnitude fewer events.

Matching names and ranking candidates

The name side is a prefix index: every name is tokenized and lowercased, and for each token prefix up to a few characters the index stores a posting list of user IDs, sharded by prefix so the "jo" list lives on a known shard, with hot two-letter prefixes split further by the next character when a single posting list grows too hot. A global posting list for "jo" holds millions of entries, so it is truncated to the globally popular tail while a parallel personalized source covers the rest: the searcher's own first-degree and second-degree names are few enough (500 friends, perhaps 250,000 second-degree) that a per-user or per-shard scan of cached friend names matches prefixes exhaustively, guaranteeing that a friend named Jordan always surfaces even though Jordan is globally obscure. Candidates from both sources are unioned, tiered by the membership tests above, scored within tiers by interaction signals such as mutual friend counts and recent contact, and privacy-filtered with block lists and visibility checks before the response leaves the backend, never on the client, because filtered-but-shipped results are a data leak regardless of what the UI hides.

Freshness rides the write path. A friend or unfriend event updates the graph store, then invalidates and rebuilds the two cached structures for both endpoints, and the second-degree rebuild can lag a few minutes without harm as long as the first-degree list updates within seconds, because the visible promise is that a new friend ranks as a friend immediately, while their friends creeping into the second tier is below anyone's perception. Name changes flow to the prefix index through the same event stream.

Scaling, failures, and operations

Each store scales by its own key: the graph store shards by user ID and replicates per shard, with the cache layer in front absorbing the read multiple, which is the TAO shape; the prefix index shards by prefix with hot prefixes subdivided and replicated, since prefix popularity follows a steep skew; and the friend-list cache shards by user ID, sized at roughly 500 entries of 8 bytes per active user, about 4 KB each, so 200 million active users fit in under a terabyte of cache memory across the cluster. The typeahead service is stateless and scales horizontally, and per-session caching helps more than usual here because each keystroke extends the previous query, letting the service refine the prior candidate set ("jor" filters the cached "jo" results) instead of re-fanning out, which cuts shard traffic several-fold for free.

The failure plan is the degraded mode promised at the start. If the friend-list cache or graph store is slow or a shard is down, the ranker times out at a fixed deadline of a few milliseconds and the service returns name-relevance ordering with the degraded flag set, so the search box keeps working with slightly worse ordering rather than spinning, and the client can choose to retry the ranking quietly. If a prefix shard is down, its replica serves; if both are gone, that prefix's global candidates vanish but personalized candidates from the friend cache still match, which fails in the direction users forgive. The metrics that matter are keystroke-to-response p99, ranker timeout rate (the degraded-mode meter), cache rebuild lag after friend events, and a privacy regression test in continuous deployment, because the one unforgivable defect in this system is a blocked user appearing in a result list.

Follow-up questions

  • Why not run BFS at query time? Branching factor 500 makes a two-hop frontier about 250,000 nodes, each hop is a cross-shard round trip, and the budget is 50 milliseconds at 150,000 queries per second. The arithmetic fails before any implementation detail matters.
  • What exactly does bidirectional BFS save? Exploration drops from bd to about 2 times bd/2: for b of 500 and d of 2, that is 1,000 nodes instead of 250,000, and the distance-2 case reduces to intersecting two cached friend lists.
  • How fresh must the cached friend lists be? First-degree within seconds, because a new friend must rank as a friend immediately; second-degree structures can rebuild lazily over minutes, since changes there sit below user perception.
  • How do you handle a user with millions of followers? Treat followers and friends differently: distance ranking runs on the bounded friend graph, while celebrity accounts surface through the global popularity tier of the prefix index, so no per-query structure ever scales with follower count.
  • Where does privacy filtering run, and why there? In the backend after tiering and before the response is built, because results that are shipped and merely hidden by the client are a leak. Block lists are small per user and cached beside the friend lists, so the check costs microseconds.
  • What degrades first under load, and what does the user see? The distance ranker hits its internal deadline first, and the service falls back to name relevance with a degraded flag: results still appear within budget, ordered a little worse, which is the designed failure rather than an outage.
