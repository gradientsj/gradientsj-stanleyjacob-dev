Design a search service

Systems design · Search and discovery · May 2025

Almost every product eventually grows a search box, and behind it sits a service that takes a few words of free text and returns the most relevant documents in well under a second. The scope here is deliberately a product's own corpus, the listings of a marketplace, the posts of a forum, the items of a store, rather than the open web, because crawling and ranking the web is a different problem with a different article's worth of machinery. The interesting question for a product search service is how to turn "blue running shoes" into a ranked result list over fifty million documents, fast, while the documents themselves keep changing.

Interviewers like this design because the database everyone already has is the wrong tool, and explaining precisely why forces real understanding. A SQL LIKE '%running%' scans every row, cannot rank, and does not know that "running" and "runs" are the same idea. The hard parts of the design live in the index structure that makes text lookup fast, in the scoring function that makes results worth reading, and in the distributed query that stitches shards together without wrecking either latency or ranking.

Scope and requirements

Functionally, the service indexes documents with text fields and some structured fields (price, category, timestamps), answers keyword queries with ranked results, supports filters such as category and price range alongside the text match, and paginates. Updates to documents should become searchable quickly, where "quickly" is worth negotiating: seconds of delay between writing a listing and finding it in search is acceptable for almost every product, and conceding that buys the entire indexing architecture. Facets, the counts-by-category sidebar, come up in most product searches, so they stay in scope at the level of a sentence or two.

Non-functionally, assume 50 million documents averaging 2 KB of indexable text, 20 million searches per day, and a latency target of about 200 ms at the 95th percentile for the search round trip. Twenty million searches over 86,400 seconds is about 230 queries per second on average, and planning for a peak of four to five times that means designing for roughly 1,000 QPS. Reads dominate writes here the way they do in most search workloads: perhaps a few hundred document updates per second against a thousand queries.

The inverted index, with a worked example

The core data structure inverts the document-to-words relationship into words-to-documents, which is why it is called an inverted index. For every term in the corpus, the index stores a postings list: the IDs of the documents containing that term, in sorted order, usually with the term's frequency in each document and often the positions where it occurs. Three tiny documents make it concrete. Document 1 is "blue trail running shoes," document 2 is "red running shoes," and document 3 is "blue suede boots." The postings list for running is [1, 2] and the postings list for blue is [1, 3]. A query for "blue running" walks both lists and, because they are sorted, intersects them in a single linear merge, finding document 1 without ever looking at document 3's text. At fifty million documents the lists are longer but the algorithm is unchanged, and the sortedness is what makes ANDs, ORs, and skipping efficient.

Terms do not come from the raw text directly; they come out of an analysis pipeline. Analysis tokenizes the text into words, lowercases them, drops stop words (extremely common words like "the" that match everything and rank nothing), and stems each word, which means reducing it to a root form so that "running," "runs," and "ran" all become run. The non-negotiable rule is that the exact same pipeline must run at index time and at query time, because the index stores run and a query that searches for the unanalyzed Running would find nothing. Most search defects in practice trace back to a mismatch between the two sides of that rule, which is also why changing an analyzer requires rebuilding the index, a point the operations section returns to.

The query API

GET /v1/search?q=blue+running+shoes&category=footwear&price_max=150&size=20
→ 200 {
  "total": 8412,
  "took_ms": 41,
  "hits": [
    { "doc_id": "listing-99412", "score": 14.2, "title": "Blue trail running shoes" },
    { "doc_id": "listing-10293", "score": 12.8, "title": "Lightweight blue runners" }
  ],
  "facets": { "brand": { "asics": 1204, "nike": 988 } },
  "search_after": ["12.8", "listing-10293"]
}

The search_after cursor in the response is the pagination mechanism, and the reason it exists instead of a page number is one of the deep dives below.

The high-level architecture

The corpus is too large and too hot for one machine, so it is split into shards, each shard holding the full index for a subset of documents, with documents assigned by hashing their ID. Writes flow through a queue: services publish document creates, updates, and deletes, and an indexer consumes them, runs analysis, and writes to the owning shard. Each shard batches incoming documents into immutable index segments and opens them to queries on a refresh interval of about a second, which is the near-real-time compromise: a freshly written document is searchable not instantly but a heartbeat later, in exchange for writes that never lock the read path. Queries take the opposite route through a coordinator that fans out to every shard and merges what comes back. This is precisely the architecture inside Elasticsearch, where each shard is a Lucene index, and naming that embodiment in the interview anchors the design in something real.

The top lane is the write path: document changes queue up, and the indexer analyzes each one and routes it to the shard owning its ID hash. The bottom lane is the read path: the coordinator scatters every query to all shards and merges their top results.

Inside a shard: segments and the index structure

Within a shard, the index is a term dictionary pointing at postings lists, and both live in immutable segment files. New documents accumulate in memory and flush as a fresh small segment on each refresh, which is why writes never modify existing files; deletes are tombstone markers checked at query time, and a background merge process periodically rewrites several small segments into one larger one, dropping tombstoned documents for good. Immutability is what lets queries run lock-free against files that the operating system can cache aggressively, and the merge process is the price, a steady background I/O load that operations has to budget for.

Each dictionary entry records its document frequency (df, how many documents contain the term) and points to a postings list whose entries pair a document ID with the term frequency (tf) inside that document. Sorted IDs make intersecting two lists a linear merge.

Relevance: from TF-IDF intuition to BM25

Matching finds candidates; scoring orders them, and the intuition has two halves. A document mentioning the query term many times is probably more about it, which is term frequency, and a term that appears in few documents carries more information than one that appears everywhere, which is inverse document frequency: "shoes" in a shoe store's corpus says almost nothing, while "waterproof" narrows things sharply. TF-IDF multiplies the two, and BM25 is the refinement that production systems actually use, adding two corrections with a knob each. The knob k1, usually around 1.2, saturates term frequency so the tenth occurrence adds far less than the second, because a keyword-stuffed document should not win on repetition alone. The knob b, usually 0.75, normalizes by document length relative to the corpus average, because three mentions in a 50-word title-and-blurb mean more than three mentions buried in a 500-word essay.

A small worked comparison shows the machinery moving. Take the term frequency part of BM25, tf times (k1 + 1) over (tf + k1 times (1 minus b plus b times dl over avgdl)), with dl the document's length and avgdl the corpus average of 100 words. Document A has tf 3 and length 50, so its denominator is 3 + 1.2 times (0.25 + 0.75 times 0.5) = 3.75, giving 6.6 over 3.75, about 1.76 times the term's IDF. Document B has the same tf 3 but length 500, so its denominator is 3 + 1.2 times (0.25 + 3.75) = 7.8, giving 6.6 over 7.8, about 0.85 times IDF. Same term frequency, half the score, purely because B's mentions are diluted across ten times the text. Real ranking then layers signals beyond text: a recency boost multiplies scores by a decaying function of age so fresh listings surface, and a popularity boost folds in click or sales counts, with the standard caution that popularity feedback loops entrench incumbents and need damping.

The distributed query: scatter, gather, and pagination

Because documents are sharded by hash, every shard holds a random slice of the corpus, and a query must visit all of them. The coordinator broadcasts the query, each shard runs it locally and returns only its top results with scores, and the coordinator merges. With two shards and a request for the top 3: shard 1 returns A at 9.1, B at 8.4, C at 7.9, and shard 2 returns D at 8.8, E at 8.0, F at 6.5; the merged top 3 is A, D, B, assembled from twelve numbers rather than two full result sets, which is what keeps the gather step cheap. The latency consequence is less friendly: the user waits for the slowest shard, so the 95th-percentile of the maximum over 10 shards is governed by each shard's worst moments, and tail-latency tricks like hedged requests, sending a duplicate query to a replica when the first copy is slow to answer, earn their complexity here.

Pagination hides a trap called deep paging. Page 100 with 20 results per page asks for results 1,981 to 2,000, but under scatter-gather each shard must return its own top 2,000 candidates, because any of them might rank globally above the cutoff, so a 10-shard cluster materializes and sorts 20,000 entries to serve 20, and the cost grows linearly with page depth. The fix is the search_after cursor from the API: instead of an offset, the client passes the sort values of the last result it saw, and each shard returns only its top 20 entries ranking strictly after that boundary, making page 100 cost the same as page 1. The product concession is that users cannot jump to an arbitrary page number, which virtually no search user does anyway. Filters ride along inexpensively because structured fields are indexed too, and facet counts are computed per shard and summed by the coordinator during the same gather.

Scaling, failures, and operations

Read throughput scales with replicas: each shard keeps one or more full copies on other nodes, the coordinator load-balances queries across them, and a thousand QPS spread over 10 shards with two replicas each is a modest few hundred shard-queries per second per node. Write throughput scales with shard count, but changing the number of shards changes every document's hash assignment, so shard count is chosen with headroom up front, ten shards of five million documents each here, and a reshard is treated as the same heavyweight operation as a reindex. The queue in the write path is what makes indexing resilient: if the indexer or a shard falls over, updates accumulate in the queue and replay afterward, and search serves slightly stale results in the meantime, which is the graceful version of that failure.

The operation that deserves its own plan is changing the index itself, because analyzers and field mappings are baked into the files at index time. A new stemmer, a new field type, or a new shard count all require rebuilding, and the safe pattern is build-alongside-and-swap: create a second index with the new configuration, bulk-reindex all fifty million documents into it from the source of record while the live index keeps serving and dual-writing both, verify counts and spot-check rankings, then atomically flip an alias, a named pointer that clients query instead of a physical index name, from old to new. Rollback is flipping the alias back, and the old index is kept for a few days before deletion. At a few thousand documents per second, the rebuild takes a few hours, which is exactly why the alias indirection exists rather than any in-place migration.

Failures otherwise follow the shard-replica pattern: a node loss promotes replicas of its primaries and rebuilds the lost copies elsewhere, a slow node is the tail-latency story above, and the coordinator tier is stateless and scales flat behind a load balancer. The metric pair worth alerting on is refresh-to-visible lag on the write side and per-shard p99 on the read side, because each one degrades quietly before anything actually breaks.

Follow-up questions

  • Why not serve search from the primary database? Text matching by table scan is linear in corpus size, there is no ranking, and no linguistic normalization, so "running" never matches "runs." The inverted index makes lookup proportional to result size rather than corpus size, and BM25 gives an ordering worth showing.
  • Why shard by document instead of by term? Term sharding sends each query to only the shards owning its terms, but multi-term queries then need postings shipped across the network to intersect, hot terms create hot shards, and a document update touches many shards. Document sharding keeps every intersection local and an update on one shard, which is why real systems choose it.
  • What does "near-real-time" actually mean here? New documents become searchable on the next refresh, about a second after indexing, because they sit in an in-memory buffer until flushed as a segment. The delay is a deliberate trade for lock-free immutable reads, and products that need read-your-own-write for the author handle that one case at the application layer.
  • How would you add typo tolerance? Index character n-grams or maintain a fuzzy term lookup that matches dictionary terms within edit distance 1 or 2, then rank exact matches above fuzzy ones. It widens candidate sets, so it is best gated to queries that returned few or no results.
  • Why is deep pagination expensive, and what is the fix? Offset N forces every shard to return N plus page-size candidates so the coordinator can find the true global slice, so cost grows with depth. A search-after cursor keyed on the last result's sort values keeps every page the price of the first, at the cost of no random page jumps.
  • How do you change an analyzer in production? Never in place: index files embed the old analysis. Build a new index alongside, reindex from the source of record while dual-writing, validate, then swap the serving alias atomically and keep the old index briefly as the rollback path.
