Design a top-K most accessed items tracker

Systems design · Distributed building blocks · Apr 2025

Every large consumer product has a surface that answers one question: what is hot right now? Trending hashtags on a social network, most-viewed videos on a streaming site, hot search terms in a search box, top products on a storefront. Behind each of those sits the same system: a service that watches an enormous stream of access events and continuously maintains, for several time windows, the K items with the highest counts. The windows are part of the product, because "trending in the last five minutes" and "most viewed today" are different features, so a real design tracks something like the last 1 minute, 5 minutes, 1 hour, and 1 day, with K between 10 and 100.

Interviewers reach for this question because the obvious answer does not survive contact with the numbers, and the standard escape, accepting approximation on the hot path, requires explaining a probabilistic data structure carefully rather than waving at it. The hard parts live in the memory math, in the count-min sketch's error guarantees, and in a subtle distributed problem: combining per-node top-K lists is not as sound as it looks.

Scope and requirements

Functionally, the system ingests access events, each carrying an item ID and a timestamp, and serves a query of the form "give me the top K items for window W," where W is one of the fixed windows above. Results should be fresh within a few seconds for the short windows, and the service should also answer historical questions such as yesterday's top videos, which implies keeping exact daily results around. K is fixed and small, and we do not need arbitrary ad hoc windows; fixed windows are what the product shows and what makes precomputation possible.

Non-functionally, assume one million events per second at peak, query latency of a few tens of milliseconds since the lists render on home pages, and high availability. The crucial relaxation to negotiate early is accuracy: the short-window lists may be approximate, because no user can tell whether the fifth trending hashtag really has 1,002,113 views or 1,001,890, but the daily results that feed reporting and payouts should be exact. That split in accuracy requirements is the design.

Sizing the problem

Start with why exact counting in memory fails. At one million events per second, a day sees 86.4 billion events, and a popular platform can plausibly touch 500 million distinct items in that day (videos, hashtags, queries). A hash table entry holding an 8-byte item identifier, an 8-byte counter, and roughly 34 bytes of pointer and bucket overhead costs about 50 bytes, so an exact table over a day's distinct items is 500 million times 50 bytes, about 25 GB, for one window on one machine. The 1-hour, 5-minute, and 1-minute windows each need their own counts, and sliding those windows means keeping per-interval tables, so the total comfortably exceeds any single node's memory, and sharding it everywhere makes every query a scatter-gather over hundreds of gigabytes of state. Exactness is affordable in batch on cheap storage, but not in RAM on the hot path.

Bandwidth, by contrast, is manageable: a million events per second at roughly 100 bytes each is 100 MB/s into the ingestion tier, which one well-provisioned queue cluster absorbs without drama. The bottleneck is memory for counting, not the pipe, and that observation points directly at the two-path architecture.

The query interface

The read side is a small HTTP API over precomputed lists, which keeps query latency independent of event volume. Writes are not an API at all from the product's perspective: services emit access events to a queue topic, and everything downstream is this system's internals.

GET /v1/top?window=5m&k=10
→ 200 {
  "window": "5m",
  "as_of": "2025-04-18T17:42:30Z",
  "items": [
    { "item_id": "video-8841", "count": 1250410 },
    { "item_id": "video-1029", "count": 983112 }
  ],
  "exact": false
}

GET /v1/top?window=1d&date=2025-04-17&k=100   → 200 { ..., "exact": true }

The exact flag is product-facing because the two paths have different guarantees, and downstream consumers such as reporting jobs should be able to insist on the exact variant.

The high-level architecture

All events flow into a message queue, which fans them out to two consumers with opposite trade-offs. The fast path is a fleet of stream counters that maintain approximate counts in fixed memory and push fresh top-K lists to a serving store every second or two. The slow path holds the same events durably in the queue (or an archive), and batch jobs aggregate them into exact per-window counts hourly and daily, overwriting the approximate results for those windows after the fact. Queries always hit the serving store, never the counters, so reads are cheap and the counting tier can be rebuilt freely. This is the lambda-architecture shape: a low-latency approximate layer and a high-latency exact layer converging on one serving view.

Events flow once into the queue and twice out of it: the stream counters publish approximate top-K lists within seconds, and the batch aggregator replaces them with exact lists hours later. Queries only ever read the store.

The fast path: count-min sketch plus a heap

The fast path needs approximate counts for every item in bounded memory, and the standard tool is the count-min sketch. A count-min sketch is a small two-dimensional array of counters, d rows by w columns, with one independent hash function per row. To record an event for item x, compute each row's hash of x, which picks one column in that row, and increment that one counter in each of the d rows. To estimate x's count, read the same d counters and take the minimum. Every counter is shared by many items, so each cell only ever over-counts (collisions add, never subtract), which is why the minimum across rows is the best single cell to trust: it is the cell where x collided with the least extra traffic. Estimates therefore never undershoot the true count and overshoot by a bounded amount.

The dimensions come from two knobs. The width controls the error: with total event count N, the estimate exceeds the truth by at most (e/w) times N, where e is Euler's number, about 2.718. The depth controls the confidence: the bound holds with probability 1 minus (1/2 raised to a power proportional to d), and the convenient form is d = ln(1/delta) for failure probability delta. Choosing an error of 0.001 percent of N and delta of 0.1 percent gives w = 2.718 divided by 0.00001, about 272,000 columns, and d = ln(1000), about 7 rows. That is 272,000 times 7, roughly 1.9 million counters, and at 4 bytes each the whole sketch is about 7.6 MB. Compare that with the 25 GB exact table: five orders of magnitude smaller, with a quantified price. Over a one-minute window of 60 million events, the error bound is 0.00001 times 60 million, about 600 counts, which cannot confuse a genuinely hot item with the long tail.

One event for item x increments exactly one counter per row, at the column chosen by that row's hash. Reading takes the minimum of the same cells, so collisions can only inflate the answer, never hide it.

The sketch alone answers "how many times was x accessed?" but not "which items were accessed most?", because you cannot enumerate a sketch. The companion structure is a min-heap of the current K best candidates: for every incoming event, estimate the item's count from the sketch, and if it exceeds the smallest count in the heap, insert or update it, evicting the minimum. A heap of K = 100 entries costs a few kilobytes, and the pair of structures per window per counter node stays under 10 MB, which is why each counting node can maintain all four windows simultaneously. Windows are handled by keeping one sketch-plus-heap per interval, for example sixty 1-second sketches whose heaps are merged for the rolling minute, or more simply one sketch per tumbling window that resets on the boundary, which is what most products actually ship.

Distribution introduces the subtle flaw worth naming unprompted. With events partitioned across, say, three counter nodes by item hash, per-node top-K lists merge cleanly, because every event for a given item lands on one node, so that node's estimate is the global estimate. But if events are partitioned by source or arrive at whatever node is nearest, an item's count is split across nodes, and merging per-node top-K lists becomes approximate in a way no sketch math fixes: an item that is number 11 on each of three nodes is invisible in every top-10 list they report, yet its summed count could place it fifth globally. Concretely, if each node's tenth item has 1,000 local accesses and item X has 900 on every node, X's true total of 2,700 beats any item that appears on only one node with 1,500, but no merger of the three top-10 lists will ever see X. The fix is to partition events by item ID so each item is counted whole on one node, and that single routing decision is the difference between a sound design and a quietly wrong one.

The slow path and the serving store

The slow path exists because the product also needs exact, auditable numbers and a recovery story. The queue retains raw events for a few days, and an archival consumer copies them into cheap blob storage partitioned by hour. Batch jobs in the MapReduce style then compute exact counts per item per window: the map phase emits (item, 1) pairs from each event file, the shuffle groups by item, and the reduce phase sums and takes the true top K with a full sort of counts, which is trivial offline even at 86 billion events because it streams from disk across many workers rather than living in RAM. Hourly jobs finalize the 1-hour windows and daily jobs finalize the days, writing results into the same serving store keyed by window, where they overwrite the fast path's approximations. Reconciliation falls out for free: the difference between the approximate and exact lists for the same window is a continuous measurement of sketch quality, and a growing gap is the alert that the fast path needs wider sketches or has a routing defect.

The serving store itself is a small key-value table, keyed by window identifier with the ordered list as the value, for example top:5m:2025-04-18T17:40 → [(video-8841, 1250410), ...]. Each entry is a few kilobytes, queries are single-key gets, and a cache in front handles the home-page read storm. An alternative framing replaces both hand-built paths with a stream processor: Flink consuming the same topic, keyed by item, with windowed aggregations and exactly-once state checkpoints can produce exact short-window counts too, holding per-key state in managed RocksDB rather than RAM. That is the better answer when the team already operates Flink and the distinct-item cardinality per window is modest; the sketch design is the better answer at extreme cardinality or when an operationally minimal fast path matters. Either way, event time needs care: events arrive late and out of order, so the aggregator tracks a watermark, a moving estimate of "no events older than T are still coming," closes windows only when the watermark passes them, and either drops later stragglers or issues corrections to the store.

Scaling, failures, and operations

Each tier scales separately. The queue scales by partitions, and partitioning by item ID is already required for correctness, so adding counter nodes means adding partitions and consumers in step. Counter nodes hold only sketches and heaps, so a crashed node restarts empty and rebuilds its current windows by replaying the last few minutes from the queue, which is the great operational virtue of keeping the fast path's state small and reconstructible. The batch tier scales like any MapReduce workload, by adding workers, and a failed job simply reruns since its inputs are immutable files. The serving store and query API scale with read replicas and caching, and they are the only tiers users can see fail, so they get the redundancy budget.

The failure modes worth rehearsing are quieter than crashes. A hot item concentrated on one queue partition can lag that partition's counter, which shows up as one window's list going stale; the mitigation is monitoring per-partition consumer lag and over-partitioning so a single hot key is a small fraction of one node's load. Watermark misconfiguration silently undercounts items whose traffic arrives late from mobile clients. And the sketch's one-sided error means collisions can only promote tail items, never demote hot ones, so the worst user-visible failure is a marginal item appearing at rank K, which is exactly the kind of defect the exact path's reconciliation catches within the hour.

Follow-up questions

  • Why not just count exactly in Redis? A day of 500 million distinct counters is about 25 GB before replication, per window, with a million increments per second against it. It can be sharded into existence, but the sketch gives the same product answer in 8 MB per window, and the exact batch path already covers every use that needs precision.
  • Why does the count-min sketch never undercount? Every occurrence of item x increments all d of x's counters, so each one is at least x's true count; collisions add other items' counts on top. The minimum is therefore a guaranteed upper-bound-free lower estimate: at least the truth, at most the truth plus bounded noise.
  • How do you get a top-K list out of a sketch at all? You cannot enumerate it, so you maintain a K-entry min-heap on the side, updated on every event using the sketch's estimate. The heap remembers identities; the sketch remembers counts.
  • What goes wrong merging top-K lists from nodes that each saw part of the traffic? An item just below the cutoff on every node is dropped from every list, even though its global sum belongs in the top K, like the item with 900 accesses on each of three nodes whose 2,700 total goes unseen. Partitioning events by item ID removes the problem by making each node's count global for its items.
  • How are late events handled in the windowed counts? The stream path closes a window when the watermark passes it, then either drops late events or emits corrections; the batch path naturally includes everything that arrived by job time. The exact daily list is the system of record, and the product copy explains that short-window trends are estimates.
  • What changes for "top K in the last N minutes" sliding continuously? Keep fine-grained tumbling sketches, for example per second, and answer a query by merging the last N times 60 sketches' counts for heap candidates. Count-min sketches merge by cell-wise addition since they share hash functions, which is the property that makes the decomposition work.
