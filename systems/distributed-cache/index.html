Design a distributed cache

Systems design · Distributed building blocks · Mar 2025

A cache keeps copies of frequently read data in memory so requests can skip the slower system behind it. The numbers explain why every large service runs one: a query that reads from a disk-backed database typically costs 10 to 20 milliseconds once disk seeks and query execution are paid, while fetching the same bytes from a memory cache over the local network costs a few hundred microseconds, roughly 100 times faster. The protection effect is just as important as the speed: at a 95 percent hit rate, only 5 percent of reads reach the database, so the database sees one twentieth of the read load it would otherwise carry, which is frequently the difference between a database tier that needs 3 machines and one that needs 60.

Interviewers ask the question because a cache looks like a hash table until it is distributed, and then every classic problem appears in miniature: how to spread keys across nodes, what to evict when memory fills, what happens when ten thousand requests miss the same key in the same instant, and how stale data is tolerated or chased out. The hard parts live in the failure and contention cases, not the happy path, and a strong answer spends its time there.

Scope and requirements

Functionally the cache offers get, set with a time-to-live, and delete, over string keys and binary values up to perhaps 1 MB, shared by many application servers rather than private to one process. It is a look-aside accelerator for a database that remains the source of truth, so the cache is allowed to lose data on a crash; what it is not allowed to do is serve wrong answers indefinitely or fall over in a way that takes the database with it.

The non-functional requirements: sub-millisecond access at the median, throughput of hundreds of thousands of operations per second across the cluster, capacity beyond one machine's memory so the data must shard, high availability because a cold cache redirects its full read load onto the database at the worst possible moment, and predictable behavior under the pathological access patterns (hot keys, stampedes, floods of misses) that production traffic eventually produces.

Sizing the problem

Take a service handling 100,000 page requests per second at peak, where rendering a page makes on average 5 cache reads: that is 500,000 gets per second against the cluster. A single well-run cache node sustains on the order of 100,000 operations per second, so 5 to 6 nodes cover the throughput before memory even enters the picture. At a 95 percent hit rate the misses are 25,000 per second, which is the read load the database must absorb in steady state, and that number doubling because the hit rate slipped to 90 percent is the kind of quiet regression worth alerting on: hit rate is the cache's single most important health metric.

Memory follows from the working set, the subset of data that is actually hot. Suppose the catalog holds 500 million objects averaging 2 KB, which is 1 TB in total: caching everything is possible but wasteful, because access skews heavily. If 10 percent of objects receive essentially all of the day's reads, the working set is 50 million objects at 2 KB, about 100 GB, and a cluster of 4 nodes with 32 GB devoted to cache each holds it with headroom. Getting this estimate wrong in the small direction shows up directly in the hit rate, because a cache smaller than the working set evicts entries that are still hot, and each such eviction converts future hits into misses.

The interface and access patterns

The client API is tiny, and showing the canonical usage pattern matters more than the signatures, because the pattern determines the consistency story. Cache-aside, the default, puts the application in charge: read the cache, fall through to the database on a miss, and refill.

def get_product(pid: str) -> Product:
    key = f"prod:{pid}"
    if (v := cache.get(key)) is not None:
        return v                          # hit
    row = db.query("SELECT ... WHERE id = %s", pid)
    cache.set(key, row, ttl=300)          # refill, 5 min TTL
    return row

def update_product(pid: str, fields: dict):
    db.update(pid, fields)                # source of truth first
    cache.delete(f"prod:{pid}")           # invalidate, next read refills

Three alternatives shift the work around. Read-through moves the miss handling into the cache itself, which fetches from the database on the application's behalf: the application code gets simpler and the cache layer needs a loader plugged in, but behavior is otherwise cache-aside's. Write-through updates the cache synchronously on every write as well as the database, keeping the cache permanently fresh at the cost of slower writes and of caching items that may never be read. Write-behind acknowledges the write after updating only the cache and flushes to the database asynchronously, which gives the fastest writes and absorbs write bursts, and in exchange a cache node crash loses acknowledged writes, so it is only appropriate when the data is reconstructible or the loss window is made tiny; for the look-aside accelerator scoped here, cache-aside with delete-on-write is the recommendation, and the eviction and invalidation sections explain the rest of its consistency story.

Topology: how keys spread across nodes

Once the data exceeds one node, some layer must map each key to a shard, and there are three places to put that logic. Client-side sharding builds the map into the client library: each application server hashes the key with consistent hashing, the scheme where nodes and keys share a hash ring so that membership changes move only about 1/N of the keys, and talks directly to the owning shard. This is the Memcached lineage (the Ketama library made it standard), it adds zero extra hops, and its weakness is that every client must hold a correct, current view of the node list, which makes rolling topology changes delicate. A proxy topology inserts a small router, Twemproxy being the well-known example, so clients speak to the proxy and the proxy owns the sharding map: clients stay simple and topology changes localize to the proxy fleet, at the price of one added network hop and a new tier to operate. A cluster protocol builds routing into the cache nodes themselves, as Redis Cluster does: the keyspace is divided into 16,384 slots, each node owns a range of slots, a key's slot is a hash of the key modulo 16,384, and a client that asks the wrong node receives a redirect naming the right one, which smart clients cache, so the steady state has no extra hops and the cluster can move slots between nodes online.

The client library hashes each key to one of three shards with consistent hashing, and each shard streams its writes asynchronously to a replica that can be promoted if the primary dies.

Eviction: choosing what to forget

Memory is the budget, so the cache must constantly decide what to discard, and the default policy nearly everywhere is least recently used: evict the entry that has gone longest without being touched, on the theory that recent access predicts future access. The classic implementation is worth being able to draw, because it achieves constant time for every operation: a hash map gives direct access to entries, and the entries are also threaded onto a doubly linked list ordered by recency, where every get unlinks the touched entry and reattaches it at the head, and eviction pops from the tail. Both structures update in a constant number of pointer operations, which is why a cache can run this on every single request. Production systems shave the bookkeeping further; Redis, for instance, approximates LRU by sampling a handful of keys and evicting the least recent among the sample, which behaves close to true LRU at a fraction of the memory overhead.

LRU has a known weakness: a burst of one-time reads, a batch job scanning the catalog, floods the recency list and evicts the genuinely hot set. Least frequently used eviction resists that by counting accesses and evicting the coldest by count, at the cost of heavier bookkeeping and slow adaptation when popularity shifts, since an item hot last week retains a high count; hybrids with decaying counters split the difference. Time-to-live is the orthogonal mechanism: every entry carries an expiry chosen at write time, which bounds staleness regardless of eviction pressure, and the working-set arithmetic from the sizing section is what keeps eviction in its lane, because when the cache comfortably fits the hot 100 GB, eviction trims the cold tail, and when it does not, eviction churns the hot set itself and the hit rate pays.

Hot keys, stampedes, and penetration

Sharding spreads the keyspace, not the popularity. When a celebrity posts, one key can draw hundreds of thousands of reads per second, all hashing to the same shard, and a shard that sustains 100,000 operations per second is now the whole system's ceiling. The fixes layer: request coalescing inside each application server collapses concurrent identical lookups into one outbound request; a small local in-process cache holding the few hottest keys for a second or two answers most reads before they touch the network, trading a bounded second of staleness for an enormous load reduction; and replicating hot keys under derived names (the key plus a random suffix from a small range, each replica readable interchangeably) spreads one key's load across several shards at the cost of multiplying its invalidations.

The cache stampede, also called the thundering herd, is the sharper version of the same emergency: a popular key expires, and every request that misses it simultaneously queries the database and tries to refill, so the database receives in one instant the very load the cache existed to absorb. With 10,000 requests per second against a key whose recompute takes 200 milliseconds, an unprotected expiry sends about 2,000 identical queries into the database before the first refill lands. Per-key locking is the direct defense: the first request to miss acquires a short-lived lock on that key (in Redis, a SET with the NX flag, which sets only if the key does not exist), performs the single database read and refill, and releases; everyone else either waits a few milliseconds and retries the cache or, better, is served the just-expired stale value while the refresh proceeds. Probabilistic early refresh removes the cliff altogether: each reader, with a probability that rises as expiry approaches, volunteers to refresh before the deadline, so hot keys practically never expire cold; the published version of this idea weights the probability by the recompute cost and provably minimizes stampedes for the noise it adds.

Step 1: thousands of requests miss the expired hot key at once. Step 2: each tries to acquire the per-key lock and exactly one wins. Step 3: the winner alone queries the database. Step 4: it refills the key with a fresh TTL and releases the lock. Step 5: the waiters are served the refilled value (or the stale value during the refresh) and the database saw one query instead of thousands.

Cache penetration is the third pathology: requests for keys that exist nowhere, from typos or attackers probing random IDs, miss the cache by definition and hit the database every time. Negative caching stores the absence itself, a short-TTL "not found" entry, so repeated probes for the same missing key stop at the cache; against attackers spraying unique keys, a Bloom filter in front of the database, a compact probabilistic structure that answers definitely-absent for most keys never written, filters the flood while occasionally letting a false positive through to the database, which is harmless.

Invalidation and the TTL backstop

Cache-aside's consistency story is delete-on-write: update the database, then delete the cache key so the next reader refills from fresh data. Deleting rather than setting the new value avoids one race (two concurrent writers setting in the wrong order), but a thinner race remains: a reader that misses, reads the database just before a writer commits, and refills just after the writer's delete will pin the stale value until something removes it. Fully closing that window requires versioned keys or coordination that costs more than it saves for most workloads, and the practical posture is to accept the small window and bound its damage with TTLs. Every entry gets one, even entries believed to be explicitly invalidated, because the TTL is the backstop that guarantees no failure mode (a missed delete, a crashed writer, the race above) can outlive a few minutes. Choosing the TTL is then a product decision stated in seconds: how stale may a price, a profile, a permission be, and the answer differs per key family, which is why the TTL belongs in the code at the call site rather than in a global config.

Scaling, failures, and operations

Scaling out means adding shards, and consistent hashing or slot migration keeps the reshuffle proportional to the capacity added rather than total; the operational caution is that every moved key arrives cold, so large rebalances should be paced to let the hit rate recover between steps. Each shard pairs with a replica receiving asynchronous updates, and when a primary dies, promotion (by Redis Sentinel, the cluster protocol, or the proxy's health checks) puts a warm copy in service within seconds; the alternative of restarting empty is the reason replication exists for a system that is allowed to lose data, because a cold shard at peak traffic converts its entire throughput into database load. The arithmetic from sizing makes that vivid: one cold shard out of four turns roughly 125,000 gets per second into misses, quintupling the database's read load until the shard rewarms, so availability work in a cache is really database-protection work. A full cache restart, after a datacenter power event, is the worst case of the same story, and mature operations pre-warm from a snapshot or replay recent keys before admitting traffic. The metrics that matter day to day are hit rate per key family, eviction rate (rising evictions with a falling hit rate mean the working set no longer fits), per-shard operation counts to spot hot keys early, and the database's read QPS, which is the cache's report card rendered on someone else's dashboard.

Follow-up questions

  • Cache-aside or write-through? Cache-aside keeps the cache optional and only caches what is read, at the cost of a stale window managed by delete-on-write and TTLs; write-through keeps the cache always fresh but slows every write and fills memory with never-read entries. Default to cache-aside unless reads must never see staleness that a TTL cannot bound.
  • Why is LRU the default, and when is it wrong? It is constant-time, simple, and recency predicts access for most workloads. It is wrong under scans, which flood the recency list, and for stable popularity skews where LFU or a sampled hybrid protects the truly hot set better.
  • How do you protect one key receiving 500,000 reads per second? Coalesce duplicate in-flight requests per app server, serve the key from a one-second local in-process cache, and if still needed replicate the key under suffixed names across shards. The shard stops being the ceiling once most reads never reach it.
  • What is the difference between a stampede and penetration? A stampede is concurrent misses on a key that exists, fixed with per-key locks, stale-while-revalidate, and early refresh; penetration is misses on keys that exist nowhere, fixed with negative caching and a Bloom filter. They overload the database through different doors.
  • Is the cache a source of truth for anything? Not in this design: every entry must be reconstructible from the database, which is what licenses eviction, crash-tolerance without persistence, and TTLs. Write-behind designs that buffer acknowledged writes change that contract and need durability machinery this design deliberately avoids.
  • Why not make cache and database consistent with transactions? Distributed transactions across a cache and a database cost latency and availability on every write to close a staleness window that TTLs already bound for cents. The proportionate tool is versioned keys for the few families that need it, not global coordination.
