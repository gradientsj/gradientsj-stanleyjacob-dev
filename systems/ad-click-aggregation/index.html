Design an ad click event aggregation system

Systems design · Data pipelines and observability · Oct 2025

When someone clicks an ad, three things hang on that event being counted correctly: the advertiser is billed for it, their campaign budget is paced against it, and the bidding system learns from it. This system ingests every click from a large ad network, aggregates counts per ad per minute, and serves those numbers to dashboards, budget pacers, and the billing pipeline. It is worth saying plainly why interviewers love it: this is a money pipeline. A metrics system that undercounts CPU usage wastes a capacity planner's afternoon, but an ad system that overcounts clicks bills customers for traffic that never happened, and the failure modes are measured in refunds, audits, and occasionally lawsuits. Correctness outranks freshness here, which inverts the priorities of most streaming designs and shapes every decision below.

The hard parts are time and duplication. Clicks arrive late and out of order from flaky mobile networks, the same click arrives twice because a phone retried, and the aggregation must nonetheless converge on one defensible number per ad per minute. The design that follows is the standard shape for this problem: a durable raw log as the source of truth, a streaming layer for fast approximate answers, and a batch layer that recomputes the truth at leisure.

Scope and requirements

Functionally, the system must return the number of clicks for a given ad in any one-minute window, return the top N ads by clicks over a recent window such as the last hour, and support queries over both recent data for pacing and historical data for invoices and disputes. Filtering out fraudulent clicks before they reach billable aggregates is in scope, because no real ad network bills raw clicks.

Non-functionally, correctness is the headline: billable numbers must be exact, deduplicated, and reproducible, in the sense that recomputing any window from raw events yields the same answer an auditor was shown. Freshness matters at the level of a minute or two for dashboards and pacing, not seconds. The system must absorb large traffic spikes, a viral moment or a holiday auction surge, without losing events, and every raw event must be retained long enough to recompute history when, not if, a defect in the aggregation logic ships.

Sizing the problem

Assume 1 billion clicks per day. A day has 86,400 seconds, so the average rate is about 11,600 clicks per second, and ad traffic is spiky enough that the system should be sized for peaks around 50,000 per second. A raw click event carries an event ID, ad ID, user identifier, timestamp, and a little context, roughly 0.1 KB, so a day of raw events is about 100 GB, a comfortably small volume by log standards. Keeping 90 days of raw events in Kafka with threefold replication costs about 100 GB times 90 times 3, or 27 TB, which is cheap insurance for a billing system. The aggregate side is smaller still: with around 2 million ads active in a day, even if every ad received clicks in every minute the day would produce 2.9 billion aggregate rows, and in practice click activity is concentrated, so daily aggregate output lands well under that and a year of it fits in a few TB of columnar storage.

The query API

Two consumers dominate: dashboards and pacing read recent windows, and billing reads reconciled history. The write side is an internal ingestion endpoint that ad servers call; end-user clicks never hit this API directly.

POST /v1/clicks                       ← from ad servers, batched
{ "events": [ { "event_id": "9f1c-...", "ad_id": 88231,
                "user_id": "u-2231", "ts": "2025-10-07T12:00:40Z" }, ... ] }

GET /v1/ads/88231/clicks?from=2025-10-07T12:00Z&to=2025-10-07T13:00Z
    &granularity=minute
→ 200 { "windows": [ { "minute": "12:00", "clicks": 1042,
                       "source": "reconciled" }, ... ] }

GET /v1/ads/top?window=1h&n=100       → 200 { "ads": [ ... ] }

The source field in the response is a deliberate piece of API design: every aggregate is labeled as streaming or reconciled, so downstream systems can choose, and billing simply refuses to read anything not yet reconciled.

The data model

The aggregate store is keyed by ad and minute, which is the access pattern for every query above, and each row carries the identity of the computation run that produced it so that recomputation can supersede old results without ever updating in place ambiguously.

CREATE TABLE clicks_per_minute (
  ad_id        BIGINT      NOT NULL,
  minute       TIMESTAMPTZ NOT NULL,   -- window start, UTC
  clicks       BIGINT      NOT NULL,
  filtered     BIGINT      NOT NULL,   -- clicks held out as suspect
  run_id       BIGINT      NOT NULL,   -- which computation produced this
  reconciled   BOOLEAN     NOT NULL DEFAULT FALSE,
  PRIMARY KEY (ad_id, minute, run_id)
);
-- readers take the highest run_id per (ad_id, minute),
-- billing additionally requires reconciled = TRUE

The high-level architecture

Ad servers log every click to Kafka, a durable, partitioned, append-only log, and that log is the source of truth for the entire system: raw, immutable, retained for 90 days, and never filtered or rewritten, because any cleverness applied before the log cannot be undone later. A stream processor consumes the log, deduplicates, applies fraud filtering, aggregates into one-minute windows, and writes rows to the aggregate store within a minute or two of real time. Independently, a nightly batch job replays the same raw log and recomputes every window with hours of hindsight, writing corrected rows that supersede the streaming ones. Dashboards read whatever is freshest; billing reads only reconciled rows. This is the two-path arrangement often called a lambda architecture, and its cost, maintaining two computations of the same logic, is the explicit price paid for putting a fast number on a dashboard without ever billing from it.

Every click lands in the raw log before anything else happens to it. The stream job publishes fast per-minute rows, the nightly batch replays the same log and writes corrections, and suspect clicks are routed aside rather than deleted.

Event time, windows, and watermarks

Aggregation here means windowed counting: the day is divided into tumbling windows, fixed, non-overlapping one-minute intervals, and each click increments the count of exactly one window. The subtlety is which window, because every event has two times. Event time is when the click actually happened on the device; processing time is when the event reached the pipeline. A click at 12:00:40 from a phone in a subway tunnel may not arrive until 12:02:30, and if the pipeline assigned it by arrival it would land in the 12:02 window, overcounting one minute and undercounting another, and replaying the log later would assign it differently again, which is fatal for reproducibility. Billing-grade aggregation must therefore window by event time, and that immediately raises the question a watermark answers: when is the 12:00 window finished?

A watermark is the pipeline's moving claim that all events with event time earlier than T have, it believes, arrived, and it is typically computed as the maximum event time seen so far minus an allowed lateness, say two minutes. When the watermark passes 12:01, the 12:00 to 12:01 window closes and its count is emitted. The setting is a trade with no free option: a tight watermark of a few seconds emits windows fast but calls more events late, while a generous one of ten minutes catches almost every straggler but delays every dashboard and pacing decision by that long. The practical middle ground emits at a two-minute watermark, accepts that a small tail will still arrive later, and lets the closed window be corrected, in the streaming layer through late-update rows, and definitively by the nightly batch, which by then has seen everything.

Step 1: clicks arriving on time count into the window holding their event time. Step 2: the watermark passes 12:01 and the first window closes and emits. Step 3: a click stamped 12:00:40 arrives during 12:02 and is routed back as a correction to the already-closed window.

Counting each click exactly once

Mobile clients retry over bad networks, so the same click arrives more than once as a matter of routine, and the defense starts at the source: the client mints a unique event ID when the click happens, not when the request is sent, so every retry carries the same ID. The stream job keeps a set of recently seen IDs in its keyed state and drops repeats; the set is bounded by the watermark, since an event older than the allowed lateness will be handled by reconciliation anyway, so a few minutes of IDs at 50,000 events per second is on the order of tens of millions of small entries, which fits in operator state without strain.

Deduplication handles duplicate inputs, but the pipeline can also duplicate its own work when it crashes mid-stream, and the guard there is the checkpoint: the stream processor periodically snapshots all operator state, the partial window counts and the dedup set, together with the exact Kafka offsets it had consumed, the positions in the log marking how far it had read. After a crash it restores the snapshot and resumes from those offsets, so no event is skipped and none is counted twice inside the processor. The last gap is the boundary with the store, because a job that crashes after writing a row but before recording the checkpoint will rewrite that row on recovery; the fix is an idempotent sink, meaning a write that can be applied many times with the same result, which the schema above provides by making the row key include the window and the run, so a rewrite of the 12:00 row for ad 88231 replaces an identical value rather than adding to it. Transactional sinks, where the write and the checkpoint commit or abort together, are the heavier alternative when the target store supports them. None of this machinery makes events arrive exactly once, which is impossible over real networks; it makes their effect on stored aggregates happen exactly once, and that is the property billing needs.

Fraud filtering and the reconciliation path

Raw clicks are not billable clicks. Some fraction comes from automated traffic, click farms, and accidental double taps, and a filtering stage sits in the stream job ahead of billable aggregation, applying cheap rules first, impossible click rates from one device, datacenter IP ranges, clicks with no matching ad impression, and then a model score for subtler patterns. Suspicious events are not deleted; they are routed to a quarantine store with the reason attached, counted in a separate filtered column, and kept for dispute resolution, because an advertiser challenging an invoice will ask exactly what was excluded and why, and because the filters themselves need auditing when their thresholds drift.

The batch path is what makes the whole system defensible. Every night, a job replays the raw log for the previous day and recomputes every window with complete information: every late event has arrived, fraud filters run with final rule versions and the day's full context, and the output is written as a new run with the reconciled flag set. The correction policy is explicit and asymmetric: dashboards and pacing read streaming rows because they need speed and tolerate small drift, while billing reads only reconciled rows, so an invoice is never built on a number that a late click could still move. Streaming and batch totals are compared every night, and a drift beyond a small threshold pages someone, since persistent disagreement means one of the two computations has a defect. The same machinery doubles as the recovery procedure after a faulty deploy: rewind the consumer offsets to before the damage, rerun the corrected job over the affected range, and publish the output under a new run ID, which supersedes the wrong rows without destroying the audit trail of what was previously reported.

Scaling, failures, and operations

Kafka partitions the click topic by ad ID, which gives each stream-processor task a disjoint slice of ads and makes window state local; 50,000 events per second at 0.1 KB is 5 MB/s, trivial for the log itself, so the partition count is set by the parallelism the stream job wants rather than by throughput. The aggregate store partitions the same way and scales reads with replicas. The stream job scales by adding tasks, and its failure story is the checkpoint replay described above, with the queue absorbing backlog while the job catches up. One ad going viral creates a hot key, a single partition receiving a disproportionate share of traffic, and the standard relief is local pre-aggregation: split the hot ad's counts across sub-keys, count each independently, and sum them at emission, trading a little merge work for an even load.

The failures worth rehearsing are the quiet ones. If the stream job lags, dashboards go stale but nothing is lost, and an alert on consumer lag catches it within minutes. If the aggregate store is briefly unavailable, the job pauses against its checkpoint rather than dropping output. If a fraud rule misfires and quarantines a legitimate campaign's clicks, the quarantine store plus reconciliation rerun makes the repair a recomputation, not an apology. The one disaster with no good recovery is losing raw events before they reach the log, which is why the ingest gateway acknowledges an ad server only after Kafka has replicated the write, and why the log's replication and retention settings are treated as billing-critical configuration rather than tunables.

Follow-up questions

  • Why window by event time instead of arrival time? Arrival-time windows change with network weather and replays, so the same log would yield different invoices on different days. Event-time windows make every recomputation land identically, which is the definition of auditable.
  • How do you choose the watermark delay? Measure the real lateness distribution: if 99.9 percent of clicks arrive within two minutes, a two-minute watermark leaves a 0.1 percent tail for reconciliation to fix. Tighter delays trade correction volume for freshness.
  • What does exactly-once actually mean here? Not that events traverse the network once, which is impossible, but that each event's effect on stored aggregates is applied once: dedup by client-minted event ID, checkpointed state with Kafka offsets, and idempotent keyed writes together give that.
  • Why keep both a streaming and a batch path? They serve different masters: pacing needs an answer in a minute, billing needs an answer that cannot move. One path could serve both only by being as slow as batch or as approximate as streaming.
  • How does top-100 ads over the last hour work? The OLAP store sums sixty per-minute rows per ad and takes the top of the ranking; at a few million active ads that is an inexpensive scan. A streaming top-K with a count-min sketch can serve the dashboard version if sub-second latency is wanted.
  • A deploy miscounted six hours of clicks. Now what? Rewind offsets to before the deploy, replay the raw log through the fixed job, publish under a new run ID, and let reconciled rows supersede the wrong ones. The raw log's immutability is what makes this a procedure instead of an incident report.
