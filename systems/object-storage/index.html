Design S3-like object storage

Systems design · Media and storage · Aug 2025

Object storage is the system that holds bytes for everyone else. A client names a bucket and a key, uploads a blob of any size from a few bytes to a few terabytes, and gets it back later with a GET, and that simple contract turns out to be the foundation under data lakes, photo services, backup products, machine learning training sets, and most of the modern web's static content. Interviewers like the question because it forces a candidate to take durability seriously as a number rather than a slogan, and because the design splits naturally into a metadata problem and a data problem that scale in completely different ways.

The hard parts are not where they first appear. Serving a GET is easy; promising that one hundred billion objects will still be readable in a decade, while disks fail every day and the fleet is being repaired, rebalanced, and upgraded underneath, is the actual product. The walkthrough below builds the system the way I would in an interview: requirements, sizing, the two-plane architecture, then the deep ends of redundancy, small objects, and metadata.

Scope and requirements

Functionally, the service stores objects in buckets under a flat keyspace. There are no real directories: a key like logs/2025/08/14/host7.gz is just a string, and the directory illusion comes from listing keys by prefix. The core operations are PUT, GET, and DELETE on a single object, listing a bucket by prefix, and multipart upload so a client can push a 100 GB file in parallel parts and have it appear atomically at the end. Versioning, lifecycle policies that move old data to cheaper storage, and per-object metadata round out the surface.

The non-functional requirements carry the design. Durability is the headline: the standard claim is eleven nines, 99.999999999 percent per object per year, and it is worth saying what that means. A loss probability of 10-11 per object-year over a fleet of 100 billion objects gives an expected loss of about one object per year across everything everyone has ever stored. Availability matters but sits a tier lower, around four nines, because a retryable 500 is an inconvenience while a lost object is forever. Throughput must scale to many gigabytes per second of aggregate ingest and egress, and cost per stored terabyte decides whether anyone uses the service at all, which is what pushes the design toward erasure coding and dense disks rather than triple replication on fast media.

Sizing the problem

Assume the service holds 100 PB of logical data with an average object size of 1 MB. That is 1017 bytes over 106 bytes per object, which is 1011 objects: one hundred billion. This single division explains most of the architecture, because if a metadata record costs about 256 bytes, the metadata alone is 100 billion times 256 bytes, roughly 25 TB before replication and about 75 TB with three copies. Metadata is therefore its own distributed storage problem, with its own sharding, replication, and failure handling, and treating it as a footnote is the classic way to fail this interview.

Traffic scales similarly. Suppose clients PUT 5,000 objects per second on average, which at 1 MB each is 5 GB/s of ingest, about 430 TB per day, and reads run ten to one over writes for 20 GB/s of egress at peak. With a 1.5x redundancy overhead the fleet stores 150 PB raw, which at 20 TB per disk is 7,500 disks, or about 250 storage nodes at 30 disks each. Those 7,500 spindles can stream over 1 TB/s in aggregate, so raw bandwidth is comfortable, and the real constraints become per-key hotspots, repair traffic, and metadata operations per second.

The API

The external interface is deliberately small, and it is HTTP because every language and tool already speaks it. Multipart upload is the one piece with real protocol shape: initiate, upload parts independently, then commit the whole list in one call so the object is never visible half-written.

PUT /photos/cats/whiskers.jpg          ← body is the object bytes
→ 200 { "etag": "9b2cf535..." }

GET /photos/cats/whiskers.jpg          → 200 + bytes (Range supported)
DELETE /photos/cats/whiskers.jpg       → 204

GET /photos?prefix=cats/&max-keys=1000 → 200 { "keys": [...], "next": "..." }

POST /videos/launch.mp4?uploads        → 200 { "uploadId": "Vqx91" }
PUT  /videos/launch.mp4?partNumber=3&uploadId=Vqx91   ← one 100 MB part
POST /videos/launch.mp4?uploadId=Vqx91 ← body lists part etags, commit

The data model

Inside the metadata service, two logical tables do the work. The object table maps a fully qualified key to the object's current version, size, checksum, and the layout of its data, and the chunk table maps each chunk to the physical nodes holding it. Writing it as SQL keeps the sketch concrete even though the real store is a purpose-built sharded system.

CREATE TABLE objects (
  bucket      VARCHAR(64)  NOT NULL,
  key         VARCHAR(1024) NOT NULL,
  version     BIGINT       NOT NULL,   -- newest wins; old rows kept if versioned
  size        BIGINT       NOT NULL,
  checksum    BYTEA        NOT NULL,   -- whole-object CRC
  chunk_ids   BIGINT[]     NOT NULL,   -- ordered layout of the bytes
  deleted     BOOLEAN      NOT NULL DEFAULT FALSE,  -- delete marker
  PRIMARY KEY (bucket, key, version)
);

CREATE TABLE chunks (
  chunk_id    BIGINT PRIMARY KEY,
  stripe_id   BIGINT,                  -- erasure-coding group, if coded
  nodes       INT[]  NOT NULL,         -- data nodes holding this chunk
  checksum    BYTEA  NOT NULL
);

The separation matters because the two tables answer different questions at different rates: the object table is hit on every request, while the chunk table is also walked by repair and garbage collection, and keeping placement out of the object row lets the system move bytes between nodes without touching user-visible metadata.

The high-level architecture

The system splits into a control plane and a data plane. A stateless API tier authenticates requests and orchestrates each operation; a metadata service owns the mapping from keys to chunk locations; a placement service decides which data nodes should receive new bytes, balancing capacity, load, and failure domains; and a large fleet of data nodes does nothing but store, serve, and verify chunks. The crucial property is that object bytes never flow through the metadata service, and metadata never lives on the data path's disks, so each plane scales on its own terms.

The PUT path: the API tier asks placement where new bytes belong, streams chunks straight to data nodes, and commits the key with its chunk locations in the metadata service. A GET runs the same path in reverse, resolving the key first and then reading chunks directly from the fleet.

The write path and durability

A PUT begins at the API tier, which computes a checksum, a small fingerprint such as a CRC derived from the bytes that later reveals any corruption, while the body streams in. The placement service returns a set of target nodes chosen across failure domains, meaning groups of hardware that tend to fail together, so no two copies of the same data share a rack, a power feed, or ideally a building. The API tier streams the chunks to those nodes, each node persists its chunk and verifies the checksum before answering, and the write is acknowledged to the client once a quorum, a minimum count of successful copies such as two of three replicas, has confirmed, with the remaining copy completed in the background. Acknowledging on quorum rather than on all copies keeps tail latency sane, because there is almost always one slow node in any set of three, while still guaranteeing the data survives a single immediate failure.

Only after the bytes are safe does the API tier commit the metadata row, and that ordering is the consistency story in miniature: the metadata commit is the moment the object exists. If the writer dies before the commit, orphaned chunks sit on data nodes referenced by nothing, and a garbage collector reclaims them later, which is a far better failure mode than metadata pointing at bytes that were never fully written. Checksums travel with the data forever: stored beside each chunk, verified on every read, and verified again by background scrubbing, so corruption is detected wherever it creeps in.

Replication against erasure coding

The simplest redundancy is replication: keep three full copies, survive any two losses, pay 3x in disks. At 100 PB logical that is 300 PB raw, and disks are most of the cost of the service, so the pressure to do better is enormous. The answer is erasure coding, a technique that splits a block of data into k equal data chunks and computes m additional parity chunks using Reed-Solomon arithmetic, with the property that any k chunks out of the k plus m total are enough to reconstruct the original block. With k equal to 8 and m equal to 4, a 32 MB block becomes eight 4 MB data chunks plus four 4 MB parity chunks spread across twelve different nodes, the storage overhead is 12 over 8, which is 1.5x, and the block survives the loss of any four chunks at once. Compared with replication, that is both less storage (1.5x against 3x) and more failure tolerance (four losses against two), which sounds like a free lunch until repair enters the picture.

Step 1: when node n3 dies, the repair worker reads any 8 of the 11 surviving chunks (two of the reads are drawn). Step 2: it recomputes the missing chunk and writes it to a spare node, restoring the stripe to full strength.

The price of erasure coding is repair amplification. When a replica is lost, replication copies one chunk to heal it, but when a coded chunk is lost the repair worker must read k chunks, eight in this scheme, to reconstruct one, so healing a dead 20 TB disk requires reading roughly 160 TB from peers. The saving grace is declustered placement: because every stripe lives on a different set of twelve nodes, the chunks of a dead disk have partners spread across thousands of other disks, and the rebuilt chunks can also be written to thousands of disks. If a thousand disks each contribute 50 MB/s of spare bandwidth, the fleet repairs at 50 GB/s and the 160 TB of reads complete in about 3,200 seconds, under an hour, whereas copying 20 TB into a single replacement disk at 150 MB/s would take a day and a half and leave the data exposed the whole time. Reads of degraded stripes also pay the k-way reconstruction cost, which is why hot data and small objects often stay on replication while the bulk of the corpus, large and cool, lives erasure coded.

Small objects, big objects, and metadata at scale

Erasure coding a 10 KB object directly would be absurd, since splitting it into twelve chunks of under a kilobyte each multiplies per-chunk fixed costs and seek overhead past the size of the data itself, and billions of small objects also threaten the filesystem beneath each data node, where every file costs an inode lookup and a directory entry. The fix, made famous by Facebook's Haystack photo store, is packing: small objects are appended into large block files of around 1 GB, the data node keeps an in-memory index from object ID to offset and length within the block, and reading a small object becomes one index lookup plus one disk read. The big blocks then become the unit of replication or erasure coding, so the redundancy machinery never knows small objects exist. Deletes write a tombstone in the index rather than touching the block, and a compaction process later rewrites blocks that have accumulated enough dead space, reclaiming it in bulk.

Large objects go the other way: a 100 GB upload arrives as multipart parts of, say, 100 MB, each part is striped and stored independently as it lands, and the final commit writes a manifest listing the parts in order. Parts upload in parallel from the client, a failed part retries alone without restarting the transfer, and a Range GET for bytes near the end touches only the stripes that hold them.

The metadata service shards the object table by a hash of bucket and key, which spreads load evenly and makes single-key operations cheap, but a hash destroys order, and listing a bucket by prefix is fundamentally an ordered scan. So listing needs an ordered secondary structure: each bucket maintains a key index in a range-partitioned store, one that splits data by sorted key ranges rather than hashes, so that prefix=cats/ becomes a seek to cats/ followed by a sequential scan. That index is updated asynchronously from the object table, which is why listings in real systems may trail writes by a moment even when single-key reads do not. For single keys the service promises read-after-write consistency for new objects, meaning a GET issued after a successful PUT sees the object, and the mechanism is direct: both the PUT's commit and the subsequent GET resolve through the same metadata partition, which is itself a small consensus group, so once the commit is durable every reader of that partition observes it.

Scaling, failures, and operations

Each tier scales by its own rule. The API tier is stateless and grows behind load balancers. The metadata service grows by splitting hot partitions, and its consensus groups failover automatically when a leader dies. Data nodes are the easy tier to grow and the interesting tier to operate, because at 7,500 disks with an annualized failure rate around 2 percent, the fleet loses roughly 150 disks per year, three per week, and disk death is a routine event that the system must metabolize without humans. When a node misses heartbeats, the placement service marks its chunks under-replicated and feeds a prioritized repair queue, with stripes that have lost the most redundancy healed first, since a stripe down to eight survivors out of twelve is one failure from data loss while a stripe at eleven is in no hurry.

Two background processes guard against quieter enemies. Scrubbing is the practice of continuously re-reading stored chunks and verifying their checksums, because disks suffer silent corruption, bit flips and misdirected writes that no error is ever raised for, and a corrupt chunk that is never read is a loss waiting to be discovered during a repair when it is needed most; a scrub pass over every byte each month costs each 20 TB disk about 8 MB/s of background reads, which is affordable. Garbage collection walks the chunk table against the object table and reclaims chunks that no live object version references, the debris of failed uploads, overwrites, and completed deletes. Deletes themselves are soft in a versioned bucket: a DELETE writes a delete marker, a metadata row saying the key now resolves to nothing, and older versions remain retrievable until lifecycle policy expires them. Lifecycle policies also drive tiering, rewriting objects untouched for months into colder, cheaper, more highly coded storage, and the eleven nines arithmetic deserves one closing dose of humility: at these redundancy levels the dominant remaining risks are correlated failures and defects in the software itself, which is why placement spreads stripes across independent failure domains and why the most careful operators in this business deploy storage code more cautiously than almost anything else they run.

Follow-up questions

  • Why not build this on a distributed filesystem? The flat keyspace deliberately drops hierarchical semantics, rename, locking, and partial overwrite, and that smaller contract is what allows hash-sharded metadata, immutable chunks, and erasure coding to work simply at 1011 objects.
  • When does replication beat erasure coding? For hot or small data: replicas serve reads from any copy with no reconstruction cost, repair by copying one chunk instead of reading eight, and avoid coding latency on the write path. The economics flip as objects get large and cool.
  • How does listing work if metadata is hashed? It cannot be served from the hash shards; a per-bucket ordered key index in a range-partitioned store serves prefix scans, maintained asynchronously, which is why listings can briefly trail writes.
  • What happens when an entire datacenter is lost? Placement never puts more chunks of one stripe in a single facility than the stripe can lose; with 8 plus 4 coding spread across at least three sites, a full site loss costs at most four chunks per stripe and repair proceeds from the survivors.
  • Why acknowledge a PUT on a quorum instead of all copies? The slowest of three nodes dominates latency, and waiting for it buys little: with two durable copies in separate failure domains plus immediate background completion of the third, the window of reduced redundancy is seconds.
  • What actually limits durability at eleven nines? Not independent disk failures, which the coding math handles with room to spare, but correlated events and software defects: a placement error that stacks a stripe in one rack, or a release that corrupts data on write, which is why scrubbing and failure-domain-aware placement are not optional extras.
