Design a visual search system

Systems design · Machine learning systems · Nov 2025

Visual search is the feature behind Pinterest Lens and Google Lens: a user points a camera at a lamp, or taps a lamp inside a photo someone else posted, and the system returns visually similar lamps they can save or buy. The query is an image rather than a string of words, which removes the user's burden of describing a shape, a texture, or a style they may not have vocabulary for, and that is exactly why commerce and discovery platforms want it. Interviewers like the question because it cannot be solved with a classifier and a lookup table; it forces the candidate through representation learning, billion-scale nearest-neighbor retrieval, and the unglamorous indexing machinery that keeps results fresh while the catalog changes underneath the model every day.

The hard parts live in three places. The first is learning an image representation where visual similarity is actually measurable. The second is searching a billion catalog items in tens of milliseconds without comparing the query to every one of them. The third is operational: new items arrive constantly, the embedding model itself gets retrained, and every model change quietly invalidates the entire index. The walkthrough below takes those in the order an interview would.

Scope and requirements

I would start by pinning the product behavior. The user submits a photo or taps a region of an existing image, and the system returns a ranked grid of visually similar catalog items, where the catalog means pins, products, or listings depending on the platform. A photo often contains several objects, a sofa and a lamp and a rug, so the system should let the user pick which detected object to search, which means object detection sits in the front of the pipeline rather than being an afterthought. Results should respect basic business constraints such as availability and region, which argues for a re-ranking stage on top of pure visual similarity.

For scale, a reasonable set of numbers to agree on is a catalog of one billion items with roughly ten million new items per day, and query traffic peaking around 5,000 searches per second. The latency budget matters because this is an interactive feature: I would target 250 milliseconds at the 95th percentile for the backend, leaving room for network and rendering on top. Out of scope, and worth saying so explicitly, are text search, deep personalization beyond light re-ranking signals, and exact-duplicate detection, which is a related but different problem solved with hashing rather than learned similarity.

Framing the problem as a machine learning task

The tempting first framing is classification: train a model that labels the query image as "mid-century table lamp" and return catalog items with that label. It breaks down quickly. The catalog gains millions of items daily and the space of visual concepts is unbounded, so any fixed label set is stale the week it ships, and human labeling can never keep up with catalog churn. Classification also throws away fine-grained visual information; two lamps in the same class can look nothing alike, and the user asked for things that look like this one.

The framing that works is representation learning plus nearest-neighbor retrieval. An embedding is a learned vector of numbers, say 256 floating point values, produced by a neural network from an image, with the defining property that distance between vectors tracks visual similarity: similar-looking images land close together and dissimilar ones land far apart. The input to the system is the query image (or a cropped object inside it), the model's output is the query's embedding, and retrieval becomes geometry: find the catalog items whose embeddings sit nearest the query embedding. New catalog items need no labels at all, only a pass through the embedding model, which is what makes the framing survive a changing catalog. The model objective, then, is not to predict a class but to shape the embedding space so that nearness means similarity, and the next two sections cover where the training signal for that comes from and how the model is trained.

Data and labels

The supervision is mostly implicit, harvested from behavior rather than annotators. When a user runs a visual search and then clicks or saves one of the results, that query-result pair is weak evidence of visual similarity. Co-engagement gives another source: images that users repeatedly save to the same boards, or product photos of the same item uploaded by different sellers, form natural positive pairs. Data augmentation supplies free positives with no logging at all, since a crop, a horizontal flip, or a mild color shift of an image should embed near the original, and these synthetic pairs teach the model invariances that engagement data underrepresents.

Each source carries its own noise. Engagement conflates similarity with attractiveness: users click pretty results and popular items, not necessarily the most visually faithful ones, and results shown higher get clicked more simply for being higher, which is position bias, the tendency of click data to reward rank rather than relevance. Same-product pairs are clean but narrow, teaching instance matching rather than style similarity. The practical answer is to mix sources, to de-bias engagement pairs by logging the position at which results were shown, and to keep a separate human-labeled evaluation set, a few thousand query images each paired with catalog items that trained raters marked as visually similar, so that model quality is judged against something cleaner than the training signal.

Embeddings and representation

The central representation is the image embedding itself: 256 dimensions is a common working point, L2-normalized so that every vector has length one, which makes cosine similarity (the angle between vectors) the distance measure and simplifies the index. Dimension is a real trade: 512 dimensions buys a little accuracy and doubles index memory and search cost, while 128 squeezes memory and gives some recall back, and 256 with normalization is the defensible middle that the sizing math below assumes.

Visual similarity alone does not make a good results page, so a second family of features serves the re-ranking stage: item category, price, availability, seller region, and engagement statistics such as historical click and save rates. These live in a feature store, a system that computes and serves the same feature values to offline training and online serving so the model sees identical inputs in both places, and the re-ranker reads them per candidate after retrieval. Keeping these out of the embedding is deliberate, because baking popularity into the vector would distort the geometry that retrieval depends on.

Model architecture

The baseline worth stating first is an off-the-shelf backbone, a ResNet or vision transformer pretrained on a large public dataset, used as a frozen feature extractor with its penultimate layer as the embedding. It works surprisingly well and it is the right first milestone, but its notion of similarity reflects the pretraining task rather than the product's, so a pretrained model thinks two photos are similar because both contain a dog, while the product needs them similar because both show low-pile geometric rugs.

The production model is trained with contrastive learning, which is the plain idea of teaching by comparison: take a pair of images that should be similar (an engagement pair or an augmented copy), take a set of images that should not be (other images in the same training batch work as these negatives), and adjust the network so the positive pair's embeddings move closer together while the negatives are pushed apart. Training on large batches gives hundreds of free negatives per example, and mining hard negatives, items that look deceptively close but were not engaged with, sharpens the space where it matters most. The backbone choice is a secondary decision and I would say so in the interview: a convolutional network is cheaper to serve and well understood, a vision transformer typically wins a few points of recall at higher compute, and either slots into the same contrastive recipe.

In front of the embedding model sits an object detector, a model that outputs bounding boxes around objects rather than one label for the whole image. Real query photos contain whole rooms, and embedding the entire scene produces a vector that matches other whole rooms instead of the lamp the user cares about. The pipeline therefore detects shoppable objects, lets the user confirm or pick one, crops it, and embeds the crop, and the same detect-crop-embed treatment is applied to catalog images at indexing time so that query and index vectors describe objects, not scenes. The detector can be small and fast since it only needs coarse boxes over a few dozen object groups.

Offline evaluation

The retrieval stage is measured with recall@K: of the items that should have been retrieved, what fraction appears in the top K results. The human-labeled set makes this concrete. Suppose a query image of a brass floor lamp has five rater-approved similar items in the catalog, and the system's top 10 contains three of them; recall@10 for that query is 3/5 = 0.6, and averaging this over, say, 2,000 labeled queries gives the model's score, so a move from 0.58 to 0.64 average recall@10 is the kind of offline win that justifies an online test. Engagement-based evaluation sets, built from logged clicks rather than raters, are larger and cheaper but inherit the biases of the logging policy: they can only contain items an earlier model already surfaced, so a genuinely better model gets penalized for retrieving good items the old system never showed. Reporting both, with the human-labeled number as the headline, is the discipline that keeps the team from optimizing into its own logging shadow.

Validation splits need the same care. Splitting by query image alone leaks near-duplicates between train and test, since the same product photographed twice can land on both sides, so the split should group by product or by user session before dividing, and evaluation queries should postdate training data when engagement pairs are involved.

The serving architecture

Serving is a short pipeline with a strict budget. The query image hits the detector, the chosen crop hits the embedding model, the embedding fans out to the ANN index shards, and the merged candidates pass through the re-ranker. ANN means approximate nearest neighbor search: instead of comparing the query vector against all one billion index vectors exactly, the index uses a precomputed structure to examine only a tiny fraction of them, trading a small loss of recall, typically a few percent, for a speedup of several orders of magnitude. HNSW builds a layered graph over the vectors and answers queries by greedily walking from node to neighboring node toward the query. IVF clusters the vectors ahead of time and searches only the handful of clusters whose centers lie closest to the query.

The memory math decides the shape of the fleet. One billion vectors at 256 dimensions in float16 (two bytes per value) is 1,000,000,000 × 256 × 2 bytes = 512 GB of raw vectors, and graph or cluster overhead adds roughly half again, call it 768 GB total. That cannot live on one machine, so the index is sharded 16 ways at about 48 GB per shard, each shard fitting comfortably in a 64 GB host, with every shard replicated twice for availability and query throughput. A query is sent to all 16 shards in parallel, each returns its local top 100, and the merger keeps the global best 400 or so for re-ranking.

The latency budget walks through cleanly: 20 ms for detection, 10 ms for embedding on a GPU host with small-batch queueing, 30 ms for the ANN fan-out (the slowest shard sets the pace, which is why per-shard tail latency matters more than the average), 15 ms to re-rank 400 candidates with metadata and engagement features, and roughly 25 ms of network hops and serialization between stages, landing near 100 ms and leaving headroom inside the 250 ms target for retries and queue spikes.

The serving path: the detector finds objects, the chosen crop becomes a query vector, all ANN shards answer in parallel, and the re-ranker orders the merged candidates with catalog metadata. The whole path fits in roughly 100 ms.

The indexing pipeline

The index is a living thing, and most of the operational difficulty lives here rather than in serving. New catalog items flow through the same detect-crop-embed treatment as queries, and their vectors are inserted into the live shards within seconds to minutes of ingest, because a marketplace where today's listings are unsearchable until tomorrow loses exactly the freshest inventory users want. HNSW supports incremental insertion naturally; IVF accepts appends into existing clusters at some quality cost until the next rebuild. Deletions for sold-out or removed items are handled with tombstones, markers that hide an entry until a rebuild physically removes it.

Periodic full rebuilds remain necessary even with streaming inserts, because incremental structures degrade and cluster assignments drift, and the rebuild cadence, weekly is typical, also compacts tombstones. The event that forces a rebuild of everything is a new embedding model version, and the reason deserves a plain statement in the interview: two separately trained models produce vector spaces that are not comparable, since each training run settles on its own arbitrary orientation of the space, so a query embedded with model vN+1 lands in meaningless places in an index built with vN vectors. Shipping a new model therefore means re-embedding the full billion-item catalog, a scheduled batch job of real cost (at 5 ms per item on a GPU, one billion items is about 58 GPU-days, so a 100-GPU fleet finishes in about 14 hours), building fresh shards alongside the live ones, and cutting traffic over atomically once the new shards pass validation, with the old shards kept warm briefly for rollback.

Online evaluation and operations

Offline recall is a proxy, so changes graduate through A/B tests measured on what users do with the results: click-through rate on the result grid, save rate, and add-to-cart or purchase where commerce applies, with save rate usually the most stable signal of genuine visual relevance. Interleaving is a useful sharper instrument for ranking changes: instead of splitting users into two groups, results from the old and new rankers are blended into a single grid and the system observes which ranker's items earn the clicks, which detects preference with far less traffic than a standard A/B split.

Retraining runs on a regular cadence, monthly is a reasonable default, with off-cycle retrains when monitoring says the world has moved. Training-serving skew, where the model sees differently processed inputs offline and online, is the classic silent failure for image systems: if training resized images with one library and serving resizes with another, embeddings shift just enough to bleed recall with no error logged anywhere, so the preprocessing code should be one shared artifact, exercised by a test that embeds a fixed image set in both stacks and asserts near-identical vectors. Production monitoring watches embedding norm and dimension statistics for drift, runs a canary set of probe queries with known expected neighbors against every shard, alerts on per-shard tail latency and recall divergence, and tracks the age distribution of the index against ingest volume so a stalled insertion pipeline is caught by a metric rather than by a seller complaint.

Follow-up questions

  • Why not classify the query image and look results up by label? The catalog changes daily and the concept space is unbounded, so labels are perpetually stale, and classification discards the fine-grained appearance the user is querying with. Embedding plus nearest neighbor needs no labels on new items, only a forward pass.
  • Why is a new model version so expensive to ship? Independently trained models produce incompatible vector spaces, so every catalog item must be re-embedded and every index shard rebuilt before the new model can serve queries. The mitigation is engineering the cutover well, not avoiding it: parallel index build, validation against a canary query set, atomic flip, fast rollback.
  • How would you handle a query photo with five objects in it? Run the detector, surface the detected boxes to the user, and search the object they pick, defaulting to the most salient box. Indexing applies the same per-object treatment so query and catalog vectors describe objects rather than scenes.
  • HNSW or IVF for the index? HNSW gives better recall-latency trade-offs and natural incremental inserts, at a higher memory overhead from the graph; IVF is more compact and pairs well with compression but wants periodic retraining of its clusters. With 64 GB hosts and a freshness requirement, HNSW per shard is the default I would pick first.
  • Where does the system quietly lose quality without erroring? Preprocessing skew between training and serving, a stalled insert pipeline aging the index, and engagement-based evaluation flattering the incumbent model. All three are monitoring problems before they are modeling problems.
  • How do you evaluate without trusting clicks? Keep a rater-labeled set of query-to-similar-item judgments as the headline recall@K metric, and treat engagement-derived sets as a large but biased supplement, since they only contain items earlier systems chose to show.
