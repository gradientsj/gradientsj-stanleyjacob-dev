Design a metrics monitoring and alerting system

Systems design · Data pipelines and observability · Oct 2025

A metrics system watches everything else. Every server, container, and service in a fleet emits numbers, CPU load, request counts, queue depths, error rates, and this system collects them, stores them as time series, draws them on dashboards, and wakes someone up when a number crosses a line. Operations teams live inside it, capacity planning is impossible without it, and during an incident it is the difference between a five-minute diagnosis and an hour of guessing. Interviewers reach for this question because it looks like simple plumbing and then turns out to contain a specialized database, a streaming pipeline, and a small state machine with pager-duty consequences, all of which must keep working precisely when the rest of the infrastructure is on fire.

The hard parts are the data model's tendency to explode combinatorially, the storage engine that must absorb a million writes per second on hardware that a general-purpose database would saturate at a fraction of that, and the alerting path, which has to be the most reliable component in the company while depending on everything else. The walkthrough below takes them in that order.

Scope and requirements

Functionally, the system collects operational metrics from a large fleet, on the order of 100,000 servers, stores them as time series, serves a query language for dashboards and ad hoc investigation, and evaluates alert rules that notify on-call engineers through pagers, chat, and email. Log and trace collection are explicitly out of scope, since they have different shapes and different systems; this design is for numbers over time.

Non-functionally, ingestion must keep up with the fleet at around one million samples per second without dropping data during traffic spikes, queries over recent data should return in well under a second because humans iterate on dashboards interactively, and a year or more of history must be retained at a cost that does not dwarf the fleet it monitors. Alert evaluation has the strictest requirement of all: it must keep functioning through partial failures of the very infrastructure it observes, and a late or lost page is the worst defect this system can have. Slightly stale data is acceptable almost everywhere, which is a freedom worth exploiting, and samples are append-only facts that are never updated in place.

Sizing the problem

The unit of storage is a series: one metric name plus one unique combination of labels. If each of 100,000 servers exposes about 100 series, the fleet carries 10 million active series, and scraping each one every 10 seconds produces 1 million samples per second, 86.4 billion samples per day. A sample is a timestamp and a 64-bit float, 16 bytes uncompressed, so a day of raw data is about 1.4 TB before compression and, as covered below, nearer 120 GB after it.

The number that actually kills these systems is cardinality, the count of distinct label combinations. Labels are key-value pairs attached to a metric name, such as http_requests_total{service="checkout", region="us-east"}, and every new combination creates a new series with its own index entry and in-memory state. Suppose one engineer adds a user_id label to a request-latency metric in a service with 2 million active users and 10 instrumented endpoints: that single decision mints up to 20 million new series, tripling the fleet-wide total, and at roughly 3 KB of index and head-block memory per active series it demands about 60 GB of additional RAM across the storage tier. Series churn does similar damage, since a label that embeds a container ID creates a fresh series on every redeploy. Guardrails on label values are a requirement, not a refinement.

The query API

Dashboards and alert rules speak the same interface: select series by name and label matchers, apply functions over time windows, and aggregate across series. The HTTP surface is small, and the expression language carries the weight, PromQL in spirit here.

GET /api/v1/query_range
    ?query=sum by (service) (rate(http_requests_total{status="500"}[5m]))
    &start=2025-10-07T10:00:00Z&end=2025-10-07T16:00:00Z&step=60s
→ 200 { "result": [ { "labels": {"service": "checkout"},
                      "values": [[1759831200, 4.2], ...] }, ... ] }

GET /api/v1/series?match=node_cpu_seconds_total{region="us-east"}
→ 200 { "series": [ ...label sets... ] }

POST /api/v1/ingest        ← push path for batch jobs and edge agents
{ "series": [ { "labels": {...}, "samples": [[ts, value], ...] } ] }

The data model in storage

Logically the store is two structures: an index from label values to series IDs, and per-series sample streams. An inverted index, one posting list per label pair, is what makes {service="checkout"} resolve to its few hundred series without scanning ten million. The relational sketch makes the shape concrete even though the real engine is purpose-built.

CREATE TABLE series (
  series_id   BIGINT PRIMARY KEY,
  metric_name TEXT NOT NULL,           -- e.g. http_requests_total
  labels      JSONB NOT NULL           -- {"service":"checkout", ...}
);
-- inverted index: label pair → series ids, one row per (pair, series)
CREATE INDEX series_by_label ON series USING gin (labels);

CREATE TABLE samples (
  series_id   BIGINT NOT NULL,
  ts          BIGINT NOT NULL,         -- ms since epoch
  value       DOUBLE PRECISION NOT NULL,
  PRIMARY KEY (series_id, ts)          -- append-mostly, time-ordered
);

The high-level architecture

Collection comes in two flavors and a serious design supports both. Pull collection, the Prometheus model, has collectors discover targets through service discovery, the registry of what is running where, and scrape each target's metrics endpoint on a schedule; it wins inside a datacenter because the scraper controls load, a dead target is detected by the failed scrape itself, and nothing is configured on the monitored host. Push collection has agents send metrics through a gateway, and it wins for short-lived batch jobs that may finish between scrapes and for edge devices behind NAT that a scraper cannot reach. Either way, collected samples flow into Kafka, a durable distributed queue, before storage, which decouples the two sides: a storage deploy or a slow shard does not push back on collectors, and an ingest spike parks safely in the queue instead of being dropped. Storage nodes consume from the queue, partitioned so that each series consistently lands on one shard, and a query layer fans out across shards while the rule evaluator and dashboards sit on top of it.

Samples flow left to right into sharded time-series storage; the query service fans out across shards for dashboards and for the rule evaluator, whose firing alerts pass through the alert manager before anyone is paged.

The time-series storage engine

A general-purpose B-tree database struggles here for a structural reason: one million samples per second arriving for ten million different series means writes land all over the key space, so the tree's pages churn randomly, write amplification climbs, and per-row overhead of dozens of bytes swamps 16-byte samples. The data, though, has exploitable shape. It is append-mostly, time-ordered within each series, and highly regular, and the engine that fits is the one Facebook described in the Gorilla paper. Timestamps are stored as delta-of-delta: scrapes arrive on a nearly fixed cadence, so instead of recording 1759831200, 1759831210, 1759831220, the engine records the difference between consecutive deltas, which is almost always zero and encodes in one or two bits. Values are stored by XOR: each float is XORed with its predecessor, and since adjacent samples of a gauge are usually close, the result is mostly zero bits that compress to a handful of meaningful ones. Gorilla reported an average around 1.37 bytes per 16-byte sample, a twelvefold reduction, and that single number is what makes the rest of the design affordable: 86.4 billion samples per day shrink from 1.4 TB to roughly 120 GB.

Each shard keeps the most recent two hours of its series in a compressed in-memory head block, writing a write-ahead log so a crash replays cleanly, and then flushes the head to an immutable, time-partitioned file on disk with its own index. Queries for recent data, which dominate dashboard traffic, are served from memory; older windows read a handful of immutable files. Compaction merges small files into larger ones in the background, and because files are immutable, deleting expired data is just removing whole files rather than rewriting anything.

Retention and downsampling

Nobody needs 10-second resolution from eight months ago, and the budget math shows why pretending otherwise fails: keeping raw samples for 5 years would cost about 120 GB times 1,825 days, around 220 TB, almost all of it never queried at full detail. Downsampling, replacing many fine-grained samples with fewer coarse aggregates over each window, is the standard escape. Raw samples are kept for 15 days, about 2 TB, which covers incident response and week-over-week comparison. A rollup job then reduces each series to per-minute aggregates, storing a sum and a count per minute so averages remain exact and rates remain computable: 10 million series times 1,440 minutes times 2 values is about 29 billion values per day, roughly 43 GB at the same compression, and 13 months of that is about 17 TB. A second tier keeps per-hour aggregates for 5 years at about 0.7 GB per day, 1.3 TB in total. The three tiers together cost around 20 TB against 220 TB for raw-everything, and the query layer picks the coarsest tier whose resolution still answers the question, so a one-year dashboard reads hourly points instead of dragging 3 billion raw samples per series-screen through the network. What is lost is real and worth stating: a sub-minute spike older than 15 days is invisible forever, and any percentile not precomputed cannot be recovered from sums and counts.

The raw tier holds immutable two-hour blocks at full resolution; rollup jobs feed the one-minute and one-hour tiers, and total storage lands near 20 TB instead of 220 TB for five years of raw data.

Alerting and its state machine

The rule evaluator runs each alert rule on a fixed clock, typically every 15 to 60 seconds, by issuing the rule's query through the same query layer the dashboards use, so an alert and a graph can never disagree about the data. Each rule's result drives a small state machine per label set: a rule is inactive while its condition is false, becomes pending when the condition first turns true, becomes firing only after the condition has held continuously for a configured duration, and resolves when it turns false again. The pending state is the flap suppressor: a CPU metric that spikes over the threshold for one scrape and drops back never pages anyone if the rule requires five minutes of sustained breach, because brief noise dies in pending. The cost is detection delay, exactly the configured duration, so rules for hard failures use short holds and rules for slow burns use long ones.

Firing alerts then pass through the alert manager, which exists because raw alerts at fleet scale would page someone hundreds of times for one outage. It deduplicates identical alerts arriving from redundant evaluators, groups related alerts so that fifty instances of one service failing produce one notification listing fifty members, honors silences during planned maintenance, and routes by severity: a page for the database primary being down, a chat message for a disk filling slowly. Escalation policies re-page up a chain when nobody acknowledges. One question closes the design: who watches the watcher? The standard answer is a dead-man's switch, an alert configured to fire continuously by definition, whose arrival a small external service expects every minute; when the heartbeat stops arriving, the external service pages directly, so silence from the monitoring system becomes the loudest signal it can send.

Scaling, failures, and operations

Each tier scales by its own rule. Collectors are stateless and partition the target list among themselves through the same service discovery they scrape from. Kafka scales by partitions, and sizing is mild: a million samples per second in batched, compressed form is on the order of 100 MB/s, well within a small cluster's comfort. Storage shards partition by hash of series, so ingest spreads evenly, and each shard replicates to a peer so a node loss costs no data and queries fail over; a shard that dies and returns replays the queue from its last consumed offset, which is the quiet payoff of buffering ingestion, since the queue absorbs hours of backlog while a shard rebuilds. Queries that span shards fan out and merge, and the expensive ones are protected by limits on the number of series a single query may touch, because one unbounded regex over label values can otherwise pull a shard's whole index into memory.

The failure stories worth rehearsing are the partial ones. If Kafka itself degrades, collectors buffer briefly in memory and then shed the oldest data first, on the argument that the newest samples matter most during an incident. If a rollup job falls behind, raw data simply lives a little longer before deletion, which costs disk rather than correctness. The rule evaluator runs as two replicas evaluating everything, with the alert manager deduplicating their output, so losing one evaluator changes nothing visible. And the alerting path is kept deliberately boring: few dependencies, its own small deployment, version pinned, because the one component that must work during a bad day is the one that tells you the day is bad.

Follow-up questions

  • Pull or push collection? Pull inside the datacenter: the scraper controls load, dead targets are detected by failed scrapes, and hosts need no config. Push for short-lived jobs and NAT-bound edge agents. Real fleets run pull with a push gateway at the edges.
  • How do you stop a cardinality explosion before it lands? Enforce per-service series quotas at ingestion, reject label keys like user or request IDs by policy, and surface a "top series growth" report so the offending deploy is found in minutes, not after the RAM is gone.
  • Why buffer ingestion through Kafka instead of writing straight to storage? It decouples failure domains: a storage deploy, a slow shard, or an ingest spike becomes queue lag instead of dropped samples, and a rebuilt shard replays from its offset.
  • Why not store metrics in a relational database? Random-key writes at a million rows per second churn B-tree pages and pay tens of bytes of per-row overhead on 16-byte facts; delta-of-delta and XOR encoding in a purpose-built engine cut storage about twelvefold and serve recent data from memory.
  • What does "pending for five minutes" actually buy? Flap suppression: threshold noise dies in the pending state and never pages. The price is exactly five minutes of detection delay, so hold durations are tuned per rule severity.
  • Who alerts when the alerting system is down? A dead-man's switch: one rule fires by construction forever, and a tiny external service pages when that heartbeat stops arriving.
