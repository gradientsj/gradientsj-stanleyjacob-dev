Design a search autocomplete system

Systems design · Search and discovery · May 2025

Autocomplete is the dropdown that appears under a search box, offering the five to ten most likely completions of whatever the user has typed so far. It looks like a garnish, but it shapes how people search: most users on major search products pick a suggestion rather than finishing their query, so the feature carries real traffic and real product weight. The defining constraint is that suggestions must appear within about 100 milliseconds of a keystroke to feel attached to the typing, and once network time to a realistic user eats 30 to 60 of those milliseconds, the server has a few tens of milliseconds to produce an answer. That budget rules everything: the design below is, from one angle, just a list of things precomputed so that the request path does almost nothing.

Interviewers like the question because the naive answer, search the query database for everything starting with the prefix and sort by popularity, is easy to propose and easy to demolish with arithmetic, and because the repair touches a classic data structure, an offline pipeline, and a caching story all in one design. The hard parts live in what to precompute, how to ship fresh precomputed data without downtime, and how to keep a deliberately stale system responsive to a world where queries trend within minutes.

Scope and requirements

Functionally, the system returns up to 10 ranked completions for the current text in the search box, restricted to prefix matches to keep scope controlled (matching mid-phrase words is a follow-up, not the core). Ranking is by historical popularity, with a mechanism for trending queries to surface quickly, and suggestions must pass a filter that keeps offensive, harmful, or policy-violating completions out of the box, because a suggestion is something the product says, not something the user said. Personalization stays out of scope beyond noting where it would attach.

Non-functionally: end-to-end suggestion latency around 100 ms with a server budget near 20 ms at the median, availability high enough that the search box never blocks on a dead suggestion service (the client must degrade to plain typing), and staleness tolerance, which is the great gift of this problem: a suggestion list computed yesterday is almost as good as one computed this second, for everything except trending topics, and that tolerance is what makes aggressive caching and offline builds legitimate.

Sizing the problem

Request volume multiplies out of search volume because every keystroke is a potential request. Take a product with 10 million searches per day and suppose debouncing and prefix-length thresholds mean an average of 4 suggestion requests fire per search. That is 40 million requests per day, and over 86,400 seconds, about 460 requests per second on average, with a peak factor of 5 giving roughly 2,300 requests per second. Each response is a kilobyte or two, so bandwidth is trivial; the work is purely lookup rate against the latency budget.

Storage sizes the precomputation decision. Suppose the query logs yield 50 million distinct queries worth suggesting after cleaning, averaging 20 characters. Storing the top-10 list at every prefix means at most 50 million times 20, or 1 billion prefix entries, and in practice far fewer because queries share prefixes heavily; call it a few hundred million. At roughly 300 bytes per entry (the prefix plus ten short strings with counts), the ceiling is 1 billion times 300 bytes, or 300 GB, with the realistic shared-prefix figure under 100 GB. That does not fit one memory node comfortably, but it shards beautifully, and it is small enough to rebuild from scratch every night, both of which the architecture leans on.

The trie, and why query time walks are too slow

The natural structure for prefix lookup is a trie, a tree where each edge adds one character, so the path from the root spelling out "ca" reaches the node owning that prefix, and every query in the subtree below it starts with "ca." Finding suggestions then means walking the subtree to collect completions and their popularity counts, and that walk is the problem: a two-letter prefix on a 50-million-query corpus can own hundreds of thousands of descendants, so the walk plus a sort costs far more than 20 ms, and it does the same work again on the next keystroke. The repair is to compute each node's answer once, offline: store the top 10 completions directly at every node, so the query path walks down exactly len(prefix) edges and reads a list, with no subtree visit at all.

class TrieNode:
    __slots__ = ("children", "top_k")
    def __init__(self):
        self.children = {}     # char -> TrieNode
        self.top_k = []        # [(count, query)] precomputed, length <= 10

def suggest(root, prefix):
    node = root
    for ch in prefix:
        node = node.children.get(ch)
        if node is None:
            return []          # no query starts with this prefix
    return node.top_k          # O(len(prefix)), no subtree walk

Filling top_k for every node is one bottom-up pass: each node merges its children's lists with its own terminal query, keeping the best 10, so the whole build is linear in the corpus with a constant for the small merges. The same idea flattens into a table, mapping each prefix string to its list, which is the form that ships to serving: a trie is the natural builder, a key-value table of prefix → suggestions is the natural servable artifact, and the storage math above already priced it. The worked example: if "calendar" was searched 1.2 million times last month and "calculator" 900 thousand, then the node for "ca" holds those two near the top, the node for "cal" holds them too, and the node for "cale" drops "calculator" because it no longer matches; every keystroke is one lookup against a ready list.

The serving path

The read path is a cache hierarchy with a sharded table at the bottom. The browser caches recent prefix results itself, the CDN caches responses for popular prefixes with a short TTL since suggestions tolerate a minute of staleness, and behind that a suggest service checks a memory cache and then the owning prefix shard. Sharding is by prefix range so that one shard owns, say, everything from "a" to "m", though hashing the first couple of characters with an explicit map for hot prefixes balances better than naive alphabetical splits, since "s" carries far more traffic than "x." Each shard holds its slice of the table fully in memory, replicated for availability, and answers in well under a millisecond, leaving the budget to the network where it belongs.

Most keystrokes never pass the CDN; misses reach the suggest service, which checks the result cache, reads the owning prefix shard, and blends in trending candidates (dashed) before responding.

The client carries two responsibilities that are part of the design, not afterthoughts. Debouncing means waiting a beat, typically 50 to 150 ms, after a keystroke before sending the request, so a fast typist generates one request for "shoes" rather than five for each prefix along the way; combined with a minimum prefix length of one or two characters, it is what turned 10 million searches into only 40 million requests in the sizing. Request cancellation handles the responses that still overlap: when the user types another character, the client aborts the in-flight request for the old prefix, because rendering stale suggestions over fresh ones is the most user-visible failure this product has, and an AbortController on the fetch makes it a one-line discipline.

The offline pipeline: from logs to versioned tries

Everything served above is an artifact, and the pipeline that builds it runs far from the request path. Search query logs accumulate in blob storage; an aggregation job, daily or weekly depending on how fast the product's queries drift, counts queries over a trailing window such as 30 days, with more recent days weighted higher; a filtering stage removes offensive and policy-violating queries against blocklists and a classifier, plus queries below a frequency floor, which both shrinks the corpus and avoids suggesting one user's idiosyncratic search back to the world (a real privacy leak if skipped: a rare query containing a person's name must never become a suggestion). The trie builder then computes per-prefix top-10 lists, flattens them into the table, and writes the result to an artifact store under a version number, sliced per shard.

Step 1: raw logs feed the aggregator. Step 2: counts pass the policy filter. Step 3: clean weighted queries reach the trie builder. Step 4: per-shard artifacts land in the store under a new version. Step 5: shards download the new version, load it beside the old one, and swap pointers atomically.

Versioning is what makes shipping safe. Each shard serves version 42 from memory while downloading and loading version 43 beside it, then swaps a pointer between requests, so deploys are invisible and rollback is the same pointer swap in reverse. A canary step, serving the new version to a slice of traffic while comparing click-through on suggestions, catches a poisoned build, and keeping the last few versions in the artifact store means the worst pipeline failure, a corrupt build at 3 a.m., degrades to serving yesterday's suggestions, which users cannot detect.

Trending is a second, much smaller pipeline layered on top rather than a faster rebuild of the big one. A streaming job counts queries over the last 10 to 30 minutes, and the suggest service blends its output with the stable lists at request time, for example by promoting a short-window query into the displayed 10 when its recent velocity clears a threshold. The blend stays at the merge step deliberately: the stable trie gives trustworthy long-run ranking, the overlay reacts within minutes, and neither pipeline needs the other's properties. Typo tolerance also lives at the merge step: when the prefix returns few or no completions, the service retries with a handful of candidates within edit distance one (defined as one character inserted, deleted, or substituted), generated against the prefix table rather than the full corpus, so the cost stays bounded and only triggers when the primary lookup comes back thin.

Scaling, failures, and operations

Every serving tier scales horizontally because every serving tier is read-only. Prefix shards split further when a slice outgrows memory or traffic, replicas absorb read load, and CDN caching means popular-prefix traffic barely reaches the origin: prefixes follow the same skew as everything else, so a cache of the few million hottest entries serves a large majority of requests. The pipeline scales like any batch system, and its latency only affects how fresh the stable rankings are, never whether the product works.

The failure modes are pleasantly contained. A dead shard fails over to its replica with seconds of staleness at worst; a dead suggest tier degrades the search box to plain typing, which the client must treat as a non-event by simply rendering no dropdown rather than blocking; a stuck pipeline freezes rankings at yesterday's truth, alarmed on artifact age rather than user reports. The genuinely sharp edge is the filter stage, because a build that lets one harmful suggestion ship turns a backend artifact into a screenshot, so blocklist updates need a fast path that does not wait for the next full build: shards consult a small deny set at serve time, updated within minutes, in front of whatever the artifact contains. The other operational habit is replaying a sample of yesterday's real prefixes against each new artifact before promotion, comparing suggestion overlap with the previous version, because a quiet aggregation defect that empties half the prefix lists looks exactly like success to every infrastructure metric while being the worst regression the product can ship.

Follow-up questions

  • Why precompute top-K at every prefix instead of walking the trie per query? A short prefix can own hundreds of thousands of completions, so collecting and sorting them blows a 20 ms budget thousands of times over, on every keystroke. Precomputation moves that cost into a nightly build and makes the read path one bounded lookup, and the storage bill, under 100 GB with prefix sharing, is cheap against the latency it buys.
  • How do new or trending queries appear if the trie rebuilds nightly? Through the short-window overlay: a streaming count over the last few minutes is merged into responses at serve time when velocity clears a threshold. The stable trie supplies trustworthy rankings, the overlay supplies recency, and no single pipeline has to provide both.
  • Where does debouncing belong and what does it actually save? In the client, before the request exists: waiting roughly 100 ms after a keystroke collapses a fast typist's burst into one request. In the sizing it is the difference between every character of 10 million searches hitting the backend and an average of 4 requests per search, roughly a 5x reduction.
  • How is an offensive suggestion prevented, and how is one removed fast? Prevention is the build-time filter of blocklists plus a classifier and a frequency floor. Removal cannot wait a build cycle, so shards also check a small serve-time deny set that propagates in minutes, and the artifact is patched at the next build.
  • What changes for multi-word and mid-word matching? Index additional entry points: each query is inserted under the prefixes of each of its words, so "weather tokyo" is reachable from "tok". The table grows by the average number of words per query, and ranking needs a tie-break preferring whole-query prefix matches, which keeps quality predictable.
  • Could you serve this from Redis sorted sets instead of custom shards? Yes, and it is a respectable mid-scale answer: one sorted set per prefix, scored by popularity, sharded by key. The custom in-memory table earns its keep when the corpus and traffic outgrow what general-purpose storage serves within the latency budget, or when version-swapped immutable artifacts simplify operations enough to matter.
