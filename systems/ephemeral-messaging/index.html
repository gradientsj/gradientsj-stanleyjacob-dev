Design ephemeral messaging

Systems design · Social and messaging · Jul 2025

Ephemeral messaging is the Snapchat-shaped product: messages that disappear after they are viewed, photos that can be opened exactly once, and stories that live for 24 hours and then vanish. Most chat questions are about getting data to people and keeping it forever, and this one inverts the second half, because here deletion is the product. Interviewers like the question because it looks like generic chat and then is not: a candidate has to reason about what "deleted" actually means in a distributed system, where copies live in databases, caches, content delivery networks, and backups, and has to be precise about which of those copies the service can truthfully promise to destroy. The hard parts live in the gap between marking data expired and physically removing it, in making a view-once photo burn on every one of the recipient's devices, and in the awkward truce between caching media for speed and erasing it on schedule.

Scope and requirements

The functional surface I would agree on: one-to-one disappearing messages with sender-chosen lifetimes, view-once photos and videos, 24-hour stories broadcast to friends or followers, screenshot notification, and delivery states similar to ordinary chat. Abuse reporting has to exist, which immediately complicates the deletion story, because reporting content the service has already destroyed is impossible, so a narrow carve-out for reported content is in scope from the start.

The non-functional requirements are where this design departs from chat. Deletion must really happen within a bounded and stated window, not merely hide content from the interface, since the product's entire promise is that the service does not keep what you sent. Delivery latency should match a normal messenger, a few hundred milliseconds when both parties are online. Durability is required only within a message's lifetime: losing a message that was going to be destroyed tomorrow is still a failure today, so the store replicates like any other. The pleasant consequence of the product is cost: because the corpus is small and rolling rather than monotonically growing, the storage bill stays roughly flat as the service ages, and the design should preserve that property instead of accidentally hoarding data in side channels like logs and caches.

Sizing the problem

Assume 30 million daily active users sending 30 messages each per day, which is 900 million messages per day, or about 10,400 per second on average and call it 50,000 per second at peak. Suppose half the messages carry media averaging 1 MB after client-side compression, so the service ingests around 450 million media objects per day, roughly 450 TB of blob traffic daily.

Now the ephemeral arithmetic. A keep-forever service ingesting 450 TB per day grows by about 165 PB a year and never stops. This service, with everything expiring within 24 hours of viewing or posting, holds a steady-state corpus of roughly one to two days of traffic, on the order of 1 PB including replication, forever. Text rows are negligible next to that: 900 million rows at 300 bytes of metadata each is 270 GB per day, and the rolling window keeps the table under a terabyte. The whole storage tier for a 30-million-user product fits in what a single day costs the keep-forever competitor, which is the business reason ephemerality is more than a gimmick.

The interface

Clients hold a WebSocket, a persistent two-way connection upgraded from HTTP, to a socket tier for real-time delivery, and use plain HTTP for media. Media never travels through the socket: the client asks the API for an upload target and receives a signed URL, which is an object-storage link carrying a cryptographic signature that grants one specific operation on one specific object for a short time, so clients talk to storage directly without holding any credentials.

POST /api/v1/media/upload-urls
{ "content_type": "image/jpeg", "bytes": 812400 }
→ 200 { "media_key": "m/7f3a9c", "put_url": "https://blobs.example.com/m/7f3a9c?sig=...&expires=300" }

// then over the socket
→ { "type": "send", "to": "u815", "media_key": "m/7f3a9c",
    "view_policy": "once", "ttl_seconds": 86400, "client_msg_id": "k21" }
← { "type": "ack", "client_msg_id": "k21", "msg_id": 99312, "state": "sent" }

POST /api/v1/stories
{ "media_key": "m/2cc41e", "ttl_seconds": 86400 }   → 201

Every send carries an explicit lifetime. The server treats the time-to-live, the number of seconds a row is allowed to exist, as a first-class column rather than an afterthought, because every downstream system from the database to the sweeper keys off it.

The data model

Messages land in a wide-column or key-value store with native row expiry, and both major candidates have it: DynamoDB lets you name a TTL attribute and Cassandra accepts a TTL on any write. The schema looks like ordinary chat with two additions, the expiry timestamp and the view state.

CREATE TABLE messages (
  conversation_id BIGINT,
  msg_id          BIGINT,
  sender_id       BIGINT,
  media_key       TEXT,          -- pointer into object storage
  view_policy     TEXT,          -- 'once' or 'timed'
  view_state      TEXT,          -- unviewed / viewed / expired
  expires_at      TIMESTAMP,     -- the native TTL column
  PRIMARY KEY ((conversation_id), msg_id)
);

It is worth saying plainly how native TTL works, because the mechanism defines what the product can promise. The store does not run around deleting rows the second they expire. Instead, expiry is lazy: a read that touches an expired row filters it out as if it were gone, and the bytes are physically reclaimed later during compaction, the background process in log-structured stores that rewrites data files and drops dead entries. That creates a gap between logical deletion, the moment no query can return the row, and physical deletion, the moment the bytes no longer exist on disk, and in a busy Cassandra cluster that gap can be hours to days depending on compaction cadence. The design has to state which one the product promise refers to, and the defensible answer is logical deletion immediately at expiry with physical deletion guaranteed within a stated window, enforced by scheduling compaction aggressively on these tables and by keeping message bodies out of long-retention systems such as backups and diagnostic logs entirely.

The high-level architecture

The shape resembles a chat system, deliberately, with a socket tier delivering frames and a store holding rows, but every storage component carries expiry and a sweeper exists as a first-class service. Media lives in object storage as encrypted blobs reachable only through signed URLs, and the push service covers recipients with no live socket.

Every row in the message store carries a TTL and the sweeper turns logical expiry into physical deletion; clients upload and download media directly against object storage using short-lived signed URLs minted through the socket tier's API.

Making deletion real

Database rows are the easy third of the problem, since TTL plus aggressive compaction handles them. Media blobs are harder because they are large, cached, and replicated for speed, and a content delivery network, the geographically distributed cache tier that serves static content from locations near users, is precisely a machine for making copies the origin does not control. There is real tension here: caching a popular story's video at the edge makes playback fast, but every cached copy is a deletion obligation, and CDN purge APIs are best-effort with propagation delays. The design resolves the tension differently per content type: view-once media skips the CDN entirely, served straight from object storage since it will be fetched exactly once anyway, while stories may use a CDN with cache lifetimes capped well below the 24-hour content lifetime so edge copies die of natural causes before the deadline.

The strongest tool is encryption as deletion. Each media object is encrypted at upload with its own random data key, the blob in object storage is ciphertext, and the data key lives in the message row, which carries the TTL. When the row expires, the key is gone, and every remaining copy of the blob, in storage, in a cache, on a misplaced disk, becomes unreadable bytes; deleting one small key is equivalent to deleting every copy of the content at once. The blob itself is still removed by a cleanup job for cost reasons, but the promise no longer depends on that job winning a race against replication. This technique, usually called crypto-shredding, converts a distributed deletion problem into a local one, which is the kind of move worth naming explicitly in an interview.

Two scope limits keep the promise truthful. First, expired does not mean unsendable history is rewritten everywhere: the recipient's phone displayed the photo, and a phone can screenshot it or another camera can photograph the screen, so the guarantee is strictly about the service's storage, never about the recipient's behavior, and screenshot detection is a notification feature, the sender gets told, not a prevention mechanism, because the platform cannot stop a determined capture. Second, reported content is the deliberate exception: when a user reports a message, the service copies it into a separate quarantine store with its own access controls and a retention clock scoped to the investigation, and a legal hold can extend that clock. The product copy should reflect both limits, and the architecture should make the quarantine path the only door out of the ephemeral world.

View-once and the multi-device wrinkle

View-once is a server-side state machine, not a client promise. The view state lives in the message row, and when a recipient opens the photo, the client reports the view, the server transitions unviewed → viewed atomically, starts a short fuse on the media key, and writes a tombstone, a small marker row that says this message existed and was consumed, so late-syncing devices learn the message is burned rather than seeing it vanish mysteriously. The atomic transition matters because two devices can race: a user with a phone and a tablet might open the same photo on both within a second, and the compare-and-set on the view state lets exactly one open succeed as the first view while the other renders the burned placeholder. That is the multi-device wrinkle in one sentence: viewing on one device must burn the message on all devices, so the burn is a server event fanned out like any other message, and clients holding a cached preview discard it on receipt.

Step 1: the message is persisted with its TTL and delivered. Step 2: the recipient opens it and the server records the first view atomically. Step 3: the view starts the burn timer and the message transitions to expired, invisible to every query. Step 4: the dashed branch covers messages whose TTL fires without a view. Step 5: compaction, key deletion, and the blob cleanup job make the deletion physical.

Stories, counters, and the sweep

A story is a broadcast with a fixed 24-hour TTL, and the delivery question is the usual fanout choice. Posting writes the story row once, then fans out a lightweight reference into each follower's story tray, the per-user list the app renders along the top of the screen; for a typical friend graph of a few hundred connections that is a few hundred 50-byte writes per post, which is cheap, while accounts with millions of followers flip to read-time merging exactly like large channels in a chat system, so the tray render pulls their active stories on demand. Tray references carry the same expiry as the story, which means the tray cleans itself: a client that opens the app at hour 25 simply gets no reference, and no synchronous mass-delete across millions of trays ever needs to happen.

View counts and viewer lists are the one place ephemerality meets a counter, and the counters are deliberately approximate in flight: view events stream through a queue and increment a per-story aggregate, the viewer list is a set keyed by story with the story's TTL, and both disappear with the story. The cleanup sweep is the service's background heartbeat: it walks expiring partitions, deletes orphaned blobs whose key rows are gone, verifies that compaction is keeping the physical-deletion window inside the published bound, and emits metrics on the lag between logical and physical deletion, because that lag is the product promise expressed as a number an operator can watch.

Scaling, failures, and operations

The socket tier scales like any chat connection tier, horizontally with a session registry mapping users to servers, and the message store scales by conversation partitions; nothing ephemeral changes those rules. What changes is the failure analysis around deletion. If the sweeper falls behind, no user-visible feature breaks, which is exactly why it needs explicit monitoring: the deletion-lag metric has to alarm on its own, because nothing else will. If compaction stalls on a store node, expired rows persist physically past the window, and the response is operational, prioritizing compaction on the affected ranges, plus the crypto-shredding backstop, which keeps the content unreadable regardless. Caches deserve suspicion everywhere: application-level caches of message bodies are simply banned, and any cache that does exist must carry TTLs no longer than the content's remaining lifetime.

Backups are the failure mode people forget. A nightly snapshot of the message store quietly extends every message's life by the backup retention period, which contradicts the product. The resolution is to back up metadata needed for recovery, accounts, social graph, configuration, while excluding message content tables, accepting that a catastrophic store loss loses in-flight messages, a defensible trade for this product since everything in the store was about to be deleted anyway. Push notifications must not leak content either, so they carry only "new message from Ana" with the actual payload fetched over the socket. And the quarantine store for reported content runs the opposite regime from everything else: durable, backed up, access-logged, and small, with every entry tied to a case and a clock.

Follow-up questions

  • What does "deleted" actually mean here? Logically deleted at expiry, meaning no query can return the row, and physically deleted within a stated window enforced by compaction scheduling and the sweeper, with crypto-shredding making stray copies unreadable in the meantime.
  • Why not just run a cron job that deletes expired rows? At hundreds of millions of rows a day, scan-and-delete competes with live traffic and always lags. Native store TTLs make expiry a property of the row, and compaction reclaims space as part of work the store does anyway.
  • How does view-once survive a user with two devices? The view state is server-side and the first-view transition is a compare-and-set, so one device wins, and the burn is fanned out as an event that all the user's devices apply, discarding any cached preview.
  • Can the service promise the recipient cannot keep a copy? No, and the design should say so: screenshots and second cameras are outside the trust boundary. The promise covers the service's storage, and screenshot detection only notifies the sender.
  • How do you cache media on a CDN if it must disappear? Cap edge cache lifetimes below the content's remaining TTL so copies expire on their own, skip the CDN for view-once media, and encrypt blobs so key expiry neutralizes any copy that outlives a purge call.
  • What happens when someone reports a message that then expires? Reporting copies the content into a quarantine store before the ephemeral clock can destroy it, with scoped access and case-bound retention; a legal hold extends retention there, never in the main store.
