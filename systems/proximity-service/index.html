Design a proximity service

Systems design · Location and maps · Sep 2025

A proximity service answers the question behind every "restaurants near me" tap: given a point on the map and a radius, return the businesses inside it, ranked sensibly. It is the engine under Yelp's nearby tab, a retailer's store locator, and the place results in a maps app. The workload has a distinctive shape that makes it a favorite interview question: it is overwhelmingly read-heavy, it tolerates slightly stale data because businesses move rarely, and the core technical obstacle is that ordinary database indexes order one dimension while the query lives in two. A candidate who can explain why a B-tree struggles here, and can then build a spatial index from first principles, demonstrates exactly the kind of reasoning interviewers want to see.

The walkthrough below runs the question the way I would in a real interview: scope first, then numbers, then the indexing problem that dominates the design, and finally the operational story of replicas, caches, and shards.

Scope and requirements

The functional core is a single query: return businesses within a user-chosen radius of a latitude and longitude, with optional filters such as category and a ranking that blends distance with rating. Business owners need to add, edit, and remove listings, and users need a detail view for a single business. I would confirm with the interviewer that full-text search, reviews, and photos belong to other services, and that this one owns the geospatial lookup.

The non-functional requirements are friendly compared to most designs. Searches should return in a couple hundred milliseconds at the 99th percentile because they sit on the critical path of a consumer app. The system should be highly available, and it can be eventually consistent: if a newly added cafe takes a few minutes or even a day to appear in search results, nobody is harmed, and that looseness is worth saying out loud because it unlocks aggressive caching and periodic index rebuilds later. Reads dwarf writes, since millions of people search while only a trickle of owners edit listings on any given day.

Sizing the problem

Concrete numbers keep the design grounded. Assume 100 million businesses and 100 million daily users who each perform about five searches a day. That is 500 million searches per day, and a day is 86,400 seconds, so the average load is roughly 5,800 queries per second; I would round to 5,000 average and plan for a peak around three times that, or 15,000 queries per second. Writes are a rounding error by comparison: if one percent of businesses are edited on a busy day, that is 1 million updates in 86,400 seconds, or about 12 writes per second.

Storage is equally tame. A business record with name, coordinates, category, rating, address, and hours fits in about 1 KB, so 100 million businesses occupy roughly 100 GB, which a single replicated relational database handles without ceremony. The geospatial index is smaller still: one entry per business holding an 8-byte ID plus a cell identifier comes to a few GB even with structural overhead, and that observation, that the entire index fits comfortably in the memory of one machine, will shape the architecture more than any other number here.

The API

The read path is one endpoint, and the write path is conventional CRUD on listings. Pagination uses an opaque token so the server can encode where the previous page ended without exposing internals.

GET /v1/search/nearby?lat=37.7749&lng=-122.4194&radius=500&category=cafe
→ 200 { "total": 312,
        "businesses": [ { "id": 8123, "name": "Blue Door Coffee",
                          "distance_m": 120, "rating": 4.6 }, ... ],
        "next_page_token": "eyJvZmZzZXQiOjIw..." }

GET    /v1/businesses/{id}        → 200 { full business detail }
POST   /v1/businesses             → 201 (owner adds a listing)
PUT    /v1/businesses/{id}        → 200 (edit; index updated asynchronously)
DELETE /v1/businesses/{id}        → 204

The data model

The source of truth is a plain relational table, because business records are small, relational tooling is mature, and nothing about the data demands anything exotic. The spatial structure lives beside it as a mapping from cell identifiers to business IDs, which can be a second table at small scale or an in-memory structure at large scale, and the next sections explain what those cells are.

CREATE TABLE businesses (
  id          BIGINT PRIMARY KEY,
  name        TEXT NOT NULL,
  lat         DOUBLE PRECISION NOT NULL,
  lng         DOUBLE PRECISION NOT NULL,
  category    TEXT,
  rating      REAL,
  address     TEXT,
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE geo_index (
  geohash      CHAR(6) NOT NULL,        -- cell of about 1.2 km by 0.6 km
  business_id  BIGINT NOT NULL REFERENCES businesses(id),
  PRIMARY KEY (geohash, business_id)
);

The high-level architecture

A load balancer fronts a stateless search service. Searches hit a geospatial index that maps cells to business IDs, candidates are hydrated from the business database, and hot cell results are cached. Because the index is a few GB, it is replicated wholesale to several read-only nodes rather than sharded, and the database feeds those replicas through incremental updates or periodic rebuilds, which the loose freshness requirement permits.

Searches resolve cells against an index replica, hydrate business details from the database, and lean on a cache for hot cells. The dashed path is the asynchronous index rebuild, which the loose freshness requirement makes possible.

Why a one-dimensional index fails

The instinctive first answer is to index latitude and longitude with ordinary B-tree indexes, which are the balanced tree structures relational databases use to answer range queries over a single ordered column, and the instinct fails in an instructive way. A B-tree gives fast answers to "all rows with lat between 37.770 and 37.780" because latitude is one ordered dimension, but a radius query is a conjunction of two ranges, and each range alone matches an enormous stripe of the planet.

Work the numbers to see it. A 500 m radius spans about 0.009 degrees of latitude, so the latitude predicate selects a thin band that nonetheless circles the entire globe. If businesses were spread evenly over the 180 degrees of latitude, the band would hold 100 million times 0.009 over 180, about 5,000 rows; in reality businesses cluster in cities, and a band passing through one metro area also passes through every other city sitting at the same latitude anywhere on Earth, so a band through Seoul also sweeps Madrid and Philadelphia and the count lands in the tens or hundreds of thousands. The longitude band behaves the same way north to south. The database can use only one of the two indexes to drive the scan, so it pulls one huge stripe and filters it row by row against the other condition to find perhaps a hundred true matches, and doing that work 15,000 times per second is not a plan. The fix is an index whose keys encode both dimensions at once, which is what geohashes and quadtrees provide.

Geohash, quadtree, and the neighbor problem

A geohash is a way of folding two dimensions into one sortable string. Halve the longitude range and write a 1 if the point falls in the upper half, halve again, and repeat; do the same for latitude; then interleave the two bit streams, longitude bit first, and encode every five bits as one character of a base32 alphabet. The decisive property is that a prefix names a rectangle: every point whose geohash starts with 9q8yy lies in one specific cell, and longer prefixes name smaller cells nested inside. The standard sizes are worth memorizing because they let you pick a precision instantly: 4 characters is about 39 km by 20 km, 5 characters about 4.9 km by 4.9 km, 6 characters about 1.2 km by 0.6 km, and 7 characters about 150 m by 150 m. For a 500 m radius search, 6 characters is the natural fit, and the geo index table above is exactly this: rows keyed by 6-character cell, which a B-tree handles perfectly because the cell ID is one ordered dimension.

One failure mode needs calling out before it bites. Two points 10 m apart can sit on opposite sides of a cell boundary and share no useful prefix at all, and a query point near a cell edge has part of its radius hanging into adjacent cells, so searching only the home cell silently drops results near every boundary. The fix is mechanical: compute the 8 neighboring cells, which is cheap bit arithmetic, and query all 9. With 6-character cells, the 3 by 3 block covers roughly 3.7 km by 1.8 km, comfortably containing any 500 m circle no matter where the point sits inside the home cell.

The query point sits near the edge of its home cell, so the radius spills into the neighbors; fetching the 3 by 3 block guarantees coverage, and businesses inside fetched cells but outside the circle are dropped by the exact distance check.

The quadtree is the other classic answer, and it adapts to density instead of using fixed cells. A quadtree is a tree that starts with one node covering the whole map and recursively splits any node into four equal quadrants until each leaf contains fewer than some threshold k businesses, so leaves are tiny in Manhattan and enormous over Montana. The build arithmetic is reassuring: with k of 100, the structure needs about 100 million over 100, or 1 million leaves; a quadtree with 1 million leaves has about a third of a million internal nodes, and at tens of bytes per node plus 8 bytes per stored business ID the whole tree lands near 1 to 2 GB, which one server builds in minutes and holds in memory easily. Updating a quadtree in place is fiddlier than upserting a geohash row, which is why many teams rebuild it periodically instead. Google's S2 library is the production-grade refinement of these ideas: it projects the sphere onto cube faces and orders cells along a Hilbert curve, which is a space-filling path that preserves locality better than geohash interleaving, and Uber's H3 does something similar with hexagons. In an interview, geohash is the easiest to defend end to end, with S2 or H3 named as what you would adopt rather than rebuild.

The query path, ranking, and caching

A search now runs in four steps. The service converts the query point and radius into a cell cover, meaning the home cell plus its 8 neighbors at the chosen precision; it fetches candidate business IDs for those cells from an index replica; it computes the exact distance from the query point to each candidate using the haversine formula, the standard great-circle distance over the Earth's surface, discarding anything outside the radius; and it ranks the survivors, typically by a blend of distance and rating, before hydrating the top page from the database or the object cache. Candidate counts stay small because 9 cells at 6-character precision cover only a few square kilometers, which even in a dense downtown holds a few thousand businesses, and a few thousand haversine evaluations cost well under a millisecond.

Pagination works best as a cursor rather than an offset: the page token encodes the rank score and ID of the last returned business, and the next request re-runs the cheap cell query and resumes after that position, which stays correct even if the underlying data shifts slightly between pages. Caching attacks the read load at two levels. Whole cell results, keyed by cell ID plus the filter set, get a TTL of a few minutes, which collapses the many users searching the same neighborhood into one index lookup; individual business objects are cached by ID with a longer TTL and invalidated on edit. Because popular areas are searched constantly, hit rates on the cell cache run high, and the staleness window is exactly the freshness the requirements already conceded.

Writes follow the quiet path. An edit lands in the business database as the source of truth, and the index absorbs it either incrementally, which for a geohash table is a delete plus insert of one row, or through a periodic rebuild, where a job walks the table, constructs a fresh index or quadtree, and swaps replicas one at a time behind the load balancer. At 12 writes per second, either approach is loafing.

Scaling, failures, and operations

Each tier scales by its own rule. The search service is stateless and scales horizontally, and 15,000 queries per second spread over a couple dozen nodes is unremarkable. The index scales by replication rather than sharding, because a few GB fits everywhere and adding a replica adds read throughput linearly; replicas register with the load balancer and the search tier round-robins across them. The database carries 12 writes per second and the read traffic the caches let through, so a primary with a couple of replicas is enough for a long time.

Sharding deserves a paragraph because interviewers push on it even when the numbers say replicate. The tempting scheme is to shard the index by region, giving each shard a contiguous slice of the world so a query touches exactly one shard, but geography is brutally skewed: the shard holding Tokyo or Manhattan sees orders of magnitude more traffic than the shard holding open ocean, and redrawing region boundaries under load is operational misery. The alternative is to shard by hashing the business ID, so every shard holds a thin slice of every cell; a query then scatter-gathers across all shards and merges, which makes tail latency the latency of the slowest shard but spreads load perfectly evenly and never needs rebalancing by geography. At this data size the right answer is to replicate the whole index and avoid the choice, while saying that hash-plus-scatter-gather is the scheme you would reach for if the index grew tenfold.

The failure stories are mild. A dead search node drops out of the load balancer; a dead index replica reduces capacity until a replacement rebuilds from a database snapshot in minutes; a cache cluster loss raises index traffic until it rewarms, and request coalescing, where concurrent misses on the same cell wait on one fill instead of stampeding, keeps that surge polite. The database failover is the only moment of real care, and since search reads come from the index and caches, a brief write outage on listings is nearly invisible to users. Operationally the metrics that matter are search latency percentiles, cell cache hit rate, index replica staleness, and a skew report of the hottest cells, which warns you early if one neighborhood is heating beyond what a single replica set handles.

Follow-up questions

  • Why not use PostGIS or Elasticsearch geo queries instead of building an index? At moderate scale you should, and saying so is good judgment. The custom path earns its keep when query volume outgrows what a database extension serves cheaply, and the interview value is showing you understand what those tools do inside, which is exactly the geohash and tree machinery above.
  • What geohash precision do you pick, and what if the radius changes? Match the cell size to the radius: 6 characters for hundreds of meters, 5 for a few kilometers, 4 for tens of kilometers. A 5 km search at 6-character precision would need dozens of cells, so the service picks the precision where the home cell plus neighbors covers the circle in 9 fetches.
  • How do results near a cell boundary stay correct? Every query fetches the 8 neighbor cells along with the home cell, because a point near an edge has part of its radius in adjacent cells and two close points can share no geohash prefix at all. The exact haversine filter then trims overfetch.
  • Geohash table or quadtree? The geohash table is simpler to update and trivially durable; the quadtree adapts cell size to density and answers variable-radius queries gracefully but is easier to rebuild than to mutate. Both fit in memory here, so the deciding factor is operational simplicity, which favors geohash.
  • What breaks first at 10x? Read volume, and the design absorbs it by adding index replicas and cache nodes, which is why keeping the index small and replicable was the central decision. The business database and write path barely notice 120 writes per second.
  • How would you rank beyond distance? Compute distance as a feature, then score candidates with rating, popularity, and openness now, either with a hand-tuned formula or a lightweight model, but keep candidate generation purely geometric so the index stays generic.
