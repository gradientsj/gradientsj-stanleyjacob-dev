Design consistent hashing

Systems design · Distributed building blocks · Feb 2025

Consistent hashing answers a question every distributed storage and caching system faces: given a key, which of N servers should hold it, in a way that survives servers joining and leaving? The naive answer, hash the key and take the remainder modulo N, works perfectly until N changes, at which point almost every key suddenly belongs somewhere else and the cluster spends hours shuffling data or, in the cache case, melts the database behind it with misses. Consistent hashing rearranges the assignment so that adding or removing one server moves only the keys that server should own, roughly a 1/N fraction, and leaves everything else untouched.

Interviewers ask this both as a standalone design and as a building block inside larger questions, because it tests whether a candidate can reason about distribution, balance, and failure with actual arithmetic rather than hand-waving. The hard parts are not the ring itself, which takes two minutes to describe, but the consequences: arcs are badly uneven with few nodes, hotspots ignore the hash entirely, and replication and membership changes interact with the ring in ways worth thinking through carefully.

Scope and requirements

The deliverable is a key-placement scheme, usually packaged as a library, with three operations: map a key to a server, add a server, and remove a server. Functionally, every client holding the same view of membership must route a given key to the same server, lookups must be fast enough to sit on the hot path of every request, and the scheme must support weighting, so a machine with twice the capacity can take twice the keys.

The non-functional requirements are where the design earns its name. When membership changes by one server, the fraction of keys that move should be close to the theoretical minimum of 1/N, because every moved key is either data to copy or a cache miss to absorb. Load should spread evenly, within a few percent of the mean per server, since the most loaded server sets the cluster's effective capacity. The scheme should need no central coordinator on the lookup path: each client computes placement locally from the membership list. And it must handle the practical sizes, hundreds of servers and billions of keys, without the lookup structure itself becoming a burden.

Sizing the problem

The numbers that matter are the cost of a lookup and the cost of a membership change. Take a cluster of 10 cache servers holding 50 million keys. With the ring scheme described below and 200 virtual positions per server, the lookup structure holds 10 × 200 = 2,000 entries; at roughly 40 bytes per entry that is 80 KB, small enough to live in every client's memory, and a binary search over 2,000 sorted positions costs log base 2 of 2,000, about 11 comparisons, well under a microsecond.

Membership changes are where the scheme pays for itself. Growing from 10 servers to 11 should move about 1/11 of the keys, which is 50 million / 11, roughly 4.5 million keys. The modulo scheme, as the worked example below shows, would move about 80 to 90 percent of them, around 40 million keys: nearly ten times the data transfer, or ten times the cache misses, for the same one-server change. If each key is a 1 KB cached object, that is the difference between copying 4.5 GB and refetching 40 GB through the database.

The interface

The whole design fits in a small class, and writing the lookup makes the data structure concrete: the ring state is nothing more than a sorted array of (position, server) pairs, and a lookup is a binary search for the first position at or after the key's hash, wrapping to the start when the search falls off the end. The bisect module is Python's binary search over a sorted list.

import bisect, hashlib

def h(s: str) -> int:
    return int.from_bytes(hashlib.md5(s.encode()).digest()[:8], "big")

class Ring:
    def __init__(self, vnodes: int = 200):
        self.vnodes = vnodes
        self.positions = []           # sorted list of (hash, server)

    def add(self, server: str):
        for i in range(self.vnodes):
            bisect.insort(self.positions, (h(f"{server}#{i}"), server))

    def remove(self, server: str):
        self.positions = [(p, s) for p, s in self.positions if s != server]

    def get(self, key: str) -> str:
        i = bisect.bisect_right(self.positions, (h(key), ""))
        return self.positions[i % len(self.positions)][1]

The hash function needs to spread values uniformly across the 64-bit space but has no security role, so a fast non-cryptographic hash such as MurmurHash or xxHash is the production choice; MD5 appears above only because it ships with the standard library.

Why hash mod N fails

The baseline deserves a worked demolition because it looks so reasonable. Assign each key to server hash(key) mod N. With N = 4, a key hashing to 17 goes to server 1, since 17 mod 4 is 1. Now add a fifth server. The same key now goes to 17 mod 5, which is server 2. Run the same check across hashes 0 through 19, one full cycle of both moduli: a key stays put only when its hash gives the same remainder mod 4 and mod 5, which happens for hashes 0, 1, 2, and 3 and for no others in the cycle. Four out of twenty keys stay, so 80 percent of all keys move for a change that logically required moving only 20 percent. In general growing from N to N+1 servers relocates about N/(N+1) of the keys, which approaches everything as the cluster grows: a 99-to-100-server change moves 99 percent of the data to gain 1 percent capacity.

For a storage system that means a rebalancing storm of copies; for a cache it means the hit rate collapses to near zero at the moment of the change, and every miss lands on the database the cache was protecting. Both versions of the disaster are self-inflicted by the placement function, which is the motivation for changing it.

The ring

Consistent hashing maps servers and keys into the same space and makes ownership a matter of proximity. Treat the output range of the hash function as a circle, so the largest value wraps around to zero. Hash each server's identifier to get its position on the circle, and hash each key the same way. A key is owned by the first server found moving clockwise from the key's position. That single rule produces the property the modulo scheme lacked: a server's arrival or departure only redraws the boundary of one arc, and keys everywhere else still find the same owner by the same clockwise walk.

Servers and keys hash onto the same circle, and each key belongs to the first server clockwise: keys u15 and u42 belong to Node B, and key u77 belongs to Node D.

The construction comes from Karger and colleagues' 1997 paper, written originally for distributed web caching, and it has since become the placement scheme inside Amazon's Dynamo and its descendants, Apache Cassandra's token ring, client libraries for Memcached such as Ketama, and request routing in CDN caches, which is a useful list to mention in an interview because each system stresses a different property of the ring.

Adding and removing nodes

The payoff shows up the first time membership changes. Suppose node E joins and hashes to a position between A and B. Every key whose position falls between A and E used to walk clockwise past that point to reach B, and now stops at E first; those keys, and only those, change owner. With five nodes placed roughly evenly, the arc between A and E holds about one fifth of the circle's keys at most and typically half an arc, so on the order of 50 million / 5 = 10 million keys move in the 50-million-key cluster, and in the general case about 1/N of the keyspace, compared with the 80 percent the modulo scheme moved for the same event. Removal is the mirror image: when a node leaves, its arc's keys continue clockwise to the next node, and nothing else changes.

Node E lands between A and B, and only the highlighted arc between A and E changes owner: key k, which belonged to B before, now stops at E on its clockwise walk. Keys on every other arc keep their owners.

Operationally the data copy follows the same arithmetic: the joining node streams its new arc from the node that previously owned it, B in the picture, while the rest of the cluster serves traffic undisturbed, and the old owner can keep serving the arc until the copy completes so that the handoff is invisible to clients.

Virtual nodes and balance

The plain ring has a balance problem that the picture hides. With only a handful of nodes, the arcs are whatever random hashing produced, and randomness is lumpy: with 4 nodes it is entirely plausible that one arc covers 40 percent of the circle and another covers 10, so one server carries four times the load of another. The expected size of the largest arc among N random positions is on the order of (ln N)/N rather than 1/N, which means imbalance gets relatively worse, not better, as a fraction of fair share for small clusters.

The fix is virtual nodes: instead of hashing each server to a single position, hash it to k positions, for example serverA#0 through serverA#199, and let the server own the union of its 200 small arcs. Each server's total share is now the sum of many independent random arc lengths, and sums of independent random variables concentrate around their mean, with the relative standard deviation shrinking roughly as 1 over the square root of k. Concretely, with one position per server the per-server load can easily deviate from fair share by 50 percent or more; with k = 100 the standard deviation falls to roughly 10 percent of the mean, and with k = 400 to roughly 5 percent, because the square roots of 100 and 400 are 10 and 20. Virtual nodes also give weighting almost for free: a server with double the capacity simply registers double the positions. The cost is a larger lookup table, which the sizing section already showed is tens of kilobytes, and a membership change now redraws k small boundaries instead of one, which actually helps, since the leaving node's load disperses across many successors instead of dumping onto one neighbor.

Replication, hotspots, and bounded loads

Storage systems need each key on more than one machine, and the ring extends naturally: store each key on the first R distinct physical servers encountered clockwise from its position, skipping virtual positions that belong to a server already in the list. With R = 3, the key u42 in the first diagram would live on B, C, and D, and the skip rule matters because adjacent virtual positions frequently belong to the same physical machine. This is exactly the preference-list construction in Dynamo and Cassandra, and it means a node failure leaves every key still reachable on its other replicas while the ring repairs.

Hotspots are the limit of what any placement scheme can fix, because hashing balances key counts, not key popularity. If one celebrity's profile receives 100,000 reads per second, the server owning that one key is overloaded no matter how evenly the keyspace is spread. Within the placement layer, the tool is bounded-load consistent hashing, the variant published by Mirrokni and colleagues: set a cap of (1 + ε) times the average load per server, for instance 1.25 times, and when the clockwise owner is already at its cap, walk on to the next server. Placement stays mostly consistent, no server exceeds the cap, and the analysis shows membership changes still move few keys. Beyond the placement layer the answers are caching the hot key closer to clients and replicating it explicitly, which are application decisions rather than ring decisions, and worth saying so in the interview.

Scaling, failures, and operations

The ring itself scales gracefully because it is just a sorted array recomputed from membership: 1,000 servers at 200 virtual nodes each is 200,000 entries, a few megabytes per client and a binary search of 18 comparisons. The real scaling question is who maintains the membership list. Cassandra and Dynamo-style stores gossip membership between nodes, meaning each node periodically exchanges its view with random peers until everyone converges; Memcached client rings typically read a static configuration pushed by deployment tooling; and systems wanting strong agreement keep membership in a coordination service such as ZooKeeper or etcd. The failure mode to call out is split views: if two clients briefly hold different membership lists, the same key routes to different servers, which a cache tolerates as extra misses but a storage system must reconcile with versioning or read repair, which is why storage systems pair the ring with quorum reads rather than trusting any single placement.

Node failure is deliberately uneventful. When a server dies, its arcs transfer to the next servers clockwise, which with virtual nodes means the load spreads thinly across many machines rather than doubling one neighbor's traffic; with 200 virtual nodes, a dead server's load lands in roughly 200 slivers. Flapping membership (a node repeatedly leaving and rejoining as health checks disagree) causes repeated small migrations, so production systems damp membership changes with timeouts before declaring a node truly gone. The last operational note is to never derive the ring position from anything mutable such as an IP address that changes on reboot, because a node that silently changes position orphans its old arcs; stable server identifiers are the rule.

Follow-up questions

  • Why not just keep a key-to-server table instead of hashing? A table gives perfect control but must be stored, replicated, and consulted on every lookup, becoming its own distributed system; at billions of keys it is also large. The ring computes placement from a few kilobytes of membership data with no lookup service. Some systems (HDFS, many SQL shard maps) do choose explicit tables when key counts are small and control matters more.
  • How many virtual nodes should each server get? Enough that load imbalance is acceptable: the deviation shrinks like 1 over the square root of k, so 100 to 200 positions brings imbalance to roughly 5 to 10 percent, and Cassandra's historical default of 256 sits in the same range. Beyond that, table growth buys little.
  • What changes if two clients disagree about membership? The same key routes differently. A cache absorbs it as misses; a store must detect the divergence, which is why ring-based databases add versioning, read repair, and quorum overlap rather than relying on placement alone.
  • Does consistent hashing help with a single hot key? No, and saying so matters: hashing balances key counts, not popularity. Bounded-load variants cap per-server load by spilling to the next node, and beyond that the fixes are replicating or caching the hot key explicitly.
  • How does a new node actually get the data for its arc? It streams the arc from the previous owner while that owner continues serving, then ownership flips atomically in the membership view; with virtual nodes the new server pulls many small slivers from many donors, which parallelizes the transfer.
  • Where would you not use consistent hashing? When range queries matter, because hashing destroys key order; range-partitioned systems such as Bigtable and HBase split sorted key ranges instead and rebalance by moving range boundaries.
