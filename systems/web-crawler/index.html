Design a web crawler

Systems design · Search and discovery · Apr 2025

A web crawler starts from a set of seed URLs, downloads those pages, extracts the links they contain, and follows them, repeating until it has visited some meaningful slice of the web. Search engines run crawlers to build their indexes, archives run them to preserve pages before they disappear, and research and monitoring systems run them to mine content at scale. The loop sounds like a weekend project, and a single-machine crawler genuinely is one, which is exactly why interviewers ask the question: the distance between the toy and the real system is made of problems that only appear at scale, and most of them are about restraint rather than speed.

That restraint is the social contract of crawling. A crawler is a guest on every server it visits, and an aggressive one is indistinguishable from a denial-of-service attack, so politeness, honoring each site's stated rules and limiting request rate per host, is a first-class design constraint rather than a feature to add later. The hard parts of this design live in three places: the URL frontier, which is the data structure that decides what to fetch next while enforcing politeness; deduplication of both URLs and content at billions of items; and the traps a hostile or merely strange web lays for any program that follows links blindly.

Scope and requirements

Functionally, the crawler should fetch HTML pages starting from seeds, extract and follow links, store raw page content for downstream consumers such as an indexer, and revisit pages to pick up changes. Scope excludes rendering JavaScript-heavy pages (a real consideration, but a separate fleet of headless browsers), and excludes non-HTML media beyond noting their links. Non-functionally, the target is one billion pages per month with the politeness rules above, resilience to individual machine failures without losing the frontier, and extensibility, since every crawl operator eventually adds new content processors.

It is worth saying that coverage and freshness fight each other for the same fetch budget. Every request spent revisiting a known page is a request not spent discovering a new one, so the requirement is not "crawl everything, always fresh" but "spend a fixed budget where it earns the most," which reframes the scheduler as the interesting component rather than the downloader.

Sizing the problem

One billion pages per month over roughly 2.6 million seconds is about 385 pages per second, call it 400. At an average page size of 500 KB, that is 400 times 500 KB, or 200 MB/s of download bandwidth, about 1.6 Gbps sustained, which a small fleet handles without exotic networking. Storage is the bigger number: one billion pages times 500 KB is 500 TB per month of raw HTML before compression, and HTML compresses well, so 100 to 150 TB per month lands in the blob store, with retention policy deciding how many months accumulate.

The politeness constraint converts throughput into concurrency. If the crawler limits itself to one request per host every two seconds, sustaining 400 pages per second requires fetches in flight against at least 800 distinct hosts at any moment, and since a fetch including DNS, connect, and transfer takes about a second on average, the fleet needs on the order of a thousand concurrent connections spread across hosts. That single calculation explains the frontier design below: the scheduler's real job is keeping hundreds of hosts in rotation, not keeping a queue full.

The crawler's interface to the web

The crawler speaks plain HTTP, and two exchanges define its manners. Before fetching anything from a host, it fetches and caches that host's robots.txt, the file where sites declare what crawlers may touch, and on revisits it uses conditional requests so unchanged pages cost almost nothing.

GET /robots.txt HTTP/1.1
Host: example.com
User-Agent: research-crawler/1.0 (+https://crawler.example.org/about)

→ 200
User-agent: *
Disallow: /private/
Crawl-delay: 2

GET /products/blue-shoes HTTP/1.1
Host: example.com
If-Modified-Since: Tue, 01 Apr 2025 08:00:00 GMT

→ 304 Not Modified              (unchanged: no body transferred)

The descriptive User-Agent with a contact URL is part of the contract: site operators seeing the crawler in their logs can find out who is fetching and ask for changes, and crawlers that hide behind browser agent strings get blocked as abusers because that is what they resemble.

The data model

Raw page bodies go to a blob store keyed by content hash, and a metadata database carries the crawl state per URL. The per-URL row is what the scheduler reads and writes, so it stays narrow.

CREATE TABLE urls (
  url_hash      BYTEA PRIMARY KEY,     -- 16-byte hash of normalized URL
  url           TEXT NOT NULL,
  host          TEXT NOT NULL,
  last_fetched  TIMESTAMPTZ,
  content_hash  BYTEA,                 -- last seen body checksum
  change_score  REAL DEFAULT 0.5,      -- estimated change frequency
  depth         SMALLINT DEFAULT 0,
  failures      SMALLINT DEFAULT 0
);

One billion URLs at roughly 200 bytes per row is about 200 GB, which shards comfortably by url_hash; the table's job is bookkeeping, while the heavy read-write traffic of "have I seen this URL" goes through an in-memory structure described below.

The high-level architecture

The system is a loop drawn as a pipeline. The URL frontier holds URLs waiting to be fetched and releases them according to priority and politeness. Fetcher workers take URLs from the frontier, resolve DNS through a local cache, download the page, and hand the body to a content-dedup check that discards pages whose content was already stored under a different URL. New content goes to the blob store, and a parser extracts outgoing links, which pass through URL normalization, filtering, and the seen-set test before the survivors re-enter the frontier, closing the loop. Traversal is breadth-first in flavor, since links are discovered level by level from the seeds, but the frontier's priorities mean it is never strictly BFS: important or fast-changing pages jump the line, which is the behavior you actually want.

The crawl loop: the frontier dispenses URLs under politeness rules, fetchers download through a DNS cache, new content is stored and parsed for links, and normalized unseen URLs flow back into the frontier.

DNS deserves its box in that picture because resolution quietly becomes a bottleneck: 400 fetches per second against fresh hostnames is 400 resolver queries per second, each costing tens of milliseconds through a typical resolver chain. Running a local caching resolver per crawler node, with the crawler's own cache keyed by host on top, turns the common case into a memory lookup and also gives one place to implement per-host connection reuse.

The frontier: front queues for priority, back queues for politeness

A naive frontier is one big FIFO queue, and it fails politeness immediately: a burst of links discovered on one site, say ten thousand URLs from example.com, would be fetched back to back, hammering that host. The classic fix, from the Mercator crawler, splits the frontier in two. A set of front queues implements priority: a prioritizer scores each incoming URL by signals like site importance, depth from seeds, and expected change rate, and places it in the queue for its priority band. A larger set of back queues implements politeness: each back queue holds URLs for exactly one host, a table maps hosts to their queues, and a delay heap orders hosts by the earliest time each may be contacted again, based on its crawl delay.

Incoming URLs are scored into front queues; the router moves them into per-host back queues, never mixing hosts. A fetcher pops the next ready host from the delay heap, dequeues one URL from that host's back queue (dashed), fetches it, and pushes the host back with ready time now plus its crawl delay.

The flow step by step: a fetcher wanting work pops the delay heap, which yields the host whose ready time has passed, takes one URL from that host's back queue, and fetches it. When the fetch completes, the host goes back on the heap with a ready time of now plus its delay, two seconds by default or whatever Crawl-delay requested. If a back queue empties, the router refills it by pulling from the front queues, biased toward the high-priority queues, and assigns any URL from a brand-new host to a free back queue, updating the host table. Priority and politeness stay cleanly separated: the front queues decide what deserves fetching, the back queues and heap decide when each host can be touched, and no burst of discoveries can ever concentrate requests on one server. Keeping roughly three times as many back queues as fetcher threads ensures a fetcher almost always finds some host ready.

Deduplication, traps, and recrawl

The seen-set answers "has this URL ever entered the frontier?" billions of times, so it cannot be a database round trip per link. URLs are first normalized, lowercasing scheme and host, resolving relative paths, sorting or stripping known-irrelevant query parameters, and removing fragments, so that trivially different spellings of the same page collapse. Then a Bloom filter, a compact bit array that answers definitely-no or probably-yes using a few hash probes per key, screens candidates in memory: 10 billion URLs at 10 bits each is about 12.5 GB for a false-positive rate near 1 percent. The trade is that a false positive silently skips a never-seen URL, which costs about 1 percent of discovered links; a crawl that cannot accept that pairs the filter with a sharded key-value confirmation for filter hits, paying the lookup only on the rare probably-yes path.

Content dedup is the same idea one level up, because the web is full of identical bodies under different URLs, mirrors, session-ID variants, and www versus bare domains, and storing each copy wastes the blob store and skews the index. An exact checksum over the body catches identical copies cheaply, and near-duplicate detection catches the boilerplate-plus-same-article case: simhash condenses a document into a 64-bit fingerprint with the property that similar documents differ in only a few bits, so checking new fingerprints against the corpus within a small Hamming distance flags near-copies at scale.

Spider traps are link structures that generate unbounded URLs, sometimes by accident: an infinite calendar whose "next month" link never ends, session IDs minting a fresh URL per visit, faceted listings multiplying parameters forever. Defenses stack: a depth limit per crawl path, URL pattern heuristics such as rejecting URLs beyond a length cap or with many repeated path segments, and most robustly a per-domain page budget so even an undetected trap costs only that domain's allotment before the crawler moves on. Budgets also bound damage from genuinely enormous sites that would otherwise absorb the whole crawl.

Recrawl scheduling closes the loop on freshness. Each fetch that returns 304 or an unchanged checksum nudges that page's change-rate estimate down, and each observed change nudges it up, with the scheduler converting estimates into next-visit times: a news front page every few minutes, a stable reference page every few weeks. The change_score column in the metadata table is exactly this estimate, and the frontier's prioritizer reads it, which is how freshness competes with discovery inside one fetch budget rather than in two separate systems.

Scaling, failures, and operations

The crawl distributes by partitioning the URL space by host across crawler nodes, so every URL for example.com lives on the same node. That one decision keeps all politeness state, the back queue, the delay timing, the robots.txt cache for the host, local to one machine with no cross-node coordination on the hot path, and it also localizes DNS cache hits and connection reuse. Links a node extracts for hosts it does not own are forwarded to the owning node, conveniently through a queue partitioned the same way. Adding nodes means re-partitioning hosts, which consistent hashing keeps proportional, and a node's frontier is recoverable because back-queue contents are checkpointed to durable storage; losing a node at worst re-fetches the few pages in flight.

The failure stories are mostly graceful. A dead fetcher's URLs time out and retry elsewhere. A slow or unreachable host accumulates failures in the metadata table and gets backed off exponentially, then parked. The blob store and metadata database scale independently, by capacity and by url_hash shards respectively. The system's two real operational hazards are external: hammering someone's site through a politeness defect, which is why per-host rate limiting deserves tests and the User-Agent carries contact information, and being blocked or rate limited by content delivery networks, which is managed by spreading requests in time rather than by evasion. A daily report of top hosts by request count is the cheapest early warning for both.

Follow-up questions

  • Why partition the crawl by host rather than by URL hash? Politeness is per host, so host-partitioning keeps the delay state, robots cache, and connection pools on one node. URL-hash partitioning would scatter one host's URLs everywhere and force the nodes to coordinate every fetch against the shared rate limit.
  • What happens when robots.txt itself is unavailable? A 404 means no restrictions, but a timeout or 5xx is treated conservatively: retry later, and skip the host's pages meanwhile. Caching robots.txt for about a day with revalidation balances respect for changes against fetch overhead.
  • How does the design avoid recrawling the same page forever via different URLs? Three layers: URL normalization collapses spelling variants, the seen-set drops known URLs, and content dedup catches the remainder where genuinely different URLs return the same body, recording an alias rather than storing a copy.
  • Is this BFS or DFS, and does it matter? BFS-flavored: discovery spreads outward from seeds, which finds important, well-linked pages early. But strict ordering is abandoned in favor of priorities, since politeness already reorders everything and pages differ enormously in value. DFS is avoided because it dives into one site, which is both rude and low-yield.
  • What breaks first at 10x scale? The seen-set and frontier memory: 100 GB of Bloom filter wants sharding across nodes, which host-partitioning already provides. Bandwidth grows to 16 Gbps, pushing the fetcher fleet across regions, and the metadata table's write rate makes batched, asynchronous updates necessary.
  • How would you crawl JavaScript-rendered pages? A separate rendering fleet of headless browsers, fed by the same frontier but budgeted independently, because a rendered fetch costs 10 to 100 times a plain one. Most operators render selectively, triggered by signals that static HTML was empty of content.
