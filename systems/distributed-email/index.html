Design a distributed email service

Systems design · Social and messaging · Jul 2025

This question asks for a webmail product at Gmail's scale: a billion accounts that can send, receive, read, search, and label mail, with spam filtered before it reaches the inbox. What makes email different from every other messaging design is that the service does not own both ends of the conversation. A chat system invents its own protocol and talks only to itself, while an email service must interoperate with every other mail server on the planet through protocols standardized decades ago, and it inherits their semantics, their retry conventions, and their abuse. Interviewers like the question because it combines a high-volume ingestion pipeline, a storage problem measured in exabytes, a per-user search engine, and an adversarial subsystem, spam, where the opponent adapts to whatever is built. The hard parts live in the receive pipeline that must never lose a message it has accepted, in the storage layout that makes a billion mailboxes affordable, and in the send path, where getting mail delivered is as much about reputation as about engineering.

Scope and requirements

The functional scope: receive mail from any server on the internet, deliver it to the right mailbox with spam scored on the way in, let users read, label, archive, and delete through the product's own web and mobile clients, search across a mailbox's full history, and send mail to any address in the world. The product's own clients talk to an HTTP API, which is simpler and richer than the legacy protocols, but the service still speaks the standard ones at its edges: SMTP, the Simple Mail Transfer Protocol, is how servers transfer mail to each other and is non-negotiable, while IMAP and POP, the older protocols that let third-party clients read a mailbox, are supported as legacy access for the minority who want them.

Non-functionally, accepted mail must never be lost, which makes durability the first constraint on the receive path: once this service tells a sending server "accepted," responsibility has transferred. Inbox reads should render in a couple hundred milliseconds, search in well under a second, and availability must be high enough that mail is effectively always receivable, since sending servers retry for only a few days before giving up. Mail content is sensitive, so encryption at rest and tight internal access controls are assumed throughout.

Sizing the problem

Take 1 billion users receiving an average of 50 emails a day, counting the spam that must be processed even when it is never shown. That is 50 billion deliveries a day, and dividing by 86,400 seconds gives about 580,000 messages per second on average, with peaks two to three times higher during weekday mornings across major time zones. Every one of those messages passes through acceptance, scanning, and delivery, so the pipeline is built for millions of messages per second of headroom.

Storage is the number that forces the architecture. At an average of 50 KB per message including attachments amortized in, 50 billion messages a day is 2.5 PB of new mail daily, which compounds to roughly 900 PB a year and several exabytes over five years, before replication multiplies it. Nobody stores that naively, and two levers make it affordable. Deduplication: identical content is stored once, which matters enormously for email because newsletters, notifications, and forwarded attachments are sent to thousands or millions of recipients each. A 5 MB attachment mailed to 1,000 recipients is 5 GB stored naively and 5 MB stored content-addressed, a 1,000-fold saving on exactly the heaviest objects. Tiering: mail older than a few months is rarely opened, so cold messages move to cheaper, denser storage with slightly slower retrieval, invisible to users beyond a moment's delay on a ten-year-old thread.

The client API

The product's own clients use an HTTP API that treats the mailbox as a database of threads and labels rather than a folder of files, with notifications pushed over a persistent channel so a new message appears without polling.

GET /api/v1/threads?label=inbox&limit=50
→ 200 { "threads": [ { "thread_id": "t8812", "subject": "Quarterly report",
                       "snippet": "Attached is the...", "unread": true } ] }

GET /api/v1/messages/m42751
→ 200 { "from": "ana@example.org", "body_ref": "blob/9f31c2...", "labels": ["inbox"] }

POST /api/v1/messages
{ "to": ["ben@elsewhere.com"], "subject": "Re: Quarterly report", "body": "..." }
→ 202 { "message_id": "m42788", "state": "queued" }

POST /api/v1/messages/m42751/labels   { "add": ["archive"], "remove": ["inbox"] }

Send returns 202 rather than 200 because sending is asynchronous by nature: the message enters an outbound queue, and actual delivery to the remote server happens, and possibly retries, long after the API call returns.

The data model

The storage split that makes the system work is metadata apart from bodies. Mailbox metadata, threads, labels, read flags, and per-message envelope fields, is small, hot, and queried in user-scoped patterns, so it lives in a distributed database partitioned by user ID with strong consistency inside a partition, since a user who archives a thread on their phone must see it archived on their laptop immediately. Bodies are large, immutable, and cold, so they live in object storage, content-addressed: the raw message, in MIME format, the standard envelope-and-parts encoding that lets one email carry text, HTML, and attachments together, is hashed, and the hash is both its name and its dedup key, so the thousandth copy of a newsletter stores nothing new and just increments a reference count.

CREATE TABLE messages (         -- partitioned by user_id
  user_id     BIGINT,
  message_id  BIGINT,
  thread_id   BIGINT,
  from_addr   TEXT,
  subject     TEXT,
  snippet     TEXT,
  labels      TEXT[],           -- inbox, sent, spam, user labels
  body_ref    CHAR(64),         -- content hash into object storage
  received_at TIMESTAMPTZ,
  PRIMARY KEY (user_id, message_id)
);

CREATE TABLE blobs (
  body_ref    CHAR(64) PRIMARY KEY,
  size_bytes  BIGINT,
  refcount    BIGINT            -- dedup bookkeeping
);

The high-level architecture: receiving mail

Receiving starts before the first byte arrives, with DNS. A domain's MX records, the DNS entries that name the mail servers willing to accept mail for that domain, point at a fleet of SMTP gateways behind regional load balancing. A gateway accepts the SMTP conversation, applies connection-level defenses such as rate limits and IP reputation checks, and writes the raw message to a durable intake log before acknowledging, because the acknowledgment is a promise. From there the pipeline is asynchronous: scanning classifies the message for spam and malware, the delivery service writes metadata into the recipient's partition and the body into object storage, the search index absorbs the new message, and a notification wakes any connected client.

The gateway durably logs each accepted message before acknowledging the sender, and everything downstream of acceptance is asynchronous: delivery writes metadata and the body, updates the index, and wakes connected clients over the dashed notification path.

Sending mail and the reputation game

The outbound path starts at the submission service, which authenticates the user, applies per-account rate limits that keep a compromised password from becoming a spam cannon, and signs the message before queueing it for the outbound MTA pool, the mail transfer agents that actually connect to remote servers. Three authentication standards govern whether the world believes the mail. SPF lets a domain publish in DNS which servers are allowed to send mail on its behalf. DKIM attaches a cryptographic signature to each message so receivers can verify it was sent by the domain and not altered in transit. DMARC lets the domain tell receivers what to do when SPF and DKIM checks fail, and asks them to send back aggregate reports. A provider at this scale enforces all three on inbound mail and maintains them meticulously on outbound mail, because its sending reputation, tracked per outbound IP and per domain by every large receiver, decides whether a billion users' legitimate mail lands in inboxes or spam folders.

Step 1: the client submits through the authenticated API. Step 2: the submission service rate-limits and DKIM-signs the message into the outbound queue. Step 3: an outbound MTA picks it up. Step 4: the MTA attempts SMTP delivery to the recipient's MX. Step 5: a temporary 4xx response parks the message in the retry queue. Step 6: after backoff it is retried, while a permanent 5xx response instead generates a bounce back to the sender.

The retry discipline comes straight from SMTP's response codes. A 4xx response means temporary failure, the remote mailbox is over quota or the server is greylisting, and the correct behavior is to retry on a backoff schedule, typically minutes at first and stretching toward hours, for up to a few days before giving up. A 5xx response means permanent failure, no such user or mail refused by policy, and retrying is worse than useless because hammering a server with rejected mail damages reputation, so the message bounces immediately with a delivery failure notice to the sender. The outbound pool also shapes traffic per destination, because large receivers throttle by source IP, so queues are organized per destination domain and drained at each domain's tolerated pace.

Search across a billion mailboxes

Email search is a different problem from web search in a way that makes it tractable: no query ever spans users, so the index can be partitioned exactly like the mailbox. Each user gets their own inverted index, the structure that maps each term to the list of messages containing it, sharded on the same user ID as the metadata so a user's mail and index live together. On delivery, the pipeline tokenizes the subject and body, updates the user's index, and the change is visible to search within seconds; a delivery rate of 580,000 messages per second is also 580,000 index updates per second, which is why the index write path is built into the pipeline rather than batched overnight. The index stores term positions for phrase queries and structured fields for the filters users actually type, from, to, has attachment, date ranges. A median mailbox of a few gigabytes yields an index of a few hundred megabytes, which at a billion users is on the order of hundreds of petabytes of index, kept affordable the same way bodies are: the index for cold mail can be rebuilt or paged from cheaper storage, while the hot recent index stays on fast media.

Spam as a layered system

Spam filtering cannot be one classifier, because the adversary probes whatever single thing is deployed. The layers run from cheap to expensive. Connection-time checks reject the worst traffic before reading a byte of content: IP reputation lists, rate anomalies, and protocol fingerprints, which matters when a majority of all connection attempts are abusive. Authentication checks, SPF, DKIM, and DMARC results, then separate forged sender claims from real ones. Content rules catch known campaign signatures and malicious attachment types, and a machine-learned classifier scores everything that survives, trained on features from headers, body text, sender history, and the user's own past behavior toward similar mail. The user feedback loop closes the system: every "report spam" and "not spam" click is a labeled example flowing back into training, hundreds of millions of labels a day at this scale, which is the data advantage that makes a large provider's filter hard to evade. Scoring places mail in the inbox or the spam folder rather than silently dropping it except at the very highest confidence, because false positives, real mail marked as spam, are the costliest outcome the system can produce.

Scaling, failures, and operations

The tiers scale on different axes. Gateways and scanners are stateless and scale with connection volume, the mailbox store scales by adding user partitions, and object storage scales by hash space, the easiest kind. The intake log is the component whose failure semantics matter most: a gateway that crashes after acknowledging a message must not lose it, so acceptance writes to a replicated log (a Kafka-style commit log or equivalent) before the SMTP 250 response goes out, and delivery consumes from that log with at-least-once semantics, deduplicating on message ID so a replay never produces two copies in an inbox.

Multi-region design follows the mailbox. Each user's partition is homed to one region, with synchronous replication inside the region for durability and asynchronous replication to a second region for disaster recovery, so a regional failure means failing over mailboxes with a bounded few seconds of replication lag rather than reconciling a globally writable store. Inbound mail tolerates this gracefully because MX records list multiple gateways across regions, and a gateway in any region can accept mail and forward it to the home region's pipeline, with the sending server's own retry behavior covering brief gaps. The operational metrics that matter are acceptance-to-inbox latency, intake log consumer lag, bounce rates per outbound IP (the early warning that reputation is slipping), spam false positive reports, and storage dedup ratio, which quietly protects the largest line item in the budget.

Follow-up questions

  • Why object storage for bodies instead of the database? Bodies are large, immutable, and mostly cold, exactly object storage's design point, and content addressing gives dedup for free. The database keeps only what queries need: envelope fields, labels, flags, and the body hash.
  • What stops a crashed gateway from losing mail? The SMTP acknowledgment is only sent after the message is in a replicated intake log, so a crash after acceptance replays from the log, and delivery dedup on message ID makes the replay harmless.
  • Why are 4xx and 5xx handled so differently on send? 4xx is the remote server saying "not now," so backing off and retrying for days is correct, while 5xx is "never," so the message bounces immediately, because retrying permanent failures burns sender reputation.
  • How does search stay fast at a billion users? By never being global: each user's inverted index is sharded with their mailbox, so a query touches one shard, and index updates ride the delivery pipeline so new mail is searchable in seconds.
  • What is the single highest-leverage cost optimization? Content-addressed dedup of bodies and attachments. Bulk mail dominates volume and is identical across recipients, so storing each unique body once collapses the largest class of data by orders of magnitude.
  • Why does the provider's own outbound mail need SPF, DKIM, and DMARC? Because other receivers judge it by the same rules it applies to them, and reputation is tracked per IP and domain. Misconfigured authentication at this scale means a billion users' mail starts landing in spam folders.
