Design a distributed rate limiter

Systems design · Distributed building blocks · Feb 2025

A rate limiter decides, for every incoming request, whether the caller has exceeded an allowed budget such as 100 requests per minute, and rejects the excess before it reaches the rest of the system. Every public API needs one, because without it a single misconfigured client retry loop can generate as much traffic as the rest of the user base combined, a scraper can vacuum a catalog at line speed, and a credential-stuffing attack can hammer a login endpoint until the database behind it falls over. Rate limiting also enforces fairness between paying tiers and keeps a noisy tenant from starving everyone else, and it caps the cloud bill for endpoints that fan out into expensive downstream work.

Interviewers like this question because the single-machine version is a data structures exercise while the distributed version forces real trade-offs: counters must be shared across many limiter nodes without making every request wait on coordination, the arithmetic of each algorithm has visible product consequences, and the failure story (what happens when the counter store is down) has no answer that is free. The hard parts live in three places: choosing and reasoning about the counting algorithm, keeping the counters atomic under concurrency, and deciding how strictly the limit must hold when the system is degraded.

Scope and requirements

I would first pin down what is being limited and where. The brief here is a server-side limiter for an API platform: it must enforce configurable rules keyed by user ID, API key, IP address, or endpoint, and a single request may be subject to several rules at once, for example 10 requests per second per user and 1,000 per hour per IP. Clients that exceed a limit should receive a clear rejection that tells them when to retry rather than a silent drop. Rules must be editable by operators without redeploying services.

The non-functional requirements are what shape the design. The limiter sits on the hot path of every request, so it must add very little latency, ideally under a couple of milliseconds. It must itself scale to the full traffic of the platform, since by definition it sees everything. It should use modest memory, because tracking millions of distinct callers with heavyweight state per caller gets expensive fast. Accuracy can be slightly relaxed: briefly admitting a few percent over the limit during a failure or a synchronization gap is usually acceptable, while rejecting legitimate traffic under the limit is worse. Finally, the limiter must not become a new single point of failure for the API it protects.

Sizing the problem

Concrete numbers keep the design grounded. Suppose the platform serves 1 billion requests per day. A day is 86,400 seconds, so that is about 11,600 requests per second on average, and traffic peaks are commonly five times the average, so the limiter must make roughly 58,000 decisions per second at peak. Each decision is one or two operations against a counter store, so the store needs to sustain on the order of 100,000 small operations per second, which is within reach of a single large Redis node and comfortable for a small cluster.

Memory stays small because per-caller state is tiny. A token bucket needs a token count and a timestamp, around 50 bytes with key overhead. If 10 million distinct callers are active in any given hour and each matches two rules, that is 20 million counters at 50 bytes, about 1 GB. Even the most expensive algorithm discussed below, which keeps a timestamp per request rather than per caller, would need 10 requests per minute per caller times 10 million callers times 8 bytes, around 800 MB per minute of window, which explains why that algorithm is reserved for low-volume, high-value endpoints.

The interface

From the client's point of view the limiter is invisible until it triggers, so the contract that matters is the rejection. The convention is HTTP status 429 Too Many Requests, defined in RFC 6585, together with a Retry-After header that says how many seconds to wait, and a set of X-RateLimit headers on every response so well-behaved clients can pace themselves before ever hitting the wall.

GET /api/v1/orders
Authorization: Bearer ...

→ 200 OK
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 23
X-RateLimit-Reset: 1714066920

GET /api/v1/orders          (the 101st request this minute)

→ 429 Too Many Requests
Retry-After: 37
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
{ "error": "rate_limited", "retry_after_seconds": 37 }

Publishing the remaining budget is a real design choice rather than decoration: clients that can see X-RateLimit-Remaining falling will back off voluntarily, which converts hard rejections into smooth self-throttling and reduces the retry storms that 429s otherwise provoke.

Rules and counter state

Two kinds of data live in the system and they have opposite shapes. Rules are small, read constantly, and change rarely, so they live in a durable store and are cached inside every limiter node with a periodic refresh or a push on change. Counters are huge in cardinality, written on every request, and worthless after the window passes, so they live in an in-memory store keyed by rule and caller, with a time-to-live so stale entries clean themselves up.

CREATE TABLE rate_limit_rules (
  id          BIGSERIAL PRIMARY KEY,
  scope       TEXT NOT NULL,        -- 'user' | 'ip' | 'api_key'
  endpoint    TEXT NOT NULL,        -- 'POST /v1/login' or '*'
  capacity    INT  NOT NULL,        -- bucket size, e.g. 100
  refill_per_sec NUMERIC NOT NULL,  -- e.g. 1.67 for 100/minute
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Counter keys in Redis, expiring automatically:
--   rl:{rule_id}:{caller}  ->  { tokens: 23, ts: 1714066883.21 }

Placement is the other scoping decision. Limiting purely in the client SDK is courtesy, not enforcement, because attackers do not run your SDK. Limiting inside each backend service works but scatters the logic and spends backend capacity on requests that will be rejected anyway. The standard answer is the API gateway, the single front door that already terminates connections and authenticates callers: a limiter middleware there sees every request with the caller's identity attached, rejects excess traffic before it consumes anything downstream, and keeps the rules in one place.

The high-level architecture

The shape that falls out is small. Clients reach an API gateway, which is the layer that routes external requests to internal services, and a limiter middleware inside the gateway consults two stores: a rules store for configuration and a Redis cluster for live counters. Redis is an in-memory key-value store chosen here because it offers sub-millisecond operations, built-in key expiry, and server-side scripting for atomicity. Requests under the limit continue to the protected services and the rest never leave the gateway.

The limiter middleware in the gateway checks Redis on every request and forwards only traffic under the limit; rules are fetched rarely and cached locally. Dashed arrows are periodic or conditional.

Choosing the algorithm

Five algorithms cover the design space, and the differences show up at window boundaries and in memory cost, so each deserves a worked number.

The token bucket is the workhorse. Picture a bucket that holds at most 100 tokens and refills at 100 tokens per minute, which is 1.67 tokens per second; each request removes one token and is rejected if the bucket is empty. A caller idle for a while accumulates a full bucket and may burst 100 requests at once, then sustain 1.67 per second, which matches how real clients behave: quiet, then a flurry. Two numbers per caller (token count and last refill time) are the entire state, and the refill needs no background job because it can be computed lazily, on the next request, from the elapsed time.

The leaky bucket is the same metaphor inverted: requests enter a queue of bounded size and drain at a fixed rate, so the output is perfectly smooth regardless of input bursts. That smoothness is exactly right when the protected resource cannot absorb bursts at all, but it means a burst of legitimate requests waits in the queue even when the system has spare capacity, and the queued requests add latency that the token bucket would not.

The fixed window counter is the simplest to implement: keep one counter per caller per clock window, increment it on each request, and reject above the limit. Its flaw lives at the boundary. With a limit of 100 per minute, a caller can send 100 requests at 11:00:59 inside the first window and 100 more at 11:01:01 inside the second, and both windows are individually legal, yet 200 requests landed within two seconds, double the intended rate over that span. For abuse protection that is a real hole, not a rounding error.

The sliding window log fixes the boundary exactly by storing the timestamp of every request and counting how many fall within the trailing window. It is perfectly accurate and brutally expensive: as the sizing section showed, timestamps for millions of callers run to hundreds of megabytes per minute of window, and every decision scans or trims a per-caller list. It earns its cost only on endpoints where precision matters more than volume, such as login or password reset.

The sliding window counter is the practical compromise, blending the two adjacent fixed windows with a weighted interpolation. Keep the counts for the previous and current fixed windows, and estimate the trailing-window count as the current count plus the previous count scaled by how much of the previous window still overlaps the trailing window. With a limit of 100 per minute, suppose the previous minute saw 84 requests and we are 15 seconds into the current minute with 36 so far. The trailing 60 seconds still overlaps 45 of the previous window's 60 seconds, so the estimate is 84 × (45/60) + 36, which is 63 + 36 = 99, just under the limit, and the next request is the last one admitted. The estimate assumes requests were spread evenly across the previous window, which is rarely exact but, as Cloudflare reported when they deployed this at scale, the error in practice is small and the state is two integers per caller.

I would recommend the token bucket as the default for its burst-friendliness and tiny state, with the sliding window counter where smooth enforcement over a window is the contract. The decision flow for a single token bucket check looks like this.

Step 1: the request reaches the gateway. Step 2: the limiter invokes a Lua script against the caller's bucket key. Step 3: the script refills tokens for the elapsed time, tries to take one, and returns the result atomically. Step 4: admitted requests continue to the service. Step 5: rejected requests answer 429 with Retry-After and never touch the service.

Making the counter atomic

With many gateway nodes sharing one Redis, the naive read-modify-write sequence has a race condition, which is the failure mode where two concurrent operations interleave and produce a result neither intended. Suppose the bucket holds 1 token and two gateway nodes each run GET (both see 1), each decide to admit, and each run DECR: both requests pass and the counter lands at -1, so the limit admitted one request too many. Under load this happens constantly, and the over-admission scales with concurrency.

The fix is to push the whole decision into the store as one atomic unit. Redis executes a Lua script as a single uninterrupted operation, so the refill arithmetic, the comparison, and the decrement happen with no interleaving. The entire limiter core is about a dozen lines.

# Lua, executed atomically inside Redis (shown with Python-style comments)
# KEYS[1] = bucket key; ARGV = capacity, refill_per_sec, now
local b = redis.call('HMGET', KEYS[1], 'tokens', 'ts')
local tokens = tonumber(b[1]) or capacity
local elapsed = now - (tonumber(b[2]) or now)
tokens = math.min(capacity, tokens + elapsed * refill_per_sec)
local allowed = 0
if tokens >= 1 then
  tokens = tokens - 1
  allowed = 1
end
redis.call('HMSET', KEYS[1], 'tokens', tokens, 'ts', now)
redis.call('EXPIRE', KEYS[1], 2 * capacity / refill_per_sec)
return { allowed, tokens }

The EXPIRE call is the memory story: a caller that goes quiet has its key deleted automatically once enough time has passed that the bucket would be full anyway, so the working set tracks active callers rather than every caller ever seen. For the simpler fixed and sliding window counters, INCR with an expiry achieves the same atomicity without a script, which is why the Redis documentation presents rate limiting as the canonical INCR pattern.

Many limiter nodes, one budget

A single shared Redis gives exact global counting at the cost of one network round trip per decision, roughly 0.5 to 1 millisecond within a datacenter, which is usually an acceptable tax. When it is not, or when limiter nodes span regions, the alternative is local counting with asynchronous synchronization: each gateway node counts in its own memory and periodically reconciles with the shared store, so decisions are instant but the global count is eventually consistent, meaning the nodes converge on the true total only after a delay.

The cost of that delay is bounded and worth computing out loud. With 10 gateway nodes syncing every 100 milliseconds and a caller blasting requests at every node at once, each node can admit traffic for up to 100 milliseconds before learning the budget is spent. At a limit of 100 per minute and an attacker pushing 1,000 requests per second spread evenly, each node sees 100 per second and may admit about 10 extra requests in its blind window, so the platform briefly admits on the order of 100 extra requests, doubling the per-minute budget once, after which all nodes know the bucket is empty. For protecting capacity that overshoot is noise; for billing enforcement or login protection it is not, which is why precision-critical rules should stay on the synchronous shared-store path while high-volume coarse rules can ride the local-counter path. A middle ground used by several gateways is to keep the shared store authoritative but answer from a local cache when Redis latency spikes, accepting brief imprecision to protect tail latency.

Sticky routing deserves a mention because it makes the problem smaller rather than solving it: if the load balancer consistently routes each caller to the same gateway node, that node's local counter is the global truth for that caller, and no synchronization is needed until the node dies or the routing rebalances. It pairs naturally with consistent hashing on caller ID, and its weakness is exactly the rebalancing moment, when counters effectively reset.

Scaling, failures, and operations

Each tier scales by a different rule. Gateway nodes are stateless with respect to limiting (the state lives in Redis), so they scale horizontally behind the load balancer. Redis scales by sharding the counter keys across a cluster, and because each key is touched only by requests from one caller, the keys distribute evenly and shard cleanly; at 58,000 peak decisions per second, three modest shards with replicas carry the load with headroom. The rules store barely registers, since rules are cached in every node and refreshed on the order of seconds.

The failure that matters is Redis becoming unreachable, and the design must pick a posture in advance. Failing open means admitting all traffic unlimited while the store is down, which preserves availability for legitimate users but drops the shield exactly when an attack may be the cause of the trouble. Failing closed means rejecting everything, which is safe for the backends and terrible for users. The defensible answer is fail open for general API traffic with an immediate page to operations, and fail closed (or fall back to conservative local-only counters) for abuse-sensitive endpoints such as authentication, stated per rule in configuration rather than hard-coded. A local in-process token bucket with a small fraction of the global budget is a good degraded mode: it cannot enforce the precise global limit, but it caps the worst case while the store recovers.

Operationally, the limiter needs its own observability: a counter of decisions per rule, the rejection rate, and Redis latency percentiles, because a rising rejection rate is often the first visible symptom of a client-side failure mode in some customer's retry logic. Rule changes should roll out in shadow mode first, logging what would have been rejected without rejecting it, since an operator typo in a capacity field is otherwise a self-inflicted outage. Hot keys are the other operational wrinkle: a rule keyed on a single shared resource (one global counter, say) concentrates all traffic on one Redis shard, so global rules should be split into per-node allowances or use a key that spreads load.

Follow-up questions

  • Why not enforce limits in each backend service instead of the gateway? It spends backend CPU and connections on requests that will be rejected, duplicates rule logic across teams, and loses the single place to observe and tune limits. Service-level limits still make sense as a second line of defense for internal callers that bypass the gateway.
  • Token bucket or sliding window counter? Token bucket when bursts are a feature and state must be minimal; sliding window counter when the contract is a smooth ceiling over a window. The fixed window is acceptable only when the boundary burst (double the rate across a window edge) is harmless.
  • What does the Lua script buy over INCR? Atomicity for multi-step logic: refill arithmetic, comparison, and decrement execute as one uninterrupted unit, eliminating the read-modify-write race where concurrent nodes both see one remaining token and both admit.
  • How would you rate limit by IP behind a corporate NAT? Carefully, because thousands of users share one address. Prefer authenticated identity when available, set IP limits high enough for shared egress points, and use IP rules mainly for unauthenticated abuse, ideally combined with other signals before blocking.
  • Fail open or fail closed when Redis is down? Per rule. General traffic fails open with degraded local counters and an alert; authentication and payment endpoints fail closed or to a strict local budget, because the cost of over-admission there is account takeover, not just load.
  • How do you avoid punishing clients with rejected-then-retried storms? Return Retry-After and the X-RateLimit headers so clients can pace themselves, document jittered exponential backoff in the SDK, and consider briefly queueing slightly-over-limit requests (a leaky bucket at the edge) for paying tiers.
